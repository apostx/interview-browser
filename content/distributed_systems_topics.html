Foundations
01

CAP Theorem

During a network partition, a distributed system can guarantee at most two of Consistency, Availability, Partition tolerance — and since partitions are unavoidable, the real choice under one is C vs A.

P isn't optional — the choice is what to do during a partition: CP or AP.

What do C, A, P actually mean here?

  • Consistency (linearizability): every read sees the latest write. Availability: every request gets a non-error response. Partition tolerance: the system keeps working despite dropped/delayed messages between nodes.

So which do you “pick”?

  • Networks fail, so P is forced. The real decision is the partition response: CP (reject/block to stay consistent) or AP (serve possibly-stale data to stay available).

Watch outCAP only applies during a partition — a healthy network gives you both C and A. Real systems are tunable per-operation, not globally “CP” or “AP.”

02

PACELC

An extension of CAP: if there's a Partition, choose Availability or Consistency; Else (normal operation), choose Latency or Consistency.

Why does PACELC matter?

  • CAP ignores the normal case. Even without a partition, stronger consistency costs latency (coordination round trips) — PACELC names that everyday trade-off.
  • Examples: Dynamo/Cassandra = PA/EL (availability + low latency); a strongly-consistent store = PC/EC.

Watch outFor most systems the interesting trade-off is the “else” branch — latency vs consistency in normal operation — not the rare partition case.

03

Consistency Models

The contract for what a read can observe after writes — from strongest to weakest.

ModelGuarantee
Linearizablereads see the latest committed write; acts like one copy (costly)
Sequentialall nodes see operations in the same order (not real-time)
Causalcausally-related ops seen in order; concurrent ops may differ
Read-your-writesyou always see your own writes
Eventualreplicas converge if writes stop; reads may be stale meanwhile

Why not always use strong consistency?

  • It requires coordination (consensus/quorum) → higher latency and lower availability. Weaker models scale better and stay available under partitions.

Watch out“Eventual” doesn't mean “no rules” — causal + read-your-writes are practical middle grounds that avoid the worst anomalies.

04

Replication

Keeping copies of data on multiple nodes for availability, durability and read scaling.

Single-leader: all writes hit the leader, then flow to followers.

The main strategies?

  • Single-leader: writes to the leader, replicated to followers. Simple, no write conflicts.
  • Multi-leader: several nodes accept writes (multi-region) — needs conflict resolution.
  • Leaderless (Dynamo): any replica takes reads/writes; uses quorums + read repair.

Sync vs async replication?

  • Sync: leader waits for followers → durable but slower, and stalls if a follower is down. Async: fast, but a leader crash can lose un-replicated writes.

Watch outAsync replication + failover can lose acknowledged writes and cause split-brain (two leaders) — which is why failover needs fencing/consensus.

05

Partitioning / Sharding

Splitting a dataset across nodes so each holds a subset — for horizontal scale beyond one machine.

Partitioning strategies?

  • Range: split by key ranges — good for range scans, but hotspots on skewed keys.
  • Hash: hash the key to a partition — even distribution, but no efficient range queries.

What's the rebalancing problem?

  • Adding/removing a node with plain key mod N remaps almost every key. Consistent hashing minimizes movement.

Watch outA bad partition key creates hotspots — one shard gets most traffic. Pick a high-cardinality, evenly-accessed key and watch for “celebrity” keys.

06

Consistent Hashing

Maps both keys and nodes onto a ring; a key belongs to the next node clockwise. Adding/removing a node moves only the keys in its arc — not everything.

A node change touches only ~1/N of keys, versus nearly all with key mod N.

Why better than key mod N?

  • With mod N, changing N remaps ~all keys (cache/DB churn). Consistent hashing moves only ~1/N on a node change.

What are virtual nodes for?

  • Without them, load is uneven and removing a node dumps all its keys on one neighbor. Each physical node owns many small ring positions (vnodes) → smoother spread and rebalancing.

Watch outPlain consistent hashing still has variance — virtual nodes (or rendezvous hashing) are what make it practical.

07

Quorums (R + W > N)

With N replicas, require W nodes to ack a write and R for a read. If R + W > N, the read and write sets overlap on ≥1 node → the read sees the latest write.

N=3, W=2, R=2 → any read set and any write set share at least one node.

How do you tune R and W?

  • W=N, R=1 → fast reads, fragile writes. W=1, R=N → fast writes, slow reads. W=R=⌈(N+1)/2⌉ (N=3 → 2/2) → balanced, tolerates one node down.

Does a quorum give strong consistency?

  • It gives read-after-write on overlap, not full linearizability — concurrent writes, sloppy quorums and read-repair timing mean Dynamo-style stores are eventually consistent by default.

Watch outR + W > N guarantees overlap, not conflict resolution — concurrent writes still need last-write-wins or version vectors to reconcile.

Coordination & Consensus
08

Consensus & Raft

Getting a group of nodes to agree on a value or order despite failures. Raft (and Paxos) provide this — for leader election, replicated logs and config.

How does Raft work at a high level?

  • Nodes elect a leader for a term. The leader appends entries to its log and replicates them; an entry commits once a majority acks. If the leader fails, a new election begins.

Why does it need a majority?

  • Any two majorities overlap, so decisions can't conflict and a committed entry survives failures. Hence odd cluster sizes (3, 5) tolerating ⌊N/2⌋ failures.

Watch outConsensus is expensive (round trips + majority). Use it for metadata/coordination (etcd, ZooKeeper), not the hot data path.

09

Leader Election

Choosing a single coordinator among nodes, and re-choosing when it fails.

How is it done safely?

  • Via a consensus system (Raft election) or a lease in ZooKeeper/etcd. The leader holds a time-bounded lease it must renew; if it can't, another node takes over.

What's split-brain, and how do you prevent it?

  • Two nodes both believe they're leader (e.g. after a partition) → conflicting writes. Prevent with quorum-based election + fencing tokens (monotonic IDs that let resources reject a stale leader).

Watch outA paused leader (GC/partition) may still think it's leader — always use fencing tokens so downstream systems reject its late writes.

10

Clocks & Ordering

There's no single global clock, so ordering events across nodes needs logical clocks.

Why not just use wall-clock timestamps?

  • Physical clocks drift and skew; “last write wins” by wall clock can silently drop the right update. NTP helps but isn't precise enough for ordering.

Lamport vs vector clocks?

  • Lamport: a counter giving a total order consistent with causality, but can't tell if two events are concurrent.
  • Vector clocks: per-node counters that can detect concurrency (and thus conflicts).

Watch outUsing server wall-clock time for ordering or conflict resolution is a classic bug — clock skew makes it non-deterministic. Use logical clocks or a sequencer.

Reliability Patterns
11

Idempotency & Exactly-Once

An idempotent operation has the same effect applied once or many times — the foundation for safe retries over an unreliable network.

Is “exactly-once delivery” real?

  • Not over an unreliable network. What you can achieve is exactly-once effect: at-least-once delivery + idempotent handling (dedup by message/operation ID).

How do you implement it?

  • Assign a unique ID per operation; the consumer records processed IDs (dedup table) and skips duplicates — ideally in the same transaction as the effect.

Watch out“Exactly-once” in messaging means exactly-once effect via dedup, not magic delivery. If someone claims exactly-once delivery, ask how they handle a broker crash after send but before ack.

12

Retries, Backoff & Jitter

Re-attempting failed operations, spaced out to avoid overloading a struggling service.

Why exponential backoff + jitter?

  • Immediate retries hammer a failing service. Exponential backoff (1s, 2s, 4s…) gives it room; jitter (randomized delay) prevents a thundering herd where all clients retry in sync.

What must you NOT retry?

  • Non-idempotent ops without an idempotency key (double-charge risk), and client errors (4xx). Only retry transient failures — timeouts, 5xx, resets.

Watch outRetries amplify load during an outage (retry storms). Cap attempts, add a deadline/budget, and combine with a circuit breaker so you stop hammering a dead dependency.

13

Circuit Breaker

A wrapper that “trips” after repeated failures, failing fast instead of piling up doomed calls — like an electrical breaker.

Open fails fast during a cooldown; half-open probes recovery before fully closing.

The three states?

  • Closed: calls pass, failures counted. Open: calls fail immediately (fallback) for a cooldown. Half-open: a few trial calls — success closes it, failure re-opens.

Why is it needed?

  • Without it, a slow/dead dependency piles up threads/connections → cascading failure. The breaker isolates the failure and lets the caller degrade gracefully.

Watch outA breaker needs a real fallback (cached data, default, queued work) — tripping without one just relocates the error. Pair with timeouts, or slow calls never count as failures.

14

Timeouts & Bulkheads

Timeouts bound how long you wait; bulkheads isolate resources so one failing dependency can't sink the whole system — like watertight ship compartments.

Why are timeouts non-negotiable?

  • Without one, a hung dependency holds a thread/connection forever; enough of them exhaust the pool → total outage. Every network call needs a timeout + deadline.

What's a bulkhead in practice?

  • Separate connection/thread pools per dependency (or tenant), so exhausting the pool for service A doesn't starve calls to service B.

Watch outSet timeouts shorter than the caller's (deadline propagation) — otherwise the client gives up while the server keeps working, wasting resources.

15

Distributed Transactions & Saga

A transaction spanning services can't use one ACID commit. A Saga breaks it into a sequence of local transactions, each with a compensating action to undo it on failure.

If a step fails, the already-completed steps are compensated in reverse order.

Choreography vs orchestration?

  • Choreography: services react to each other's events (no coordinator) — decoupled, hard to trace.
  • Orchestration: a central orchestrator drives each step and triggers compensations — clearer, but a coordinator to run.

What guarantee do you get?

  • Not isolation — intermediate states are visible (eventual consistency). You get atomicity via compensation: all steps complete, or completed ones are undone.

Watch outCompensation isn't a rollback — the money really moved and must be really refunded. Design each compensating action explicitly, and make steps idempotent (they'll be retried).

16

Two-Phase Commit

An atomic-commit protocol across nodes: a coordinator asks all participants to prepare (phase 1); if all vote yes, it tells them to commit (phase 2), else abort.

All-or-nothing across participants — but blocking if the coordinator dies mid-protocol.

Why is it problematic at scale?

  • It's blocking: if the coordinator crashes after “prepare,” participants hold locks indefinitely, unsure whether to commit. It also lowers availability — any participant down blocks the commit.

2PC vs Saga?

  • 2PC: strong consistency, synchronous, locks, blocking — okay within one trust boundary (XA over a few DBs).
  • Saga: eventual consistency, non-blocking, scales across services — preferred in microservices.

Watch out2PC is largely avoided in microservices because of its blocking failure mode and lock-holding — Sagas + idempotency are the usual answer.

17

Outbox Pattern

A fix for the dual-write problem — you can't atomically write to a DB and publish to a broker. Write the event to an “outbox” table in the same DB transaction; a relay reads it and publishes.

The write and the event share one local transaction, so they can't diverge.

What problem does it solve?

  • Without it, a crash between “commit to DB” and “publish event” makes the DB and message log diverge (lost event, or event with no DB change). The outbox makes the write + event atomic.

How does the relay work?

  • A poller or CDC (change-data-capture, e.g. Debezium reading the DB log) reads new outbox rows and publishes them, marking them sent.

Watch outThe relay publishes at-least-once (it may re-send after a crash before marking sent) — consumers must be idempotent. The outbox guarantees eventual publish, not exactly once.

Messaging & Events
18

Message Queues & Pub/Sub

Async messaging decouples producers from consumers. Queues (one consumer per message, work distribution) vs pub/sub (topics, every subscriber gets a copy, fan-out).

Why use them?

  • Decoupling (producer doesn't wait on consumers), buffering/load-leveling (absorb spikes), and resilience (consumer down ≠ lost work). Enables async, event-driven workflows.

Queue vs log (Kafka) model?

  • Queue (RabbitMQ/SQS): message removed once consumed. Log (Kafka): an ordered, retained log; consumers track their own offset and can replay — favoring high-throughput streaming.

Watch outA queue adds a component to operate — dead-letter queues, backlog, poison messages. And async breaks request/response: you need correlation IDs and idempotent consumers.

19

Delivery Semantics

The guarantee about how many times a message is delivered/processed.

SemanticMeaningUse
At-most-oncemay be lost, never duplicatedmetrics, best-effort
At-least-oncenever lost, may be duplicatedmost systems (needs idempotent consumers)
Exactly-onceno loss, no duplicate effectat-least-once + dedup/idempotency

Which do most systems use, and why?

  • At-least-once is the practical default: retry on a missing ack, and make consumers idempotent (dedup by ID) to tolerate the resulting duplicates.

Watch out“Exactly-once” means exactly-once processing (via idempotency), not delivery. Ack timing — before vs after processing — is what makes a system at-most vs at-least-once.

20

Event Sourcing & CQRS

Event Sourcing stores state as an append-only log of events (current state = replay the log). CQRS separates the write model from query-optimized read models.

Why event sourcing?

  • Full audit history, time-travel/replay, and the ability to build new read projections from past events. The event log becomes the source of truth.

Why pair it with CQRS?

  • The log is great for writes but awkward to query; CQRS builds separate read models (projections) updated from the events, so reads and writes scale independently.

Watch outBoth add real complexity — eventual consistency between write and read models, event schema/versioning, replay cost. Use them where audit/history or read/write asymmetry justifies it, not by default.

21

Failure Detection & Heartbeats

Detecting that a node has failed so the system can react (failover, rebalance). Nodes send periodic heartbeats; a missed one suggests failure.

Why is failure detection fundamentally hard?

  • You can't distinguish a crashed node from a slow one or a network partition — a missed heartbeat could be any of them. Detectors trade speed vs false positives.

How is it done in practice?

  • Heartbeats/health checks with timeouts, often gossiped for scale. Phi-accrual detectors give a suspicion level rather than a binary up/down.

Watch outAggressive timeouts evict healthy nodes (needless failover, thrash); lax ones detect failures slowly. There's no perfect setting — tune for your latency profile.
