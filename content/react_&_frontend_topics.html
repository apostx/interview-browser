Virtual DOM & Reconciliation

React renders a lightweight in-memory tree, diffs it against the previous one (reconciliation), and applies only the minimal real-DOM mutations.A React egy könnyű, memóriabeli fát renderel, összeveti az előzővel (reconciliation), és csak a minimálisan szükséges valódi DOM-módosításokat hajtja végre.

Same type + key → reuse and patch. Different type → tear down and rebuild.Azonos típus + key → újrahasznosítás és patch. Eltérő típus → lebontás és újraépítés.

What's the diffing heuristic?Mi a diffing heurisztika?

  • An O(n) heuristic, not a full tree diff. Two assumptions: different element types produce different trees; stable keys identify which children moved.Egy O(n) heurisztika, nem teljes fa-diff. Két feltevés: eltérő elemtípusok eltérő fát eredményeznek; a stabil key-ek azonosítják, mely gyerekek mozdultak el.

What actually causes a re-render?Mi okoz ténylegesen re-rendert?

  • A state change, a parent re-render, a consumed context value changing, or a subscribing hook firing.State-változás, a szülő re-rendere, egy fogyasztott context-érték változása, vagy egy feliratkozó hook eseménye.
  • Re-render ≠ DOM update — reconciliation may find nothing changed and touch no DOM.A re-render ≠ DOM-frissítés — a reconciliation azt is találhatja, hogy semmi nem változott, és hozzá sem nyúl a DOM-hoz.

Render vs commit phase?Render vs commit fázis?

  • Render: React calls your components and builds the tree — pure, and pausable in concurrent mode.Render: a React meghívja a komponenseidet és felépíti a fát — tiszta, és concurrent módban megszakítható.
  • Commit: applies DOM changes and runs effects — synchronous.Commit: alkalmazza a DOM-változásokat és lefuttatja az effekteket — szinkron.

Watch outBuktatóA re-render is relatively cheap; the cost is your render logic. Don't optimize re-renders you haven't profiled.A re-render viszonylag olcsó; a költség a te render-logikád. Ne optimalizálj olyan re-rendert, amit nem profiloztál.

Keys in ListsKey-ek listákban

key is the stable identity React uses to match elements across renders.A key az a stabil identitás, amivel a React renderek között összepárosítja az elemeket.

With index keys, inserting at the top shifts every key and misassigns DOM/state.Index-key esetén a lista elejére szúrás minden key-t elcsúsztat, és rosszul rendeli hozzá a DOM-ot/state-et.

Why not use the array index as a key?Miért ne használjuk a tömb indexét key-ként?

  • On insert/reorder/delete, indexes shift, so React reuses the wrong DOM node and component state — e.g. an input's value sticks to the wrong row.Beszúrásnál/átrendezésnél/törlésnél az indexek elcsúsznak, így a React rossz DOM-node-ot és komponens-state-et használ újra — pl. egy input értéke rossz sorhoz ragad.

What makes a good key?Mitől jó egy key?

  • Stable, unique among siblings, and tied to the data's identity (an ID) — not its position. It only needs to be unique within the list, not globally.Stabil, a testvérek között egyedi, és az adat identitásához (egy ID-hez) kötött — nem a pozíciójához. Csak a listán belül kell egyedinek lennie, globálisan nem.

Watch outBuktatóIndex keys are safe only for static, never-reordered lists. The classic bug: uncontrolled inputs in a reorderable list keyed by index.Az index-key csak statikus, soha nem átrendezett listáknál biztonságos. A klasszikus hiba: uncontrolled inputok index szerint key-elt, átrendezhető listában.

Rules of Hooks

Call hooks only at the top level — never in loops, conditions or nested functions — and only from React functions (components or custom hooks).A hookokat csak a legfelső szinten hívd — soha ne ciklusban, feltételben vagy beágyazott függvényben — és csak React függvényekből (komponensek vagy custom hookok).

Why the top-level rule?Miért van a top-level szabály?

  • React tracks hook state by call order, not by name. A conditional hook shifts the order between renders, so state gets misassigned to the wrong hook.A React a hook-state-et hívási sorrend alapján tartja nyilván, nem név szerint. Egy feltételes hook elcsúsztatja a sorrendet a renderek között, így a state rossz hookhoz kerül.

How do you do conditional logic then?Akkor hogyan csinálsz feltételes logikát?

  • Put the condition inside the hook (e.g. inside useEffect), or split into separate components — never wrap the hook call itself.A feltételt tedd a hookon belülre (pl. a useEffect-en belül), vagy bontsd külön komponensekre — soha ne magát a hook-hívást csomagold feltételbe.

Watch outBuktatóeslint-plugin-react-hooks catches most violations — don't disable it to “make it work.”Az eslint-plugin-react-hooks a legtöbb szabálysértést elkapja — ne kapcsold ki, hogy „működjön”.

04

State & Batching

useState returns a value + setter; React batches multiple setState calls in one event into a single re-render.A useState egy értéket és egy settert ad vissza; a React egy eseményen belül több setState hívást egyetlen re-renderbe von össze (batching).

Why does calling setCount(count + 1) three times only add 1?Miért ad hozzá csak 1-et, ha háromszor hívod a setCount(count + 1)-et?

  • All three read the same stale count from the render's closure. Use the functional updater setCount(c => c + 1) so each builds on the latest value.Mindhárom ugyanazt az elavult count-ot olvassa a render closure-jéből. Használd a funkcionális updatert: setCount(c => c + 1), így mindegyik a legfrissebb értékre épül.

Is state synchronous?Szinkron a state?

  • No — setters schedule a re-render; the variable only changes on the next render. React 18 batches across all contexts, including promises and timeouts (automatic batching).Nem — a setterek re-rendert ütemeznek; a változó csak a következő renderben változik meg. A React 18 minden kontextusban batchel, beleértve a promise-okat és timeoutokat is (automatic batching).

Why immutability?Miért kell immutability?

  • React detects change by reference. Mutating state in place doesn't change the reference, so the update is missed. Always return new objects/arrays.A React referencia alapján észleli a változást. A state helyben módosítása nem változtatja meg a referenciát, így a frissítés elveszik. Mindig új objektumot/tömböt adj vissza.

Watch outBuktatóReading state right after setting it gives the old value — updates aren't applied synchronously.Ha közvetlenül a beállítás után olvasod a state-et, a régi értéket kapod — a frissítések nem szinkron módon érvényesülnek.

05

useEffect & Dependencies

useEffect runs a side effect after render, re-running whenever a value in its dependency array changes.A useEffect render után futtat egy mellékhatást, és újrafut, valahányszor a dependency tömbjében lévő érték megváltozik.

What belongs in the dependency array?Mi tartozik a dependency tömbbe?

  • Every reactive value used inside the effect (props, state, and anything derived from them). Missing deps → stale values; the lint rule enforces the full set.Minden reaktív érték, amit az effekten belül használsz (propok, state, és bármi, ami ezekből származik). Hiányzó dep → elavult érték; a lint szabály kikényszeríti a teljes listát.

When does the cleanup function run?Mikor fut le a cleanup függvény?

  • Before the next effect run and on unmount. Use it for subscriptions, timers and listeners to avoid leaks and race conditions.A következő effekt-futás előtt és unmountkor. Használd subscription-ökhöz, timerekhez és listenerekhez, hogy elkerüld a leakeket és a race condition-öket.

[] vs no array vs [deps]?[] vs nincs tömb vs [deps]?

  • No array → after every render. [] → once after mount. [deps] → when deps change.Nincs tömb → minden render után. [] → egyszer, mount után. [deps] → amikor a depek változnak.

Watch outBuktatóDon't use an effect to derive state from props/state — compute it during render. Effects are for synchronizing with external systems, not reacting to your own state. Object/function deps recreated each render cause infinite loops → memoize or move them.Ne használj effektet arra, hogy propokból/state-ből state-et származtass — számold ki render közben. Az effektek külső rendszerekkel való szinkronizálásra valók, nem a saját state-edre való reagálásra. A minden renderben újra létrehozott objektum/függvény depek végtelen ciklust okoznak → memoizáld vagy mozgasd ki őket.

Stale Closures

A closure that captured a value from an earlier render and keeps using that old snapshot.Olyan closure, ami egy korábbi renderből fogott be egy értéket, és továbbra is azt a régi pillanatképet használja.

Where does it bite?Hol üt vissza?

  • Event handlers, setTimeout/setInterval, and effects with missing deps — they all close over the render's snapshot of state and props.Event handlerekben, setTimeout/setInterval-ban, és hiányzó depekkel rendelkező effektekben — mindegyik a render state- és prop-pillanatképét zárja be.

How do you fix it?Hogyan javítod?

  • Add the value to the deps (re-create the closure), use the functional updater, or stash the latest value in a ref and read ref.current.Vedd fel az értéket a depek közé (újra létrejön a closure), használd a funkcionális updatert, vagy tedd a legfrissebb értéket egy refbe és olvasd a ref.current-et.

Watch outBuktatóA setInterval created in useEffect([]) that reads state will forever see the initial value — the textbook stale closure.Egy useEffect([])-ben létrehozott setInterval, ami state-et olvas, örökre a kezdeti értéket látja — ez a tankönyvi stale closure.

07

Memoization

Caching to skip work: React.memo skips a re-render on shallow-equal props; useMemo caches a computed value; useCallback caches a function reference.Cache-elés a munka megspórolására: a React.memo kihagyja a re-rendert, ha a propok shallow-egyenlők; a useMemo egy kiszámolt értéket cache-el; a useCallback egy függvény-referenciát.

A new object/function passed as a prop defeats memo's shallow comparison.Egy propként átadott új objektum/függvény kivédi a memo shallow összehasonlítását.

When is each worth it?Mikor éri meg melyik?

  • useMemo — a genuinely expensive computation, or a stable value fed to memoized children / effect deps.useMemo — valóban drága számításnál, vagy ha stabil értéket adsz memoizált gyerekeknek / effekt-depeknek.
  • useCallback — a function passed to a memo child or used as an effect dep.useCallback — ha a függvényt memo-zott gyereknek adod át, vagy effekt-depként használod.
  • React.memo — a component that re-renders often with the same props and has a non-trivial render.React.memo — olyan komponensnél, ami gyakran renderel újra ugyanazokkal a propokkal, és nem triviális a renderje.

Why do useMemo/useCallback often do nothing?Miért nem csinál gyakran semmit a useMemo/useCallback?

  • If the memoized value still feeds a non-memoized child, or the deps change every render, you pay the overhead for no benefit.Ha a memoizált érték továbbra is nem-memoizált gyerekhez megy, vagy a depek minden renderben változnak, csak az overheadet fizeted haszon nélkül.

Watch outBuktatóMemoization costs memory + comparison. Default to not memoizing; add it when profiling shows a real problem. (The React Compiler auto-memoizes, reducing the manual need.)A memoizáció memóriába és összehasonlításba kerül. Alapból ne memoizálj; akkor tedd, ha a profilozás valós problémát mutat. (A React Compiler automatikusan memoizál, csökkentve a kézi igényt.)

useRef

A mutable container ({ current }) that persists across renders without triggering one when it changes.Egy módosítható konténer ({ current }), ami megmarad a renderek között, és változáskor nem vált ki re-rendert.

Two main uses?Két fő felhasználás?

  • Accessing DOM nodes (ref={myRef}), and holding mutable values that shouldn't cause renders (timer IDs, the previous value, a “latest state” ref).DOM-node-ok elérése (ref={myRef}), és olyan módosítható értékek tárolása, amiknek nem szabad rendert okozniuk (timer ID-k, az előző érték, egy „legfrissebb state” ref).

useRef vs useState?

  • Changing state re-renders; changing a ref doesn't. Use a ref when the value isn't part of the rendered output.A state változása re-rendert okoz; a refé nem. Akkor használj refet, ha az érték nem része a renderelt kimenetnek.

Watch outBuktatóDon't read/write ref.current during render for rendered data — it's outside React's data flow and won't reflect in the UI.Ne olvasd/írd a ref.current-et render közben renderelt adathoz — kívül esik a React adatfolyamán, és nem jelenik meg a UI-ban.

09

Context API

Passes data through the tree without prop drilling; consumers re-render when the provider's value changes.Adatot ad át a fán keresztül prop drilling nélkül; a fogyasztók újrarenderelnek, amikor a provider értéke változik.

The performance gotcha?Mi a performance-buktató?

  • Every consumer re-renders on any value change, even if it uses only part of it. A fresh object literal as value re-renders all consumers on every parent render.Minden fogyasztó újrarenderel bármilyen értékváltozásra, még ha csak egy részét használja is. Egy friss objektum-literál value-ként minden szülő-renderkor újrarendereli az összes fogyasztót.

How do you mitigate it?Hogyan enyhíted?

  • Memoize the value, split contexts by update frequency, or move to a store (Zustand/Redux) with selector-based subscriptions for high-frequency state.Memoizáld az értéket, bontsd külön a contextet frissítési gyakoriság szerint, vagy válts store-ra (Zustand/Redux) selector-alapú feliratkozással a gyakran változó state-hez.

Watch outBuktatóContext is a dependency-injection / low-frequency-state tool (theme, auth, locale) — not a general state manager for rapidly-changing data.A Context dependency-injection / ritkán változó state eszköz (theme, auth, locale) — nem általános state manager gyorsan változó adathoz.

10

Custom Hooks

A function starting with use that composes built-in hooks to extract and reuse stateful logic.Egy use-szal kezdődő függvény, ami beépített hookokat komponál, hogy stateful logikát emeljen ki és tegyen újrahasznosíthatóvá.

What do they share — and not share?Mit osztanak meg — és mit nem?

  • They share logic, not state. Each call gets its own independent state — two components using useCounter() don't share a counter.Logikát osztanak meg, nem state-et. Minden hívás saját, független state-et kap — két komponens, ami useCounter()-t használ, nem osztozik egy számlálón.

When should you extract one?Mikor érdemes kiemelni egyet?

  • Repeated stateful logic — fetching, form state, subscriptions, debouncing — to keep components declarative and testable.Ismétlődő stateful logikánál — fetch, form-state, subscription-ök, debounce —, hogy a komponensek deklaratívak és tesztelhetők maradjanak.

Watch outBuktatóA custom hook is just a function — the Rules of Hooks still apply (top-level calls, React-functions only).A custom hook is csak egy függvény — a Rules of Hooks továbbra is érvényes (top-level hívások, csak React függvényekből).

Controlled vs Uncontrolled

Controlled: React state is the source of truth (value + onChange). Uncontrolled: the DOM holds the value, read via a ref (defaultValue).Controlled: a React state a source of truth (value + onChange). Uncontrolled: a DOM tartja az értéket, refen keresztül olvasod (defaultValue).

Trade-offs?Trade-offok?

  • Controlled: full control and live validation, but a re-render per keystroke.Controlled: teljes kontroll és élő validáció, viszont billentyűleütésenként egy re-render.
  • Uncontrolled: less code, good for simple/large forms and file inputs, but harder to validate or transform on the fly.Uncontrolled: kevesebb kód, jó egyszerű/nagy formokhoz és file inputokhoz, viszont nehezebb menet közben validálni vagy transzformálni.

Watch outBuktatóFlipping a field between controlled and uncontrolled (value going undefined → defined) triggers React's warning — always initialize with a defined value like ''.Ha egy mező controlled és uncontrolled között billen (a value undefined → definiált), a React figyelmeztet — mindig definiált értékkel inicializálj, pl. ''.

Client vs Server State

Client state is UI state you own (modals, inputs, toggles). Server state is remote data — fetched async, shared, and able to go stale.A client state a saját UI-állapotod (modalok, inputok, kapcsolók). A server state távoli adat — aszinkron lekérve, megosztott, és el tud avulni.

Why treat them differently?Miért kezeljük őket másképp?

  • Server state needs caching, background refetch, dedup, retry, invalidation and staleness handling — none of which a plain useState or Redux slice gives you. That's what React Query / SWR / RTK Query solve.A server state-hez caching, háttérbeli refetch, dedup, retry, invalidáció és elavultság-kezelés kell — ezt egy sima useState vagy Redux slice nem adja meg. Erre valók a React Query / SWR / RTK Query.

What does a server-state library give you?Mit ad egy server-state library?

  • Request dedup, cache + stale-while-revalidate, background updates, pagination/infinite scroll, optimistic updates, and loading/error state out of the box.Request-dedupot, cache-t + stale-while-revalidate-et, háttérfrissítést, pagination/infinite scrollt, optimistic update-eket, valamint loading/error state-et out of the box.

Watch outBuktatóDumping fetched data into Redux/Context and hand-rolling caching is the source of most “why is my data stale” bugs. Keep client and server state separate.Ha a lekért adatot Reduxba/Contextbe öntöd és kézzel írod a cachinget, abból jön a legtöbb „miért elavult az adatom” bug. Tartsd külön a client és a server state-et.

Data Fetching Patterns

How and when you load remote data — on render, on route load, or on the server.Hogyan és mikor töltesz be távoli adatot — renderkor, route betöltésekor, vagy a szerveren.

Parallelize independent requests; only chain when B truly depends on A.A független kéréseket futtasd párhuzamosan; csak akkor láncold őket, ha B tényleg A-tól függ.

Common problems and fixes?Gyakori problémák és javításuk?

  • Race conditions: a slow earlier request resolves after a newer one — cancel with AbortController or ignore stale results in cleanup.Race condition: egy lassú, korábbi kérés az újabb után fut be — szakítsd meg AbortController-rel, vagy hagyd figyelmen kívül az elavult eredményt a cleanupban.
  • Waterfalls: sequential dependent fetches — parallelize independent ones or fetch higher up.Waterfall: egymásra épülő, soros fetchek — futtasd párhuzamosan a függetleneket, vagy kérd le feljebb.

Where should data be fetched?Hol érdemes lekérni az adatot?

  • Close to where it's used, but avoid per-component waterfalls — route-level or server-side fetching (RSC/loaders) cuts round trips.Közel oda, ahol használod, de kerüld a komponensenkénti waterfallt — a route-szintű vagy szerver-oldali lekérés (RSC/loaderek) csökkenti a körutakat.

Watch outBuktatóuseEffect fetching has no built-in dedup or caching — two components fetching the same thing means two requests. Use a server-state library or hoist the fetch.A useEffect-es fetchnek nincs beépített dedupja vagy cache-e — ha két komponens ugyanazt kéri le, az két kérés. Használj server-state libraryt, vagy emeld feljebb a fetchet.

Suspense & Error Boundaries

Suspense lets a component wait for something (lazy code, data) and show a fallback. Error Boundaries catch render-time errors in their subtree and show a fallback UI.A Suspense lehetővé teszi, hogy egy komponens várjon valamire (lazy kód, adat) és fallbacket mutasson. Az Error Boundary elkapja a részfájában keletkező render-idejű hibákat és fallback UI-t jelenít meg.

What can an Error Boundary catch — and not?Mit kap el egy Error Boundary — és mit nem?

  • Catches: errors during render, lifecycle and child constructors.Elkapja: a render, a lifecycle és a gyerek-konstruktorok hibáit.
  • Doesn't catch: event handlers, async code, SSR, or errors in the boundary itself — use try/catch for those.Nem kapja el: event handlereket, aszinkron kódot, SSR-t, vagy magában a boundaryben keletkező hibákat — ezekhez try/catch kell.

Why is Suspense useful?Miért hasznos a Suspense?

  • Declarative loading states, coordinating multiple async dependencies, and enabling streaming SSR and Server Components.Deklaratív loading state-ek, több aszinkron függőség összehangolása, valamint a streaming SSR és a Server Components lehetővé tétele.

Watch outBuktatóError Boundaries are still class components (or a library like react-error-boundary) — there's no hook that catches render errors.Az Error Boundary továbbra is class komponens (vagy egy library, pl. react-error-boundary) — nincs hook, ami render-hibát kapna el.

Code Splitting & Lazy Loading

Splitting the bundle so the browser downloads only what it needs, loading the rest on demand.A bundle felbontása úgy, hogy a böngésző csak azt töltse le, amire szüksége van, a többit igény szerint.

How in React?Hogyan Reactben?

  • React.lazy(() => import('./X')) + <Suspense fallback>. Split by route first (the biggest win), then heavy components.React.lazy(() => import('./X')) + <Suspense fallback>. Először route szerint bonts (ez a legnagyobb nyereség), utána a nehéz komponenseknél.

Why does it matter?Miért számít?

  • A smaller initial bundle means faster first load (TTI/LCP). The bundler emits a separate chunk per dynamic import.A kisebb kezdeti bundle gyorsabb első betöltést jelent (TTI/LCP). A bundler dinamikus importonként külön chunkot állít elő.

Watch outBuktatóOver-splitting creates many tiny chunks and request overhead. Split at meaningful boundaries (routes, rarely-used heavy features), not everything.A túlzott bontás sok apró chunkot és kérés-overheadet szül. Értelmes határoknál bonts (route-ok, ritkán használt nehéz feature-ök), ne mindenhol.

List Virtualization

Rendering only the rows currently in the viewport (plus a small buffer) instead of the whole list.Csak a viewportban éppen látható sorok (plusz egy kis buffer) renderelése az egész lista helyett.

The DOM node count stays roughly constant no matter how long the list is.A DOM-node-ok száma nagyjából állandó marad, bármilyen hosszú is a lista.

Why and when?Miért és mikor?

  • Large lists (thousands of rows) create thousands of DOM nodes → slow render, high memory, janky scroll. Virtualization keeps the node count constant. Libraries: react-window, TanStack Virtual.A nagy listák (több ezer sor) több ezer DOM-node-ot hoznak létre → lassú render, magas memóriahasználat, akadozó görgetés. A virtualizáció állandón tartja a node-ok számát. Libraryk: react-window, TanStack Virtual.

The catch?Mi a bökkenő?

  • Off-screen content isn't in the DOM → breaks Ctrl+F and screen readers if unhandled, and needs known or estimated row heights (variable heights are trickier).A képernyőn kívüli tartalom nincs a DOM-ban → kezeletlenül elrontja a Ctrl+F-et és a képernyőolvasókat, továbbá ismert vagy becsült sormagasság kell hozzá (a változó magasság trükkösebb).

Watch outBuktatóDon't virtualize small lists — the machinery costs more than it saves. It's strictly for the long-list case.Kis listát ne virtualizálj — a gépezet többe kerül, mint amennyit megspórol. Kizárólag hosszú listákra való.

Re-render Optimization

Reducing unnecessary re-renders and the cost of each one.A felesleges re-renderek és az egyes renderek költségének csökkentése.

The main strategies, in order?A fő stratégiák, sorrendben?

  • Don't re-render: push state down, split components, pass children as props so a re-rendering parent doesn't re-render static subtrees.Ne rendereljen újra: told le a state-et, bontsd szét a komponenseket, add át a children-t propként, hogy az újrarenderelő szülő ne rendereljen újra statikus részfákat.
  • Then memoize: React.memo + stable props via useMemo/useCallback.Utána memoizálj: React.memo + stabil propok useMemo/useCallback segítségével.
  • For global state: selector-based subscriptions so only components using a changed slice update.Globális state-hez: selector-alapú feliratkozás, hogy csak a megváltozott szeletet használó komponensek frissüljenek.

How do you find the problem?Hogyan találod meg a problémát?

  • The React DevTools Profiler (highlight updates / flamegraph). Measure before optimizing.A React DevTools Profilerrel (highlight updates / flamegraph). Előbb mérj, aztán optimalizálj.

Watch outBuktatóWrapping everything in memo is an anti-pattern — it adds comparison cost and hides the real fix, which is usually state placement or component structure.Mindent memo-ba csomagolni antipattern — összehasonlítási költséget ad, és elfedi a valódi megoldást, ami általában a state elhelyezése vagy a komponens-struktúra.

Browser Event Loop

The single-threaded loop that runs JS: empty the call stack, drain all microtasks, render, then take one macrotask — and repeat.Az egyszálú ciklus, ami a JS-t futtatja: kiüríti a call stacket, lefuttatja az összes microtaskot, renderel, majd vesz egy macrotaskot — és ismétli.

Microtasks drain fully before the next macrotask — so a resolved Promise runs before setTimeout(0).A microtaskok teljesen lefutnak a következő macrotask előtt — ezért egy feloldott Promise előbb fut, mint a setTimeout(0).

Microtasks vs macrotasks?Microtask vs macrotask?

  • Microtasks (Promise callbacks, queueMicrotask, MutationObserver) drain fully after each task, before rendering.A microtaskok (Promise callbackek, queueMicrotask, MutationObserver) minden task után teljesen lefutnak, a renderelés előtt.
  • Macrotasks (setTimeout, events, I/O) run one per loop iteration — so a Promise resolves before setTimeout(0).A macrotaskokból (setTimeout, események, I/O) ciklusonként egy fut le — ezért a Promise előbb oldódik fel, mint a setTimeout(0).

Why does this matter for UI?Miért számít ez a UI-nak?

  • Long synchronous tasks or microtask floods block rendering and input → jank and poor INP. Break work up with chunking, setTimeout, or requestIdleCallback.A hosszú szinkron taskok vagy a microtask-áradat blokkolja a renderelést és az inputot → akadozás és rossz INP. Darabold fel a munkát chunkinggal, setTimeout-tal vagy requestIdleCallback-kel.

Watch outBuktatóThis is the browser loop (task → microtasks → render). Node's event loop has different phases (timers, poll, check…) — don't conflate them in a full-stack interview.Ez a böngésző ciklusa (task → microtaskok → render). A Node event loopjának más fázisai vannak (timers, poll, check…) — full-stack interjún ne keverd össze a kettőt.

Reflow & Repaint

Reflow (layout) recomputes element geometry and can cascade; repaint redraws pixels without a layout change. Reflow is the more expensive one.A reflow (layout) újraszámolja az elemek geometriáját, és tovagyűrűzhet; a repaint pixeleket rajzol újra layout-változás nélkül. A reflow a drágább.

Animating transform/opacity jumps to the composite step and avoids layout.A transform/opacity animálása a composite lépésre ugrik, és elkerüli a layoutot.

What triggers reflow, and what's “layout thrashing”?Mi vált ki reflow-t, és mi a „layout thrashing”?

  • Triggers: DOM changes, size/position style changes, and reading layout props (offsetHeight, getBoundingClientRect).Kiváltók: DOM-változások, méret/pozíció stílusváltozások, és a layout-propok olvasása (offsetHeight, getBoundingClientRect).
  • Thrashing: interleaving reads and writes in a loop forces a synchronous reflow each time. Fix: batch reads, then writes.Thrashing: ha ciklusban váltogatod az olvasást és az írást, az minden alkalommal szinkron reflow-t kényszerít ki. Javítás: előbb minden olvasás, aztán az írások.

Watch outBuktatóAnimate with transform/opacity, not top/left/width — the latter trigger reflow every frame.transform/opacity-vel animálj, ne top/left/width-tel — utóbbiak minden frame-ben reflow-t váltanak ki.

CORS

A browser security mechanism: cross-origin requests are only readable if the server opts in via response headers. Origin = scheme + host + port.Böngésző-oldali biztonsági mechanizmus: a cross-origin kérések válasza csak akkor olvasható, ha a szerver ezt response headerekkel engedélyezi. Az origin = scheme + host + port.

How does it actually work?Hogyan működik valójában?

  • The browser adds an Origin header; the server responds with Access-Control-Allow-Origin. If it doesn't match, the browser blocks the JS from reading the response.A böngésző hozzáad egy Origin headert; a szerver Access-Control-Allow-Origin-nal válaszol. Ha nem egyezik, a böngésző megakadályozza, hogy a JS olvassa a választ.

What's a preflight?Mi az a preflight?

  • For “non-simple” requests (custom headers, methods like PUT/DELETE, certain content types), the browser first sends an OPTIONS request; the server must approve with Allow-Methods/Allow-Headers.A „nem egyszerű” kéréseknél (egyedi headerek, PUT/DELETE metódusok, bizonyos content type-ok) a böngésző előbb egy OPTIONS kérést küld; a szervernek jóvá kell hagynia Allow-Methods/Allow-Headers-szel.

Watch outBuktatóCORS is enforced by the browser, not the server — the request often still reaches the server. It protects the user, not the server, and it isn't authentication. Curl and Postman ignore it entirely.A CORS-t a böngésző kényszeríti ki, nem a szerver — a kérés gyakran így is eljut a szerverhez. A felhasználót védi, nem a szervert, és nem authentication. A curl és a Postman teljesen figyelmen kívül hagyja.

TypeScript + React

Static typing over JS — in React it types props, state, hooks and events, catching errors at compile time.Statikus típusosság a JS fölött — Reactben a propokat, state-et, hookokat és eseményeket típusozza, így fordítási időben elkapja a hibákat.

Key features to know?Mely funkciókat érdemes tudni?

  • Generics (reusable typed components/hooks), utility types (Partial, Pick, Omit, Record, ReturnType), unions + narrowing, and discriminated unions for variant props.Generikusok (újrahasznosítható, típusos komponensek/hookok), utility típusok (Partial, Pick, Omit, Record, ReturnType), unionok + narrowing, és discriminated unionok a variáns propokhoz.
  • type vs interface: interfaces merge/extend (good for public APIs); types do unions, mapped and conditional types. Mostly interchangeable for object shapes.type vs interface: az interface-ek merge-elhetők/kiterjeszthetők (jó publikus API-khoz); a type-ok tudnak uniont, mapped és conditional típusokat. Objektum-alakokhoz nagyrészt felcserélhetők.

Common typing patterns?Gyakori típusozási patternek?

  • Props via interface Props {}; children as React.ReactNode; events as React.ChangeEvent<HTMLInputElement>; generic components as function List<T>(props: { items: T[] }).Propok interface Props {}-szal; children mint React.ReactNode; események mint React.ChangeEvent<HTMLInputElement>; generikus komponensek: function List<T>(props: { items: T[] }).

Watch outBuktatóAvoid any (it disables checking) and overusing as casts (they bypass safety) — prefer narrowing and generics.Kerüld az any-t (kikapcsolja az ellenőrzést) és az as castok túlzott használatát (megkerülik a biztonságot) — inkább narrowing és generikusok.
