Node Runtime
01

Event Loop (Node)

Node runs your JS on a single thread; the event loop (via libuv) processes callbacks in ordered phases, one cycle at a time.

Each phase runs its callback queue; nextTick + Promise microtasks drain between them.

How does it differ from the browser event loop?

  • Browser: task → drain microtasks → render. Node has multiple phases (timers, poll, check…), no rendering, and adds process.nextTick as a separate, higher-priority queue.

Where do Promises and nextTick run?

  • After each operation/phase transition, Node drains the process.nextTick queue first, then the Promise microtask queue — before continuing.

Watch out“Single-threaded” refers to your JS. libuv and the OS handle the actual I/O, so one Node process serves thousands of concurrent connections without a thread per request.

02

libuv & Thread Pool

libuv is the C library behind Node's event loop and async I/O. Most I/O is OS-level async; a few operations use a small thread pool.

Only pool-backed ops (fs, DNS, some crypto/zlib) are limited by the thread count.

What actually uses the thread pool?

  • File system, DNS lookups (dns.lookup), some crypto (e.g. pbkdf2) and zlib. Network sockets do not — they're OS-async (epoll/kqueue/IOCP).
  • Default pool size is 4 (UV_THREADPOOL_SIZE, up to 1024).

Why does this matter?

  • CPU-heavy pool work (many crypto hashes) can exhaust the 4 threads and stall unrelated fs/DNS work — tune the pool or move to worker threads.

Watch outNetwork requests scale far beyond the pool because sockets are OS-async — only pool-backed operations are capped by UV_THREADPOOL_SIZE.

03

nextTick / setImmediate / setTimeout

Three ways to defer work, each with different timing and priority.

APIRunsPriority
process.nextTickbefore the loop continues (after current op)highest
Promise.thenmicrotask, right after the nextTick queuehigh
setImmediatecheck phase (just after poll)per-loop
setTimeout(_, 0)timers phase (≥ ~1ms)per-loop

setImmediate vs setTimeout(0) — which fires first?

  • Inside an I/O callback, setImmediate always wins (check comes right after poll). At the top level, the order is non-deterministic.

Watch outRecursive process.nextTick can starve the event loop — I/O never gets a turn. Prefer setImmediate for “run after I/O” deferral.

04

Blocking the Event Loop

Any long synchronous work blocks the single JS thread — heavy loops, sync crypto, parsing huge payloads, sync fs. No other request is served until it finishes.

How do you avoid it?

  • Never use *Sync fs/crypto methods in request handlers; offload CPU-bound work to Worker Threads or a separate service; stream large data instead of buffering; paginate and limit.

How do you detect it?

  • Event-loop lag metrics (perf_hooks.monitorEventLoopDelay), an APM tool, or clinic.js.

Watch outOne slow synchronous handler degrades the whole process for every user — the single most common Node performance pitfall.

05

Workers, Cluster & Child Process

Three ways to use more than one thread or process — Worker Threads (CPU work), Cluster (scale a server across cores), Child Process (run other programs).

Cluster workers share the listening port; the OS/master load-balances connections.

CPU-bound vs scaling — which do you reach for?

  • CPU-bound task (image processing, crypto) → Worker Threads (real threads, same process).
  • Use all cores for a web server → Cluster (forked processes sharing a port) — or run N containers behind a load balancer, common in production.

Why not just use threads everywhere?

  • Node's model is built for I/O-bound concurrency on one thread; threads/processes add memory + coordination cost. Use them for CPU work, not I/O.

Watch outCluster workers don't share memory — shared state needs an external store (Redis). In k8s, horizontal pod scaling usually replaces cluster.

06

CommonJS vs ES Modules

Two module systems. CJS (require/module.exports) — synchronous, Node's original. ESM (import/export) — the standard, static, async.

Key differences?

  • CJS loads synchronously and resolves at runtime; ESM is static (imports hoisted, analyzable, tree-shakeable).
  • ESM has top-level await and import.meta.url; CJS has __dirname/__filename. ESM exports are live bindings; CJS exports are a value snapshot.

How do you pick per file?

  • .mjs or "type": "module" → ESM; .cjs or default → CJS. ESM can import CJS; CJS importing ESM needs dynamic import().

Watch outYou can't require() an ESM module — mixing the two trips people up. New projects default to ESM.

07

Streams & Backpressure

Streams process data in chunks instead of loading it all into memory. Four types: Readable, Writable, Duplex, Transform.

When the sink can't keep up, the source pauses until the buffer drains.

Why streams?

  • Constant memory for arbitrarily large data (files, HTTP bodies, uploads), start processing before all data arrives, and composability via pipe/pipeline.

What is backpressure?

  • When a writable can't keep up, its buffer fills. pipeline() pauses the source until the destination drains — preventing unbounded memory growth.

Watch outUse stream.pipeline(), not raw .pipe() — pipeline handles backpressure and propagates errors + cleans up. Plain .pipe() silently drops errors.

08

EventEmitter

Node's pub/sub primitive: objects emit named events, listeners subscribe with .on(). Much of core (streams, HTTP server, process) is built on it.

Is it sync or async?

  • Listeners run synchronously in registration order when emit() fires — it's not a queue; emit blocks until every listener returns.

Common pitfalls?

  • Leaks from listeners never removed (the “MaxListenersExceededWarning” at >10). Use .once() for one-shot and remove listeners on cleanup.

Watch outAn 'error' event with no listener throws and can crash the process — always attach an error listener to emitters and streams.

Async & Errors
09

Callbacks → Promises → async/await

The evolution of async in Node: error-first callbacks → Promises → async/await syntax over Promises.

What problem did each solve?

  • Callbacks → “callback hell” (nesting, inversion of control). Promises → chaining and a single error path. async/await → synchronous-looking code with try/catch.

How do async functions work?

  • An async function always returns a Promise; await pauses it until the Promise settles — without blocking the event loop, so other work runs meanwhile.

Watch outawait in a loop serializes independent operations — use Promise.all for parallelism. And forgetting to await returns a pending Promise (a silent bug).

10

Async Error Handling

Errors must be caught on the right path — a sync throw, a rejected Promise, or an error-first callback.

How do you catch each?

  • async/await → try/catch. Promises → .catch(). Callbacks → check the first arg. Streams/emitters → an 'error' listener.

The dangerous cases?

  • Unhandled rejections (process.on('unhandledRejection')) — which crash by default in current Node — and uncaught exceptions.

Watch outA throw inside a setTimeout/callback isn't caught by an outer try/catch — the try already returned. Let a top-level handler log & exit; don't try to “resume” after an uncaught exception — process state is unknown.

11

Concurrency Control

Running multiple async operations with the right amount of parallelism.

Promise combinators?

  • all — all succeed, or reject on first failure. allSettled — wait for all, get each result/error. race — first to settle. any — first to fulfill.

Why limit concurrency?

  • Firing 10,000 requests/queries at once exhausts sockets, connections and memory. Cap in-flight work with a limiter (p-limit) or batching.

Watch outPromise.all rejects on the first failure, but the other promises keep running — they aren't cancelled. Use allSettled when you need every result regardless of individual failures.

Data & Persistence
12

SQL vs NoSQL

Relational (SQL) vs non-relational (NoSQL) databases — chosen by data model and access pattern.

TypeModelBest forExamples
SQLrelational, schema, joins, ACIDstructured data, complex queries, transactionsPostgres, MySQL
Documentflexible JSON documentsevolving schemas, nested dataMongoDB
Key-valuekey → value, very fastcaching, sessions, high throughputRedis, DynamoDB
Graphnodes + edgesrelationship-heavy queriesNeo4j

How do you choose?

  • SQL by default for most apps (relations, transactions, mature tooling). Reach for NoSQL for a specific access pattern: schema flexibility, massive horizontal scale, or key-based/high-write workloads.

Watch out“NoSQL scales, SQL doesn't” is outdated — modern Postgres scales far. Choose by data model, not hype; many systems use both (polyglot persistence).

13

Database Indexing

A data structure (usually a B-tree) that lets the DB find rows without scanning the whole table — like a book's index.

When does an index help, and when does it hurt?

  • Helps: WHERE, JOIN, ORDER BY on the indexed column(s) — turns an O(n) scan into an O(log n) lookup.
  • Hurts: every write must update the index → slower writes + storage. Unused indexes are pure cost.

Composite index / column order?

  • A multi-column index (a, b) serves queries on a or a + b (leftmost-prefix rule), but not on b alone.

Watch outIndexes don't help low-selectivity columns (e.g. a boolean) or leading-wildcard LIKE '%x'. Use EXPLAIN ANALYZE to confirm one is actually used.

14

Transactions & ACID

A transaction groups operations so they succeed or fail together. ACID = Atomicity, Consistency, Isolation, Durability.

Isolation levelPrevents
Read Uncommitted— (allows dirty reads)
Read Committeddirty reads · Postgres default
Repeatable Read+ non-repeatable reads
Serializable+ phantom reads · strictest

What does each ACID letter mean?

  • Atomicity — all-or-nothing. Consistency — constraints stay valid. Isolation — concurrent txns don't corrupt each other. Durability — committed data survives crashes.

Watch outHigher isolation trades throughput for safety (more locking). And long-running transactions hold locks and bloat — keep them short.

15

N+1 Query Problem

Fetching N parent rows, then firing one query per parent for its children → 1 + N queries instead of one.

Invisible with 10 rows in dev; catastrophic with 10,000 in prod.

Where does it come from, and how do you fix it?

  • Classic with ORMs + lazy loading: loop over users, access user.posts → a query each.
  • Fix: eager-load with a JOIN, or batch into one WHERE ... IN (...) (DataLoader batches + caches per request).

Watch outInspect query counts and use EXPLAIN — the fix is ORM eager-loading (include/join), not more indexes.

16

Connection Pooling

Reusing a fixed set of open DB connections across requests instead of opening/closing one per query.

The pool caps total connections and skips the per-query TCP + auth handshake.

Why pool?

  • Opening a DB connection is expensive (TCP + auth). A pool keeps connections warm; requests borrow and return them, capping the total the DB must handle.

How do you size it?

  • Bounded by the DB's max connections and CPU — bigger isn't better. Too many connections thrash the DB; a modest pool per instance often wins.

Watch outPool exhaustion (all borrowed, none returned) makes requests hang — usually from leaked connections or long queries. In serverless, use a pooler like PgBouncer to avoid connection storms.

17

Caching Strategies

Storing hot data in a fast store (Redis/in-memory) to cut DB load and latency. This is application-level caching — distinct from HTTP caching.

Cache-aside: read cache first; on a miss, read the DB and populate the cache.

Cache-aside (lazy) pattern?

  • Read: check cache → hit returns; miss → read DB, write to cache, return. Write: update DB, then invalidate/update the cache entry.

Other patterns, and the hard part?

  • Write-through (cache + DB together), write-behind (async). The hard part is invalidation + staleness — use TTLs and explicit invalidation on writes.

Watch outCache stampede: when a hot key expires, many misses hit the DB at once. Mitigate with a lock, staggered TTLs, or stale-while-revalidate.

Communication & Reliability
18

WebSockets vs SSE

Ways to push server data to clients in real time — polling, Server-Sent Events, and WebSockets.

SSE is one-way server→client over HTTP; WebSockets are full-duplex over a persistent TCP connection.

How do you choose?

  • One-way updates (notifications, live scores, log streams) → SSE — simpler, plain HTTP, auto-reconnect, text only.
  • Bidirectional / low-latency (chat, multiplayer, collab editing) → WebSockets — full-duplex, binary or text.

Watch outA WebSocket connection lives on one server — scaling across instances needs sticky sessions or a shared pub/sub (Redis).

19

gRPC

A high-performance RPC framework over HTTP/2 with Protocol Buffers (binary, schema-defined) for service-to-service communication.

Why gRPC over REST for internal services?

  • Binary protobuf (smaller/faster than JSON), HTTP/2 multiplexing + streaming, strongly-typed contracts with codegen, and bidirectional streaming.

Trade-offs / when not to?

  • Not natively browser-friendly (needs grpc-web + a proxy), harder to debug (binary), less human-readable. REST/JSON stays better for public/browser-facing APIs.

Watch outgRPC shines for internal microservice-to-microservice traffic; for public APIs consumed by browsers or third parties, REST or GraphQL is usually the better fit.

20

Graceful Shutdown

On a termination signal, stop taking new work, finish in-flight requests, close resources, then exit — instead of dropping connections mid-flight.

What's the sequence?

  • Catch SIGTERM → server.close() (stop accepting new connections) → let in-flight requests finish (with a timeout) → close DB pools / flush queues → exit(0). Force-exit if it hangs.

Why does it matter?

  • In k8s/containers, deploys send SIGTERM. Without graceful shutdown you drop live requests and corrupt in-flight work on every rollout.

Watch outAlways include a hard timeout — a hung connection shouldn't block shutdown forever (the orchestrator will SIGKILL after the grace period anyway).

21

Logging & Observability

The three pillars: logs (events), metrics (numbers over time), traces (a request's path across services).

Logging best practices in Node?

  • Structured JSON logs (pino/winston), log levels, a correlation/request ID per request, and no secrets/PII. Write to stdout and let the platform ship them.

What are traces for?

  • Following one request across services (OpenTelemetry) — essential for debugging latency and errors in a distributed system.

Watch outconsole.log is synchronous and can block under load, and unstructured logs are hard to query — use a proper structured logger to stdout.

22

Env Config & 12-Factor

Config (secrets, URLs, flags) belongs in the environment, not in code — one build runs unchanged in every environment.

Key principles?

  • Config in env vars (never committed); secrets in a secret manager (Vault / AWS Secrets Manager / KMS); validate config at startup and fail fast on missing vars.

Why?

  • The same artifact promotes dev → staging → prod with only env differences, keeps secrets out of source control, and makes the app portable/containerizable.

Watch outCommitting a .env with real secrets is a classic leak — gitignore it, rotate anything exposed, and inject secrets at deploy time.
