React

A mutable container ({ current }) that persists across renders without triggering one when it changes.Egy módosítható konténer ({ current }), ami megmarad a renderek között, és változáskor nem vált ki re-rendert.

Two main uses?Két fő felhasználás?

useRef vs useState?

  • Changing state re-renders; changing a ref doesn't. Use a ref when the value isn't part of the rendered output.A state változása re-rendert okoz; a refé nem. Akkor használj refet, ha az érték nem része a renderelt kimenetnek.

Suspense & Error Boundaries

React

Suspense lets a component wait for something (lazy code, data) and show a fallback. Error Boundaries catch render-time errors in their subtree and show a fallback UI.A Suspense lehetővé teszi, hogy egy komponens várjon valamire (lazy kód, adat) és fallbacket mutasson. Az Error Boundary elkapja a részfájában keletkező render-idejű hibákat és fallback UI-t jelenít meg.

What can an Error Boundary catch — and not?Mit kap el egy Error Boundary — és mit nem?

  • Catches: errors during render, lifecycle and child constructors.Elkapja: a render, a lifecycle és a gyerek-konstruktorok hibáit.
  • Doesn't catch: event handlers, async code, SSR, or errors in the boundary itself — use try/catch for those.Nem kapja el: event handlereket, aszinkron kódot, SSR-t, vagy magában a boundaryben keletkező hibákat — ezekhez try/catch kell.

Why is Suspense useful?Miért hasznos a Suspense?

  • Declarative loading states, coordinating multiple async dependencies, and enabling streaming SSR and Server Components.Deklaratív loading state-ek, több aszinkron függőség összehangolása, valamint a streaming SSR és a Server Components lehetővé tétele.

Watch outBuktatóError Boundaries are still class components (or a library like react-error-boundary) — there's no hook that catches render errors.Az Error Boundary továbbra is class komponens (vagy egy library, pl. react-error-boundary) — nincs hook, ami render-hibát kapna el.

React

How in React?Hogyan Reactben?

  • React.lazy(() => import('./X')) + <Suspense fallback>. Split by route first (the biggest win), then heavy components.React.lazy(() => import('./X')) + <Suspense fallback>. Először route szerint bonts (ez a legnagyobb nyereség), utána a nehéz komponenseknél.

Why does it matter?Miért számít?

  • A smaller initial bundle means faster first load (TTI/LCP). The bundler emits a separate chunk per dynamic import.A kisebb kezdeti bundle gyorsabb első betöltést jelent (TTI/LCP). A bundler dinamikus importonként külön chunkot állít elő.

Watch outBuktatóOver-splitting creates many tiny chunks and request overhead. Split at meaningful boundaries (routes, rarely-used heavy features), not everything.A túlzott bontás sok apró chunkot és kérés-overheadet szül. Értelmes határoknál bonts (route-ok, ritkán használt nehéz feature-ök), ne mindenhol.

Reflow & Repaint

React

Reflow (layout) recomputes element geometry and can cascade; repaint redraws pixels without a layout change. Reflow is the more expensive one.A reflow (layout) újraszámolja az elemek geometriáját, és tovagyűrűzhet; a repaint pixeleket rajzol újra layout-változás nélkül. A reflow a drágább.

Animating transform/opacity jumps to the composite step and avoids layout.A transform/opacity animálása a composite lépésre ugrik, és elkerüli a layoutot.

What triggers reflow, and what's “layout thrashing”?Mi vált ki reflow-t, és mi a „layout thrashing”?

  • Triggers: DOM changes, size/position style changes, and reading layout props (offsetHeight, getBoundingClientRect).Kiváltók: DOM-változások, méret/pozíció stílusváltozások, és a layout-propok olvasása (offsetHeight, getBoundingClientRect).
  • Thrashing: interleaving reads and writes in a loop forces a synchronous reflow each time. Fix: batch reads, then writes.Thrashing: ha ciklusban váltogatod az olvasást és az írást, az minden alkalommal szinkron reflow-t kényszerít ki. Javítás: előbb minden olvasás, aztán az írások.

Watch outBuktatóAnimate with transform/opacity, not top/left/width — the latter trigger reflow every frame.transform/opacity-vel animálj, ne top/left/width-tel — utóbbiak minden frame-ben reflow-t váltanak ki.

Workers, Cluster & Child Process

Node

Three ways to use more than one thread or process — Worker Threads (CPU work), Cluster (scale a server across cores), Child Process (run other programs).Három módszer több szál vagy processz használatára — Worker Threads (CPU-munka), Cluster (szerver skálázása magokra), Child Process (más programok futtatása).

Cluster workers share the listening port; the OS/master load-balances connections.A cluster workerek közösen használják a figyelő portot; az OS/master osztja el a kapcsolatokat.

CPU-bound vs scaling — which do you reach for?CPU-igényes munka vs skálázás — melyikhez nyúlsz?

  • CPU-bound task (image processing, crypto) → Worker Threads (real threads, same process).CPU-igényes feladat (képfeldolgozás, crypto) → Worker Threads (valódi szálak, ugyanabban a processzben).
  • Use all cores for a web server → Cluster (forked processes sharing a port) — or run N containers behind a load balancer, common in production.Az összes mag kihasználása webszerverhez → Cluster (forkolt processzek közös porton) — vagy N konténer egy load balancer mögött, ez a gyakoribb éles környezetben.

Why not just use threads everywhere?Miért ne használjunk mindenhol szálakat?

  • Node's model is built for I/O-bound concurrency on one thread; threads/processes add memory + coordination cost. Use them for CPU work, not I/O.A Node modellje I/O-igényes párhuzamosságra épül egyetlen szálon; a szálak/processzek memória- és koordinációs költséget adnak. CPU-munkára használd őket, ne I/O-ra.

Watch outBuktatóCluster workers don't share memory — shared state needs an external store (Redis). In k8s, horizontal pod scaling usually replaces cluster.A cluster workerek nem osztanak memóriát — a közös állapothoz külső tár kell (Redis). Kubernetesben a horizontális pod-skálázás általában kiváltja a clustert.

CommonJS vs ES Modules

Node

Two module systems. CJS (require/module.exports) — synchronous, Node's original. ESM (import/export) — the standard, static, async.Két modulrendszer. CJS (require/module.exports) — szinkron, a Node eredeti megoldása. ESM (import/export) — a szabvány, statikus, aszinkron.

Key differences?Fő különbségek?

  • CJS loads synchronously and resolves at runtime; ESM is static (imports hoisted, analyzable, tree-shakeable).A CJS szinkron tölt be és futásidőben oldja fel; az ESM statikus (az importok hoistolódnak, elemezhetők, tree-shakelhetők).
  • ESM has top-level await and import.meta.url; CJS has __dirname/__filename. ESM exports are live bindings; CJS exports are a value snapshot.Az ESM-ben van top-level await és import.meta.url; a CJS-ben __dirname/__filename. Az ESM exportok élő kötések; a CJS exportok érték-pillanatképek.

How do you pick per file?Hogyan választasz fájlonként?

  • .mjs or "type": "module" → ESM; .cjs or default → CJS. ESM can import CJS; CJS importing ESM needs dynamic import()..mjs vagy "type": "module" → ESM; .cjs vagy alapértelmezés → CJS. Az ESM tud CJS-t importálni; a CJS-ből ESM-hez dinamikus import() kell.

Watch outBuktatóYou can't require() an ESM module — mixing the two trips people up. New projects default to ESM.Egy ESM modult nem tudsz require()-tel behúzni — a kettő keverése sokakat megfog. Új projektek alapból ESM-et használnak.

EventEmitter

Node

Node's pub/sub primitive: objects emit named events, listeners subscribe with .on(). Much of core (streams, HTTP server, process) is built on it.A Node pub/sub primitívje: az objektumok névvel ellátott eseményeket bocsátanak ki, a listenerek .on()-nal iratkoznak fel. A core nagy része (streamek, HTTP szerver, process) erre épül.

Is it sync or async?Szinkron vagy aszinkron?

  • Listeners run synchronously in registration order when emit() fires — it's not a queue; emit blocks until every listener returns.A listenerek szinkron futnak, regisztrációs sorrendben, amikor az emit() elsül — ez nem sor; az emit blokkol, amíg minden listener vissza nem tér.

Common pitfalls?Gyakori buktatók?

  • Leaks from listeners never removed (the “MaxListenersExceededWarning” at >10). Use .once() for one-shot and remove listeners on cleanup.Leakek a soha el nem távolított listenerekből (a „MaxListenersExceededWarning” 10 fölött). Egyszeri eseményre .once(), és cleanupkor távolítsd el a listenereket.

Watch outBuktatóAn 'error' event with no listener throws and can crash the process — always attach an error listener to emitters and streams.Egy listener nélküli 'error' esemény dob, és összeomlaszthatja a processzt — mindig tegyél error listenert az emitterekre és a streamekre.

WebSockets vs SSE

Node

Ways to push server data to clients in real time — polling, Server-Sent Events, and WebSockets.Módok arra, hogy a szerver valós időben toljon adatot a klienseknek — polling, Server-Sent Events és WebSocketek.

SSE is one-way server→client over HTTP; WebSockets are full-duplex over a persistent TCP connection.Az SSE egyirányú, szerver→kliens HTTP fölött; a WebSocket full-duplex, tartós TCP kapcsolaton.

How do you choose?Hogyan választasz?

  • One-way updates (notifications, live scores, log streams) → SSE — simpler, plain HTTP, auto-reconnect, text only.Egyirányú frissítések (értesítések, élő eredmények, log streamek) → SSE — egyszerűbb, sima HTTP, automatikus újracsatlakozás, csak szöveg.
  • Bidirectional / low-latency (chat, multiplayer, collab editing) → WebSockets — full-duplex, binary or text.Kétirányú / alacsony latencia (chat, multiplayer, közös szerkesztés) → WebSocket — full-duplex, bináris vagy szöveges.

Watch outBuktatóA WebSocket connection lives on one server — scaling across instances needs sticky sessions or a shared pub/sub (Redis).Egy WebSocket kapcsolat egyetlen szerveren él — a több instance-ra skálázáshoz sticky session vagy közös pub/sub kell (Redis).

Graceful Shutdown

Node

On a termination signal, stop taking new work, finish in-flight requests, close resources, then exit — instead of dropping connections mid-flight.Leállító szignálra ne vegyél fel új munkát, fejezd be a folyamatban lévő kéréseket, zárd le az erőforrásokat, majd lépj ki — ahelyett, hogy menet közben ejtenéd a kapcsolatokat.

What's the sequence?Mi a sorrend?

  • Catch SIGTERM → server.close() (stop accepting new connections) → let in-flight requests finish (with a timeout) → close DB pools / flush queues → exit(0). Force-exit if it hangs.Kapd el a SIGTERM-et → server.close() (ne fogadj új kapcsolatot) → hagyd befejeződni a folyamatban lévő kéréseket (timeouttal) → zárd le a DB poolokat / ürítsd a sorokat → exit(0). Ha beragad, kényszerítsd a kilépést.

Why does it matter?Miért számít?

  • In k8s/containers, deploys send SIGTERM. Without graceful shutdown you drop live requests and corrupt in-flight work on every rollout.Kubernetesben/konténerekben a deploy SIGTERM-et küld. Graceful shutdown nélkül minden rolloutnál élő kéréseket ejtesz el, és megsérül a folyamatban lévő munka.

Watch outBuktatóAlways include a hard timeout — a hung connection shouldn't block shutdown forever (the orchestrator will SIGKILL after the grace period anyway).Mindig tegyél bele kemény timeoutot — egy beragadt kapcsolat ne blokkolja örökre a leállást (az orchestrator a grace period után úgyis SIGKILL-t küld).

Logging best practices in Node?Logging best practice-ek Node-ban?

What are traces for?Mire valók a trace-ek?

Watch outBuktatóconsole.log is synchronous and can block under load, and unstructured logs are hard to query — use a proper structured logger to stdout.A console.log szinkron, és terhelés alatt blokkolhat, a strukturálatlan logokat pedig nehéz lekérdezni — használj rendes strukturált loggert stdout-ra.

Env Config & 12-Factor

Node

Config (secrets, URLs, flags) belongs in the environment, not in code — one build runs unchanged in every environment.A konfiguráció (titkok, URL-ek, flagek) a környezetbe való, nem a kódba — ugyanaz a build fut változtatás nélkül minden környezetben.

Key principles?Fő elvek?

Why?Miért?

Watch outBuktatóCommitting a .env with real secrets is a classic leak — gitignore it, rotate anything exposed, and inject secrets at deploy time.Valódi titkokat tartalmazó .env commitolása klasszikus szivárgás — tedd gitignore-ba, rotálj mindent, ami kikerült, és deploy időben injektáld a titkokat.

Webhooks

API

Why is it needed?Miért van rá szükség?

  • Without it, a slow/dead dependency piles up threads/connections → cascading failure. The breaker isolates the failure and lets the caller degrade gracefully.Nélküle egy lassú/halott függőség szálakat/kapcsolatokat halmoz → cascading failure. A breaker izolálja a hibát, és hagyja, hogy a hívó gracefully degradáljon.

Watch outBuktatóA breaker needs a real fallback (cached data, default, queued work) — tripping without one just relocates the error. Pair with timeouts, or slow calls never count as failures.A breakerhez valódi fallback kell (cache-elt adat, default, sorba tett munka) — nélküle a kioldás csak áthelyezi a hibát. Párosítsd timeouttal, különben a lassú hívások sosem számítanak hibának.

Timeouts & Bulkheads

Timeouts bound how long you wait; bulkheads isolate resources so one failing dependency can't sink the whole system — like watertight ship compartments.A timeout korlátozza, meddig vársz; a bulkhead izolálja az erőforrásokat hogy egy hibázó függőség ne süllyessze el az egész rendszert — mint egy hajó vízzáró rekeszei.

Why are timeouts non-negotiable?Miért nem alku tárgya a timeout?

Every network call needs a timeout + deadline.Nélküle egy beragadt függőség örökre foglal egy szálat/kapcsolatot; elég sok belőlük kimeríti a poolt → teljes kiesés.   • Separate connection/thread pools per dependency (or tenant), so exhausting the pool for service A doesn't starve calls to service B.Külön connection-/thread-pool függőségenként (vagy tenantenként), hogy az A szolgáltatás pooljának kimerítése ne éheztesse ki a B szolgáltatás hívásait.

Watch outBuktatóSet timeouts shorter than the caller's (deadline propagation) — otherwise the client gives up while the server keeps working, wasting resources.Állítsd a timeoutot rövidebbre, mint a hívóé (deadline propagation) — különben a kliens feladja, miközben a szerver tovább dolgozik, erőforrást pazarolva.

Distributed Transactions & Saga

Distributed

A transaction spanning services can't use one ACID commit. A Saga breaks it into a sequence of local transactions, each with a compensating action to undo it on failure.Egy szolgáltatásokon átívelő tranzakció nem használhat egyetlen ACID commitot. A Saga lokális tranzakciók sorozatára bontja, mindegyikhez egy kompenzáló akcióval, amely hiba esetén visszavonja.

If a step fails, the already-completed steps are compensated in reverse order.Ha egy lépés elbukik, a már befejezett lépéseket fordított sorrendben kompenzálják.

Choreography vs orchestration?Choreography vs orchestration?

  • Choreography: services react to each other's events (no coordinator) — decoupled, hard to trace.Choreography: a szolgáltatások egymás eseményeire reagálnak (nincs koordinátor) — lazán csatolt, nehéz nyomon követni.
  • Orchestration: a central orchestrator drives each step and triggers compensations — clearer, but a coordinator to run.Orchestration: egy központi orchestrator vezérli a lépéseket és indítja a kompenzációkat — átláthatóbb, de egy koordinátort kell üzemeltetni.

What guarantee do you get?Milyen garanciát kapsz?

  • Not isolation — intermediate states are visible (eventual consistency). You get atomicity via compensation: all steps complete, or completed ones are undone.Nem isolationt — a köztes állapotok láthatók (eventual consistency). Atomicitást kapsz kompenzáción keresztül: vagy minden lépés befejeződik, vagy a befejezetteket visszavonják.

Watch outBuktatóCompensation isn't a rollback — the money really moved and must be really refunded. Outbox Pattern

Distributed

A fix for the dual-write problem — you can't atomically write to a DB and publish to a broker. Write the event to an “outbox” table in the same DB transaction; a relay reads it and publishes.Megoldás a dual-write problémára — nem tudsz atomikusan írni egy DB-be és publikálni egy brokerre. Írd az eseményt egy „outbox" táblába ugyanabban a DB-tranzakcióban; egy relay beolvassa és publikálja.

The write and the event share one local transaction, so they can't diverge.Az írás és az esemény egy lokális tranzakción osztozik, így nem térhetnek el.

What problem does it solve?Milyen problémát old meg?

  • Without it, a crash between “commit to DB” and “publish event” makes the DB and message log diverge (lost event, or event with no DB change). The outbox makes the write + event atomic.Nélküle egy összeomlás a „DB-be commit" és az „esemény-publikálás" között eltéríti a DB-t és az üzenetlogot (elveszett esemény, vagy esemény DB-változás nélkül). Az outbox atomivá teszi az írás + esemény párost.

How does the relay work?Hogyan működik a relay?

  • A poller or CDC (change-data-capture, e.g. Debezium reading the DB log) reads new outbox rows and publishes them, marking them sent.Egy poller vagy CDC (change-data-capture, pl. a DB-logot olvasó Debezium) beolvassa az új outbox-sorokat és publikálja őket, elküldöttre jelölve.

Watch outBuktatóThe relay publishes at-least-once (it may re-send after a crash before marking sent) — consumers must be idempotent. The outbox guarantees eventual publish, not exactly once.A relay at-least-once publikál (összeomlás után újraküldhet, mielőtt elküldöttre jelölné) — a consumereknek idempotensnek kell lenniük. Az outbox az eventual publish-t garantálja, nem az exactly once-t.

Why event sourcing?Miért event sourcing?

  • Full audit history, time-travel/replay, and the ability to build new read projections from past events. Watch outBuktatóBoth add real complexity — eventual consistency between write and read models, event schema/versioning, replay cost. Use them where audit/history or read/write asymmetry justifies it, not by default.Mindkettő valódi komplexitást ad — eventual consistency az írás- és olvasásmodell között, event schema/versioning, replay-költség. Ott használd, ahol az audit/history vagy az olvasás/írás aszimmetria indokolja, ne alapból.

Presence, Rooms & Slow ConsumersJelenlét, szobák & lassú fogyasztók

Realtime

Realtime features that look simple and aren't: who is online, which clients receive which messages, and what to do when a client can't keep up with the stream you're pushing at it.Realtime funkciók, amelyek egyszerűnek látszanak, de nem azok: ki van online, mely kliensek mely üzeneteket kapják, és mit tegyél, ha egy kliens nem bírja azt a folyamot, amit rátolsz.

Why is presence harder than it looks?Miért nehezebb a jelenlét, mint amilyennek látszik?

  • Because "online" is a guess about something you can't observe directly. A crashed client sends no goodbye, so presence is only as accurate as your heartbeat timeout — a user can appear online for 30 more seconds after their phone died. Across instances the state must be shared (a Redis key per user with a TTL, refreshed by heartbeat, is the common shape) and it must survive an instance dying without stranding its users as permanently online.Mert az „online" egy tipp valamiről, amit nem tudsz közvetlenül megfigyelni. Az összeomlott kliens nem küld búcsút, így a jelenlét csak annyira pontos, amennyire a heartbeat timeoutod — egy felhasználó 30 másodperccel a telefonja halála után is online-nak látszhat. Instance-ok között az állapotot meg kell osztani (felhasználónkénti Redis kulcs TTL-lel, heartbeattel frissítve a bevett forma), és túl kell élnie egy instance halálát anélkül, hogy örökre online-ként hagyná a felhasználóit.

How do rooms and subscriptions actually work?Hogyan működnek valójában a szobák és a feliratkozások?

  • A room is just a set of socket ids the server keeps, so publishing means iterating that set. The design question is where the set lives: in-process is fast but invisible to other instances, so cross-instance rooms need either the bus to carry every room message everywhere (simple, wasteful at scale) or a directory of which instance holds which member (efficient, more state to keep correct). Authorization belongs at subscribe time, not publish time — checking on every message multiplies the cost by the fan-out.A szoba csak socket-azonosítók halmaza, amit a szerver tart, így a publikálás e halmaz bejárása. A tervezési kérdés az, hol lakik a halmaz: a folyamaton belül gyors, de a többi instance számára láthatatlan, így az instance-ok közti szobákhoz vagy a busznak kell minden szobaüzenetet mindenhová vinnie (egyszerű, méretben pazarló), vagy nyilvántartás kell arról, melyik instance melyik tagot tartja (hatékony, több karbantartandó állapot). A jogosultság-ellenőrzés a feliratkozáshoz tartozik, nem a publikáláshoz — üzenetenként ellenőrizni a fan-outtal szorozza a költséget.

What do you do about a slow consumer?Mit teszel egy lassú fogyasztóval?

  • Decide explicitly, because the default is bad: if you keep writing to a socket the client isn't draining, the kernel and application buffers grow until that one client consumes the memory of the whole process. The three honest options are to drop messages (fine for tick data — the next update supersedes it), coalesce them (keep only the latest state per key), or disconnect the client and let it re-sync on reconnect. Monitor the outbound buffer size per connection and act on a threshold.Dönts kifejezetten, mert az alapértelmezett rossz: ha tovább írsz egy socketre, amit a kliens nem ürít, a kernel- és alkalmazáspufferek addig nőnek, amíg az az egy kliens felemészti az egész folyamat memóriáját. A három őszinte lehetőség: eldobás (rendben van tick-adatnál — a következő frissítés úgyis felülírja), összevonás (kulcsonként csak a legfrissebb állapot megtartása), vagy a kliens bontása, hogy újracsatlakozáskor újraszinkronizáljon. Figyeld a kimenő puffer méretét kapcsolatonként, és lépj küszöbérték felett.

Watch outBuktatóBroadcasting full state on every change is the pattern that scales worst: it's the largest payload multiplied by the largest audience. Send deltas, and give clients a way to request a full snapshot when they detect a gap — that also gives reconnection a defined recovery path.A teljes állapot szórása minden változáskor a legrosszabbul skálázódó minta: a legnagyobb payload szorozva a legnagyobb közönséggel. Küldj deltákat, és adj módot a klienseknek teljes pillanatkép kérésére, ha rést észlelnek — ez az újracsatlakozásnak is definiált helyreállítási utat ad.

Consumer Groups & PartitionsFogyasztói csoportok & partíciók

Realtime

A consumer group is a set of consumers that share a subscription — each message goes to exactly one member, so adding members adds throughput. In log-based systems the unit of assignment is a partition, which is also what bounds your parallelism.A fogyasztói csoport olyan fogyasztók halmaza, amelyek osztoznak egy feliratkozáson — minden üzenet pontosan egy taghoz megy, így a tagok növelése áteresztőképességet ad. Log-alapú rendszerekben a hozzárendelés egysége a partíció, ami egyben a párhuzamosságod korlátja is.

A partition is assigned to at most one consumer per group — so a fourth consumer against three partitions does nothing.Egy partíció csoportonként legfeljebb egy fogyasztóhoz van rendelve — így a negyedik fogyasztó három partícióval szemben nem csinál semmit.

Why can't you just add consumers to go faster?Miért nem lehet egyszerűen fogyasztókat hozzáadni a gyorsításhoz?

  • Because in a partitioned log, parallelism is capped by partition count — extra consumers sit idle. That makes partition count a capacity decision made up front, and an awkward one to change later: increasing it rebalances which key lands where, breaking the ordering guarantee for existing keys. A plain queue (SQS, RabbitMQ) has no such cap, but also no per-key ordering — that's the trade.Mert partícionált logban a párhuzamosságot a partíciószám korlátozza — a további fogyasztók tétlenül ülnek. Ettől a partíciószám előre meghozott kapacitásdöntés, és később kellemetlen megváltoztatni: a növelése újraosztja, melyik kulcs hová kerül, megtörve a meglévő kulcsok sorrendi garanciáját. Egy egyszerű sornak (SQS, RabbitMQ) nincs ilyen plafonja, de kulcsonkénti sorrendje sincs — ez a csere.

What is a rebalance, and why does it hurt?Mi az a rebalance, és miért fáj?

  • When a consumer joins, leaves or is declared dead, partitions are reassigned across the group. Classically this stops all consumption briefly — a stop-the-world pause proportional to group size. Worse, a consumer that is merely slow (blocked on a long poll interval or a long-running handler) gets declared dead and triggers a rebalance, which slows everything further. Cooperative/incremental rebalancing and correctly tuned session and poll timeouts are the mitigations.Amikor egy fogyasztó csatlakozik, kilép vagy halottnak nyilvánítják, a partíciók újraosztódnak a csoportban. Klasszikusan ez rövid időre leállítja az összes fogyasztást — a csoportmérettel arányos stop-the-world szünet. Ami rosszabb: egy pusztán lassú fogyasztót (hosszú poll intervallumon vagy hosszan futó handleren blokkolva) halottnak nyilvánítanak, ami rebalance-t vált ki, ami tovább lassít mindent. A kooperatív/inkrementális rebalance és a helyesen hangolt session- és poll-timeoutok az enyhítés.

What's the difference between a group and multiple subscribers?Mi a különbség egy csoport és több feliratkozó között?

  • Within a group, consumers share the work — one copy of each message total. Different groups each receive their own copy of everything, which is how one topic feeds the order service, the analytics pipeline and the audit log independently, each at its own pace and with its own offset. Getting these confused is why someone occasionally finds their analytics consumer stealing messages from production.Egy csoporton belül a fogyasztók megosztják a munkát — összesen egy példány üzenetenként. A különböző csoportok mindegyike a saját példányát kapja mindenből, így táplál egy topic egymástól függetlenül egy rendelésszolgáltatást, egy analitikai pipeline-t és egy audit logot, mindegyiket saját tempóban és saját offsettel. E kettő összekeverése miatt találja néha valaki azon az analitikai fogyasztóját, hogy üzeneteket lop a produkciótól.

Watch outBuktatóCommitting offsets before processing gives you at-most-once and silent data loss on a crash; committing after gives at-least-once and duplicates. Auto-commit does the former on a timer, which is why it's the wrong default for anything that matters — it can commit messages you haven't finished processing.Az offsetek feldolgozás előtti commitolása at-most-once-t és csendes adatvesztést ad összeomláskor; az utána commitolás at-least-once-t és duplikátumokat. Az auto-commit időzítővel az előbbit teszi, ezért rossz alapértelmezés bármihez, ami számít — olyan üzeneteket is commitolhat, amelyekkel még nem végeztél.

Message OrderingÜzenet-sorrendezés

Realtime

Ordering is guaranteed only within a partition or a single-consumer queue — never globally across a scaled-out system. The practical question is not "how do I get global order" but "which messages actually need to be ordered relative to each other".A sorrend csak egy partíción vagy egyfogyasztós soron belül garantált — sosem globálisan egy kiskálázott rendszerben. A gyakorlati kérdés nem az, „hogyan érek el globális sorrendet", hanem az, „mely üzeneteknek kell valójában egymáshoz képest sorrendben lenniük".

How do you get ordering where you need it?Hogyan érsz el sorrendet ott, ahol kell?

  • Partition by the entity the order matters for — all events for one orderId or userId hash to the same partition, so they're processed in sequence by one consumer, while different entities proceed in parallel. SQS FIFO calls this a message group id; Kafka calls it the partition key. This gives per-key ordering with horizontal scaling, which is almost always what the requirement actually was.Partícionálj az alapján az entitás alapján, amire a sorrend számít — egy orderId vagy userId összes eseménye ugyanabba a partícióba hashelődik, így egy fogyasztó dolgozza fel őket sorban, miközben a különböző entitások párhuzamosan haladnak. Az SQS FIFO ezt message group id-nek hívja; a Kafka partíciókulcsnak. Ez kulcsonkénti sorrendet ad vízszintes skálázással — szinte mindig ez volt a valódi követelmény.

What breaks ordering even when you've partitioned correctly?Mi töri el a sorrendet akkor is, ha helyesen partícionáltál?

  • Retries, mostly. A failed message that goes to a retry queue and comes back later arrives after messages that were behind it — so a partial ordering guarantee quietly becomes none. Concurrency inside the consumer does the same: pulling in order and then handing off to a worker pool destroys the sequence you just preserved. Ordering has to hold end to end or it doesn't hold.Főként az újrapróbálkozások. Egy hibázó üzenet, ami retry sorba kerül és később tér vissza, olyan üzenetek után érkezik, amelyek mögötte voltak — így a részleges sorrendi garancia csendben semmivé válik. A fogyasztón belüli párhuzamosság ugyanezt teszi: sorrendben behúzni, majd worker-poolnak átadni tönkreteszi az épp megőrzött sorozatot. A sorrendnek végponttól végpontig kell tartania, különben nem tart.

What's the alternative to relying on order?Mi az alternatívája annak, hogy a sorrendre támaszkodsz?

  • Make handlers order-insensitive. Carry a version or timestamp in the event and ignore anything older than what you've already applied — a last-write-wins check that makes a late message harmless instead of corrupting. Or model events as facts that commute (balanceDelta: -10 rather than balance: 90), so applying them in any order converges. This is usually cheaper than enforcing strict ordering across a distributed system.Tedd a handlereket sorrend-érzéketlenné. Vigyél verziót vagy időbélyeget az eseményben, és hagyd figyelmen kívül, ami régebbi a már alkalmazottnál — egy last-write-wins ellenőrzés, ami ártalmatlanná teszi a késve érkező üzenetet ahelyett, hogy rombolna. Vagy modellezd az eseményeket felcserélhető tényekként (balanceDelta: -10 a balance: 90 helyett), így bármilyen sorrendben alkalmazva konvergálnak. Ez általában olcsóbb, mint szigorú sorrendet kikényszeríteni egy elosztott rendszerben.

Watch outBuktatóStrict ordering and high throughput are in direct tension: SQS FIFO caps throughput per message group, and a Kafka key that's too coarse (partitioning by tenant instead of by entity) creates a hot partition that one consumer must process alone. Order costs parallelism — buy only as much as you need.A szigorú sorrend és a nagy áteresztőképesség közvetlen feszültségben áll: az SQS FIFO üzenetcsoportonként korlátozza az áteresztőképességet, egy túl durva Kafka-kulcs pedig (tenant szerinti partícionálás entitás helyett) forró partíciót hoz létre, amit egyetlen fogyasztónak kell egyedül feldolgoznia. A sorrend párhuzamosságba kerül — csak annyit vegyél belőle, amennyi kell.

Backpressure & BacklogBackpressure & backlog

Realtime

Backpressure is the signal that a consumer can't keep up, propagated back to the producer. A queue absorbs bursts by design — the danger is treating an unbounded queue as if that made the mismatch disappear.A backpressure az a jelzés, hogy a fogyasztó nem bírja a tempót, visszaterjesztve a termelőhöz. A sor terv szerint nyeli el a csúcsokat — a veszély az, ha egy korlátlan sort úgy kezelsz, mintha ettől eltűnne az eltérés.

How do you tell a temporary spike from a real problem?Hogyan különbözteted meg az átmeneti csúcsot a valódi problémától?

  • By the derivative, not the depth. A backlog that grows and then drains is the queue doing its job. A backlog whose growth rate stays positive means arrival rate exceeds processing rate, and it will never recover on its own — no amount of waiting fixes it. The metric worth alerting on is consumer lag or queue age (how old the oldest unprocessed message is), because that translates directly into user-visible delay; raw depth doesn't.A deriváltból, nem a mélységből. Az a backlog, ami nő, majd leürül, a sor rendeltetésszerű működése. Az a backlog, amelynek növekedési üteme pozitív marad, azt jelenti, hogy az érkezési ütem meghaladja a feldolgozásit, és magától sosem áll helyre — semennyi várakozás nem javítja. A riasztásra érdemes metrika a fogyasztói lag vagy a sor kora (mennyi idős a legrégebbi feldolgozatlan üzenet), mert ez közvetlenül felhasználó által látott késleltetéssé fordul; a nyers mélység nem.

What are the actual levers when a backlog won't drain?Mik a valódi eszközök, ha egy backlog nem ürül le?

  • Increase processing rate (more consumers — up to the partition ceiling; make the handler faster; batch database writes), or reduce arrival rate (rate-limit producers, shed low-priority work, sample). The third option people forget is to drop deliberately: for time-sensitive data, a message that's an hour old may be worthless, and expiring it is better than delaying everything behind it. Scaling consumers without checking whether the bottleneck is downstream just moves the queue into the database.Növeld a feldolgozási ütemet (több fogyasztó — a partícióplafonig; gyorsabb handler; kötegelt adatbázisírás), vagy csökkentsd az érkezési ütemet (termelők rate-limitelése, alacsony prioritású munka elhagyása, mintavételezés). A harmadik, elfelejtett lehetőség a szándékos eldobás: időérzékeny adatnál egy óra régi üzenet értéktelen lehet, és a lejáratása jobb, mint mindent késleltetni mögötte. A fogyasztók skálázása annak ellenőrzése nélkül, hogy a szűk keresztmetszet nem lejjebb van-e, csak az adatbázisba tolja a sort.

Where does backpressure apply outside queues?Hol érvényes a backpressure a sorokon kívül?

  • Anywhere producer and consumer speeds differ. Node streams implement it directly — write() returning false means "stop until drain", and ignoring that return value is how a fast source fills memory with buffered chunks. The same shape appears in WebSocket send buffers and in HTTP clients without connection limits. Recognising it as one pattern rather than three separate problems is the useful part.Bárhol, ahol a termelő és a fogyasztó sebessége eltér. Architecture

Both halves are more dangerous than they look — dual writes can't be made atomic, and backfills run against a system that is still changing.Az adat új tárolóba költöztetése azt jelenti, hogy egy ideig mindkettőbe írsz, majd feltöltöd a történetet. Mindkét fél veszélyesebb, mint amilyennek látszik — a dual write nem tehető atomivá, a backfill pedig egy még változó rendszeren fut.

Why can't dual writes be made safe with retries?Miért nem tehető biztonságossá a dual write újrapróbálkozással?

  • Because the process can die between the two writes, and a retry after restart doesn't know how far it got. You can narrow the window but not close it — there is no distributed transaction across two stores you'd actually want to run. The fix is structural: make one store authoritative, write there transactionally together with an event, and derive the second store from that event stream. That's the outbox pattern applied to migration.Mert a folyamat meghalhat a két írás között, és az újraindítás utáni újrapróbálkozás nem tudja, meddig jutott. Az ablakot szűkítheted, de bezárni nem tudod — nincs olyan elosztott tranzakció két tároló között, amit valóban futtatni akarnál. A javítás szerkezeti: legyen az egyik tároló a mérvadó, oda írj tranzakcionálisan egy eseménnyel együtt, és a második tárolót ebből az eseményfolyamból származtasd. Ez a migrációra alkalmazott outbox minta.

How do you run a backfill safely?Hogyan futtatsz biztonságosan egy backfillt?

  • In batches, idempotently, and resumably — a checkpoint per batch so a failure restarts where it stopped rather than at the beginning. Throttle it: a backfill that saturates the database is an outage you scheduled yourself. And it must interleave correctly with live traffic, which means the write path has to win — a backfill that overwrites a newer live value with older historical data is the classic way to lose data during a migration that "went fine".Kötegekben, idempotensen és folytathatóan — kötegenkénti checkpointtal, hogy egy hiba ott induljon újra, ahol megállt, ne az elejéről. Fojtsd le: az adatbázist telítő backfill egy magadnak ütemezett üzemszünet. És helyesen kell összefésülődnie az élő forgalommal, ami azt jelenti, hogy az írási útnak kell nyernie — az a backfill, ami egy újabb élő értéket ír felül régebbi történeti adattal, a klasszikus módja az adatvesztésnek egy „jól sikerült" migráció közben.

How do you know the two stores actually agree?Honnan tudod, hogy a két tároló tényleg egyezik?

  • You verify continuously rather than assuming. A reconciliation job samples or fully compares both sides and reports drift as a metric — with an alert, because drift that nobody sees is indistinguishable from correctness until the cutover. Verification is what earns the right to switch reads over; without it, the cutover is a guess dressed up as a plan.Folyamatosan ellenőrzöd, nem feltételezed. Egy egyeztető job mintavételez vagy teljesen összehasonlítja a két oldalt, és metrikaként jelenti az eltérést — riasztással, mert a senki által nem látott eltérés megkülönböztethetetlen a helyességtől egészen az átállásig. Watch outBuktatóDeletes are the case backfills forget. Copying rows brings across everything that exists, but a row deleted in the source after the backfill read it stays alive in the target forever unless deletions are replicated too — usually as tombstone events rather than absence.A törlések azok, amiket a backfillek elfelejtenek. A sorok másolása mindent áthoz, ami létezik, de egy a forrásban a backfill olvasása után törölt sor örökre életben marad a célban, hacsak a törléseket is nem replikálod — általában tombstone eseményként, nem hiányként.

What is a tolerant reader, concretely?Mi a tolerant reader konkrétan?

Why is adding an enum value a breaking change?Miért töréssel járó változtatás egy enum-érték hozzáadása?

What do consumer-driven contract tests give you?Mit adnak a fogyasztóvezérelt kontraktus-tesztek?

  • Each consumer publishes the subset of the API it actually relies on; the producer runs all of those in its own pipeline, so a change that would break a real consumer fails the producer's build instead of their production.   out.push(node.val);
  inorder(node.right, out);

function levels(root) {
  if (!root) return [];
  const res = [], q = [root];
  let head = 0;
  while (head < q.length) {
    const size = q.length - head, level = [];
    for (let i = 0; i < size; i++) {
      const n = q[head++];
      level.push(n.val);
      if (n.left) q.push(n.left);
      if (n.right) q.push(n.right);
    }
    res.push(level);

  }
  return -1;
}

Watch outBuktatóMark nodes visited when you enqueue, not when you dequeue — otherwise the same node enters the queue many times and the complexity degrades badly. Forgetting the visited set entirely means infinite loops on any cyclic graph.A node-okat enqueue-kor jelöld látogatottnak, ne dequeue-kor — különben ugyanaz a node sokszor bekerül a sorba, és a komplexitás súlyosan romlik. A visited halmaz teljes elfelejtése végtelen ciklust jelent bármely ciklikus gráfon.

What are the parts of a correct recursion?Mik egy helyes rekurzió részei?

  • A base case that returns without recursing, a recursive case that moves strictly toward the base case, and correct state handling — either pass copies or mutate and undo.Egy báziseset, amely rekurzió nélkül tér vissza, egy rekurzív eset, amely szigorúan a báziseset felé halad, és helyes állapotkezelés — vagy másolatokat adsz át, vagy mutálsz és visszavonsz.

What problems are backtracking?Mely feladatok backtrackingek?

  • Generating permutations, combinations and subsets; N-queens; sudoku; word search on a grid; partitioning. The giveaway is “find all …” or “generate every valid …”.Permutációk, kombinációk és részhalmazok generálása; N-királynő; sudoku; szókeresés rácson; particionálás. Subsets via backtrackingRészhalmazok backtrackinggelO(n·2ⁿ) time / O(n) space (excl. output)
function subsets(nums) {
  const res = [], path = [];
  function backtrack(start) {
    res.push([...path]);              // copy — path keeps mutating
    for (let i = start; i < nums.length; i++) {
      path.push(nums[i]);             // choose
      backtrack(i + 1);               // explore
      path.pop();                     // un-choose
    }
  }
  backtrack(0);
  return res;
}

Watch outBuktatóPushing path instead of [...path] stores a reference to an array that keeps mutating — every entry in the result ends up identical (usually empty). And forgetting the pop() leaks state into sibling branches.Ha a path-t pusholod a [...path] helyett, egy folyamatosan mutálódó tömbre való referenciát tárolsz — az eredmény minden eleme azonos lesz (általában üres). És a pop() elfelejtése átszivárogtatja az állapotot a testvér-ágakba.

Reverse a linked list — iterativeLáncolt lista megfordítása — iteratívO(n) time / O(1) space
function reverse(head) {
  let prev = null, cur = head;
  while (cur) {
    const next = cur.next;   // save before overwriting
    cur.next = prev;         // flip the pointer
    prev = cur;              // advance
    cur = next;
  }
  return prev;               // new head
}

Watch outBuktatóAlways save cur.next before you overwrite it, or you lose the rest of the list.
