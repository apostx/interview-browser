List Virtualization

React

Rendering only the rows currently in the viewport (plus a small buffer) instead of the whole list.Csak a viewportban éppen látható sorok (plusz egy kis buffer) renderelése az egész lista helyett.

The DOM node count stays roughly constant no matter how long the list is.A DOM-node-ok száma nagyjából állandó marad, bármilyen hosszú is a lista.

Why and when?Miért és mikor?

  • Large lists (thousands of rows) create thousands of DOM nodes → slow render, high memory, janky scroll. Virtualization keeps the node count constant. Libraries: react-window, TanStack Virtual.A nagy listák (több ezer sor) több ezer DOM-node-ot hoznak létre → lassú render, magas memóriahasználat, akadozó görgetés. A virtualizáció állandón tartja a node-ok számát. Libraryk: react-window, TanStack Virtual.

The catch?Mi a bökkenő?

  • Off-screen content isn't in the DOM → breaks Ctrl+F and screen readers if unhandled, and needs known or estimated row heights (variable heights are trickier).A képernyőn kívüli tartalom nincs a DOM-ban → kezeletlenül elrontja a Ctrl+F-et és a képernyőolvasókat, továbbá ismert vagy becsült sormagasság kell hozzá (a változó magasság trükkösebb).

Watch outBuktatóDon't virtualize small lists — the machinery costs more than it saves. It's strictly for the long-list case.Kis listát ne virtualizálj — a gépezet többe kerül, mint amennyit megspórol. Kizárólag hosszú listákra való.

gRPC

Node

A high-performance RPC framework over HTTP/2 with Protocol Buffers (binary, schema-defined) for service-to-service communication.Nagy teljesítményű RPC keretrendszer HTTP/2 fölött, Protocol Buffersszel (bináris, sémával definiált) szolgáltatások közötti kommunikációhoz.

Why gRPC over REST for internal services?Miért gRPC REST helyett belső szolgáltatásokhoz?

  • Binary protobuf (smaller/faster than JSON), HTTP/2 multiplexing + streaming, strongly-typed contracts with codegen, and bidirectional streaming.Bináris protobuf (kisebb/gyorsabb a JSON-nál), HTTP/2 multiplexálás + streaming, erősen típusos szerződések codegennel, és kétirányú streaming.

Trade-offs / when not to?Trade-offok / mikor ne?

  • Not natively browser-friendly (needs grpc-web + a proxy), harder to debug (binary), less human-readable. REST/JSON stays better for public/browser-facing APIs.Natívan nem böngészőbarát (grpc-web + proxy kell hozzá), nehezebb debugolni (bináris), kevésbé olvasható emberi szemmel. Publikus/böngésző felé néző API-khoz a REST/JSON marad a jobb.

Watch outBuktatógRPC shines for internal microservice-to-microservice traffic; for public APIs consumed by browsers or third parties, REST or GraphQL is usually the better fit.A gRPC belső, microservice-microservice forgalomnál ragyog; böngészők vagy harmadik felek által használt publikus API-khoz általában a REST vagy a GraphQL illik jobban.

PostgreSQL vs OraclePostgreSQL vs Oracle

Node

Both are mature relational databases with MVCC, but they differ in defaults and dialect in ways that bite during a migration — and those differences, not licensing, are what an interview asks about.Mindkettő érett, MVCC-alapú relációs adatbázis, de az alapértelmezéseik és a dialektusuk úgy térnek el, ami migráció közben visszaüt — és interjún ezek a különbségek jönnek elő, nem a licencelés.

Which difference causes the most damage in practice?Melyik különbség okozza a legtöbb kárt a gyakorlatban?

  • Oracle treating '' as NULL. Logic that distinguishes "not provided" from "provided but blank" silently changes meaning on migration, and WHERE col = '' matches nothing in Oracle. It's a data-semantics difference, not a syntax one, so it survives a mechanical dialect translation and surfaces later as wrong results rather than errors.Az, hogy az Oracle NULL-ként kezeli a ''-t. Az a logika, ami megkülönbözteti a „nincs megadva"-t a „megadva, de üres"-től, csendben jelentést vált migrációkor, és a WHERE col = '' semmire nem illeszkedik Oracle-ben. Ez adatszemantikai különbség, nem szintaktikai, így túléli a gépies dialektus-fordítást, és később rossz eredményként bukkan fel, nem hibaként.

And the transaction defaults?És a tranzakciós alapértelmezések?

  • Oracle opens a transaction implicitly and requires an explicit COMMIT; Postgres autocommits each statement unless you open a block. DDL is the sharper edge: Postgres can roll back a CREATE TABLE inside a transaction, Oracle commits implicitly around DDL — so a migration script that assumes all-or-nothing behaves very differently on each.Az Oracle implicit nyit tranzakciót, és explicit COMMIT-ot vár; a Postgres minden utasítást autocommitál, hacsak nem nyitsz blokkot. A DDL az élesebb különbség: a Postgres vissza tud görgetni egy tranzakción belüli CREATE TABLE-t, az Oracle implicit commitál a DDL körül — így egy mindent-vagy-semmit feltételező migrációs script nagyon eltérően viselkedik a kettőn.

Do you need to rewrite application code, or just SQL?Az alkalmazáskódot is át kell írni, vagy csak az SQL-t?

  • Usually more than SQL. Identifier case folding differs (Postgres folds to lower, Oracle to upper, both fold unless quoted), so a driver returning ID instead of id breaks object mapping across the whole codebase. Date and NUMBER handling differ. And anything in PL/SQL — packages, triggers, stored logic — has no automatic equivalent; that's typically the largest and least visible part of the estimate.Általában többet az SQL-nél. Az azonosítók kis/nagybetű-kezelése eltér (a Postgres kisbetűre, az Oracle nagybetűre hajt, idézőjel nélkül mindkettő hajt), így egy id helyett ID-t visszaadó driver az egész kódbázisban töri az objektum-leképezést. A dátum- és NUMBER-kezelés eltér. És bármi, ami PL/SQL-ben van — csomagok, triggerek, tárolt logika — nem rendelkezik automatikus megfelelővel; jellemzően ez a becslés legnagyobb és legkevésbé látható része.

Watch outBuktatóBoth use MVCC, but Oracle's default isolation is Read Committed with statement-level consistency and no reader-blocks-writer, while Postgres implements Repeatable Read as snapshot isolation — so a workload tuned against one may see different anomalies on the other even at a nominally identical isolation level.Mindkettő MVCC-t használ, de az Oracle alapértelmezett izolációja Read Committed utasítás-szintű konzisztenciával és olvasó-nem-blokkol-írót viselkedéssel, míg a Postgres a Repeatable Read-et snapshot izolációként valósítja meg — így egy az egyikre hangolt terhelés a másikon névlegesen azonos izolációs szinten is más anomáliákat láthat.

Realtime

The choice is mostly about retention and routing: whether messages disappear once consumed, and how much the broker does to decide who gets what.A választás főként a megőrzésről és az útválasztásról szól: eltűnnek-e az üzenetek fogyasztás után, és mennyit tesz a broker azért, hogy eldöntse, ki mit kap.

When does the log model actually matter?Mikor számít valójában a log-modell?

  • When you need replay — adding a new consumer that must process history, rebuilding a projection after a bug, or feeding several independent pipelines from one stream. With a queue, a message consumed by the wrong service is simply gone; with a retained log, each group has its own offset and history is still there. If you never replay and have one consumer per message, the log's operational cost buys you nothing.Amikor visszajátszásra van szükséged — új fogyasztó hozzáadása, aminek a történetet is fel kell dolgoznia, projekció újraépítése egy hiba után, vagy több független pipeline táplálása egy streamből. Sornál egy rossz szolgáltatás által fogyasztott üzenet egyszerűen odavan; megőrzött lognál minden csoportnak saját offsetje van, és a történet még megvan. Ha sosem játszol vissza, és üzenetenként egy fogyasztód van, a log üzemeltetési költsége semmit nem vesz neked.

What does RabbitMQ's routing buy you?Mit ad neked a RabbitMQ útválasztása?

  • The broker decides delivery, not the consumer. Topic exchanges with patterns like order.*.eu, priority queues, per-message TTL and delayed delivery all live in the broker, so consumers stay simple. Kafka pushes that logic to the consumer — it reads a partition and filters — which is more work but scales differently. Broker-side routing is the better fit for complex enterprise integration; consumer-side is better for high-volume streaming.A broker dönt a kézbesítésről, nem a fogyasztó. A topic exchange-ek olyan mintákkal, mint az order.*.eu, a prioritásos sorok, az üzenetenkénti TTL és a késleltetett kézbesítés mind a brokerben laknak, így a fogyasztók egyszerűek maradnak. A Kafka ezt a logikát a fogyasztóra tolja — olvas egy partíciót és szűr —, ami több munka, de másképp skálázódik. A broker-oldali útválasztás komplex vállalati integrációhoz illik jobban; a fogyasztó-oldali nagy volumenű streaminghez.

How would you justify the choice in an interview?Hogyan indokolnád a választást egy interjún?

  • Lead with the requirement, not the product. "We need replay and multiple independent consumers of the same stream" points at a log. "We need complex routing and per-message priority" points at RabbitMQ. "We need a queue and have no platform team" points at SQS. Naming the trade-off you're accepting — Kafka's operational weight, SQS's limited ordering, RabbitMQ's lack of replay — is what makes the answer credible.A követelménnyel kezdj, ne a termékkel. „Visszajátszás kell, és ugyanannak a streamnek több független fogyasztója" — ez logra mutat. „Komplex útválasztás és üzenetenkénti prioritás kell" — ez RabbitMQ-ra. „Sor kell, és nincs platformcsapat" — ez SQS-re. Az teszi hitelessé a választ, ha megnevezed az elfogadott trade-offot is — a Kafka üzemeltetési súlyát, az SQS korlátozott sorrendjét, a RabbitMQ visszajátszás-hiányát.

Watch outBuktatóKafka is frequently chosen for problems a queue would solve, and then run by a team with no capacity to operate it. Managed offerings remove much of that, but partition planning, retention sizing and consumer-lag monitoring remain yours regardless of who runs the brokers.A Kafkát gyakran választják olyan problémákra, amiket egy sor is megoldana, majd olyan csapat üzemelteti, amelynek nincs rá kapacitása. A menedzselt szolgáltatások ebből sokat levesznek, de a partíciótervezés, a megőrzés méretezése és a fogyasztói lag figyelése attól függetlenül a tiéd marad, hogy ki üzemelteti a brokereket.

OpenAPI

API

A standard, language-agnostic spec (YAML/JSON) that describes a REST API — its endpoints, params, schemas and auth. Formerly Swagger.Szabványos, nyelvfüggetlen specifikáció (YAML/JSON), ami leír egy REST API-t — endpointok, paraméterek, sémák és auth. Korábbi neve: Swagger.

What do you get from it?Mit nyersz vele?

Auto-generated docs, client/server SDK codegen, request validation, and contract/mock testing.Automatikusan generált dokumentációt, kliens/szerver SDK codegent, request validációt, valamint contract- és mock-tesztelést.

Contract-first vs code-first?

  • Contract-first: write the spec, generate stubs — teams agree on the interface before coding.Contract-first: megírod a specifikációt, generálsz stubokat — a csapatok kódolás előtt megegyeznek az interfészben.
  • Code-first: generate the spec from annotated code — convenient, but the spec can drift if it isn't enforced.Code-first: a specifikációt annotált kódból generálod — kényelmes, de a spec elcsúszhat, ha nincs kikényszerítve.

Watch outBuktatóAn OpenAPI doc is only useful if it stays in sync — validate the running API against the spec in CI.Egy OpenAPI dokumentáció csak akkor hasznos, ha szinkronban marad — validáld a futó API-t a spec ellen a CI-ban.

PACELC

Distributed

An extension of CAP: if there's a Partition, choose Availability or Consistency; Else (normal operation), choose Latency or Consistency.A CAP kiterjesztése: ha van Partíció, válassz Availability vagy Consistency; Else (normál működés), válassz Latency vagy Consistency.

Why does PACELC matter?Miért számít a PACELC?

  • CAP ignores the normal case. Even without a partition, stronger consistency costs latency (coordination round trips) — PACELC names that everyday trade-off.A CAP a normál esetet figyelmen kívül hagyja. Partíció nélkül is: az erősebb konzisztencia latenciába kerül (koordinációs körök) — a PACELC ezt a hétköznapi trade-offot nevezi meg.
  • Examples: Dynamo/Cassandra = PA/EL (availability + low latency); a strongly-consistent store = PC/EC.Példák: Dynamo/Cassandra = PA/EL (availability + alacsony latencia); egy erősen konzisztens tár = PC/EC.

Watch outBuktatóFor most systems the interesting trade-off is the “else” branch — latency vs consistency in normal operation — not the rare partition case.A legtöbb rendszernél az érdekes trade-off az „else" ág — latencia vs konzisztencia normál működésben —, nem a ritka partíció-eset.

Consistent Hashing

Distributed

Maps both keys and nodes onto a ring; a key belongs to the next node clockwise. Adding/removing a node moves only the keys in its arc — not everything.A kulcsokat és a node-okat is egy gyűrűre képezi; egy kulcs az óramutató szerinti következő node-hoz tartozik. Egy node hozzáadása/eltávolítása csak az ívében lévő kulcsokat mozgatja — nem az összeset.

A node change touches only ~1/N of keys, versus nearly all with key mod N.Egy node-változás csak a kulcsok ~1/N-ét érinti, szemben a key mod N szinte-mindenével.

Why better than key mod N?Miért jobb, mint a key mod N?

  • With mod N, changing N remaps ~all keys (cache/DB churn). Consistent hashing moves only ~1/N on a node change.A mod N-nél az N változtatása ~minden kulcsot áthelyez (cache/DB churn). A consistent hashing egy node-változásnál csak ~1/N-et mozgat.

What are virtual nodes for?Mire valók a virtual node-ok?

  • Without them, load is uneven and removing a node dumps all its keys on one neighbor. Each physical node owns many small ring positions (vnodes) → smoother spread and rebalancing.Nélkülük a terhelés egyenetlen, és egy node eltávolítása az összes kulcsát egy szomszédra zúdítja. Minden fizikai node sok kis gyűrűpozíciót birtokol (vnode) → simább eloszlás és rebalancing.

Watch outBuktatóPlain consistent hashing still has variance — virtual nodes (or rendezvous hashing) are what make it practical.A sima consistent hashingnek is van szórása — a virtual node-ok (vagy a rendezvous hashing) teszik praktikussá.

08

Quorums (R + W > N)

Distributed

With N replicas, require W nodes to ack a write and R for a read. If R + W > N, the read and write sets overlap on ≥1 node → the read sees the latest write.N replikánál egy íráshoz W node visszaigazolása kell, egy olvasáshoz R. Ha R + W > N, az olvasási és írási halmaz ≥1 node-on átfed → az olvasás a legfrissebb írást látja.

N=3, W=2, R=2 → any read set and any write set share at least one node.N=3, W=2, R=2 → bármely olvasási és bármely írási halmaz legalább egy node-on osztozik.

How do you tune R and W?Hogyan hangolod az R-t és W-t?

  • W=N, R=1 → fast reads, fragile writes. W=1, R=N → fast writes, slow reads. W=R=⌈(N+1)/2⌉ (N=3 → 2/2) → balanced, tolerates one node down.W=N, R=1 → gyors olvasás, törékeny írás. W=1, R=N → gyors írás, lassú olvasás. W=R=⌈(N+1)/2⌉ (N=3 → 2/2) → kiegyensúlyozott, elvisel egy kieső node-ot.

Does a quorum give strong consistency?Ad a quorum strong consistency-t?

  • It gives read-after-write on overlap, not full linearizability — concurrent writes, sloppy quorums and read-repair timing mean Dynamo-style stores are eventually consistent by default.Read-after-write-ot ad az átfedésen, nem teljes linearizability-t — a párhuzamos írások, a sloppy quorum és a read-repair időzítése miatt a Dynamo-stílusú tárak alapból eventually consistent-ek.

Watch outBuktatóR + W > N guarantees overlap, not conflict resolution — concurrent writes still need last-write-wins or version vectors to reconcile.Az R + W > N az átfedést garantálja, nem a konfliktusfeloldást — a párhuzamos írásokhoz továbbra is last-write-wins vagy version vector kell az összefésüléshez.

Consensus & Raft

Distributed

Getting a group of nodes to agree on a value or order despite failures. Raft (and Paxos) provide this — for leader election, replicated logs and config.Egy node-csoport rábírása, hogy hibák ellenére megegyezzen egy értékben vagy sorrendben. A Raft (és a Paxos) ezt biztosítja — leader electionhöz, replikált loghoz és konfighoz.

How does Raft work at a high level?Hogyan működik a Raft magas szinten?

  • Nodes elect a leader for a term. The leader appends entries to its log and replicates them; an entry commits once a majority acks. If the leader fails, a new election begins.A node-ok egy termre leadert választanak. A leader bejegyzéseket fűz a logjához és replikálja őket; egy bejegyzés akkor commitál, ha a többség visszaigazolja. Ha a leader kiesik, új választás indul.

Why does it need a majority?Miért kell hozzá többség?

  • Any two majorities overlap, so decisions can't conflict and a committed entry survives failures. Hence odd cluster sizes (3, 5) tolerating ⌊N/2⌋ failures.Bármely két többség átfed, így a döntések nem ütközhetnek, és egy committált bejegyzés túléli a hibákat. Innen a páratlan klaszterméret (3, 5), amely ⌊N/2⌋ hibát tolerál.

Watch outBuktatóConsensus is expensive (round trips + majority). Use it for metadata/coordination (etcd, ZooKeeper), not the hot data path.A consensus drága (körök + többség). Metaadathoz/koordinációhoz használd (etcd, ZooKeeper), ne a forró adatútra.

Leader Election

Distributed

Choosing a single coordinator among nodes, and re-choosing when it fails.Egyetlen koordinátor kiválasztása a node-ok közül, és újraválasztása, ha kiesik.

How is it done safely?Hogyan csinálják biztonságosan?

  • Via a consensus system (Raft election) or a lease in ZooKeeper/etcd. The leader holds a time-bounded lease it must renew; if it can't, another node takes over.Consensus-rendszerrel (Raft election) vagy lease-szel ZooKeeperben/etcd-ben. A leader időkorlátos lease-t tart, amit meg kell újítania; ha nem tudja, egy másik node veszi át.

What's split-brain, and how do you prevent it?Mi a split-brain, és hogyan előzöd meg?

  • Two nodes both believe they're leader (e.g. after a partition) → conflicting writes. Prevent with quorum-based election + fencing tokens (monotonic IDs that let resources reject a stale leader).Két node is leadernek hiszi magát (pl. partíció után) → ütköző írások. Megelőzés quorum-alapú választással + fencing tokenekkel (monoton ID-k, amelyekkel a resource-ok elutasíthatják az elavult leadert).

Watch outBuktatóA paused leader (GC/partition) may still think it's leader — always use fencing tokens so downstream systems reject its late writes.Egy megállt leader (GC/partíció) még mindig leadernek hiheti magát — mindig használj fencing tokent, hogy a downstream rendszerek elutasítsák a késői írásait.

11

Clocks & Ordering

Distributed

There's no single global clock, so ordering events across nodes needs logical clocks.Nincs egyetlen globális óra, így a node-ok közti események rendezéséhez logikai órák kellenek.

Why not just use wall-clock timestamps?Miért ne csak wall-clock időbélyeget használjunk?

  • Physical clocks drift and skew; “last write wins” by wall clock can silently drop the right update. NTP helps but isn't precise enough for ordering.A fizikai órák driftelnek és eltérnek; a wall-clock szerinti „last write wins" csendben eldobhatja a helyes frissítést. Az NTP segít, de rendezéshez nem elég pontos.

Lamport vs vector clocks?Lamport vs vector clock?

  • Lamport: a counter giving a total order consistent with causality, but can't tell if two events are concurrent.Lamport: egy számláló, amely az okságival konzisztens teljes rendezést ad, de nem mondja meg, ha két esemény párhuzamos.
  • Vector clocks: per-node counters that can detect concurrency (and thus conflicts).Vector clock: node-onkénti számlálók, amelyek képesek párhuzamosságot (és így konfliktust) észlelni.

Watch outBuktatóUsing server wall-clock time for ordering or conflict resolution is a classic bug — clock skew makes it non-deterministic. Use logical clocks or a sequencer.A szerver wall-clock idejének használata rendezésre vagy konfliktusfeloldásra klasszikus bug — az óraeltérés nemdeterminisztikussá teszi. Használj logikai órát vagy sequencert.

Two-Phase Commit

Distributed

An atomic-commit protocol across nodes: a coordinator asks all participants to prepare (phase 1); if all vote yes, it tells them to commit (phase 2), else abort.Egy atomi-commit protokoll node-ok között: egy koordinátor megkéri az összes résztvevőt, hogy prepare (1. fázis); ha mind igennel szavaz, szól, hogy commit (2. fázis), különben abort.

All-or-nothing across participants — but blocking if the coordinator dies mid-protocol.Mindent-vagy-semmit a résztvevők között — de blokkol, ha a koordinátor a protokoll közben kiesik.

Why is it problematic at scale?Miért problémás nagyban?

  • It's blocking: if the coordinator crashes after “prepare,” participants hold locks indefinitely, unsure whether to commit. It also lowers availability — any participant down blocks the commit.Blokkoló: ha a koordinátor a „prepare" után összeomlik, a résztvevők határozatlan ideig tartják a lockokat, nem tudva, commitáljanak-e. Csökkenti az elérhetőséget is — bármelyik résztvevő kiesése blokkolja a commitot.

2PC vs Saga?2PC vs Saga?

  • 2PC: strong consistency, synchronous, locks, blocking — okay within one trust boundary (XA over a few DBs).2PC: strong consistency, szinkron, lockok, blokkoló — rendben egy trust boundary-n belül (XA néhány DB felett).
  • Saga: eventual consistency, non-blocking, scales across services — preferred in microservices.Saga: eventual consistency, nem blokkoló, szolgáltatásokon át skálázódik — microservicesben ez az ajánlott.

Watch outBuktató2PC is largely avoided in microservices because of its blocking failure mode and lock-holding — Sagas + idempotency are the usual answer.A 2PC-t microservicesben többnyire kerülik a blokkoló hibamódja és a lock-tartás miatt — a Saga + idempotency a szokásos válasz.

Failure Detection & Heartbeats

Distributed

Detecting that a node has failed so the system can react (failover, rebalance). Nodes send periodic heartbeats; a missed one suggests failure.Egy node kiesésének észlelése, hogy a rendszer reagálhasson (failover, rebalance). A node-ok periodikus heartbeatet küldenek; egy kimaradó hibát jelez.

Why is failure detection fundamentally hard?Miért alapvetően nehéz a failure detection?

  • You can't distinguish a crashed node from a slow one or a network partition — a missed heartbeat could be any of them. Detectors trade speed vs false positives.Nem tudod megkülönböztetni az összeomlott node-ot a lassútól vagy a hálózati partíciótól — egy kimaradó heartbeat bármelyik lehet. A detektorok a sebesség és a téves riasztás közt egyensúlyoznak.

How is it done in practice?Hogyan csinálják a gyakorlatban?

  • Heartbeats/health checks with timeouts, often gossiped for scale. Phi-accrual detectors give a suspicion level rather than a binary up/down.Heartbeat/health check timeouttal, skálázáshoz gyakran gossip-elve. A phi-accrual detektorok gyanú-szintet adnak bináris up/down helyett.

Watch outBuktatóAggressive timeouts evict healthy nodes (needless failover, thrash); lax ones detect failures slowly. There's no perfect setting — tune for your latency profile.A túl agresszív timeout egészséges node-okat zár ki (felesleges failover, kapkodás); a túl engedékeny lassan észleli a hibát. Nincs tökéletes beállítás — a saját latencia-profilodhoz hangold.

Infrastructure as CodeInfrastructure as Code

DevOps

Defining infrastructure in version-controlled code rather than clicking through a console — making environments reproducible, reviewable and diffable.Az infrastruktúra definiálása verziókövetett kódban ahelyett, hogy egy konzolon kattintgatnál — reprodukálhatóvá, átnézhetővé és összehasonlíthatóvá téve a környezeteket.

Declarative vs imperative IaC?Deklaratív vs imperatív IaC?

  • Declarative (Terraform, CloudFormation): you describe the desired end state and the tool computes the diff. Imperative (scripts): you specify the steps. Declarative wins because it's idempotent — applying it twice changes nothing the second time.Deklaratív (Terraform, CloudFormation): leírod a kívánt végállapotot és az eszköz kiszámolja a különbséget. Imperatív (szkriptek): megadod a lépéseket. A deklaratív nyer, mert idempotens — kétszer alkalmazva másodszor semmit nem változtat.

What is state, and why is it dangerous?Mi a state, és miért veszélyes?

  • Terraform records what it created in a state file, and diffs against it. If that file is lost, corrupted or edited by two people at once, the tool loses track of reality — so keep state in a remote backend with locking, never in git.A Terraform egy state fájlban rögzíti, mit hozott létre, és ahhoz képest számol különbséget. Ha ez a fájl elvész, sérül, vagy ketten egyszerre szerkesztik, az eszköz elveszti a kapcsolatot a valósággal — ezért tartsd a state-et távoli backendben lockolással, sosem gitben.

What is configuration drift?Mi a configuration drift?

  • When someone changes infrastructure manually, reality diverges from the code. The next apply may revert their fix or fail unexpectedly. The discipline is: no manual changes in production — the console is read-only.Amikor valaki kézzel változtat az infrastruktúrán, a valóság eltér a kódtól. A következő apply visszavonhatja a javítását vagy váratlanul elbukhat. A fegyelem: semmi kézi változtatás a productionben — a konzol csak olvasható.

Watch outBuktatóNever run a destructive apply without reading the plan — Terraform will happily replace a database if a change forces recreation. Review the plan output, and protect stateful resources with lifecycle rules that prevent destruction.Sose futtass destruktív apply-t a plan elolvasása nélkül — a Terraform készséggel lecserél egy adatbázist, ha egy változtatás újralétrehozást kényszerít. Nézd át a plan kimenetét, és védd a stateful erőforrásokat lifecycle szabályokkal, amelyek megakadályozzák a megsemmisítést.

Pipeline & Runtime SecurityPipeline- & futásidő-biztonság

DevOps

Securing the pipeline and the runtime — secrets, image provenance, least privilege, and keeping credentials out of the places they habitually leak.A pipeline és a futásidő biztonsága — titkok, image-eredet, legkisebb jogosultság, és a hitelesítő adatok távoltartása azoktól a helyektől, ahonnan rendszeresen kiszivárognak.

Where do secrets belong in a deployment?Hol a helyük a titkoknak egy deployban?

  • In a secrets manager or the platform's secret store, injected at runtime as environment variables or mounted files — never in the image, the repo, or CI logs. Prefer short-lived credentials issued per workload over long-lived static keys.Egy secrets managerben vagy a platform titok-tárában, futásidőben injektálva környezeti változóként vagy csatolt fájlként — sosem az image-ben, a repóban vagy a CI-logokban. Inkább rövid életű, terhelésenként kiadott hitelesítő adatok, mint hosszú életű statikus kulcsok.

How do you secure the CI pipeline itself?Hogyan véded magát a CI pipeline-t?

  • It holds production credentials, so treat it as production. Scope tokens to the minimum, don't expose secrets to pull-request builds from forks, pin third-party actions to a commit SHA rather than a mutable tag, and require review for changes to the pipeline definition.Production-hitelesítő adatokat tart, szóval kezeld productionként. Szűkítsd a tokeneket a minimumra, ne tedd elérhetővé a titkokat fork-okból érkező pull-request buildeknek, rögzítsd a harmadik féltől való action-öket commit SHA-hoz mutálható tag helyett, és követelj review-t a pipeline-definíció változtatásaihoz.

What does least privilege mean for workloads?Mit jelent a legkisebb jogosultság a terhelésekre?

  • Each service gets its own identity with only the permissions it needs — not a shared admin role. Combined with network policies restricting which services may talk to which, this limits what a single compromised container can reach.Minden szolgáltatás saját identitást kap, csak a szükséges jogosultságokkal — nem közös admin szerepet. Hálózati szabályzatokkal kombinálva, amelyek korlátozzák, mely szolgáltatások beszélhetnek melyikkel, ez korlátozza, mit érhet el egy kompromittált konténer.

NoteMegjegyzésThis extends the Web Security deck's secrets and supply-chain sections into the deployment layer.Ez kiterjeszti a Web Security deck titok- és supply-chain szakaszait a deploy-rétegre.

Watch outBuktatóEchoing an environment variable for debugging, or a tool printing its config on error, puts secrets into CI logs that are often world-readable and retained for months. Mask secrets in the CI configuration and rotate anything that ever appeared in a log.Egy környezeti változó kiírása debughoz, vagy egy eszköz, amely hibánál kinyomtatja a konfigját, titkokat tesz a CI-logokba, amelyek gyakran bárki által olvashatók és hónapokig megmaradnak. Maszkold a titkokat a CI konfigurációjában, és forgass mindent, ami valaha megjelent egy logban.

Heaps & Priority QueuesHeapek & prioritási sorok

DS&A

A complete binary tree keeping the min (or max) at the root — O(log n) insert and extract, O(1) peek. The structure behind “top k” and “next smallest” problems.Egy teljes bináris fa, amely a minimumot (vagy maximumot) tartja a gyökérben — O(log n) beszúrás és kivétel, O(1) peek. A „top k” és „következő legkisebb” feladatok mögötti szerkezet.

peek O(1)peek O(1)insert/extract O(log n)beszúrás/kivétel O(log n)heapify O(n)heapify O(n)space O(n)tár O(n)

When is a heap the right answer?Mikor a heap a helyes válasz?

  • “Top k” / “k-th largest” (keep a size-k heap → O(n log k), better than sorting's O(n log n)), merging k sorted lists, running median (two heaps), and any scheduler picking the next-smallest item.„Top k” / „k-adik legnagyobb” (tarts k méretű heapet → O(n log k), jobb, mint a rendezés O(n log n)-je), k rendezett lista összefésülése, futó medián (két heap), és bármely ütemező, amely a következő legkisebb elemet választja.

Why does a size-k heap beat sorting?Miért veri a k méretű heap a rendezést?

  • Sorting does full work on all n elements: O(n log n). A heap capped at k only ever holds k items, so each of the n insertions costs O(log k): total O(n log k). When k ≪ n this is a large win, and space drops from O(n) to O(k).A rendezés teljes munkát végez mind az n elemen: O(n log n). Egy k-ra korlátozott heap sosem tart k-nál többet, így az n beszúrás mindegyike O(log k): összesen O(n log k). Ha k ≪ n, ez nagy nyereség, és a tár O(n)-ről O(k)-ra csökken.

Watch outBuktatóJavaScript has no built-in heap — say this out loud and either write a compact one or state that you'd use a library. Reaching for sort() on every insertion instead is O(n² log n) and a red flag.A JavaScriptben nincs beépített heap — mondd ki ezt hangosan, és vagy írj egy tömöret, vagy jelezd, hogy library-t használnál. Ha helyette minden beszúrásnál sort()-ot hívsz, az O(n² log n) és vörös zászló.

Dynamic ProgrammingDinamikus programozás

DS&A

Solving overlapping subproblems once and reusing the results — either top-down with memoization or bottom-up with a table. It turns exponential recursion into polynomial time.Átfedő részfeladatok egyszeri megoldása és az eredmények újrafelhasználása — vagy top-down memoizációval, vagy bottom-up táblával. Exponenciális rekurziót alakít polinomiális idejűvé.

typically O(states × transition)tipikusan O(állapotok × átmenet)O(states), often reducibleO(állapotok), gyakran csökkenthető

When is DP applicable?Mikor alkalmazható a DP?

  • Two conditions: optimal substructure (the optimal answer is built from optimal answers to subproblems) and overlapping subproblems (the same subproblem recurs). Without overlap, plain recursion or greedy is enough.Két feltétel: optimális részszerkezet (az optimális válasz részfeladatok optimális válaszaiból épül) és átfedő részfeladatok (ugyanaz a részfeladat ismétlődik). Átfedés nélkül elég a sima rekurzió vagy a mohó megoldás.

Memoization vs tabulation?Memoizáció vs tabuláció?

  • Memoization (top-down): write the natural recursion, cache results — easier to derive, but uses the call stack. Tabulation (bottom-up): fill a table iteratively — no stack risk and often allows space optimization.Memoizáció (top-down): írd meg a természetes rekurziót, cache-eld az eredményeket — könnyebb levezetni, de a hívási vermet használja. Tabuláció (bottom-up): töltsd a táblát iteratívan — nincs verem-kockázat és gyakran engedi a tár-optimalizálást.

How do you approach a DP problem in an interview?Hogyan állsz neki egy DP feladatnak interjún?

  • Define the state in words first (“dp[i] = the best answer using the first i items”), write the recurrence, identify the base cases, then decide the iteration order. Start with brute-force recursion and add memoization — deriving the table directly is harder and error-prone.Előbb definiáld az állapotot szavakban („dp[i] = a legjobb válasz az első i elemmel”), írd fel a rekurziót, azonosítsd a báziseseteket, majd döntsd el az iterációs sorrendet. Indulj nyers erejű rekurzióval és add hozzá a memoizációt — a tábla közvetlen levezetése nehezebb és hibalehetőség.
Climbing stairs — three stages of the same ideaLépcsőzés — ugyanaz az ötlet három szakaszbanO(2ⁿ) → O(n)/O(n) → O(n)/O(1)
// 1. naive recursion — O(2ⁿ)
const naive = n => n <= 2 ? n : naive(n-1) + naive(n-2);

// 2. memoized — O(n) time, O(n) space
function memo(n, cache = new Map()) {
  if (n <= 2) return n;
  if (cache.has(n)) return cache.get(n);
  const r = memo(n-1, cache) + memo(n-2, cache);
  cache.set(n, r);
  return r;
}

// 3. tabulated + space-optimized — O(n) time, O(1) space
function best(n) {
  if (n <= 2) return n;
  let a = 1, b = 2;
  for (let i = 3; i <= n; i++) [a, b] = [b, a + b];
  return b;
}

Watch outBuktatóOnly the last row (or two) of a DP table is usually needed — say “I can reduce space from O(n·m) to O(m)” even if you don't implement it; interviewers listen for that. And a memo key must capture every varying parameter, or you'll return cached answers for different states.Egy DP-táblából általában csak az utolsó sor (vagy kettő) kell — mondd ki, hogy „a tárat O(n·m)-ről O(m)-re tudom csökkenteni”, akkor is, ha nem implementálod; az interjúztatók erre figyelnek. És a memo-kulcsnak minden változó paramétert meg kell fognia, különben eltérő állapotokra adsz vissza cache-elt válaszokat.

Greedy AlgorithmsMohó algoritmusok

DS&A

Making the locally optimal choice at each step and never reconsidering — fast and simple when it works, but it needs proof that local optimality yields a global optimum.A lokálisan optimális választás minden lépésben, újragondolás nélkül — gyors és egyszerű, amikor működik, de bizonyítani kell, hogy a lokális optimalitás globális optimumot ad.

often O(n log n) (sort + pass)gyakran O(n log n) (rendezés + végigmenés)wrong without the exchange argumenthibás az exchange argument nélkül

Greedy vs DP — how do you tell?Greedy vs DP — hogyan döntöd el?

  • If a locally best choice can be proven never to hurt the final answer (an exchange argument), greedy works and is cheaper. If a choice's value depends on future choices, you need DP. Coin change with arbitrary denominations is the classic case where greedy fails and DP is required.Ha egy lokálisan legjobb választásról bizonyítható, hogy sosem ront a végső válaszon (exchange argument), a greedy működik és olcsóbb. Ha egy választás értéke a jövőbeli választásoktól függ, DP kell. A tetszőleges címletű pénzváltás a klasszikus eset, ahol a greedy elbukik és DP kell.

Typical greedy problems?Tipikus greedy feladatok?

  • Interval scheduling (sort by end time), merging intervals (sort by start), jump game, assigning tasks by deadline, and Huffman coding. Nearly all begin with a sort that establishes the greedy order.Intervallum-ütemezés (végidő szerint rendezve), intervallumok összevonása (kezdet szerint rendezve), jump game, taszkok kiosztása határidő szerint, és Huffman-kódolás. Szinte mind egy rendezéssel kezdődik, amely megállapítja a mohó sorrendet.
Merge overlapping intervalsÁtfedő intervallumok összevonásaO(n log n) time / O(n) space
function merge(intervals) {
  if (!intervals.length) return [];
  intervals.sort((a, b) => a[0] - b[0]);        // by start
  const out = [intervals[0].slice()];
  for (const [s, e] of intervals.slice(1)) {
    const last = out[out.length - 1];
    if (s <= last[1]) last[1] = Math.max(last[1], e);  // overlap
    else out.push([s, e]);
  }
  return out;
}

Watch outBuktatóNever assert a greedy solution without justifying why the local choice is safe — “it seems to work on the examples” is exactly how greedy fails. If you can't argue it, reach for DP.Sose állíts mohó megoldást annak indoklása nélkül, hogy miért biztonságos a lokális választás — a „a példákon működni látszik” pontosan az, ahogy a greedy elbukik. Ha nem tudod megindokolni, nyúlj a DP-hez.
