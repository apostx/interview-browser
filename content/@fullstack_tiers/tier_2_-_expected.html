Stale Closures

React

A closure that captured a value from an earlier render and keeps using that old snapshot.Olyan closure, ami egy korábbi renderből fogott be egy értéket, és továbbra is azt a régi pillanatképet használja.

Where does it bite?Hol üt vissza?

  • Event handlers, setTimeout/setInterval, and effects with missing deps — they all close over the render's snapshot of state and props.Event handlerekben, setTimeout/setInterval-ban, és hiányzó depekkel rendelkező effektekben — mindegyik a render state- és prop-pillanatképét zárja be.

How do you fix it?Hogyan javítod?

  • Add the value to the deps (re-create the closure), use the functional updater, or stash the latest value in a ref and read ref.current.Vedd fel az értéket a depek közé (újra létrejön a closure), használd a funkcionális updatert, vagy tedd a legfrissebb értéket egy refbe és olvasd a ref.current-et.

Watch outBuktatóA setInterval created in useEffect([]) that reads state will forever see the initial value — the textbook stale closure.Egy useEffect([])-ben létrehozott setInterval, ami state-et olvas, örökre a kezdeti értéket látja — ez a tankönyvi stale closure.

Context API

React

Passes data through the tree without prop drilling; consumers re-render when the provider's value changes.Adatot ad át a fán keresztül prop drilling nélkül; a fogyasztók újrarenderelnek, amikor a provider értéke változik.

The performance gotcha?Mi a performance-buktató?

  • Every consumer re-renders on any value change, even if it uses only part of it. A fresh object literal as value re-renders all consumers on every parent render.Minden fogyasztó újrarenderel bármilyen értékváltozásra, még ha csak egy részét használja is. Egy friss objektum-literál value-ként minden szülő-renderkor újrarendereli az összes fogyasztót.

How do you mitigate it?Hogyan enyhíted?

  • Memoize the value, split contexts by update frequency, or move to a store (Zustand/Redux) with selector-based subscriptions for high-frequency state.Memoizáld az értéket, bontsd külön a contextet frissítési gyakoriság szerint, vagy válts store-ra (Zustand/Redux) selector-alapú feliratkozással a gyakran változó state-hez.

Watch outBuktatóContext is a dependency-injection / low-frequency-state tool (theme, auth, locale) — not a general state manager for rapidly-changing data.A Context dependency-injection / ritkán változó state eszköz (theme, auth, locale) — nem általános state manager gyorsan változó adathoz.

Custom Hooks

React

A function starting with use that composes built-in hooks to extract and reuse stateful logic.Egy use-szal kezdődő függvény, ami beépített hookokat komponál, hogy stateful logikát emeljen ki és tegyen újrahasznosíthatóvá.

What do they share — and not share?Mit osztanak meg — és mit nem?

  • They share logic, not state. Each call gets its own independent state — two components using useCounter() don't share a counter.Logikát osztanak meg, nem state-et. Minden hívás saját, független state-et kap — két komponens, ami useCounter()-t használ, nem osztozik egy számlálón.

When should you extract one?Mikor érdemes kiemelni egyet?

  • Repeated stateful logic — fetching, form state, subscriptions, debouncing — to keep components declarative and testable.Ismétlődő stateful logikánál — fetch, form-state, subscription-ök, debounce —, hogy a komponensek deklaratívak és tesztelhetők maradjanak.

Watch outBuktatóA custom hook is just a function — the Rules of Hooks still apply (top-level calls, React-functions only).A custom hook is csak egy függvény — a Rules of Hooks továbbra is érvényes (top-level hívások, csak React függvényekből).

Client vs Server State

React

Client state is UI state you own (modals, inputs, toggles). Server state is remote data — fetched async, shared, and able to go stale.A client state a saját UI-állapotod (modalok, inputok, kapcsolók). A server state távoli adat — aszinkron lekérve, megosztott, és el tud avulni.

Why treat them differently?Miért kezeljük őket másképp?

  • Server state needs caching, background refetch, dedup, retry, invalidation and staleness handling — none of which a plain useState or Redux slice gives you. That's what React Query / SWR / RTK Query solve.A server state-hez caching, háttérbeli refetch, dedup, retry, invalidáció és elavultság-kezelés kell — ezt egy sima useState vagy Redux slice nem adja meg. Erre valók a React Query / SWR / RTK Query.

What does a server-state library give you?Mit ad egy server-state library?

  • Request dedup, cache + stale-while-revalidate, background updates, pagination/infinite scroll, optimistic updates, and loading/error state out of the box.Request-dedupot, cache-t + stale-while-revalidate-et, háttérfrissítést, pagination/infinite scrollt, optimistic update-eket, valamint loading/error state-et out of the box.

Watch outBuktatóDumping fetched data into Redux/Context and hand-rolling caching is the source of most “why is my data stale” bugs. Keep client and server state separate.Ha a lekért adatot Reduxba/Contextbe öntöd és kézzel írod a cachinget, abból jön a legtöbb „miért elavult az adatom” bug. Tartsd külön a client és a server state-et.

Data Fetching Patterns

React

How and when you load remote data — on render, on route load, or on the server.Hogyan és mikor töltesz be távoli adatot — renderkor, route betöltésekor, vagy a szerveren.

Parallelize independent requests; only chain when B truly depends on A.A független kéréseket futtasd párhuzamosan; csak akkor láncold őket, ha B tényleg A-tól függ.

Common problems and fixes?Gyakori problémák és javításuk?

  • Race conditions: a slow earlier request resolves after a newer one — cancel with AbortController or ignore stale results in cleanup.Race condition: egy lassú, korábbi kérés az újabb után fut be — szakítsd meg AbortController-rel, vagy hagyd figyelmen kívül az elavult eredményt a cleanupban.
  • Waterfalls: sequential dependent fetches — parallelize independent ones or fetch higher up.Waterfall: egymásra épülő, soros fetchek — futtasd párhuzamosan a függetleneket, vagy kérd le feljebb.

Where should data be fetched?Hol érdemes lekérni az adatot?

  • Close to where it's used, but avoid per-component waterfalls — route-level or server-side fetching (RSC/loaders) cuts round trips.Közel oda, ahol használod, de kerüld a komponensenkénti waterfallt — a route-szintű vagy szerver-oldali lekérés (RSC/loaderek) csökkenti a körutakat.

Watch outBuktatóuseEffect fetching has no built-in dedup or caching — two components fetching the same thing means two requests. Use a server-state library or hoist the fetch.A useEffect-es fetchnek nincs beépített dedupja vagy cache-e — ha két komponens ugyanazt kéri le, az két kérés. Használj server-state libraryt, vagy emeld feljebb a fetchet.

Re-render Optimization

React

Reducing unnecessary re-renders and the cost of each one.A felesleges re-renderek és az egyes renderek költségének csökkentése.

The main strategies, in order?A fő stratégiák, sorrendben?

  • Don't re-render: push state down, split components, pass children as props so a re-rendering parent doesn't re-render static subtrees.Ne rendereljen újra: told le a state-et, bontsd szét a komponenseket, add át a children-t propként, hogy az újrarenderelő szülő ne rendereljen újra statikus részfákat.
  • Then memoize: React.memo + stable props via useMemo/useCallback.Utána memoizálj: React.memo + stabil propok useMemo/useCallback segítségével.
  • For global state: selector-based subscriptions so only components using a changed slice update.Globális state-hez: selector-alapú feliratkozás, hogy csak a megváltozott szeletet használó komponensek frissüljenek.

How do you find the problem?Hogyan találod meg a problémát?

  • The React DevTools Profiler (highlight updates / flamegraph). Measure before optimizing.A React DevTools Profilerrel (highlight updates / flamegraph). Előbb mérj, aztán optimalizálj.

Watch outBuktatóWrapping everything in memo is an anti-pattern — it adds comparison cost and hides the real fix, which is usually state placement or component structure.Mindent memo-ba csomagolni antipattern — összehasonlítási költséget ad, és elfedi a valódi megoldást, ami általában a state elhelyezése vagy a komponens-struktúra.

Browser Event Loop

React

The single-threaded loop that runs JS: empty the call stack, drain all microtasks, render, then take one macrotask — and repeat.Az egyszálú ciklus, ami a JS-t futtatja: kiüríti a call stacket, lefuttatja az összes microtaskot, renderel, majd vesz egy macrotaskot — és ismétli.

Microtasks drain fully before the next macrotask — so a resolved Promise runs before setTimeout(0).A microtaskok teljesen lefutnak a következő macrotask előtt — ezért egy feloldott Promise előbb fut, mint a setTimeout(0).

Microtasks vs macrotasks?Microtask vs macrotask?

  • Microtasks (Promise callbacks, queueMicrotask, MutationObserver) drain fully after each task, before rendering.A microtaskok (Promise callbackek, queueMicrotask, MutationObserver) minden task után teljesen lefutnak, a renderelés előtt.
  • Macrotasks (setTimeout, events, I/O) run one per loop iteration — so a Promise resolves before setTimeout(0).A macrotaskokból (setTimeout, események, I/O) ciklusonként egy fut le — ezért a Promise előbb oldódik fel, mint a setTimeout(0).

Why does this matter for UI?Miért számít ez a UI-nak?

  • Long synchronous tasks or microtask floods block rendering and input → jank and poor INP. Break work up with chunking, setTimeout, or requestIdleCallback.A hosszú szinkron taskok vagy a microtask-áradat blokkolja a renderelést és az inputot → akadozás és rossz INP. Darabold fel a munkát chunkinggal, setTimeout-tal vagy requestIdleCallback-kel.

Watch outBuktatóThis is the browser loop (task → microtasks → render). Node's event loop has different phases (timers, poll, check…) — don't conflate them in a full-stack interview.Ez a böngésző ciklusa (task → microtaskok → render). A Node event loopjának más fázisai vannak (timers, poll, check…) — full-stack interjún ne keverd össze a kettőt.

libuv & Thread Pool

Node

libuv is the C library behind Node's event loop and async I/O. Most I/O is OS-level async; a few operations use a small thread pool.A libuv az a C könyvtár, ami a Node event loopja és aszinkron I/O-ja mögött áll. Az I/O nagy része OS-szintű aszinkron; néhány művelet egy kis thread poolt használ.

Only pool-backed ops (fs, DNS, some crypto/zlib) are limited by the thread count.Csak a pool-alapú műveleteket (fs, DNS, egyes crypto/zlib) korlátozza a szálak száma.

What actually uses the thread pool?Mi használja ténylegesen a thread poolt?

  • File system, DNS lookups (dns.lookup), some crypto (e.g. pbkdf2) and zlib. Network sockets do not — they're OS-async (epoll/kqueue/IOCP).A fájlrendszer, a DNS lookupok (dns.lookup), egyes crypto műveletek (pl. pbkdf2) és a zlib. A hálózati socketek nem — azok OS-aszinkronok (epoll/kqueue/IOCP).
  • Default pool size is 4 (UV_THREADPOOL_SIZE, up to 1024).Az alapértelmezett pool-méret 4 (UV_THREADPOOL_SIZE, akár 1024-ig).

Why does this matter?Miért számít ez?

  • CPU-heavy pool work (many crypto hashes) can exhaust the 4 threads and stall unrelated fs/DNS work — tune the pool or move to worker threads.A CPU-igényes pool-munka (sok crypto hash) kimerítheti a 4 szálat, és megakaszthat tőle független fs/DNS műveleteket — hangold a poolt vagy válts worker threadekre.

Watch outBuktatóNetwork requests scale far beyond the pool because sockets are OS-async — only pool-backed operations are capped by UV_THREADPOOL_SIZE.A hálózati kérések jóval a poolon túl skálázódnak, mert a socketek OS-aszinkronok — csak a pool-alapú műveleteket korlátozza az UV_THREADPOOL_SIZE.

Streams & Backpressure

Node

Streams process data in chunks instead of loading it all into memory. Four types: Readable, Writable, Duplex, Transform.A streamek darabokban dolgozzák fel az adatot ahelyett, hogy az egészet memóriába töltenék. Négy típus: Readable, Writable, Duplex, Transform.

When the sink can't keep up, the source pauses until the buffer drains.Ha a nyelő nem bírja a tempót, a forrás szünetel, amíg a buffer ki nem ürül.

Why streams?Miért streamek?

  • Constant memory for arbitrarily large data (files, HTTP bodies, uploads), start processing before all data arrives, and composability via pipe/pipeline.Állandó memóriahasználat tetszőlegesen nagy adatnál (fájlok, HTTP bodyk, feltöltések), a feldolgozás megkezdhető, mielőtt az összes adat megérkezne, és komponálhatóság a pipe/pipeline révén.

What is backpressure?Mi az a backpressure?

  • When a writable can't keep up, its buffer fills. pipeline() pauses the source until the destination drains — preventing unbounded memory growth.Amikor egy writable nem bírja a tempót, megtelik a buffere. A pipeline() szünetelteti a forrást, amíg a cél ki nem ürül — így nem nő korlátlanul a memóriahasználat.

Watch outBuktatóUse stream.pipeline(), not raw .pipe() — pipeline handles backpressure and propagates errors + cleans up. Plain .pipe() silently drops errors.stream.pipeline()-t használj, ne sima .pipe()-ot — a pipeline kezeli a backpressure-t és továbbítja a hibákat + takarít. A sima .pipe() csendben elnyeli a hibákat.

Concurrency Control

Node

Running multiple async operations with the right amount of parallelism.Több aszinkron művelet futtatása a megfelelő mértékű párhuzamossággal.

Promise combinators?Promise kombinátorok?

  • all — all succeed, or reject on first failure. allSettled — wait for all, get each result/error. race — first to settle. any — first to fulfill.all — mind sikerül, vagy az első hibánál elutasít. allSettled — megvárja mindet, minden eredményt/hibát megkapsz. race — az első, ami lezárul. any — az első, ami sikeresen teljesül.

Why limit concurrency?Miért korlátozzuk a párhuzamosságot?

  • Firing 10,000 requests/queries at once exhausts sockets, connections and memory. Cap in-flight work with a limiter (p-limit) or batching.Ha egyszerre indítasz 10 000 kérést/lekérdezést, kimeríted a socketeket, a kapcsolatokat és a memóriát. Korlátozd a folyamatban lévő munkát limiterrel (p-limit) vagy batcheléssel.

Watch outBuktatóPromise.all rejects on the first failure, but the other promises keep running — they aren't cancelled. Use allSettled when you need every result regardless of individual failures.A Promise.all az első hibánál elutasít, de a többi promise tovább fut — nem szakadnak meg. Használj allSettled-et, ha minden eredmény kell, függetlenül az egyedi hibáktól.

Connection Pooling

Node

Reusing a fixed set of open DB connections across requests instead of opening/closing one per query.Egy fix készletnyi nyitott adatbázis-kapcsolat újrahasznosítása a kérések között ahelyett, hogy lekérdezésenként nyitnánk/zárnánk egyet.

The pool caps total connections and skips the per-query TCP + auth handshake.A pool korlátozza az összes kapcsolatot, és megspórolja a lekérdezésenkénti TCP + auth handshake-et.

Why pool?Miért poolozunk?

  • Opening a DB connection is expensive (TCP + auth). A pool keeps connections warm; requests borrow and return them, capping the total the DB must handle.Egy adatbázis-kapcsolat megnyitása drága (TCP + auth). A pool melegen tartja a kapcsolatokat; a kérések kölcsönzik és visszaadják őket, így korlátozódik, mennyit kell az adatbázisnak kezelnie.

How do you size it?Hogyan méretezed?

  • Bounded by the DB's max connections and CPU — bigger isn't better. Too many connections thrash the DB; a modest pool per instance often wins.Az adatbázis max kapcsolatszáma és CPU-ja határolja — a nagyobb nem jobb. A túl sok kapcsolat megfojtja az adatbázist; instance-onként egy mérsékelt pool gyakran jobb.

Watch outBuktatóPool exhaustion (all borrowed, none returned) makes requests hang — usually from leaked connections or long queries. In serverless, use a pooler like PgBouncer to avoid connection storms.A pool kimerülése (minden kikölcsönözve, semmi visszaadva) függő kéréseket okoz — általában elszivárgott kapcsolatok vagy hosszú lekérdezések miatt. Serverlessben használj poolert (pl. PgBouncer), hogy elkerüld a kapcsolat-viharokat.

Locking & DeadlocksLockolás & deadlockok

Node

Locks are how a database enforces isolation: a writer blocks other writers on the same row until it commits. A deadlock is two transactions each holding what the other needs — a cycle no amount of waiting resolves.A lockok révén kényszeríti ki az adatbázis az izolációt: egy író blokkolja a többi írót ugyanazon a soron, amíg nem commitol. A deadlock az, amikor két tranzakció mindegyike azt tartja, amire a másiknak szüksége van — olyan kör, amit semennyi várakozás nem old fel.

Deadlocks are detected, not prevented: the database picks a victim and aborts it. Handling that error is the application's job.A deadlockokat észlelik, nem előzik meg: az adatbázis kiválaszt egy áldozatot, és megszakítja. A hiba kezelése az alkalmazás dolga.

Optimistic or pessimistic locking — how do you choose?Optimista vagy pesszimista lockolás — hogyan választasz?

  • Pessimistic takes the lock up front (SELECT … FOR UPDATE) — correct under real contention, but holds the lock for the whole transaction and can queue everyone behind a slow request. Optimistic takes no lock and checks a version column on write (WHERE id = ? AND version = ?); if zero rows updated, someone beat you and you retry. Optimistic wins when conflicts are rare; pessimistic wins when they're common, because retry storms cost more than waiting.A pesszimista előre megszerzi a lockot (SELECT … FOR UPDATE) — valós versengésnél helyes, de a teljes tranzakció idejére tartja a lockot, és mindenkit besorolhat egy lassú kérés mögé. Az optimista nem lockol, hanem íráskor ellenőriz egy version oszlopot (WHERE id = ? AND version = ?); ha nulla sor frissült, valaki megelőzött, és újrapróbálsz. Az optimista akkor nyer, ha a konfliktus ritka; a pesszimista akkor, ha gyakori, mert az újrapróbálkozás-viharok többe kerülnek, mint a várakozás.

How do you actually avoid deadlocks?Hogyan kerülöd el ténylegesen a deadlockokat?

  • The one reliable rule: acquire locks in a consistent order everywhere — if every transaction touches accounts in ascending id order, the cycle can't form. Beyond that: keep transactions short, don't do network calls or user interaction inside one, and touch the fewest rows you can. Then still handle the deadlock error and retry, because you cannot eliminate them entirely.Az egyetlen megbízható szabály: mindenhol konzisztens sorrendben szerezd meg a lockokat — ha minden tranzakció növekvő id-sorrendben nyúl a számlákhoz, a kör nem alakulhat ki. Ezen túl: tartsd rövidre a tranzakciókat, ne legyen bennük hálózati hívás vagy felhasználói interakció, és a lehető legkevesebb sort érintsd. Aztán a deadlock-hibát így is kezeld és próbáld újra, mert teljesen kiiktatni nem lehet.

What are FOR UPDATE and SKIP LOCKED for?Mire való a FOR UPDATE és a SKIP LOCKED?

  • SELECT … FOR UPDATE locks the selected rows so a concurrent transaction can't change them before you write — the standard read-modify-write guard. FOR UPDATE SKIP LOCKED steps over rows someone else already locked, which turns a table into a work queue: many workers each grab different jobs with no coordination and no double-processing. It's the simplest correct job queue you can build without adding a broker.A SELECT … FOR UPDATE lockolja a kiválasztott sorokat, hogy egy párhuzamos tranzakció ne módosíthassa őket az írásod előtt — ez a bevett read-modify-write védelem. A FOR UPDATE SKIP LOCKED átlép a más által már lockolt sorokon, ami munkasorrá alakítja a táblát: sok worker mindegyike más feladatot ragad meg, koordináció és kettős feldolgozás nélkül. Ez a legegyszerűbb helyes job queue, amit broker hozzáadása nélkül építhetsz.

Watch outBuktatóAn ORM can create lock ordering you never wrote — saveAll on a collection may update rows in hash order, differing per process. Also, a deadlock error is retryable and a lock timeout usually isn't; treating them identically either loses work or hammers the database. And an unindexed foreign key makes a delete on the parent lock far more of the child table than expected.Egy ORM olyan lock-sorrendet hozhat létre, amit sosem írtál — egy kollekción futó saveAll hash-sorrendben frissíthet, ami processzenként eltér. Továbbá a deadlock-hiba újrapróbálható, a lock timeout általában nem; ha azonosan kezeled őket, vagy munkát veszítesz, vagy az adatbázist ostromolod. Egy indexeletlen idegen kulcs pedig a szülőn futó törléskor a vártnál sokkal nagyobb részét lockolja a gyermektáblának.

How do you authenticate a WebSocket?Hogyan autentikálsz egy WebSocketet?

Why do you need ping/pong if TCP already detects failure?Miért kell ping/pong, ha a TCP már észleli a hibát?

What breaks first as connection count grows?Mi törik el először, ahogy nő a kapcsolatszám?

What happens to connections during a deploy?Mi történik a kapcsolatokkal deploy közben?

Addig az üzenet nincs elveszve — foglalt, és visszajön, ha a fogyasztó meghal. Az, hogy mikor ackelsz, dönti el a kézbesítési szemantikádat.

What is a visibility timeout?Mi a visibility timeout?

  • The window during which a received message is hidden from other consumers. Ack inside it and the message is deleted; let it expire and the broker assumes you died and hands the message to someone else. It's SQS's term, but the mechanism is universal — RabbitMQ ties it to the channel's lifetime, Kafka to the consumer group's session timeout.Az az ablak, amíg egy fogadott üzenet rejtve van a többi fogyasztó elől. Ez az SQS terminológiája, de a mechanizmus általános — a RabbitMQ a csatorna élettartamához, a Kafka a fogyasztói csoport session timeoutjához köti.

What goes wrong if the timeout is too short?Mi romlik el, ha túl rövid a timeout?

  • You get duplicate processing that looks like a bug in your code. A job that usually takes 20 seconds against a 30-second timeout will, on a slow day, get redelivered while the first consumer is still working — so two workers process the same order and neither has crashed. Either raise the timeout above your worst case, or extend the lease periodically from the consumer (a heartbeat for the message) while work continues.Duplikált feldolgozást kapsz, ami a kódod hibájának látszik. Egy általában 20 másodperces feladat 30 másodperces timeouttal egy lassú napon újrakézbesítődik, miközben az első fogyasztó még dolgozik — így két worker dolgozza fel ugyanazt a rendelést, és egyik sem omlott össze. Vagy emeld a timeoutot a legrosszabb eseted fölé, vagy hosszabbítsd a bérletet periodikusan a fogyasztóból (heartbeat az üzenethez), amíg tart a munka.

Nack, or just let it time out?Nack, vagy hagyd lejárni?

  • Nack when you know immediately that this consumer can't handle it — it returns the message straight away instead of blocking a slot for the full timeout. But an immediate nack-and-retry on a message that always fails becomes a hot loop hammering the broker, so nack should either carry a delay or be paired with a retry counter that routes to a dead-letter queue. Letting it time out is the correct behaviour only for actual crashes.Nackelj, ha azonnal tudod, hogy ez a fogyasztó nem tudja kezelni — így az üzenet rögtön visszakerül, ahelyett hogy a teljes timeoutra lefoglalna egy helyet. De az azonnali nack-és-újrapróbálás egy mindig hibázó üzeneten forró ciklussá válik, ami ostromolja a brokert, ezért a nack vagy késleltetést hordozzon, vagy párosuljon újrapróbálkozás-számlálóval, ami dead-letter queue-ba irányít. A lejáratni hagyás csak valódi összeomlásoknál helyes viselkedés.

Watch outBuktatóPrefetch (how many unacked messages a consumer holds) defaults high in some clients, which means one slow worker reserves a large batch it can't get through while other idle workers starve. Dead-Letter Queues & Poison MessagesDead-letter queue-k & mérgezett üzenetek

Realtime

A poison message is one that fails every time it's processed. With at-least-once delivery and unlimited retries it never leaves the queue, blocking or degrading everything behind it. A dead-letter queue is where it goes instead.A mérgezett üzenet az, amely minden feldolgozáskor hibázik. At-least-once kézbesítéssel és korlátlan újrapróbálkozással sosem hagyja el a sort, blokkolva vagy rontva mindent, ami mögötte van. A dead-letter queue az a hely, ahová helyette kerül.

Retry a bounded number of times, then divert — so one bad message can't consume the consumer's capacity indefinitely.Próbáld újra korlátozott számban, aztán tereld el — így egy rossz üzenet nem foglalhatja le korlátlanul a fogyasztó kapacitását.

Which failures should retry, and which shouldn't?Mely hibák próbálkozzanak újra, és melyek ne?

  • Retry transient failures — a timeout, a 503, a deadlock, a briefly unavailable dependency. Don't retry deterministic ones: malformed payload, validation failure, a referenced record that will never exist. Retrying those burns capacity to reach the same conclusion five times. Distinguishing them in the consumer, and dead-lettering the second class immediately, is what separates a queue that self-heals from one that just accumulates.Az átmeneti hibákat próbáld újra — timeout, 503, deadlock, rövid ideig elérhetetlen függőség. A determinisztikusakat ne: hibás formátumú payload, validációs hiba, olyan hivatkozott rekord, ami sosem fog létezni. Ezek újrapróbálása kapacitást éget, hogy ötször ugyanarra a következtetésre jusson. A kettő megkülönböztetése a fogyasztóban, és a második osztály azonnali dead-letterezése az, ami elválasztja az öngyógyító sort a puszta halmozótól.

What do you do with the DLQ afterwards?Mit kezdesz utána a DLQ-val?

  • It needs an owner, an alert on depth, and a replay path — otherwise it's just where messages go to be forgotten, which is data loss with extra steps. The workflow that actually functions: alert fires, someone inspects the message and the failure reason it was stored with, fixes the bug or the data, and replays the batch back to the main queue. That last step must be idempotent, since some of those messages may have partially succeeded.Kell hozzá gazda, riasztás a mélységre, és visszajátszási út — különben csak az a hely, ahová az üzenetek elfelejtődni mennek, ami adatvesztés extra lépésekkel. A ténylegesen működő folyamat: megszólal a riasztás, valaki megnézi az üzenetet és a vele eltárolt hibaokot, javítja a hibát vagy az adatot, és visszajátssza a köteget a fő sorba. Ez az utolsó lépés legyen idempotens, mert néhány üzenet részben sikeres lehetett.

Where does the retry count live?Hol él az újrapróbálkozás-számláló?

  • Ideally in the broker — SQS tracks ApproximateReceiveCount and moves the message after a configured maximum, RabbitMQ uses a DLX with a per-queue policy. Tracking it in the message body works too but is fragile, since a consumer that crashes before writing the increment loses the count and the message retries forever. Broker-side is both simpler and harder to get wrong.Ideális esetben a brokerben — az SQS követi az ApproximateReceiveCount-ot, és beállított maximum után átmozgatja az üzenetet, a RabbitMQ DLX-et használ soronkénti szabállyal. Az üzenettörzsben követés is működik, de törékeny, mert egy a növelés kiírása előtt összeomló fogyasztó elveszíti a számlálót, és az üzenet örökké újrapróbálkozik. A broker-oldal egyszerre egyszerűbb és nehezebben elrontható.

Watch outBuktatóStore the failure reason alongside the message when dead-lettering. A DLQ full of payloads with no indication of why each failed turns every investigation into a re-run of the bug, and by then the surrounding state has usually changed.Dead-letterezéskor tárold a hiba okát az üzenet mellett. Egy payloadokkal teli DLQ, amiből nem derül ki, miért hibázott mindegyik, minden vizsgálatot a hiba újrajátszásává tesz — és addigra a környező állapot általában megváltozott.

Architecture

Organizing code into horizontal layers — typically routes/controllers, services (business logic), and repositories (data access) — with dependencies pointing in one direction only.A kód vízszintes rétegekbe szervezése — tipikusan route-ok/controllerek, szolgáltatások (üzleti logika), és repository-k (adathozzáférés) — a függőségek csak egy irányba mutatnak.

What is an ADR?Mi az ADR?

  • An Architecture Decision Record: a short document capturing the context, the decision, the alternatives considered and the consequences. It answers the question future maintainers actually ask — “why is it like this?”Architecture Decision Record: egy rövid dokumentum, amely rögzíti a kontextust, a döntést, a mérlegelt alternatívákat és a következményeket. Arra a kérdésre válaszol, amit a jövőbeli karbantartók ténylegesen feltesznek — „miért így van?”

Watch outBuktatóAnswering with an unconditional “microservices” or “clean architecture” signals pattern-matching rather than judgement. Every structure is right for some context and wrong for others — say which context you're assuming.A feltétel nélküli „microservices” vagy „clean architecture” válasz minta-illesztést jelez, nem ítélőképességet. Minden szerkezet helyes valamely kontextusban és helytelen másokban — mondd ki, melyik kontextust feltételezed.

Strangler Fig MigrationStrangler fig migráció

Architecture

Replace a legacy system incrementally by routing traffic away from it, feature by feature, until nothing is left. The alternative — a rewrite that ships in one release — is the failure mode this pattern exists to avoid.A legacy rendszert fokozatosan váltod ki: funkcióról funkcióra tereled el róla a forgalmat, míg semmi nem marad belőle. Az alternatíva — egyetlen release-ben szállított újraírás — az a hibamód, aminek elkerülésére ez a minta létezik.

The facade is the only thing clients see, so each feature can move independently — and move back if it misbehaves.A homlokzat az egyetlen, amit a kliensek látnak, így minden funkció önállóan költözhet — és vissza is, ha rosszul viselkedik.

Why not just rewrite it?Miért ne írjuk egyszerűen újra?

  • Because a big-bang rewrite delivers no value until the very end, while the legacy system keeps changing underneath you — so you're chasing a moving target with a growing divergence and a single, unrehearsed cutover at the end. The strangler approach ships value continuously, keeps rollback cheap because only one feature moved, and forces you to understand the legacy behaviour piece by piece rather than all at once.Mert a big-bang újraírás egészen a végéig nem szállít értéket, miközben a legacy rendszer közben folyamatosan változik alattad — így mozgó célpontot üldözöl növekvő eltéréssel, és a végén egyetlen, be nem gyakorolt átállással. A strangler megközelítés folyamatosan szállít értéket, olcsón tartja a visszagörgetést, mert csak egy funkció költözött, és arra kényszerít, hogy darabonként értsd meg a legacy viselkedést, ne egyszerre.

Where do you make the first cut?Hol vágsz először?

  • Somewhere with a clear boundary and low blast radius — ideally a read-mostly capability that owns little shared data. Starting with the hardest, most entangled core is a common mistake: you get no early wins, no proven cutover process, and the team's confidence in the migration erodes before it produces anything. The first slice's real job is to prove the routing, monitoring and rollback machinery works.Ott, ahol tiszta a határ és kicsi a robbanási sugár — ideálisan egy jórészt olvasó képesség, ami kevés megosztott adatot birtokol. Gyakori hiba a legnehezebb, legjobban összegabalyodott maggal kezdeni: nincs korai siker, nincs bizonyított átállási folyamat, és a csapat bizalma a migrációban elfogy, mielőtt bármit termelne. Az első szelet valódi dolga az, hogy bizonyítsa: az útválasztás, a monitorozás és a visszagörgetés gépezete működik.

What makes a strangler migration stall?Mitől akad el egy strangler migráció?

  • Running out of political will before the legacy system is switched off — leaving two systems to maintain forever, which is worse than either alone. The defences are: a named owner, a decommissioning date treated as a deliverable rather than an aspiration, and refusing to add new features to the legacy path. If everything easy has moved and the last 20% never does, you've paid the full cost and collected half the benefit.Ha a politikai akarat elfogy, mielőtt a legacy rendszert kikapcsolnák — így örökre két karbantartandó rendszer marad, ami rosszabb bármelyiknél önmagában. A védekezés: nevesített gazda, szállítandóként (nem vágyként) kezelt kivezetési dátum, és annak megtagadása, hogy új funkció kerüljön a legacy útra. Ha minden könnyű átköltözött, és az utolsó 20% soha, akkor a teljes költséget kifizetted, és a haszon felét szedted be.

Watch outBuktatóThe facade becomes a single point of failure and a place where logic quietly accumulates. Keep it dumb — routing and nothing else. Once it contains business rules, it's a third system you now have to migrate too.A homlokzat egyetlen hibaponttá válik, és olyan hellyé, ahol csendben logika gyűlik fel. Tartsd butának — útválasztás és semmi más. Amint üzleti szabályokat tartalmaz, egy harmadik rendszer lesz, amit szintén migrálnod kell.

Zero-Downtime Schema MigrationLeállás nélküli séma-migráció

Architecture

Changing a schema while the old and new code are both running requires every intermediate state to be valid for both. The technique is expand–migrate–contract: add, backfill, switch, then remove — never in one step.Séma megváltoztatása úgy, hogy a régi és az új kód is fut, megköveteli, hogy minden köztes állapot érvényes legyen mindkettőnek. A technika: expand–migrate–contract — hozzáad, feltölt, átkapcsol, majd eltávolít — sosem egy lépésben.

Why can't you just rename the column?Miért nem lehet egyszerűen átnevezni az oszlopot?

  • Because during a rolling deploy both versions of the code run simultaneously, and a rename is instantly breaking for whichever half hasn't updated. The same applies to changing a type, adding a NOT NULL constraint, or removing a field from an API response. Every one of those has an expand–contract equivalent that takes several deploys instead of one — the extra deploys are the price of not having downtime.Mert a rolling deploy alatt a kód mindkét verziója egyszerre fut, és az átnevezés azonnal töri azt a felet, amelyik még nem frissült. Which migrations lock the table?Mely migrációk lockolják a táblát?

  • This is where the theory meets the specific database. In Postgres, adding a nullable column is instant, but adding one with a volatile default rewrites the table on older versions; creating an index locks writes unless you use CREATE INDEX CONCURRENTLY; adding a NOT NULL constraint scans the whole table. Knowing that a migration which passed in staging on 10,000 rows will lock production for minutes on 100 million is the difference between a planned change and an incident.Itt találkozik az elmélet a konkrét adatbázissal. Annak ismerete, hogy egy stagingen 10 000 soron átment migráció produkcióban 100 millión percekre lockol, a különbség egy tervezett változtatás és egy incidens között.

How does this interact with rollback?Hogyan viszonyul ez a visszagörgetéshez?

  • It's the main reason for the sequence. Because each step leaves the schema compatible with the previous code version, you can roll back a deploy without rolling back the database — which matters, since restoring a database is slow and lossy. Only the final contract step is irreversible, which is why it happens last and only after the new path has been running successfully for a while.Ez a fő oka a sorrendnek. Mivel minden lépés a korábbi kódverzióval kompatibilis sémát hagy hátra, visszagörgethetsz egy deployt anélkül, hogy az adatbázist görgetnéd vissza — ami számít, mert egy adatbázis visszaállítása lassú és veszteséges. Csak az utolsó contract lépés visszafordíthatatlan, ezért történik utoljára, és csak azután, hogy az új út egy ideje sikeresen fut.

Watch outBuktatóA long-running migration inside one transaction holds locks for its entire duration and can block the application while appearing to be "just running". Batch it, commit per batch, and check whether your migration tool wraps everything in a single transaction by default — many do.Egy hosszan futó migráció egyetlen tranzakción belül a teljes időtartamra tartja a lockokat, és blokkolhatja az alkalmazást, miközben úgy tűnik, „csak fut". Kötegeld, kötegenként commitolj, és nézd meg, hogy a migrációs eszközöd alapból egyetlen tranzakcióba csomagol-e mindent — sok igen.

Rollout, Flags & RollbackRollout, flagek & visszagörgetés

Architecture

Separating deployment from release: ship the code dark, then turn it on for a slice of traffic. The goal is to make every change reversible in seconds rather than requiring a redeploy.A deployment és a release szétválasztása: szállítsd a kódot sötéten, majd kapcsold be a forgalom egy szeletére. A cél az, hogy minden változtatás másodpercek alatt visszafordítható legyen, ne új deployt igényeljen.

What is shadow traffic for?Mire való a shadow traffic?

  • Sending a copy of real production requests to the new implementation while still serving responses from the old one, then comparing outputs. It's the only way to test a migration against genuine traffic shape — the volume, the weird inputs, the long tail of cases nobody wrote a test for — with zero user risk. The catch is side effects: the shadow path must not write, send email or charge cards, so it usually needs a read-only mode built in rather than bolted on.Valós produkciós kérések másolatát küldöd az új implementációnak, miközben a válaszokat még a régi szolgáltatja, majd összehasonlítod a kimeneteket. Ez az egyetlen mód a migráció tesztelésére valódi forgalmi alakon — a volumenen, a fura inputokon, azon eseteken, amikre senki nem írt tesztet — nulla felhasználói kockázattal. A bökkenő a mellékhatás: az árnyék-út nem írhat, nem küldhet e-mailt, nem terhelhet kártyát, tehát általában beépített, nem utólag ráaggatott csak-olvasható módra van szüksége.

Canary or blue-green?Canary vagy blue-green?

  • Canary shifts a small percentage of traffic to the new version and increases it as metrics stay healthy — cheap, gradual, and it exposes problems that only appear under partial load. Blue-green runs two full environments and switches all at once, giving instant rollback but requiring double the capacity and offering no gradual signal. Canary is the usual default; blue-green suits changes that can't be partially applied.A canary a forgalom kis százalékát tereli az új verzióra, és növeli, amíg a metrikák egészségesek maradnak — olcsó, fokozatos, és olyan problémákat hoz elő, amelyek csak részleges terhelésnél jelentkeznek. A blue-green két teljes környezetet futtat, és egyszerre kapcsol át: azonnali visszagörgetést ad, de dupla kapacitást igényel, és nem ad fokozatos jelzést. A canary a szokásos alapértelmezés; a blue-green olyan változtatásokhoz illik, amiket nem lehet részlegesen alkalmazni.

What makes a rollback plan real rather than theoretical?Mitől valódi egy visszagörgetési terv, nem elméleti?

  • Having rehearsed it, and knowing what it can't undo. Code rolls back cleanly; data written by the new version does not, which is why schema changes are sequenced so the old code can still run. Define the trigger in advance — the specific metric and threshold that means abort — because deciding mid-incident, with people watching, reliably takes longer than it should.Az, hogy begyakoroltad, és tudod, mit nem tud visszacsinálni. A kód tisztán visszagörög; az új verzió által írt adat nem, ezért van úgy sorrendezve a séma-változtatás, hogy a régi kód még futhasson. Definiáld előre a kiváltó okot — azt a konkrét metrikát és küszöböt, ami leállítást jelent —, mert incidens közben, közönség előtt dönteni megbízhatóan tovább tart a kelleténél.

Watch outBuktatóFeature flags are debt with an expiry date. Each one doubles the paths through the code, and a codebase with hundreds of stale flags has a combinatorial test surface nobody understands. Give every flag an owner and a removal date at creation, and delete it once the rollout is complete.A feature flag lejárati dátummal rendelkező adósság. Mindegyik megduplázza a kódon átvezető utakat, és egy több száz elavult flaget tartalmazó kódbázisnak olyan kombinatorikus tesztfelülete lesz, amit senki nem ért. Adj minden flagnek gazdát és eltávolítási dátumot már létrehozáskor, és töröld, amint a rollout lezárult.

Why do integration tests catch what unit tests miss?Miért fogják meg az integrációs tesztek azt, amit a unit tesztek nem?

What should E2E tests cover?Mit fedjenek le az E2E tesztek?

Containers & DockerKonténerek & Docker

DevOps

Packaging an application with its dependencies into an isolated, portable unit that runs identically anywhere — solving “works on my machine” by shipping the environment with the code.Egy alkalmazás becsomagolása a függőségeivel egy izolált, hordozható egységbe, amely mindenhol azonosan fut — a „nálam működik” problémát oldja meg azzal, hogy a környezetet a kóddal együtt szállítja.

Container vs virtual machine?Konténer vs virtuális gép?

  • A VM virtualizes hardware and runs a full guest OS — heavy (GBs), slow to boot, strongly isolated. A container shares the host kernel and isolates only the process namespace and filesystem — light (MBs), starts in milliseconds, weaker isolation boundary.Egy VM hardvert virtualizál és teljes vendég OS-t futtat — nehéz (GB-ok), lassan indul, erősen izolált. Egy konténer osztozik a host kernelén és csak a folyamat-névteret és fájlrendszert izolálja — könnyű (MB-ok), ezredmásodpercek alatt indul, gyengébb izolációs határ.

What is an image versus a container?Mi a különbség image és konténer között?

  • An image is the immutable, layered template built from a Dockerfile. A container is a running instance of that image with a writable layer on top. One image, many containers — like a class and its instances.Az image az immutábilis, rétegzett sablon, amely egy Dockerfile-ból épül. A konténer ennek az image-nek egy futó példánya, egy írható réteggel a tetején. Egy image, sok konténer — mint egy osztály és a példányai.

function longestUnique(s) {
  const lastSeen = new Map();
  let best = 0, start = 0;
  for (let end = 0; end < s.length; end++) {
    const c = s[end];
    // only jump forward — never move start backwards
    if (lastSeen.has(c) && lastSeen.get(c) >= start) {
      start = lastSeen.get(c) + 1;
    }
    lastSeen.set(c, end);
    best = Math.max(best, end - start + 1);
  }
  return best;
}

What is “binary search on the answer”?Mi a „bináris keresés a válaszon”?

  • When the answer is a number in a range and you can cheaply test “is x feasible?”, and feasibility is monotonic (once true, always true above), binary search the range instead of the array. Classic uses: minimum capacity, minimum days, minimum speed.Amikor a válasz egy szám egy tartományban, és olcsón tesztelhető, hogy „megvalósítható-e x?”, és a megvalósíthatóság monoton (ha egyszer igaz, felette mindig igaz), a tömb helyett a tartományon keress binárisan. Klasszikus használat: minimális kapacitás, minimális napok, minimális sebesség.

How do you avoid the classic off-by-one and infinite loops?Hogyan kerülöd el a klasszikus off-by-one-t és a végtelen ciklust?

  • Pick one template and keep it. With while (l < r) and mid = (l+r)>>1, always move at least one pointer past mid (l = mid+1), or the loop can stall when l and r are adjacent.Válassz egy sablont és tartsd magad hozzá. A while (l < r) és mid = (l+r)>>1 mellett mindig mozgass legalább egy mutatót mid-en túlra (l = mid+1), különben a ciklus megakadhat, amikor az l és r szomszédos.
Leftmost insertion point (lower bound)Bal szélső beszúrási pont (lower bound)O(log n) time / O(1) space
function lowerBound(nums, target) {
  let l = 0, r = nums.length;          // r is exclusive
  while (l < r) {
    const mid = (l + r) >> 1;
    if (nums[mid] < target) l = mid + 1;
    else r = mid;                      // keep mid as a candidate
  }
  return l;   // first index with nums[i] >= target
}

Watch outBuktatóIn other languages (l+r)/2 can overflow; in JS the real trap is different — (l+r)>>1 uses 32-bit ints, so it breaks above ~2³¹. Use Math.floor((l+r)/2) or l + ((r-l)>>1) for very large ranges (common in binary-search-on-answer problems).Más nyelvekben az (l+r)/2 túlcsordulhat; JS-ben a valódi csapda más — az (l+r)>>1 32 bites egészeket használ, szóval ~2³¹ felett elromlik. Nagyon nagy tartományokhoz (gyakori a válaszon-bináris-keresés feladatoknál) használj Math.floor((l+r)/2)-t vagy l + ((r-l)>>1)-et.

How do you implement an efficient queue in JS?Hogyan implementálsz hatékony sort JS-ben?

  • Not with shift() (O(n)). Keep an array plus a head index and advance the index on dequeue — O(1) per operation, at the cost of not reclaiming memory until you compact.Nem shift()-tel (O(n)). Tarts egy tömböt plusz egy head indexet, és léptesd az indexet dequeue-nál — O(1) műveletenként, azon az áron, hogy nem szabadítasz fel memóriát, amíg nem tömörítesz.
O(1) queue via head pointerO(1) sor head-mutatóvalenqueue/dequeue O(1) amortized
class Queue {
  #items = []; #head = 0;
  enqueue(x) { this.#items.push(x); }
  dequeue() {
    if (this.#head >= this.#items.length) return undefined;
    const x = this.#items[this.#head++];
    // compact occasionally to free memory
    if (this.#head > 50 && this.#head * 2 > this.#items.length) {
      this.#items = this.#items.slice(this.#head);
      this.#head = 0;
    }
    return x;
  }
  get size() { return this.#items.length - this.#head; }
}
