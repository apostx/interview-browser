Virtual DOM & Reconciliation

React

React renders a lightweight in-memory tree, diffs it against the previous one (reconciliation), and applies only the minimal real-DOM mutations.A React egy könnyű, memóriabeli fát renderel, összeveti az előzővel (reconciliation), és csak a minimálisan szükséges valódi DOM-módosításokat hajtja végre.

Same type + key → reuse and patch. Different type → tear down and rebuild.Azonos típus + key → újrahasznosítás és patch. Eltérő típus → lebontás és újraépítés.

What's the diffing heuristic?Mi a diffing heurisztika?

  • An O(n) heuristic, not a full tree diff. Two assumptions: different element types produce different trees; stable keys identify which children moved.Egy O(n) heurisztika, nem teljes fa-diff. Két feltevés: eltérő elemtípusok eltérő fát eredményeznek; a stabil key-ek azonosítják, mely gyerekek mozdultak el.

What actually causes a re-render?Mi okoz ténylegesen re-rendert?

  • A state change, a parent re-render, a consumed context value changing, or a subscribing hook firing.State-változás, a szülő re-rendere, egy fogyasztott context-érték változása, vagy egy feliratkozó hook eseménye.
  • Re-render ≠ DOM update — reconciliation may find nothing changed and touch no DOM.A re-render ≠ DOM-frissítés — a reconciliation azt is találhatja, hogy semmi nem változott, és hozzá sem nyúl a DOM-hoz.

Render vs commit phase?Render vs commit fázis?

  • Render: React calls your components and builds the tree — pure, and pausable in concurrent mode.Render: a React meghívja a komponenseidet és felépíti a fát — tiszta, és concurrent módban megszakítható.
  • Commit: applies DOM changes and runs effects — synchronous.Commit: alkalmazza a DOM-változásokat és lefuttatja az effekteket — szinkron.

Watch outBuktatóA re-render is relatively cheap; the cost is your render logic. Don't optimize re-renders you haven't profiled.A re-render viszonylag olcsó; a költség a te render-logikád. Ne optimalizálj olyan re-rendert, amit nem profiloztál.

Keys in ListsKey-ek listákban

React

key is the stable identity React uses to match elements across renders.A key az a stabil identitás, amivel a React renderek között összepárosítja az elemeket.

With index keys, inserting at the top shifts every key and misassigns DOM/state.Index-key esetén a lista elejére szúrás minden key-t elcsúsztat, és rosszul rendeli hozzá a DOM-ot/state-et.

Why not use the array index as a key?Miért ne használjuk a tömb indexét key-ként?

  • On insert/reorder/delete, indexes shift, so React reuses the wrong DOM node and component state — e.g. an input's value sticks to the wrong row.Beszúrásnál/átrendezésnél/törlésnél az indexek elcsúsznak, így a React rossz DOM-node-ot és komponens-state-et használ újra — pl. egy input értéke rossz sorhoz ragad.

What makes a good key?Mitől jó egy key?

  • Stable, unique among siblings, and tied to the data's identity (an ID) — not its position. It only needs to be unique within the list, not globally.Stabil, a testvérek között egyedi, és az adat identitásához (egy ID-hez) kötött — nem a pozíciójához. Csak a listán belül kell egyedinek lennie, globálisan nem.

Watch outBuktatóIndex keys are safe only for static, never-reordered lists. The classic bug: uncontrolled inputs in a reorderable list keyed by index.Az index-key csak statikus, soha nem átrendezett listáknál biztonságos. A klasszikus hiba: uncontrolled inputok index szerint key-elt, átrendezhető listában.

Rules of Hooks

React

Call hooks only at the top level — never in loops, conditions or nested functions — and only from React functions (components or custom hooks).A hookokat csak a legfelső szinten hívd — soha ne ciklusban, feltételben vagy beágyazott függvényben — és csak React függvényekből (komponensek vagy custom hookok).

Why the top-level rule?Miért van a top-level szabály?

  • React tracks hook state by call order, not by name. A conditional hook shifts the order between renders, so state gets misassigned to the wrong hook.A React a hook-state-et hívási sorrend alapján tartja nyilván, nem név szerint. Egy feltételes hook elcsúsztatja a sorrendet a renderek között, így a state rossz hookhoz kerül.

How do you do conditional logic then?Akkor hogyan csinálsz feltételes logikát?

  • Put the condition inside the hook (e.g. inside useEffect), or split into separate components — never wrap the hook call itself.A feltételt tedd a hookon belülre (pl. a useEffect-en belül), vagy bontsd külön komponensekre — soha ne magát a hook-hívást csomagold feltételbe.

Watch outBuktatóeslint-plugin-react-hooks catches most violations — don't disable it to “make it work.”Az eslint-plugin-react-hooks a legtöbb szabálysértést elkapja — ne kapcsold ki, hogy „működjön”.

04

State & Batching

React

useState returns a value + setter; React batches multiple setState calls in one event into a single re-render.A useState egy értéket és egy settert ad vissza; a React egy eseményen belül több setState hívást egyetlen re-renderbe von össze (batching).

Why does calling setCount(count + 1) three times only add 1?Miért ad hozzá csak 1-et, ha háromszor hívod a setCount(count + 1)-et?

  • All three read the same stale count from the render's closure. Use the functional updater setCount(c => c + 1) so each builds on the latest value.Mindhárom ugyanazt az elavult count-ot olvassa a render closure-jéből. Használd a funkcionális updatert: setCount(c => c + 1), így mindegyik a legfrissebb értékre épül.

Is state synchronous?Szinkron a state?

  • No — setters schedule a re-render; the variable only changes on the next render. React 18 batches across all contexts, including promises and timeouts (automatic batching).Nem — a setterek re-rendert ütemeznek; a változó csak a következő renderben változik meg. A React 18 minden kontextusban batchel, beleértve a promise-okat és timeoutokat is (automatic batching).

Why immutability?Miért kell immutability?

  • React detects change by reference. Mutating state in place doesn't change the reference, so the update is missed. Always return new objects/arrays.A React referencia alapján észleli a változást. A state helyben módosítása nem változtatja meg a referenciát, így a frissítés elveszik. Mindig új objektumot/tömböt adj vissza.

Watch outBuktatóReading state right after setting it gives the old value — updates aren't applied synchronously.Ha közvetlenül a beállítás után olvasod a state-et, a régi értéket kapod — a frissítések nem szinkron módon érvényesülnek.

05

useEffect & Dependencies

React

useEffect runs a side effect after render, re-running whenever a value in its dependency array changes.A useEffect render után futtat egy mellékhatást, és újrafut, valahányszor a dependency tömbjében lévő érték megváltozik.

What belongs in the dependency array?Mi tartozik a dependency tömbbe?

  • Every reactive value used inside the effect (props, state, and anything derived from them). Missing deps → stale values; the lint rule enforces the full set.Minden reaktív érték, amit az effekten belül használsz (propok, state, és bármi, ami ezekből származik). Hiányzó dep → elavult érték; a lint szabály kikényszeríti a teljes listát.

When does the cleanup function run?Mikor fut le a cleanup függvény?

  • Before the next effect run and on unmount. Use it for subscriptions, timers and listeners to avoid leaks and race conditions.A következő effekt-futás előtt és unmountkor. Használd subscription-ökhöz, timerekhez és listenerekhez, hogy elkerüld a leakeket és a race condition-öket.

[] vs no array vs [deps]?[] vs nincs tömb vs [deps]?

  • No array → after every render. [] → once after mount. [deps] → when deps change.Nincs tömb → minden render után. [] → egyszer, mount után. [deps] → amikor a depek változnak.

Watch outBuktatóDon't use an effect to derive state from props/state — compute it during render. Effects are for synchronizing with external systems, not reacting to your own state. Object/function deps recreated each render cause infinite loops → memoize or move them.Ne használj effektet arra, hogy propokból/state-ből state-et származtass — számold ki render közben. Az effektek külső rendszerekkel való szinkronizálásra valók, nem a saját state-edre való reagálásra. A minden renderben újra létrehozott objektum/függvény depek végtelen ciklust okoznak → memoizáld vagy mozgasd ki őket.

Memoization

React

Caching to skip work: React.memo skips a re-render on shallow-equal props; useMemo caches a computed value; useCallback caches a function reference.Cache-elés a munka megspórolására: a React.memo kihagyja a re-rendert, ha a propok shallow-egyenlők; a useMemo egy kiszámolt értéket cache-el; a useCallback egy függvény-referenciát.

A new object/function passed as a prop defeats memo's shallow comparison.Egy propként átadott új objektum/függvény kivédi a memo shallow összehasonlítását.

When is each worth it?Mikor éri meg melyik?

  • useMemo — a genuinely expensive computation, or a stable value fed to memoized children / effect deps.useMemo — valóban drága számításnál, vagy ha stabil értéket adsz memoizált gyerekeknek / effekt-depeknek.
  • useCallback — a function passed to a memo child or used as an effect dep.useCallback — ha a függvényt memo-zott gyereknek adod át, vagy effekt-depként használod.
  • React.memo — a component that re-renders often with the same props and has a non-trivial render.React.memo — olyan komponensnél, ami gyakran renderel újra ugyanazokkal a propokkal, és nem triviális a renderje.

Why do useMemo/useCallback often do nothing?Miért nem csinál gyakran semmit a useMemo/useCallback?

  • If the memoized value still feeds a non-memoized child, or the deps change every render, you pay the overhead for no benefit.Ha a memoizált érték továbbra is nem-memoizált gyerekhez megy, vagy a depek minden renderben változnak, csak az overheadet fizeted haszon nélkül.

Watch outBuktatóMemoization costs memory + comparison. Default to not memoizing; add it when profiling shows a real problem. (The React Compiler auto-memoizes, reducing the manual need.)A memoizáció memóriába és összehasonlításba kerül. Alapból ne memoizálj; akkor tedd, ha a profilozás valós problémát mutat. (A React Compiler automatikusan memoizál, csökkentve a kézi igényt.)

Event Loop (Node)

Node

Node runs your JS on a single thread; the event loop (via libuv) processes callbacks in ordered phases, one cycle at a time.A Node egyetlen szálon futtatja a JS-edet; az event loop (a libuv-on keresztül) rendezett fázisokban dolgozza fel a callbackeket, ciklusonként.

Each phase runs its callback queue; nextTick + Promise microtasks drain between them.Minden fázis lefuttatja a saját callback-sorát; a nextTick és a Promise microtaskok közöttük futnak le.

How does it differ from the browser event loop?Miben tér el a böngésző event loopjától?

  • Browser: task → drain microtasks → render. Node has multiple phases (timers, poll, check…), no rendering, and adds process.nextTick as a separate, higher-priority queue.Böngésző: task → microtaskok lefutnak → render. A Node-nak több fázisa van (timers, poll, check…), nincs renderelés, és a process.nextTick külön, magasabb prioritású sorként jelenik meg.

Where do Promises and nextTick run?Hol futnak a Promise-ok és a nextTick?

  • After each operation/phase transition, Node drains the process.nextTick queue first, then the Promise microtask queue — before continuing.Minden művelet/fázisváltás után a Node először a process.nextTick sort futtatja le, utána a Promise microtask-sort — csak azután megy tovább.

Watch outBuktató“Single-threaded” refers to your JS. libuv and the OS handle the actual I/O, so one Node process serves thousands of concurrent connections without a thread per request.Az „egyszálú” a te JS-edre vonatkozik. A tényleges I/O-t a libuv és az OS kezeli, így egyetlen Node process több ezer párhuzamos kapcsolatot szolgál ki anélkül, hogy kérésenként szálat indítana.

Blocking the Event LoopAz event loop blokkolása

Node

Any long synchronous work blocks the single JS thread — heavy loops, sync crypto, parsing huge payloads, sync fs. No other request is served until it finishes.Bármilyen hosszú szinkron munka blokkolja az egyetlen JS szálat — nehéz ciklusok, szinkron crypto, hatalmas payloadok parse-olása, szinkron fs. Amíg be nem fejeződik, egyetlen másik kérés sem szolgálódik ki.

How do you avoid it?Hogyan kerülöd el?

  • Never use *Sync fs/crypto methods in request handlers; offload CPU-bound work to Worker Threads or a separate service; stream large data instead of buffering; paginate and limit.Soha ne használj *Sync fs/crypto metódusokat request handlerben; a CPU-igényes munkát told Worker Threadekre vagy külön szolgáltatásra; a nagy adatot streameld ahelyett, hogy pufferelnéd; lapozz és limitálj.

How do you detect it?Hogyan észleled?

  • Event-loop lag metrics (perf_hooks.monitorEventLoopDelay), an APM tool, or clinic.js.Event-loop lag metrikákkal (perf_hooks.monitorEventLoopDelay), APM eszközzel vagy a clinic.js-szel.

Watch outBuktatóOne slow synchronous handler degrades the whole process for every user — the single most common Node performance pitfall.Egyetlen lassú szinkron handler az egész processzt lerontja minden felhasználó számára — ez a leggyakoribb Node performance-buktató.

09

Callbacks → Promises → async/await

Node

The evolution of async in Node: error-first callbacks → Promises → async/await syntax over Promises.Az async fejlődése a Node-ban: error-first callbackek → Promise-ok → async/await szintaxis a Promise-ok fölött.

What problem did each solve?Melyik milyen problémát oldott meg?

  • Callbacks → “callback hell” (nesting, inversion of control). Promises → chaining and a single error path. async/await → synchronous-looking code with try/catch.Callbackek → „callback hell” (egymásba ágyazás, inversion of control). Promise-ok → láncolás és egyetlen hibaút. async/await → szinkronnak kinéző kód try/catch-csel.

How do async functions work?Hogyan működnek az async függvények?

  • An async function always returns a Promise; await pauses it until the Promise settles — without blocking the event loop, so other work runs meanwhile.Egy async függvény mindig Promise-t ad vissza; az await szünetelteti, amíg a Promise le nem zárul — az event loop blokkolása nélkül, így közben más munka is fut.

Watch outBuktatóawait in a loop serializes independent operations — use Promise.all for parallelism. And forgetting to await returns a pending Promise (a silent bug).A ciklusban lévő await sorosítja a független műveleteket — párhuzamossághoz Promise.all. Ha pedig elfelejted az await-et, egy függőben lévő Promise-t kapsz vissza (csendes bug).

Async Error HandlingAsync hibakezelés

Node

Errors must be caught on the right path — a sync throw, a rejected Promise, or an error-first callback.A hibákat a megfelelő úton kell elkapni — szinkron throw, elutasított Promise, vagy error-first callback.

How do you catch each?Melyiket hogyan kapod el?

  • async/await → try/catch. Promises → .catch(). Callbacks → check the first arg. Streams/emitters → an 'error' listener.async/await → try/catch. Promise-ok → .catch(). Callbackek → ellenőrizd az első argumentumot. Streamek/emitterek → 'error' listener.

The dangerous cases?A veszélyes esetek?

  • Unhandled rejections (process.on('unhandledRejection')) — which crash by default in current Node — and uncaught exceptions.A kezeletlen rejectek (process.on('unhandledRejection')) — amik a mai Node-ban alapból összeomlást okoznak — és az elkapatlan exceptionök.

Watch outBuktatóA throw inside a setTimeout/callback isn't caught by an outer try/catch — the try already returned. Let a top-level handler log & exit; don't try to “resume” after an uncaught exception — process state is unknown.Egy setTimeout-ban/callbackben dobott hibát nem kap el a külső try/catch — a try már visszatért. Hagyd, hogy egy top-level handler logoljon és kilépjen; ne próbálj „folytatni” egy elkapatlan exception után — a processz állapota ismeretlen.

API

Methods are the verbs that define what action a request performs on a resource.A metódusok azok az igék, amelyek megadják, hogy a kérés milyen műveletet végez a resource-on.

Safe vs idempotent — what's the difference?Safe vs idempotent — mi a különbség?

  • Safe = no server state change (GET, HEAD).Safe = nem változtat szerver-oldali állapotot (GET, HEAD).

API

Three-digit codes signalling the outcome — 1xx info, 2xx success, 3xx redirect, 4xx client error, 5xx server error.Háromjegyű kódok, amelyek a kimenetet jelzik — 1xx info, 2xx siker, 3xx átirányítás, 4xx kliens-hiba, 5xx szerver-hiba.

200 OK201 Created204 No Content 301/302304 Not Modified 400401403404409422429 500502503504

401 vs 403?

  • 401 Unauthorized = not authenticated — missing/invalid credentials (“who are you?”).401 Unauthorized = nincs authentikálva — hiányzó vagy érvénytelen credential („ki vagy te?”).
  • 403 Forbidden = authenticated, but not allowed to do this.403 Forbidden = authentikálva van, de nincs joga ehhez a művelethez.

400 vs 422?

  • 400 — malformed request (bad JSON, wrong types).400 — hibás formátumú kérés (rossz JSON, rossz típusok).
  • 422 — syntactically valid but semantically invalid (validation failed).422 — szintaktikailag helyes, de szemantikailag érvénytelen (validáció bukott).

201 vs 200 vs 204?

  • 201 after a create (often with a Location header).201 létrehozás után (gyakran Location headerrel).
  • 200 success with a body; 204 success with no body (e.g. a DELETE).200 siker body-val; 204 siker body nélkül (pl. egy DELETE).

Watch outBuktatóNever return 200 with an error inside the body — clients, proxies and monitoring all trust the status line.Soha ne adj vissza 200-at úgy, hogy a hiba a body-ban van — a kliensek, a proxyk és a monitoring mind a status sorra támaszkodnak.

Request–Response

API

A stateless exchange: the client sends a request (method, path, headers, optional body); the server returns a response (status, headers, optional body).Stateless csere: a kliens kérést küld (metódus, path, headerek, opcionális body), a szerver választ ad vissza (status, headerek, opcionális body).

A single request/response round trip.Egyetlen request/response körút.

Why is HTTP stateless, and how do we keep sessions?Miért stateless a HTTP, és hogyan tartunk fenn session-t?

  • Each request is independent — the server keeps no memory of the client between them.Minden kérés független — a szerver nem őriz emléket a kliensről a kérések között.
  • State is carried by the client (cookie / token) or looked up server-side by a session ID.Az állapotot a kliens hordozza (cookie / token), vagy a szerver session ID alapján keresi ki.

Content-Type vs Accept?

  • Content-Type describes the body you're sending.A Content-Type azt írja le, amit küldesz a body-ban.
  • Accept tells the server which formats you can handle (content negotiation).Az Accept megmondja a szervernek, milyen formátumot tudsz kezelni (content negotiation).

Error Handling

API

Returning consistent, actionable errors so clients can react programmatically and humans can debug.Konzisztens, kezelhető hibák visszaadása, hogy a kliensek programozottan reagálhassanak, az emberek pedig debugolni tudjanak.

What does a good error response look like?Hogy néz ki egy jó hibaválasz?

The correct HTTP status + a consistent JSON body: a stable machine-readable code, a human message, and field-level details for validation. application/problem+json (RFC 7807) is a common shape.A helyes HTTP status + konzisztens JSON body: stabil, géppel olvasható code, emberi message, és mezőszintű részletek a validációhoz. Gyakori formátum az application/problem+json (RFC 7807).

What should you avoid?Mit kerülj el?

  • Leaking stack traces / internals (security risk).Stack trace / belső részletek kiszivárogtatását (biztonsági kockázat).
  • Returning 200 for failures.200 visszaadását hibák esetén.
  • Inconsistent error shapes across endpoints.Endpointonként eltérő hibaformátumot.

Why a slow hash (bcrypt, scrypt, Argon2)?Miért lassú hash (bcrypt, scrypt, Argon2)?

  • Fast hashes are designed for throughput — a GPU tries billions per second, so a leaked table is cracked quickly. Argon2id is the current recommendation; bcrypt remains fine.A gyors hashek áteresztésre tervezettek — egy GPU milliárdokat próbál másodpercenként, így egy kiszivárgott tábla gyorsan törik. What does the salt do?Mit csinál a só?

  • A unique random salt per password means identical passwords hash differently, so an attacker can't use precomputed rainbow tables or crack many accounts at once. Modern libraries generate and store it for you inside the hash string.Egy egyedi véletlen só jelszavanként azt jelenti, az azonos jelszavak eltérően hashelődnek, így a támadó nem használhat előszámított rainbow table-t, és nem törhet sok fiókot egyszerre. Building a string — the two versionsString építése — a két verzióO(n²) → O(n)
// ✗ O(n²) — new string every iteration
let out = '';
for (const c of chars) out += c;

// ✓ O(n) — one join at the end
const parts = [];
for (const c of chars) parts.push(c);
const out2 = parts.join('');
