SQL & NoSQL InjectionSQL & NoSQL injection

Untrusted input concatenated into a SQL query, letting an attacker change the query's meaning — reading, modifying or destroying data they shouldn't reach.Nem megbízható input SQL query-be fűzve, amivel a támadó megváltoztathatja a query jelentését — olyan adatot olvasva, módosítva vagy törölve, amihez nem szabadna hozzáférnie.

VulnerableSebezhető
// input: "'; DROP TABLE users; --"
db.query(
  "SELECT * FROM users WHERE email = '"
  + email + "'"
);
FixedJavítva
// value is sent separately from the SQL
db.query(
  'SELECT * FROM users WHERE email = $1',
  [email]
);

Why do parameterized queries fix it?Miért javítják a parametrizált query-k?

  • The SQL text and the data travel separately — the driver never lets the value become part of the query's syntax. So input like ' OR 1=1 -- is treated as a literal string to compare, not as SQL.Az SQL-szöveg és az adat külön utazik — a driver sosem engedi, hogy az érték a query szintaxisának részévé váljon. Így egy ' OR 1=1 -- input összehasonlítandó literálként kezelődik, nem SQL-ként.

Is escaping input enough?Elég az input escape-elése?

  • No — manual escaping is easy to get wrong (encodings, edge cases, different contexts) and misses places you forgot. Parameterized queries (or a query builder / ORM that uses them) are the actual fix.Nem — a kézi escape-elést könnyű elrontani (kódolások, peremesetek, eltérő kontextusok), és kimarad, amit elfelejtettél. A parametrizált query-k (vagy egy azokat használó query builder / ORM) a valódi javítás.

What else limits the damage?Mi korlátozza még a kárt?

  • Least-privilege DB users (the app account shouldn't be able to DROP), input validation as a second layer, and not exposing raw DB errors to the client.Legkisebb jogosultságú DB-userek (az app account ne tudjon DROP-olni), input-validáció második rétegként, és a nyers DB-hibák elrejtése a kliens elől.

Watch outBuktatóORMs are not automatically safe — raw/literal query escapes (knex.raw, sequelize.query, template literals in Prisma's $queryRawUnsafe) reintroduce the hole. The same flaw exists in NoSQL: passing a user-supplied object into a Mongo filter allows operator injection like {$ne: null}.Az ORM-ek nem automatikusan biztonságosak — a nyers/literál query-kijáratok (knex.raw, sequelize.query, template literálok a Prisma $queryRawUnsafe-jában) visszahozzák a lyukat. Ugyanez a hiba létezik NoSQL-ben is: egy user által küldött objektum Mongo-filterbe adása operátor-injectiont enged, mint a {$ne: null}.

02

Cross-Site Scripting (XSS)Cross-Site Scripting (XSS)

Injecting attacker-controlled script into a page so it runs in another user's browser with that user's session — stealing tokens, performing actions, or defacing the page.Támadó által vezérelt script beinjektálása egy oldalba, hogy egy másik user böngészőjében fusson annak session-jével — tokent lopva, műveleteket végezve vagy az oldalt megrongálva.

The three types?A három típus?

  • Stored: the payload is saved server-side (a comment) and served to every viewer — the most damaging. Reflected: the payload comes from the request (a query param) and is echoed back. DOM-based: client-side JS writes untrusted data into the DOM; the server never sees it.Stored: a payload szerver-oldalon tárolódik (egy komment) és minden megtekintőnek kiszolgálódik — a legkárosabb. Reflected: a payload a kérésből jön (egy query paraméter) és visszatükröződik. DOM-based: kliens-oldali JS ír nem megbízható adatot a DOM-ba; a szerver sosem látja.
VulnerableSebezhető
// React: bypasses escaping
<div dangerouslySetInnerHTML={{
  __html: comment
}} />

// DOM: parses and runs <script>
el.innerHTML = userInput;
FixedJavítva
// React escapes by default
<div>{comment}</div>

// or sanitize if HTML is required
<div dangerouslySetInnerHTML={{
  __html: DOMPurify.sanitize(comment)
}} />

el.textContent = userInput;

How does React protect you, and where does it stop?Hogyan véd a React, és hol áll meg?

  • JSX escapes interpolated values by default, so {userInput} is safe. Protection ends at dangerouslySetInnerHTML, at href/src attributes (a javascript: URL still executes), and anywhere you touch the DOM directly.A JSX alapból escape-eli az interpolált értékeket, szóval a {userInput} biztonságos. A védelem véget ér a dangerouslySetInnerHTML-nél, a href/src attribútumoknál (egy javascript: URL még mindig lefut), és bárhol, ahol közvetlenül nyúlsz a DOM-hoz.

What's the defense in depth?Mi a mélységi védelem?

  • Context-aware output encoding (HTML body vs attribute vs URL vs JS differ), sanitize HTML you must render (DOMPurify), a strict Content-Security-Policy, and HttpOnly cookies so a successful XSS still can't read the session token.Kontextus-tudatos output-kódolás (a HTML-törzs vs attribútum vs URL vs JS eltér), a megjelenítendő HTML sanitizálása (DOMPurify), szigorú Content-Security-Policy, és HttpOnly cookie-k, hogy egy sikeres XSS se tudja kiolvasni a session tokent.

Watch outBuktatóStoring a JWT in localStorage makes any XSS a full account takeover — JS can read it. This is the strongest argument for HttpOnly cookies. Also: sanitizing on input instead of encoding on output breaks when the same data is rendered into a different context later.Egy JWT localStorage-ban tárolása bármely XSS-t teljes fiók-átvétellé tesz — a JS ki tudja olvasni. Ez a legerősebb érv a HttpOnly cookie-k mellett. Továbbá: az input-oldali sanitizálás output-oldali kódolás helyett elromlik, amikor ugyanaz az adat később más kontextusba renderelődik.

03

Cross-Site Request Forgery (CSRF)Cross-Site Request Forgery (CSRF)

Tricking a logged-in user's browser into sending an unwanted authenticated request to your site — the browser attaches cookies automatically, so the request looks legitimate.Egy bejelentkezett user böngészőjének rávétele, hogy nem kívánt autentikált kérést küldjön az oldaladra — a böngésző automatikusan csatolja a cookie-kat, így a kérés legitimnek tűnik.

The victim's browser sends your cookie to your server, even though the request originated on the attacker's page.Az áldozat böngészője elküldi a cookie-dat a szerverednek, pedig a kérés a támadó oldaláról indult.

Why does it work at all?Miért működik egyáltalán?

  • Because cookies are attached by the browser based on the destination, not on who initiated the request. A form auto-submitted from evil.com to yourbank.com still carries the victim's session cookie.Mert a cookie-kat a böngésző a célállomás alapján csatolja, nem az alapján, ki kezdeményezte a kérést. Egy evil.com-ról yourbank.com-ra auto-submitolt form is viszi az áldozat session cookie-ját.

The defenses?A védelmek?

  • SameSite cookies (Lax is the modern browser default and blocks cross-site POSTs; Strict is tighter). Anti-CSRF tokens: a per-session random token in the form/header that the attacker can't read or guess. Verify Origin/Referer on state-changing requests.SameSite cookie-k (a Lax a modern böngésző-alapérték és blokkolja a cross-site POST-okat; a Strict szigorúbb). Anti-CSRF tokenek: session-önkénti véletlen token a formban/headerben, amit a támadó nem tud olvasni vagy kitalálni. Origin/Referer ellenőrzés az állapotváltoztató kéréseknél.

Do token-based (Authorization header) APIs need CSRF protection?Kell CSRF-védelem a token-alapú (Authorization header) API-knak?

  • Generally no — the browser doesn't attach an Authorization header automatically, so the attacker's page can't forge it. CSRF is fundamentally a cookie (ambient credential) problem. If you put the token in a cookie, you're back to needing CSRF defenses.Általában nem — a böngésző nem csatolja automatikusan az Authorization headert, szóval a támadó oldala nem tudja hamisítani. A CSRF alapvetően cookie (ambient credential) probléma. Ha cookie-ba teszed a tokent, újra kellenek a CSRF-védelmek.

Watch outBuktatóNever make GET requests state-changing — GET /delete?id=1 is trivially triggered by an <img> tag and bypasses many protections. And don't rely on SameSite alone: older browsers and certain cross-origin flows still need tokens.Sose tegyél GET kérést állapotváltoztatóvá — a GET /delete?id=1-et triviálisan kiváltja egy <img> tag, és megkerül sok védelmet. És ne támaszkodj csak a SameSite-ra: a régebbi böngészők és bizonyos cross-origin folyamatok még igényelnek tokent.

04

CORS & Same-Origin PolicyCORS & Same-Origin Policy

A browser mechanism that relaxes the Same-Origin Policy, letting a server declare which other origins may read its responses. It's a controlled exception to a security default — not a security feature you add.Egy böngésző-mechanizmus, amely lazítja a Same-Origin Policy-t, lehetővé téve a szervernek, hogy megmondja, mely más originek olvashatják a válaszait. Ez egy kontrollált kivétel egy biztonsági alapérték alól — nem egy biztonsági feature, amit hozzáadsz.

What is the Same-Origin Policy protecting?Mit véd a Same-Origin Policy?

  • It stops a page on evil.com from reading responses from yourbank.com using the victim's cookies. Note it blocks reading the response — the request may still be sent (which is why CSRF exists).Megakadályozza, hogy egy evil.com-on lévő oldal olvassa a yourbank.com válaszait az áldozat cookie-jaival. Vedd észre, hogy a válasz olvasását blokkolja — a kérés még elmehet (ezért létezik a CSRF).

What's a preflight request?Mi a preflight kérés?

  • For non-simple requests (custom headers, methods beyond GET/POST/HEAD, JSON content-type), the browser first sends an OPTIONS request asking whether the real request is allowed, then proceeds only if the server approves.Nem egyszerű kéréseknél (egyedi headerek, GET/POST/HEAD-en túli metódusok, JSON content-type) a böngésző előbb egy OPTIONS kérést küld, megkérdezve, engedélyezett-e a valódi kérés, és csak akkor folytatja, ha a szerver jóváhagyja.
VulnerableSebezhető
// reflects any origin + allows cookies
app.use(cors({
  origin: true,
  credentials: true
}));
FixedJavítva
const allowed = new Set([
  'https://app.example.com'
]);
app.use(cors({
  origin: (o, cb) =>
    cb(null, !o || allowed.has(o)),
  credentials: true
}));

Watch outBuktatóAccess-Control-Allow-Origin: * combined with credentials is rejected by browsers — so people “fix” it by reflecting the request's Origin header back, which allows every site and is strictly worse. Always use an allowlist. Also remember CORS is enforced by browsers only; it protects users, not your API from scripts or servers.Az Access-Control-Allow-Origin: * credentials-szel kombinálva elutasított a böngészők által — így az emberek úgy „javítják”, hogy visszatükrözik a kérés Origin headerét, ami minden oldalt enged, és szigorúan rosszabb. Mindig allowlistet használj. És ne feledd: a CORS-t csak böngészők kényszerítik ki; a usereket védi, nem az API-dat scriptektől vagy szerverektől.

Security Headers & CSPSecurity headerek & CSP

HTTP response headers that instruct the browser to enforce security restrictions — most importantly Content-Security-Policy, which limits where scripts and other resources may load from.HTTP válasz-headerek, amelyek utasítják a böngészőt biztonsági korlátozások betartatására — legfontosabb a Content-Security-Policy, amely korlátozza, honnan tölthetők scriptek és más erőforrások.

HeaderHeaderWhat it doesMit csinál
Content-Security-PolicyContent-Security-Policyrestricts script/style/img sources; the main XSS mitigationkorlátozza a script/style/img forrásokat; a fő XSS-mitigáció
Strict-Transport-SecurityStrict-Transport-Securityforces HTTPS for future visits (HSTS)kikényszeríti a HTTPS-t a jövőbeli látogatásokra (HSTS)
X-Content-Type-OptionsX-Content-Type-Optionsnosniff — stops MIME-type guessingnosniff — megállítja a MIME-típus találgatást
X-Frame-Options / frame-ancestorsX-Frame-Options / frame-ancestorsblocks framing → prevents clickjackingblokkolja a frame-elést → megelőzi a clickjackinget
Referrer-PolicyReferrer-Policylimits how much URL info leaks to other siteskorlátozza, mennyi URL-infó szivárog más oldalakra

How does CSP actually stop XSS?Hogyan állítja meg a CSP ténylegesen az XSS-t?

  • Even if an attacker injects a <script>, the browser refuses to execute it unless it matches the policy — e.g. it carries the right nonce or hash. It turns a code-execution bug into a blocked resource.Még ha a támadó be is injektál egy <script>-et, a böngésző megtagadja a futtatását, hacsak nem felel meg a policy-nek — pl. hordozza a helyes nonce-ot vagy hasht. Egy kódfuttatási bugot blokkolt erőforrássá alakít.

Why is unsafe-inline a problem?Miért probléma az unsafe-inline?

  • It permits any inline script — exactly what an XSS injects — which defeats the policy's main purpose. Use nonces or hashes for the inline scripts you genuinely need.Bármilyen inline scriptet enged — pontosan azt, amit egy XSS injektál —, ami meghiúsítja a policy fő célját. Használj nonce-okat vagy hasheket a valóban szükséges inline scriptekhez.

Watch outBuktatóCSP is a second line of defense, not a substitute for output encoding. Roll it out with Content-Security-Policy-Report-Only first and watch the reports — a strict policy deployed blind will break your app.A CSP második védelmi vonal, nem az output-kódolás helyettesítője. Előbb Content-Security-Policy-Report-Only-val vezesd be és figyeld a jelentéseket — egy vakon deployolt szigorú policy eltöri az appod.

Broken Access Control & IDORHibás hozzáférés-vezérlés & IDOR

Failures where a user can act outside their intended permissions — accessing other users' records, escalating privileges, or reaching admin functions. Consistently the highest-ranked risk in the OWASP Top 10.Hibák, ahol egy user a szándékolt jogosultságain kívül tud cselekedni — más userek rekordjaihoz férve, jogosultságot emelve, vagy admin funkciókat elérve. Következetesen a legmagasabbra rangsorolt kockázat az OWASP Top 10-ben.

VulnerableSebezhető
// IDOR: no ownership check
app.get('/api/orders/:id',
  requireAuth,
  async (req, res) => {
    const o = await Order.findById(
      req.params.id);
    res.json(o);
  });
FixedJavítva
app.get('/api/orders/:id',
  requireAuth,
  async (req, res) => {
    const o = await Order.findOne({
      _id: req.params.id,
      userId: req.user.id   // scope it
    });
    if (!o) return res.sendStatus(404);
    res.json(o);
  });

What is IDOR?Mi az IDOR?

  • Insecure Direct Object Reference: the app authenticates you but doesn't check that the object you requested is yours. Changing /orders/123 to /orders/124 returns someone else's data.Insecure Direct Object Reference: az app autentikál téged, de nem ellenőrzi, hogy a kért objektum a tiéd-e. A /orders/123 átírása /orders/124-re valaki más adatát adja vissza.

What are the design principles?Mik a tervezési elvek?

  • Deny by default — everything requires explicit permission. Enforce server-side, always: hiding a button in the UI is not access control. Scope every query by the actor, rather than fetching then checking.Alapból tiltás — mindenhez explicit engedély kell. Mindig szerver-oldalon kényszerítsd ki: egy gomb elrejtése a UI-ban nem hozzáférés-vezérlés. Szűkítsd minden query-t a cselekvőre ahelyett, hogy lekérnél, majd ellenőriznél.

Authentication vs authorization?Autentikáció vs autorizáció?

  • Authentication = who you are (proving identity). Authorization = what you're allowed to do. Most access-control bugs are authorization failures in an otherwise correctly authenticated request.Autentikáció = ki vagy (az identitás bizonyítása). Autorizáció = mit szabad tenned. A legtöbb hozzáférés-vezérlési bug autorizációs hiba egy egyébként helyesen autentikált kérésben.

Watch outBuktatóAccess control can't be enforced in the frontend — anyone can call your API directly. Also beware mass assignment: blindly spreading req.body into a model update lets a user set role: 'admin'. Allowlist the fields you accept.A hozzáférés-vezérlés nem kényszeríthető ki a frontendben — bárki hívhatja közvetlenül az API-dat. És óvakodj a mass assignmenttől: a req.body vak szétterítése egy model-frissítésbe engedi a usernek a role: 'admin' beállítását. Allowlisteld a mezőket, amiket elfogadsz.

07

Server-Side Request Forgery (SSRF)Server-Side Request Forgery (SSRF)

Tricking your server into making a request to an address the attacker chooses — reaching internal services, cloud metadata endpoints, or the local network that the attacker can't reach directly.A szervered rávétele, hogy a támadó által választott címre küldjön kérést — belső szolgáltatásokat, cloud metadata endpointokat vagy a helyi hálózatot érve el, amit a támadó közvetlenül nem tud.

Why is it dangerous?Miért veszélyes?

  • Your server sits inside the trust boundary. A URL like http://169.254.169.254/ (cloud instance metadata) can leak credentials; http://localhost:6379 can reach an unauthenticated internal Redis.A szervered a bizalmi határon belül ül. Egy http://169.254.169.254/-hez hasonló URL (cloud instance metadata) hitelesítő adatokat szivárogtathat; a http://localhost:6379 elérhet egy autentikálatlan belső Redist.

How do you defend?Hogyan védekezel?

  • Allowlist permitted destinations (scheme, host) rather than blocklisting; resolve the DNS name and validate the resulting IP isn't private/link-local; disable redirects or re-validate after each hop; and isolate egress at the network level.Allowlisteld az engedélyezett célokat (séma, host) blocklistázás helyett; oldd fel a DNS-nevet és validáld, hogy a kapott IP nem privát/link-local; tiltsd le a redirecteket vagy validálj újra minden ugrás után; és izoláld a kimenő forgalmat hálózati szinten.

Watch outBuktatóValidating the URL string then fetching it is a TOCTOU bug — DNS can resolve differently the second time (DNS rebinding), and a redirect can send you somewhere new. Validate the resolved IP at connection time, and block redirects by default.Az URL-string validálása majd lehívása TOCTOU bug — a DNS másodszorra máshogy oldódhat fel (DNS rebinding), és egy redirect új helyre küldhet. A feloldott IP-t a kapcsolódáskor validáld, és alapból blokkold a redirecteket.

Dependency & Supply-Chain SecurityFüggőségi & supply-chain biztonság

Risk inherited from third-party packages — known CVEs in dependencies, malicious or typosquatted packages, and compromised build tooling. Most of a modern app's code isn't code you wrote.Harmadik féltől örökölt kockázat — ismert CVE-k a függőségekben, rosszindulatú vagy typosquattolt csomagok, és kompromittált build-eszközök. Egy modern app kódjának nagy része nem az általad írt kód.

How do you manage it in a Node project?Hogyan kezeled egy Node projektben?

  • Commit the lockfile so builds are reproducible; run npm audit / Dependabot / Snyk in CI; use npm ci in pipelines; and prefer fewer, well-maintained dependencies over many trivial ones.Commitold a lockfile-t, hogy a buildek reprodukálhatók legyenek; futtass npm audit-ot / Dependabotot / Snyket a CI-ban; használj npm ci-t a pipeline-okban; és inkább kevesebb, jól karbantartott függőség, mint sok triviális.

What are install-script and typosquatting attacks?Mik az install-script és typosquatting támadások?

  • A package can run arbitrary code in a postinstall hook the moment it's installed — on a developer machine or CI, with those credentials. Typosquatting publishes a package with a name close to a popular one hoping for a typo. Use --ignore-scripts where feasible and verify package names.Egy csomag tetszőleges kódot futtathat egy postinstall hookban abban a pillanatban, ahogy települ — fejlesztői gépen vagy CI-ban, azokkal a jogosultságokkal. A typosquatting egy népszerűhöz közeli nevű csomagot publikál, elgépelésre számítva. Használj --ignore-scripts-et, ahol kivitelezhető, és ellenőrizd a csomagneveket.

Watch outBuktatóA vulnerability in a transitive dependency is still your vulnerability — audit the whole tree, not just direct dependencies. And an unmaintained package with no recent releases is a risk even without a current CVE.Egy tranzitív függőségben lévő sebezhetőség is a te sebezhetőséged — auditáld a teljes fát, ne csak a közvetlen függőségeket. És egy karbantartás nélküli, régóta kiadás nélküli csomag akkor is kockázat, ha most nincs CVE-je.

Secrets ManagementTitkok kezelése

Keeping credentials out of code and limiting what a leak exposes — API keys, DB passwords, signing keys, and the configuration that governs them.A hitelesítő adatok kódon kívül tartása és a szivárgás hatásának korlátozása — API kulcsok, DB jelszavak, aláíró kulcsok, és a rájuk vonatkozó konfiguráció.

Where should secrets live?Hol éljenek a titkok?

  • In a secrets manager (Vault, AWS Secrets Manager, cloud KMS) or injected environment variables — never in the repo, never in the frontend bundle. Rotate them, and scope each key to the minimum permissions it needs.Egy secrets managerben (Vault, AWS Secrets Manager, cloud KMS) vagy injektált környezeti változókban — sosem a repóban, sosem a frontend bundle-ben. Forgasd őket, és szűkítsd minden kulcsot a szükséges minimum jogosultságra.

What happens when a secret leaks into git history?Mi történik, ha egy titok bekerül a git historybe?

  • Deleting it in a later commit doesn't help — it's still in history and likely already scraped. You must rotate the credential; purging history is secondary cleanup.A későbbi commitban törlés nem segít — még mindig a historyben van, és valószínűleg már lescrape-elték. Forgatnod kell a hitelesítő adatot; a history tisztítása másodlagos takarítás.

Watch outBuktatóAnything shipped to the browser is public — a “secret” in a React env var (REACT_APP_*, NEXT_PUBLIC_*) is baked into the bundle and readable by anyone. Frontend code cannot hold secrets; proxy through your backend instead.Bármi, ami a böngészőbe kerül, nyilvános — egy „titok” egy React env változóban (REACT_APP_*, NEXT_PUBLIC_*) beleég a bundle-be és bárki olvashatja. A frontend kód nem tarthat titkot; helyette proxyzz a backendeden át.

10

Error Handling, Logging & DoSHibakezelés, logolás & DoS

Failing safely and observing attacks — not leaking internals through errors, logging enough to detect abuse, and surviving traffic designed to exhaust you.Biztonságos hibázás és a támadások megfigyelése — nem szivárogtatva belső részleteket hibákon át, eleget logolva a visszaélés észleléséhez, és túlélve a kimerítésre tervezett forgalmat.

What should errors expose?Mit fedjen fel egy hiba?

  • A generic message and a correlation ID to the client; full detail only in server logs. Stack traces, SQL fragments and library versions handed to an attacker are free reconnaissance.Egy általános üzenetet és egy correlation ID-t a kliensnek; a teljes részletet csak a szerver-logokban. A stack trace-ek, SQL-töredékek és library-verziók a támadónak ingyen felderítés.

What must never be logged?Mit nem szabad sosem logolni?

  • Passwords, tokens, session IDs, card numbers, and personal data beyond what you need. Logs are widely readable, often shipped to third parties, and retained far longer than you expect.Jelszavakat, tokeneket, session ID-kat, kártyaszámokat, és a szükségesnél több személyes adatot. A logokat sokan olvashatják, gyakran harmadik félhez kerülnek, és sokkal tovább megőrződnek, mint gondolnád.

How do you resist abuse and DoS?Hogyan állsz ellen a visszaélésnek és DoS-nak?

  • Rate limit per user/IP (especially auth endpoints), cap request body size and upload size, set timeouts on every outbound call, and paginate expensive queries. Put a WAF/CDN in front for volumetric attacks.Rate limitelj user/IP szerint (különösen az auth endpointokon), korlátozd a kérés-törzs és feltöltés méretét, állíts timeoutot minden kimenő hívásra, és lapozd a drága query-ket. Tegyél WAF-ot/CDN-t elé a volumetrikus támadásokhoz.

Watch outBuktatóDetailed error messages on login (“user not found” vs “wrong password”) enable user enumeration — return one generic failure. And logging a full request body is a classic way to accidentally store passwords and tokens in plaintext.A részletes hibaüzenetek a loginon („nincs ilyen user” vs „rossz jelszó”) user-enumerációt tesznek lehetővé — adj egyetlen általános hibát. És a teljes request body logolása klasszikus módja annak, hogy véletlenül jelszavakat és tokeneket tárolj plaintextben.
