React Core
01

Virtual DOM & Reconciliation

React renders a lightweight in-memory tree, diffs it against the previous one (reconciliation), and applies only the minimal real-DOM mutations.

Same type + key → reuse and patch. Different type → tear down and rebuild.

What's the diffing heuristic?

  • An O(n) heuristic, not a full tree diff. Two assumptions: different element types produce different trees; stable keys identify which children moved.

What actually causes a re-render?

  • A state change, a parent re-render, a consumed context value changing, or a subscribing hook firing.
  • Re-render ≠ DOM update — reconciliation may find nothing changed and touch no DOM.

Render vs commit phase?

  • Render: React calls your components and builds the tree — pure, and pausable in concurrent mode.
  • Commit: applies DOM changes and runs effects — synchronous.

Watch outA re-render is relatively cheap; the cost is your render logic. Don't optimize re-renders you haven't profiled.

02

Keys in Lists

key is the stable identity React uses to match elements across renders.

With index keys, inserting at the top shifts every key and misassigns DOM/state.

Why not use the array index as a key?

  • On insert/reorder/delete, indexes shift, so React reuses the wrong DOM node and component state — e.g. an input's value sticks to the wrong row.

What makes a good key?

  • Stable, unique among siblings, and tied to the data's identity (an ID) — not its position. It only needs to be unique within the list, not globally.

Watch outIndex keys are safe only for static, never-reordered lists. The classic bug: uncontrolled inputs in a reorderable list keyed by index.

03

Rules of Hooks

Call hooks only at the top level — never in loops, conditions or nested functions — and only from React functions (components or custom hooks).

Why the top-level rule?

  • React tracks hook state by call order, not by name. A conditional hook shifts the order between renders, so state gets misassigned to the wrong hook.

How do you do conditional logic then?

  • Put the condition inside the hook (e.g. inside useEffect), or split into separate components — never wrap the hook call itself.

Watch outeslint-plugin-react-hooks catches most violations — don't disable it to “make it work.”

04

State & Batching

useState returns a value + setter; React batches multiple setState calls in one event into a single re-render.

Why does calling setCount(count + 1) three times only add 1?

  • All three read the same stale count from the render's closure. Use the functional updater setCount(c => c + 1) so each builds on the latest value.

Is state synchronous?

  • No — setters schedule a re-render; the variable only changes on the next render. React 18 batches across all contexts, including promises and timeouts (automatic batching).

Why immutability?

  • React detects change by reference. Mutating state in place doesn't change the reference, so the update is missed. Always return new objects/arrays.

Watch outReading state right after setting it gives the old value — updates aren't applied synchronously.

05

useEffect & Dependencies

useEffect runs a side effect after render, re-running whenever a value in its dependency array changes.

What belongs in the dependency array?

  • Every reactive value used inside the effect (props, state, and anything derived from them). Missing deps → stale values; the lint rule enforces the full set.

When does the cleanup function run?

  • Before the next effect run and on unmount. Use it for subscriptions, timers and listeners to avoid leaks and race conditions.

[] vs no array vs [deps]?

  • No array → after every render. [] → once after mount. [deps] → when deps change.

Watch outDon't use an effect to derive state from props/state — compute it during render. Effects are for synchronizing with external systems, not reacting to your own state. Object/function deps recreated each render cause infinite loops → memoize or move them.

06

Stale Closures

A closure that captured a value from an earlier render and keeps using that old snapshot.

Where does it bite?

  • Event handlers, setTimeout/setInterval, and effects with missing deps — they all close over the render's snapshot of state and props.

How do you fix it?

  • Add the value to the deps (re-create the closure), use the functional updater, or stash the latest value in a ref and read ref.current.

Watch outA setInterval created in useEffect([]) that reads state will forever see the initial value — the textbook stale closure.

07

Memoization

Caching to skip work: React.memo skips a re-render on shallow-equal props; useMemo caches a computed value; useCallback caches a function reference.

A new object/function passed as a prop defeats memo's shallow comparison.

When is each worth it?

  • useMemo — a genuinely expensive computation, or a stable value fed to memoized children / effect deps.
  • useCallback — a function passed to a memo child or used as an effect dep.
  • React.memo — a component that re-renders often with the same props and has a non-trivial render.

Why do useMemo/useCallback often do nothing?

  • If the memoized value still feeds a non-memoized child, or the deps change every render, you pay the overhead for no benefit.

Watch outMemoization costs memory + comparison. Default to not memoizing; add it when profiling shows a real problem. (The React Compiler auto-memoizes, reducing the manual need.)

08

useRef

A mutable container ({ current }) that persists across renders without triggering one when it changes.

Two main uses?

  • Accessing DOM nodes (ref={myRef}), and holding mutable values that shouldn't cause renders (timer IDs, the previous value, a “latest state” ref).

useRef vs useState?

  • Changing state re-renders; changing a ref doesn't. Use a ref when the value isn't part of the rendered output.

Watch outDon't read/write ref.current during render for rendered data — it's outside React's data flow and won't reflect in the UI.

09

Context API

Passes data through the tree without prop drilling; consumers re-render when the provider's value changes.

The performance gotcha?

  • Every consumer re-renders on any value change, even if it uses only part of it. A fresh object literal as value re-renders all consumers on every parent render.

How do you mitigate it?

  • Memoize the value, split contexts by update frequency, or move to a store (Zustand/Redux) with selector-based subscriptions for high-frequency state.

Watch outContext is a dependency-injection / low-frequency-state tool (theme, auth, locale) — not a general state manager for rapidly-changing data.

10

Custom Hooks

A function starting with use that composes built-in hooks to extract and reuse stateful logic.

What do they share — and not share?

  • They share logic, not state. Each call gets its own independent state — two components using useCounter() don't share a counter.

When should you extract one?

  • Repeated stateful logic — fetching, form state, subscriptions, debouncing — to keep components declarative and testable.

Watch outA custom hook is just a function — the Rules of Hooks still apply (top-level calls, React-functions only).

11

Controlled vs Uncontrolled

Controlled: React state is the source of truth (value + onChange). Uncontrolled: the DOM holds the value, read via a ref (defaultValue).

Trade-offs?

  • Controlled: full control and live validation, but a re-render per keystroke.
  • Uncontrolled: less code, good for simple/large forms and file inputs, but harder to validate or transform on the fly.

Watch outFlipping a field between controlled and uncontrolled (value going undefined → defined) triggers React's warning — always initialize with a defined value like ''.

State & Data
12

Client vs Server State

Client state is UI state you own (modals, inputs, toggles). Server state is remote data — fetched async, shared, and able to go stale.

Why treat them differently?

  • Server state needs caching, background refetch, dedup, retry, invalidation and staleness handling — none of which a plain useState or Redux slice gives you. That's what React Query / SWR / RTK Query solve.

What does a server-state library give you?

  • Request dedup, cache + stale-while-revalidate, background updates, pagination/infinite scroll, optimistic updates, and loading/error state out of the box.

Watch outDumping fetched data into Redux/Context and hand-rolling caching is the source of most “why is my data stale” bugs. Keep client and server state separate.

13

Data Fetching Patterns

How and when you load remote data — on render, on route load, or on the server.

Parallelize independent requests; only chain when B truly depends on A.

Common problems and fixes?

  • Race conditions: a slow earlier request resolves after a newer one — cancel with AbortController or ignore stale results in cleanup.
  • Waterfalls: sequential dependent fetches — parallelize independent ones or fetch higher up.

Where should data be fetched?

  • Close to where it's used, but avoid per-component waterfalls — route-level or server-side fetching (RSC/loaders) cuts round trips.

Watch outuseEffect fetching has no built-in dedup or caching — two components fetching the same thing means two requests. Use a server-state library or hoist the fetch.

14

Suspense & Error Boundaries

Suspense lets a component wait for something (lazy code, data) and show a fallback. Error Boundaries catch render-time errors in their subtree and show a fallback UI.

What can an Error Boundary catch — and not?

  • Catches: errors during render, lifecycle and child constructors.
  • Doesn't catch: event handlers, async code, SSR, or errors in the boundary itself — use try/catch for those.

Why is Suspense useful?

  • Declarative loading states, coordinating multiple async dependencies, and enabling streaming SSR and Server Components.

Watch outError Boundaries are still class components (or a library like react-error-boundary) — there's no hook that catches render errors.

Performance
15

Code Splitting & Lazy Loading

Splitting the bundle so the browser downloads only what it needs, loading the rest on demand.

How in React?

  • React.lazy(() => import('./X')) + <Suspense fallback>. Split by route first (the biggest win), then heavy components.

Why does it matter?

  • A smaller initial bundle means faster first load (TTI/LCP). The bundler emits a separate chunk per dynamic import.

Watch outOver-splitting creates many tiny chunks and request overhead. Split at meaningful boundaries (routes, rarely-used heavy features), not everything.

16

List Virtualization

Rendering only the rows currently in the viewport (plus a small buffer) instead of the whole list.

The DOM node count stays roughly constant no matter how long the list is.

Why and when?

  • Large lists (thousands of rows) create thousands of DOM nodes → slow render, high memory, janky scroll. Virtualization keeps the node count constant. Libraries: react-window, TanStack Virtual.

The catch?

  • Off-screen content isn't in the DOM → breaks Ctrl+F and screen readers if unhandled, and needs known or estimated row heights (variable heights are trickier).

Watch outDon't virtualize small lists — the machinery costs more than it saves. It's strictly for the long-list case.

17

Re-render Optimization

Reducing unnecessary re-renders and the cost of each one.

The main strategies, in order?

  • Don't re-render: push state down, split components, pass children as props so a re-rendering parent doesn't re-render static subtrees.
  • Then memoize: React.memo + stable props via useMemo/useCallback.
  • For global state: selector-based subscriptions so only components using a changed slice update.

How do you find the problem?

  • The React DevTools Profiler (highlight updates / flamegraph). Measure before optimizing.

Watch outWrapping everything in memo is an anti-pattern — it adds comparison cost and hides the real fix, which is usually state placement or component structure.

Browser & Language
20

Browser Event Loop

The single-threaded loop that runs JS: empty the call stack, drain all microtasks, render, then take one macrotask — and repeat.

Microtasks drain fully before the next macrotask — so a resolved Promise runs before setTimeout(0).

Microtasks vs macrotasks?

  • Microtasks (Promise callbacks, queueMicrotask, MutationObserver) drain fully after each task, before rendering.
  • Macrotasks (setTimeout, events, I/O) run one per loop iteration — so a Promise resolves before setTimeout(0).

Why does this matter for UI?

  • Long synchronous tasks or microtask floods block rendering and input → jank and poor INP. Break work up with chunking, setTimeout, or requestIdleCallback.

Watch outThis is the browser loop (task → microtasks → render). Node's event loop has different phases (timers, poll, check…) — don't conflate them in a full-stack interview.

21

Reflow & Repaint

Reflow (layout) recomputes element geometry and can cascade; repaint redraws pixels without a layout change. Reflow is the more expensive one.

Animating transform/opacity jumps to the composite step and avoids layout.

What triggers reflow, and what's “layout thrashing”?

  • Triggers: DOM changes, size/position style changes, and reading layout props (offsetHeight, getBoundingClientRect).
  • Thrashing: interleaving reads and writes in a loop forces a synchronous reflow each time. Fix: batch reads, then writes.

Watch outAnimate with transform/opacity, not top/left/width — the latter trigger reflow every frame.

22

CORS

A browser security mechanism: cross-origin requests are only readable if the server opts in via response headers. Origin = scheme + host + port.

How does it actually work?

  • The browser adds an Origin header; the server responds with Access-Control-Allow-Origin. If it doesn't match, the browser blocks the JS from reading the response.

What's a preflight?

  • For “non-simple” requests (custom headers, methods like PUT/DELETE, certain content types), the browser first sends an OPTIONS request; the server must approve with Allow-Methods/Allow-Headers.

Watch outCORS is enforced by the browser, not the server — the request often still reaches the server. It protects the user, not the server, and it isn't authentication. Curl and Postman ignore it entirely.

23

TypeScript + React

Static typing over JS — in React it types props, state, hooks and events, catching errors at compile time.

Key features to know?

  • Generics (reusable typed components/hooks), utility types (Partial, Pick, Omit, Record, ReturnType), unions + narrowing, and discriminated unions for variant props.
  • type vs interface: interfaces merge/extend (good for public APIs); types do unions, mapped and conditional types. Mostly interchangeable for object shapes.

Common typing patterns?

  • Props via interface Props {}; children as React.ReactNode; events as React.ChangeEvent<HTMLInputElement>; generic components as function List<T>(props: { items: T[] }).

Watch outAvoid any (it disables checking) and overusing as casts (they bypass safety) — prefer narrowing and generics.
