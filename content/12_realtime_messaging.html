WebSocket Handshake & LifecycleWebSocket handshake & életciklus

A WebSocket starts as an HTTP request that asks to switch protocols. The server answers 101 Switching Protocols, and from that point the same TCP connection carries frames in both directions instead of request/response.A WebSocket HTTP-kérésként indul, amely protokollváltást kér. A szerver 101 Switching Protocols-szal válaszol, és onnantól ugyanaz a TCP-kapcsolat kereteket szállít mindkét irányban, request/response helyett.

The handshake is ordinary HTTP, which is why WebSockets pass through most proxies — and why cookies and auth headers are available exactly once, at upgrade time.A handshake közönséges HTTP, ezért mennek át a WebSocketek a legtöbb proxyn — és ezért érhetők el a sütik és auth headerek pontosan egyszer, az upgrade pillanatában.

How do you authenticate a WebSocket?Hogyan autentikálsz egy WebSocketet?

  • At the handshake, because the browser WebSocket API can't set custom headers — so it's a cookie, or a short-lived ticket in the query string that the server exchanges for a session. After the upgrade there are no more HTTP requests to attach a token to, which means expiry needs handling in-band: the server tracks the session's lifetime and closes the socket when it lapses, rather than relying on a per-request check that no longer happens.A handshake-nél, mert a böngésző WebSocket API-ja nem tud egyedi headereket beállítani — tehát süti, vagy egy rövid életű ticket a query stringben, amit a szerver session-re vált. Az upgrade után nincs több HTTP-kérés, amihez tokent lehetne csatolni, így a lejáratot a csatornán belül kell kezelni: a szerver követi a session élettartamát, és lezárja a socketet, amikor lejár — nem támaszkodhat kérésenkénti ellenőrzésre, ami már nem történik meg.

Why do you need ping/pong if TCP already detects failure?Miért kell ping/pong, ha a TCP már észleli a hibát?

  • Because TCP only notices when someone actively closes. A laptop that sleeps, a phone that loses signal, or a proxy that silently drops an idle connection leaves a half-open socket: your server still thinks the client is there and keeps queueing messages for it. Protocol-level ping/pong with a timeout is what turns that into a detected disconnect, usually within 30–60 seconds.Mert a TCP csak azt veszi észre, ha valaki aktívan lezár. Egy elalvó laptop, egy térerőt vesztő telefon, vagy egy tétlen kapcsolatot csendben eldobó proxy félig nyitott socketet hagy maga után: a szervered még azt hiszi, ott a kliens, és tovább sorakoztatja neki az üzeneteket. A protokoll-szintű ping/pong timeouttal ezt alakítja észlelt bontássá, jellemzően 30–60 másodpercen belül.

What does a correct reconnect strategy look like?Hogyan néz ki egy helyes újracsatlakozási stratégia?

  • Exponential backoff with jitter, capped — without jitter, a server restart brings every client back simultaneously and knocks it over again. On reconnect the client must also resynchronise: it missed everything sent while offline, so either it re-fetches current state or replays from a last-seen message id. Treating reconnection as merely reopening the socket is the common bug; the connection comes back, the state is silently stale.Exponenciális backoff jitterrel, felső korláttal — jitter nélkül egy szerver-újraindítás után minden kliens egyszerre tér vissza, és megint ledönti. Újracsatlakozáskor a kliensnek újra is kell szinkronizálnia: kimaradt minden, amit offline töltött ideje alatt küldtek, tehát vagy újratölti az aktuális állapotot, vagy egy utoljára látott üzenet-azonosítótól visszajátszik. A gyakori hiba az, ha az újracsatlakozást puszta socket-újranyitásnak tekintik: a kapcsolat visszajön, az állapot csendben elavult marad.

Watch outBuktatóLoad balancers and proxies close idle connections on their own timer — often 60 seconds — regardless of your application. If your heartbeat interval is longer than theirs, connections drop mysteriously in production and never in local development, where no proxy sits in between.A load balancerek és proxyk saját időzítővel zárják a tétlen kapcsolatokat — gyakran 60 másodperc —, függetlenül az alkalmazásodtól. Ha a heartbeat intervallumod hosszabb az övéknél, a kapcsolatok rejtélyesen szakadnak meg produkcióban, lokálisan viszont soha, ahol nincs köztes proxy.

Scaling WebSocketsWebSocketek skálázása

A WebSocket lives on exactly one server process. The moment you run two instances, a message published on one has no way to reach clients connected to the other — unless you add something between them.Egy WebSocket pontosan egy szerverfolyamaton él. Amint két instance fut, az egyiken publikált üzenet sehogy nem éri el a másikhoz csatlakozott klienseket — hacsak nem teszel közéjük valamit.

The bus carries messages between instances; each instance owns only the sockets it accepted.A busz az instance-ok között szállítja az üzeneteket; minden instance csak az általa elfogadott socketeket birtokolja.

Sticky sessions or a shared bus?Sticky session vagy közös busz?

  • They solve different halves. Sticky sessions keep a client pinned to the instance holding its socket — necessary if you use a transport that falls back to HTTP polling, since those requests must land on the same node. A shared bus solves the actual problem: getting a message to whichever instance holds the recipient. Stickiness alone doesn't give you cross-instance delivery, and it makes deployments worse by concentrating reconnect storms.Két különböző felét oldják meg. A sticky session az adott instance-hoz szögezi a klienst, amelyik a socketjét tartja — szükséges, ha a transzport HTTP-pollingra vált vissza, mert azoknak a kéréseknek ugyanarra a node-ra kell érkezniük. A közös busz a valódi problémát oldja meg: eljuttatni az üzenetet ahhoz az instance-hoz, amelyik a címzettet tartja. Önmagában a stickiness nem ad instance-ok közti kézbesítést, és a deploymentet is rontja azzal, hogy koncentrálja az újracsatlakozás-viharokat.

What breaks first as connection count grows?Mi törik el először, ahogy nő a kapcsolatszám?

  • Memory and file descriptors, not CPU — each connection holds a socket plus its send buffer, so tens of thousands of mostly-idle clients are a memory problem. Then the fan-out multiplier: broadcasting one message to 10,000 subscribers is 10,000 writes, and doing that per event is what saturates a node. The usual answers are batching, sending deltas instead of full state, and sharding rooms across instances so no single node owns a hot topic.A memória és a fájlleírók, nem a CPU — minden kapcsolat egy socketet és a küldőpufferét tartja, így több tízezer, jórészt tétlen kliens memóriaprobléma. Utána a fan-out szorzó: egy üzenet szórása 10 000 feliratkozónak 10 000 írás, és ha ez eseményenként megtörténik, az telíti a node-ot. A szokásos válaszok: kötegelés, teljes állapot helyett delták küldése, és a szobák szétosztása az instance-ok között, hogy egy node se birtokoljon forró topicot.

What happens to connections during a deploy?Mi történik a kapcsolatokkal deploy közben?

  • They all drop at once, and every client reconnects simultaneously — a thundering herd against the instances that are still starting. Mitigations: drain gracefully by sending close frames with a reason before shutdown, stagger rollout across instances, and rely on client-side jittered backoff. Rolling deploys make this routine rather than exceptional, so it has to be designed for rather than treated as an incident.Egyszerre mind megszakad, és minden kliens egyidejűleg csatlakozik újra — thundering herd a még induló instance-ok ellen. Enyhítés: szabályos leürítés close keretekkel és indokkal leállás előtt, a rollout lépcsőzetes elosztása az instance-ok között, és kliensoldali jitteres backoff. A rolling deploy ezt rutinná teszi, nem kivétellé, tehát tervezni kell rá, nem incidensként kezelni.

Watch outBuktatóRedis pub/sub is fire-and-forget: a subscriber that is momentarily disconnected simply misses those messages, with no replay. That's fine for presence or typing indicators and wrong for anything that must arrive — those need a durable stream (Redis Streams, Kafka, or a queue) behind the socket layer.A Redis pub/sub fire-and-forget: egy pillanatra lecsatlakozott feliratkozó egyszerűen lemarad azokról az üzenetekről, visszajátszás nélkül. Ez rendben van jelenlét- vagy gépelésjelzéshez, és hibás bármihez, aminek meg kell érkeznie — ahhoz tartós stream kell (Redis Streams, Kafka vagy queue) a socket-réteg mögött.

03

Presence, Rooms & Slow ConsumersJelenlét, szobák & lassú fogyasztók

Realtime features that look simple and aren't: who is online, which clients receive which messages, and what to do when a client can't keep up with the stream you're pushing at it.Realtime funkciók, amelyek egyszerűnek látszanak, de nem azok: ki van online, mely kliensek mely üzeneteket kapják, és mit tegyél, ha egy kliens nem bírja azt a folyamot, amit rátolsz.

Why is presence harder than it looks?Miért nehezebb a jelenlét, mint amilyennek látszik?

  • Because "online" is a guess about something you can't observe directly. A crashed client sends no goodbye, so presence is only as accurate as your heartbeat timeout — a user can appear online for 30 more seconds after their phone died. Across instances the state must be shared (a Redis key per user with a TTL, refreshed by heartbeat, is the common shape) and it must survive an instance dying without stranding its users as permanently online.Mert az „online" egy tipp valamiről, amit nem tudsz közvetlenül megfigyelni. Az összeomlott kliens nem küld búcsút, így a jelenlét csak annyira pontos, amennyire a heartbeat timeoutod — egy felhasználó 30 másodperccel a telefonja halála után is online-nak látszhat. Instance-ok között az állapotot meg kell osztani (felhasználónkénti Redis kulcs TTL-lel, heartbeattel frissítve a bevett forma), és túl kell élnie egy instance halálát anélkül, hogy örökre online-ként hagyná a felhasználóit.

How do rooms and subscriptions actually work?Hogyan működnek valójában a szobák és a feliratkozások?

  • A room is just a set of socket ids the server keeps, so publishing means iterating that set. The design question is where the set lives: in-process is fast but invisible to other instances, so cross-instance rooms need either the bus to carry every room message everywhere (simple, wasteful at scale) or a directory of which instance holds which member (efficient, more state to keep correct). Authorization belongs at subscribe time, not publish time — checking on every message multiplies the cost by the fan-out.A szoba csak socket-azonosítók halmaza, amit a szerver tart, így a publikálás e halmaz bejárása. A tervezési kérdés az, hol lakik a halmaz: a folyamaton belül gyors, de a többi instance számára láthatatlan, így az instance-ok közti szobákhoz vagy a busznak kell minden szobaüzenetet mindenhová vinnie (egyszerű, méretben pazarló), vagy nyilvántartás kell arról, melyik instance melyik tagot tartja (hatékony, több karbantartandó állapot). A jogosultság-ellenőrzés a feliratkozáshoz tartozik, nem a publikáláshoz — üzenetenként ellenőrizni a fan-outtal szorozza a költséget.

What do you do about a slow consumer?Mit teszel egy lassú fogyasztóval?

  • Decide explicitly, because the default is bad: if you keep writing to a socket the client isn't draining, the kernel and application buffers grow until that one client consumes the memory of the whole process. The three honest options are to drop messages (fine for tick data — the next update supersedes it), coalesce them (keep only the latest state per key), or disconnect the client and let it re-sync on reconnect. Monitor the outbound buffer size per connection and act on a threshold.Dönts kifejezetten, mert az alapértelmezett rossz: ha tovább írsz egy socketre, amit a kliens nem ürít, a kernel- és alkalmazáspufferek addig nőnek, amíg az az egy kliens felemészti az egész folyamat memóriáját. A három őszinte lehetőség: eldobás (rendben van tick-adatnál — a következő frissítés úgyis felülírja), összevonás (kulcsonként csak a legfrissebb állapot megtartása), vagy a kliens bontása, hogy újracsatlakozáskor újraszinkronizáljon. Figyeld a kimenő puffer méretét kapcsolatonként, és lépj küszöbérték felett.

Watch outBuktatóBroadcasting full state on every change is the pattern that scales worst: it's the largest payload multiplied by the largest audience. Send deltas, and give clients a way to request a full snapshot when they detect a gap — that also gives reconnection a defined recovery path.A teljes állapot szórása minden változáskor a legrosszabbul skálázódó minta: a legnagyobb payload szorozva a legnagyobb közönséggel. Küldj deltákat, és adj módot a klienseknek teljes pillanatkép kérésére, ha rést észlelnek — ez az újracsatlakozásnak is definiált helyreállítási utat ad.

Acks, Nacks & Visibility TimeoutAck, nack & visibility timeout

An acknowledgement is a consumer telling the broker "this message is done". Until then the message isn't gone — it's reserved, and it comes back if the consumer dies. When you ack is what decides your delivery semantics.Az ack az, amikor a fogyasztó közli a brokerrel, hogy „ez az üzenet kész". Addig az üzenet nincs elveszve — foglalt, és visszajön, ha a fogyasztó meghal. Az, hogy mikor ackelsz, dönti el a kézbesítési szemantikádat.

Ack first — at-most-onceElőbb ack — at-most-once
const msg = await queue.receive();
await msg.ack();        // ← released
await processOrder(msg); // crash here
// message is gone. work never done.
Ack last — at-least-onceUtoljára ack — at-least-once
const msg = await queue.receive();
try {
  await processOrder(msg); // crash here
  await msg.ack();         // ← only now
} catch { await msg.nack(); }
// redelivered. consumer must be idempotent.

What is a visibility timeout?Mi a visibility timeout?

  • The window during which a received message is hidden from other consumers. Ack inside it and the message is deleted; let it expire and the broker assumes you died and hands the message to someone else. It's SQS's term, but the mechanism is universal — RabbitMQ ties it to the channel's lifetime, Kafka to the consumer group's session timeout.Az az ablak, amíg egy fogadott üzenet rejtve van a többi fogyasztó elől. Ackelj ezen belül, és az üzenet törlődik; hagyd lejárni, és a broker feltételezi, hogy meghaltál, és másnak adja az üzenetet. Ez az SQS terminológiája, de a mechanizmus általános — a RabbitMQ a csatorna élettartamához, a Kafka a fogyasztói csoport session timeoutjához köti.

What goes wrong if the timeout is too short?Mi romlik el, ha túl rövid a timeout?

  • You get duplicate processing that looks like a bug in your code. A job that usually takes 20 seconds against a 30-second timeout will, on a slow day, get redelivered while the first consumer is still working — so two workers process the same order and neither has crashed. Either raise the timeout above your worst case, or extend the lease periodically from the consumer (a heartbeat for the message) while work continues.Duplikált feldolgozást kapsz, ami a kódod hibájának látszik. Egy általában 20 másodperces feladat 30 másodperces timeouttal egy lassú napon újrakézbesítődik, miközben az első fogyasztó még dolgozik — így két worker dolgozza fel ugyanazt a rendelést, és egyik sem omlott össze. Vagy emeld a timeoutot a legrosszabb eseted fölé, vagy hosszabbítsd a bérletet periodikusan a fogyasztóból (heartbeat az üzenethez), amíg tart a munka.

Nack, or just let it time out?Nack, vagy hagyd lejárni?

  • Nack when you know immediately that this consumer can't handle it — it returns the message straight away instead of blocking a slot for the full timeout. But an immediate nack-and-retry on a message that always fails becomes a hot loop hammering the broker, so nack should either carry a delay or be paired with a retry counter that routes to a dead-letter queue. Letting it time out is the correct behaviour only for actual crashes.Nackelj, ha azonnal tudod, hogy ez a fogyasztó nem tudja kezelni — így az üzenet rögtön visszakerül, ahelyett hogy a teljes timeoutra lefoglalna egy helyet. De az azonnali nack-és-újrapróbálás egy mindig hibázó üzeneten forró ciklussá válik, ami ostromolja a brokert, ezért a nack vagy késleltetést hordozzon, vagy párosuljon újrapróbálkozás-számlálóval, ami dead-letter queue-ba irányít. A lejáratni hagyás csak valódi összeomlásoknál helyes viselkedés.

Watch outBuktatóPrefetch (how many unacked messages a consumer holds) defaults high in some clients, which means one slow worker reserves a large batch it can't get through while other idle workers starve. Setting prefetch to a small number is often the single biggest throughput fix in a queue-based system.A prefetch (hány ackeletlen üzenetet tart egy fogyasztó) egyes kliensekben magas alapértékű, ami azt jelenti, hogy egy lassú worker olyan nagy köteget foglal le, amivel nem végez, miközben más tétlen workerek éheznek. A prefetch kis értékre állítása gyakran a legnagyobb áteresztőképesség-javítás egy queue-alapú rendszerben.

05

Dead-Letter Queues & Poison MessagesDead-letter queue-k & mérgezett üzenetek

A poison message is one that fails every time it's processed. With at-least-once delivery and unlimited retries it never leaves the queue, blocking or degrading everything behind it. A dead-letter queue is where it goes instead.A mérgezett üzenet az, amely minden feldolgozáskor hibázik. At-least-once kézbesítéssel és korlátlan újrapróbálkozással sosem hagyja el a sort, blokkolva vagy rontva mindent, ami mögötte van. A dead-letter queue az a hely, ahová helyette kerül.

Retry a bounded number of times, then divert — so one bad message can't consume the consumer's capacity indefinitely.Próbáld újra korlátozott számban, aztán tereld el — így egy rossz üzenet nem foglalhatja le korlátlanul a fogyasztó kapacitását.

Which failures should retry, and which shouldn't?Mely hibák próbálkozzanak újra, és melyek ne?

  • Retry transient failures — a timeout, a 503, a deadlock, a briefly unavailable dependency. Don't retry deterministic ones: malformed payload, validation failure, a referenced record that will never exist. Retrying those burns capacity to reach the same conclusion five times. Distinguishing them in the consumer, and dead-lettering the second class immediately, is what separates a queue that self-heals from one that just accumulates.Az átmeneti hibákat próbáld újra — timeout, 503, deadlock, rövid ideig elérhetetlen függőség. A determinisztikusakat ne: hibás formátumú payload, validációs hiba, olyan hivatkozott rekord, ami sosem fog létezni. Ezek újrapróbálása kapacitást éget, hogy ötször ugyanarra a következtetésre jusson. A kettő megkülönböztetése a fogyasztóban, és a második osztály azonnali dead-letterezése az, ami elválasztja az öngyógyító sort a puszta halmozótól.

What do you do with the DLQ afterwards?Mit kezdesz utána a DLQ-val?

  • It needs an owner, an alert on depth, and a replay path — otherwise it's just where messages go to be forgotten, which is data loss with extra steps. The workflow that actually functions: alert fires, someone inspects the message and the failure reason it was stored with, fixes the bug or the data, and replays the batch back to the main queue. That last step must be idempotent, since some of those messages may have partially succeeded.Kell hozzá gazda, riasztás a mélységre, és visszajátszási út — különben csak az a hely, ahová az üzenetek elfelejtődni mennek, ami adatvesztés extra lépésekkel. A ténylegesen működő folyamat: megszólal a riasztás, valaki megnézi az üzenetet és a vele eltárolt hibaokot, javítja a hibát vagy az adatot, és visszajátssza a köteget a fő sorba. Ez az utolsó lépés legyen idempotens, mert néhány üzenet részben sikeres lehetett.

Where does the retry count live?Hol él az újrapróbálkozás-számláló?

  • Ideally in the broker — SQS tracks ApproximateReceiveCount and moves the message after a configured maximum, RabbitMQ uses a DLX with a per-queue policy. Tracking it in the message body works too but is fragile, since a consumer that crashes before writing the increment loses the count and the message retries forever. Broker-side is both simpler and harder to get wrong.Ideális esetben a brokerben — az SQS követi az ApproximateReceiveCount-ot, és beállított maximum után átmozgatja az üzenetet, a RabbitMQ DLX-et használ soronkénti szabállyal. Az üzenettörzsben követés is működik, de törékeny, mert egy a növelés kiírása előtt összeomló fogyasztó elveszíti a számlálót, és az üzenet örökké újrapróbálkozik. A broker-oldal egyszerre egyszerűbb és nehezebben elrontható.

Watch outBuktatóStore the failure reason alongside the message when dead-lettering. A DLQ full of payloads with no indication of why each failed turns every investigation into a re-run of the bug, and by then the surrounding state has usually changed.Dead-letterezéskor tárold a hiba okát az üzenet mellett. Egy payloadokkal teli DLQ, amiből nem derül ki, miért hibázott mindegyik, minden vizsgálatot a hiba újrajátszásává tesz — és addigra a környező állapot általában megváltozott.

Consumer Groups & PartitionsFogyasztói csoportok & partíciók

A consumer group is a set of consumers that share a subscription — each message goes to exactly one member, so adding members adds throughput. In log-based systems the unit of assignment is a partition, which is also what bounds your parallelism.A fogyasztói csoport olyan fogyasztók halmaza, amelyek osztoznak egy feliratkozáson — minden üzenet pontosan egy taghoz megy, így a tagok növelése áteresztőképességet ad. Log-alapú rendszerekben a hozzárendelés egysége a partíció, ami egyben a párhuzamosságod korlátja is.

A partition is assigned to at most one consumer per group — so a fourth consumer against three partitions does nothing.Egy partíció csoportonként legfeljebb egy fogyasztóhoz van rendelve — így a negyedik fogyasztó három partícióval szemben nem csinál semmit.

Why can't you just add consumers to go faster?Miért nem lehet egyszerűen fogyasztókat hozzáadni a gyorsításhoz?

  • Because in a partitioned log, parallelism is capped by partition count — extra consumers sit idle. That makes partition count a capacity decision made up front, and an awkward one to change later: increasing it rebalances which key lands where, breaking the ordering guarantee for existing keys. A plain queue (SQS, RabbitMQ) has no such cap, but also no per-key ordering — that's the trade.Mert partícionált logban a párhuzamosságot a partíciószám korlátozza — a további fogyasztók tétlenül ülnek. Ettől a partíciószám előre meghozott kapacitásdöntés, és később kellemetlen megváltoztatni: a növelése újraosztja, melyik kulcs hová kerül, megtörve a meglévő kulcsok sorrendi garanciáját. Egy egyszerű sornak (SQS, RabbitMQ) nincs ilyen plafonja, de kulcsonkénti sorrendje sincs — ez a csere.

What is a rebalance, and why does it hurt?Mi az a rebalance, és miért fáj?

  • When a consumer joins, leaves or is declared dead, partitions are reassigned across the group. Classically this stops all consumption briefly — a stop-the-world pause proportional to group size. Worse, a consumer that is merely slow (blocked on a long poll interval or a long-running handler) gets declared dead and triggers a rebalance, which slows everything further. Cooperative/incremental rebalancing and correctly tuned session and poll timeouts are the mitigations.Amikor egy fogyasztó csatlakozik, kilép vagy halottnak nyilvánítják, a partíciók újraosztódnak a csoportban. Klasszikusan ez rövid időre leállítja az összes fogyasztást — a csoportmérettel arányos stop-the-world szünet. Ami rosszabb: egy pusztán lassú fogyasztót (hosszú poll intervallumon vagy hosszan futó handleren blokkolva) halottnak nyilvánítanak, ami rebalance-t vált ki, ami tovább lassít mindent. A kooperatív/inkrementális rebalance és a helyesen hangolt session- és poll-timeoutok az enyhítés.

What's the difference between a group and multiple subscribers?Mi a különbség egy csoport és több feliratkozó között?

  • Within a group, consumers share the work — one copy of each message total. Different groups each receive their own copy of everything, which is how one topic feeds the order service, the analytics pipeline and the audit log independently, each at its own pace and with its own offset. Getting these confused is why someone occasionally finds their analytics consumer stealing messages from production.Egy csoporton belül a fogyasztók megosztják a munkát — összesen egy példány üzenetenként. A különböző csoportok mindegyike a saját példányát kapja mindenből, így táplál egy topic egymástól függetlenül egy rendelésszolgáltatást, egy analitikai pipeline-t és egy audit logot, mindegyiket saját tempóban és saját offsettel. E kettő összekeverése miatt találja néha valaki azon az analitikai fogyasztóját, hogy üzeneteket lop a produkciótól.

Watch outBuktatóCommitting offsets before processing gives you at-most-once and silent data loss on a crash; committing after gives at-least-once and duplicates. Auto-commit does the former on a timer, which is why it's the wrong default for anything that matters — it can commit messages you haven't finished processing.Az offsetek feldolgozás előtti commitolása at-most-once-t és csendes adatvesztést ad összeomláskor; az utána commitolás at-least-once-t és duplikátumokat. Az auto-commit időzítővel az előbbit teszi, ezért rossz alapértelmezés bármihez, ami számít — olyan üzeneteket is commitolhat, amelyekkel még nem végeztél.

Message OrderingÜzenet-sorrendezés

Ordering is guaranteed only within a partition or a single-consumer queue — never globally across a scaled-out system. The practical question is not "how do I get global order" but "which messages actually need to be ordered relative to each other".A sorrend csak egy partíción vagy egyfogyasztós soron belül garantált — sosem globálisan egy kiskálázott rendszerben. A gyakorlati kérdés nem az, „hogyan érek el globális sorrendet", hanem az, „mely üzeneteknek kell valójában egymáshoz képest sorrendben lenniük".

How do you get ordering where you need it?Hogyan érsz el sorrendet ott, ahol kell?

  • Partition by the entity the order matters for — all events for one orderId or userId hash to the same partition, so they're processed in sequence by one consumer, while different entities proceed in parallel. SQS FIFO calls this a message group id; Kafka calls it the partition key. This gives per-key ordering with horizontal scaling, which is almost always what the requirement actually was.Partícionálj az alapján az entitás alapján, amire a sorrend számít — egy orderId vagy userId összes eseménye ugyanabba a partícióba hashelődik, így egy fogyasztó dolgozza fel őket sorban, miközben a különböző entitások párhuzamosan haladnak. Az SQS FIFO ezt message group id-nek hívja; a Kafka partíciókulcsnak. Ez kulcsonkénti sorrendet ad vízszintes skálázással — szinte mindig ez volt a valódi követelmény.

What breaks ordering even when you've partitioned correctly?Mi töri el a sorrendet akkor is, ha helyesen partícionáltál?

  • Retries, mostly. A failed message that goes to a retry queue and comes back later arrives after messages that were behind it — so a partial ordering guarantee quietly becomes none. Concurrency inside the consumer does the same: pulling in order and then handing off to a worker pool destroys the sequence you just preserved. Ordering has to hold end to end or it doesn't hold.Főként az újrapróbálkozások. Egy hibázó üzenet, ami retry sorba kerül és később tér vissza, olyan üzenetek után érkezik, amelyek mögötte voltak — így a részleges sorrendi garancia csendben semmivé válik. A fogyasztón belüli párhuzamosság ugyanezt teszi: sorrendben behúzni, majd worker-poolnak átadni tönkreteszi az épp megőrzött sorozatot. A sorrendnek végponttól végpontig kell tartania, különben nem tart.

What's the alternative to relying on order?Mi az alternatívája annak, hogy a sorrendre támaszkodsz?

  • Make handlers order-insensitive. Carry a version or timestamp in the event and ignore anything older than what you've already applied — a last-write-wins check that makes a late message harmless instead of corrupting. Or model events as facts that commute (balanceDelta: -10 rather than balance: 90), so applying them in any order converges. This is usually cheaper than enforcing strict ordering across a distributed system.Tedd a handlereket sorrend-érzéketlenné. Vigyél verziót vagy időbélyeget az eseményben, és hagyd figyelmen kívül, ami régebbi a már alkalmazottnál — egy last-write-wins ellenőrzés, ami ártalmatlanná teszi a késve érkező üzenetet ahelyett, hogy rombolna. Vagy modellezd az eseményeket felcserélhető tényekként (balanceDelta: -10 a balance: 90 helyett), így bármilyen sorrendben alkalmazva konvergálnak. Ez általában olcsóbb, mint szigorú sorrendet kikényszeríteni egy elosztott rendszerben.

Watch outBuktatóStrict ordering and high throughput are in direct tension: SQS FIFO caps throughput per message group, and a Kafka key that's too coarse (partitioning by tenant instead of by entity) creates a hot partition that one consumer must process alone. Order costs parallelism — buy only as much as you need.A szigorú sorrend és a nagy áteresztőképesség közvetlen feszültségben áll: az SQS FIFO üzenetcsoportonként korlátozza az áteresztőképességet, egy túl durva Kafka-kulcs pedig (tenant szerinti partícionálás entitás helyett) forró partíciót hoz létre, amit egyetlen fogyasztónak kell egyedül feldolgoznia. A sorrend párhuzamosságba kerül — csak annyit vegyél belőle, amennyi kell.

Backpressure & BacklogBackpressure & backlog

Backpressure is the signal that a consumer can't keep up, propagated back to the producer. A queue absorbs bursts by design — the danger is treating an unbounded queue as if that made the mismatch disappear.A backpressure az a jelzés, hogy a fogyasztó nem bírja a tempót, visszaterjesztve a termelőhöz. A sor terv szerint nyeli el a csúcsokat — a veszély az, ha egy korlátlan sort úgy kezelsz, mintha ettől eltűnne az eltérés.

How do you tell a temporary spike from a real problem?Hogyan különbözteted meg az átmeneti csúcsot a valódi problémától?

  • By the derivative, not the depth. A backlog that grows and then drains is the queue doing its job. A backlog whose growth rate stays positive means arrival rate exceeds processing rate, and it will never recover on its own — no amount of waiting fixes it. The metric worth alerting on is consumer lag or queue age (how old the oldest unprocessed message is), because that translates directly into user-visible delay; raw depth doesn't.A deriváltból, nem a mélységből. Az a backlog, ami nő, majd leürül, a sor rendeltetésszerű működése. Az a backlog, amelynek növekedési üteme pozitív marad, azt jelenti, hogy az érkezési ütem meghaladja a feldolgozásit, és magától sosem áll helyre — semennyi várakozás nem javítja. A riasztásra érdemes metrika a fogyasztói lag vagy a sor kora (mennyi idős a legrégebbi feldolgozatlan üzenet), mert ez közvetlenül felhasználó által látott késleltetéssé fordul; a nyers mélység nem.

What are the actual levers when a backlog won't drain?Mik a valódi eszközök, ha egy backlog nem ürül le?

  • Increase processing rate (more consumers — up to the partition ceiling; make the handler faster; batch database writes), or reduce arrival rate (rate-limit producers, shed low-priority work, sample). The third option people forget is to drop deliberately: for time-sensitive data, a message that's an hour old may be worthless, and expiring it is better than delaying everything behind it. Scaling consumers without checking whether the bottleneck is downstream just moves the queue into the database.Növeld a feldolgozási ütemet (több fogyasztó — a partícióplafonig; gyorsabb handler; kötegelt adatbázisírás), vagy csökkentsd az érkezési ütemet (termelők rate-limitelése, alacsony prioritású munka elhagyása, mintavételezés). A harmadik, elfelejtett lehetőség a szándékos eldobás: időérzékeny adatnál egy óra régi üzenet értéktelen lehet, és a lejáratása jobb, mint mindent késleltetni mögötte. A fogyasztók skálázása annak ellenőrzése nélkül, hogy a szűk keresztmetszet nem lejjebb van-e, csak az adatbázisba tolja a sort.

Where does backpressure apply outside queues?Hol érvényes a backpressure a sorokon kívül?

  • Anywhere producer and consumer speeds differ. Node streams implement it directly — write() returning false means "stop until drain", and ignoring that return value is how a fast source fills memory with buffered chunks. The same shape appears in WebSocket send buffers and in HTTP clients without connection limits. Recognising it as one pattern rather than three separate problems is the useful part.Bárhol, ahol a termelő és a fogyasztó sebessége eltér. A Node streamjei közvetlenül megvalósítják — a write() false visszatérése azt jelenti, „állj meg a drain-ig", és e visszatérési érték figyelmen kívül hagyása az, ahogy egy gyors forrás pufferelt darabokkal tölti meg a memóriát. Ugyanez az alak jelenik meg a WebSocket küldőpuffereiben és a kapcsolatkorlát nélküli HTTP-kliensekben. A hasznos rész az, ha ezt egy mintaként ismered fel, nem három külön problémaként.

Watch outBuktatóAutoscaling consumers on queue depth can make things worse: if the bottleneck is a shared database, more consumers means more contention and lower total throughput. Scale on lag, verify the constraint is actually the consumer, and cap the maximum.A fogyasztók sormélység alapú autoscale-je ronthat a helyzeten: ha a szűk keresztmetszet egy közös adatbázis, több fogyasztó több versengést és alacsonyabb összesített áteresztőképességet jelent. Skálázz lag alapján, ellenőrizd, hogy a korlát tényleg a fogyasztó-e, és maximáld a felső határt.

When does the log model actually matter?Mikor számít valójában a log-modell?

  • When you need replay — adding a new consumer that must process history, rebuilding a projection after a bug, or feeding several independent pipelines from one stream. With a queue, a message consumed by the wrong service is simply gone; with a retained log, each group has its own offset and history is still there. If you never replay and have one consumer per message, the log's operational cost buys you nothing.Amikor visszajátszásra van szükséged — új fogyasztó hozzáadása, aminek a történetet is fel kell dolgoznia, projekció újraépítése egy hiba után, vagy több független pipeline táplálása egy streamből. Sornál egy rossz szolgáltatás által fogyasztott üzenet egyszerűen odavan; megőrzött lognál minden csoportnak saját offsetje van, és a történet még megvan. Ha sosem játszol vissza, és üzenetenként egy fogyasztód van, a log üzemeltetési költsége semmit nem vesz neked.

What does RabbitMQ's routing buy you?Mit ad neked a RabbitMQ útválasztása?

  • The broker decides delivery, not the consumer. Topic exchanges with patterns like order.*.eu, priority queues, per-message TTL and delayed delivery all live in the broker, so consumers stay simple. Kafka pushes that logic to the consumer — it reads a partition and filters — which is more work but scales differently. Broker-side routing is the better fit for complex enterprise integration; consumer-side is better for high-volume streaming.A broker dönt a kézbesítésről, nem a fogyasztó. A topic exchange-ek olyan mintákkal, mint az order.*.eu, a prioritásos sorok, az üzenetenkénti TTL és a késleltetett kézbesítés mind a brokerben laknak, így a fogyasztók egyszerűek maradnak. A Kafka ezt a logikát a fogyasztóra tolja — olvas egy partíciót és szűr —, ami több munka, de másképp skálázódik. A broker-oldali útválasztás komplex vállalati integrációhoz illik jobban; a fogyasztó-oldali nagy volumenű streaminghez.

How would you justify the choice in an interview?Hogyan indokolnád a választást egy interjún?

  • Lead with the requirement, not the product. "We need replay and multiple independent consumers of the same stream" points at a log. "We need complex routing and per-message priority" points at RabbitMQ. "We need a queue and have no platform team" points at SQS. Naming the trade-off you're accepting — Kafka's operational weight, SQS's limited ordering, RabbitMQ's lack of replay — is what makes the answer credible.A követelménnyel kezdj, ne a termékkel. „Visszajátszás kell, és ugyanannak a streamnek több független fogyasztója" — ez logra mutat. „Komplex útválasztás és üzenetenkénti prioritás kell" — ez RabbitMQ-ra. „Sor kell, és nincs platformcsapat" — ez SQS-re. Az teszi hitelessé a választ, ha megnevezed az elfogadott trade-offot is — a Kafka üzemeltetési súlyát, az SQS korlátozott sorrendjét, a RabbitMQ visszajátszás-hiányát.

Watch outBuktatóKafka is frequently chosen for problems a queue would solve, and then run by a team with no capacity to operate it. Managed offerings remove much of that, but partition planning, retention sizing and consumer-lag monitoring remain yours regardless of who runs the brokers.A Kafkát gyakran választják olyan problémákra, amiket egy sor is megoldana, majd olyan csapat üzemelteti, amelynek nincs rá kapacitása. A menedzselt szolgáltatások ebből sokat levesznek, de a partíciótervezés, a megőrzés méretezése és a fogyasztói lag figyelése attól függetlenül a tiéd marad, hogy ki üzemelteti a brokereket.

What's the most common design mistake here?Mi itt a leggyakoribb tervezési hiba?

  • Using the socket as the source of truth. The connection is a delivery mechanism, not storage — anything sent while a client was disconnected is gone unless it also exists somewhere durable. The robust shape is: write the change, publish the event, and let the socket deliver a notification that state changed; the client can always fall back to fetching. That also makes reconnection trivial, because there's a defined way to catch up.Ha a socketet tekinted az igazság forrásának. A kapcsolat kézbesítési mechanizmus, nem tároló — bármi, amit egy lecsatlakozott kliensnek küldtek, odavan, hacsak nem létezik valahol tartósan is. A robusztus forma: írd le a változást, publikáld az eseményt, és hagyd, hogy a socket értesítést kézbesítsen az állapotváltozásról; a kliens mindig visszaeshet lekérésre. Ettől az újracsatlakozás is triviális lesz, mert van definiált mód a felzárkózásra.

When is polling the right answer?Mikor a polling a helyes válasz?

  • More often than realtime enthusiasm suggests. If updates are infrequent, or a few seconds of staleness is acceptable, polling with a sensible interval and conditional requests (ETag, 304) costs almost nothing and removes the entire class of problems in this file — no connection state, no sticky sessions, no fan-out bus, no reconnection logic. Choosing WebSockets for a dashboard that updates once a minute is paying a large operational bill for nothing.Gyakrabban, mint azt a realtime iránti lelkesedés sugallja. Ha a frissítések ritkák, vagy néhány másodperc elavultság elfogadható, az értelmes intervallumú polling feltételes kérésekkel (ETag, 304) szinte semmibe nem kerül, és eltünteti az ebben a fájlban tárgyalt problémák teljes osztályát — nincs kapcsolatállapot, nincs sticky session, nincs fan-out busz, nincs újracsatlakozási logika. WebSocketet választani egy percenként frissülő dashboardhoz nagy üzemeltetési számla a semmiért.

How do you test any of this?Hogyan teszteled mindezt?

  • The failure modes are the point, so test them explicitly: kill a consumer mid-processing and assert the message is redelivered; disconnect a client and assert it resynchronises rather than silently missing data; send duplicates and assert the handler is idempotent; and load-test fan-out at realistic subscriber counts, since that's where the cost is non-linear. A test suite that only covers the happy path tells you nothing about a realtime system.A hibamódok a lényeg, tehát teszteld őket kifejezetten: ölj meg egy fogyasztót feldolgozás közben, és ellenőrizd, hogy az üzenet újrakézbesítődik; bonts egy klienst, és ellenőrizd, hogy újraszinkronizál, nem pedig csendben adatot veszít; küldj duplikátumokat, és ellenőrizd, hogy a handler idempotens; és terheléstesztelj fan-outot realisztikus feliratkozószámmal, mert ott nemlineáris a költség. A csak boldog utat lefedő tesztkészlet semmit nem mond egy realtime rendszerről.

Watch outBuktató"Realtime" in a requirements document usually means "fast enough that it feels live", not milliseconds. Establishing the actual latency budget early frequently turns a WebSocket architecture into a two-second poll — and the honest version of that conversation is itself a strong interview answer.A követelménydokumentumban a „realtime" általában azt jelenti, „elég gyors ahhoz, hogy élőnek érződjön", nem ezredmásodperceket. A tényleges késleltetési keret korai tisztázása gyakran kétmásodperces pollinggá alakít egy WebSocket-architektúrát — és ennek a beszélgetésnek az őszinte változata önmagában erős interjús válasz.
