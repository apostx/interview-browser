01

Endpoint

An endpoint is a specific URL that exposes a resource or collection a client can act on — base URL + a resource path (e.g. https://api.shop.com/orders/42).Az endpoint egy konkrét URL, ami egy resource-t vagy collection-t tesz elérhetővé a kliens számára — base URL + resource path (pl. https://api.shop.com/orders/42).

How does REST resource design differ from RPC-style endpoints?Miben tér el a REST resource-design az RPC-stílusú endpointoktól?

  • REST models nouns: /orders/42, and the HTTP method carries the verb.A REST főneveket modellez: /orders/42, az igét pedig a HTTP metódus hordozza.
  • RPC bakes the action into the path: /getOrder?id=42, /cancelOrder.Az RPC a műveletet a path-ba égeti: /getOrder?id=42, /cancelOrder.

How should endpoints be named?Hogyan érdemes elnevezni az endpointokat?

  • Plural nouns for collections (/orders), path params for identity (/orders/42).Többes számú főnév a collection-ökhöz (/orders), path param az azonosításhoz (/orders/42).
  • No verbs in the path; filtering / sorting go in the query string (/orders?status=paid&sort=-created).Ige ne legyen a path-ban; a szűrés és a rendezés a query stringbe megy (/orders?status=paid&sort=-created).
  • Nest to show relationships (/users/7/orders), but avoid going much deeper than two levels.A kapcsolatokat nesteléssel jelezd (/users/7/orders), de két szintnél sokkal mélyebbre ne menj.

Watch outBuktatóThe endpoint is the URL; the operation is endpoint + method. GET /orders and POST /orders are two different operations on one endpoint.Az endpoint maga az URL; a művelet az endpoint + metódus. A GET /orders és a POST /orders két külön művelet ugyanazon az endpointon.

HTTP Methodsmetódusok

Methods are the verbs that define what action a request performs on a resource.A metódusok azok az igék, amelyek megadják, hogy a kérés milyen műveletet végez a resource-on.

MethodPurposeCélSafeIdempotent
GETRead a resourceResource olvasása✓✓
POSTCreate / non-idempotent actionLétrehozás / nem idempotens művelet——
PUTFull replaceTeljes csere—✓
PATCHPartial updateRészleges módosítás——*
DELETERemove a resourceResource törlése—✓

Safe vs idempotent — what's the difference?Safe vs idempotent — mi a különbség?

  • Safe = no server state change (GET, HEAD).Safe = nem változtat szerver-oldali állapotot (GET, HEAD).
  • Idempotent = repeating N times has the same effect as once (GET, PUT, DELETE).Idempotent = N-szer megismételve ugyanaz a hatás, mint egyszer (GET, PUT, DELETE).
  • Every safe method is idempotent; the reverse isn't true.Minden safe metódus idempotens; fordítva nem igaz.

PUT vs PATCH?

  • PUT replaces the whole resource — fields you omit get cleared/defaulted.A PUT a teljes resource-t lecseréli — a kihagyott mezők törlődnek vagy default értéket kapnak.
  • PATCH applies a partial change. *It isn't inherently idempotent (e.g. an increment op).A PATCH részleges módosítást alkalmaz. *Önmagában nem idempotens (pl. egy increment művelet).

Watch outBuktatóPOST is the catch-all for non-idempotent creates and RPC-ish actions that don't map cleanly onto the other verbs.A POST a gyűjtőmetódus a nem idempotens létrehozásokhoz és azokhoz az RPC-szerű műveletekhez, amelyek nem illenek tisztán a többi igére.

03

Request–Response

A stateless exchange: the client sends a request (method, path, headers, optional body); the server returns a response (status, headers, optional body).Stateless csere: a kliens kérést küld (metódus, path, headerek, opcionális body), a szerver választ ad vissza (status, headerek, opcionális body).

A single request/response round trip.Egyetlen request/response körút.

Why is HTTP stateless, and how do we keep sessions?Miért stateless a HTTP, és hogyan tartunk fenn session-t?

  • Each request is independent — the server keeps no memory of the client between them.Minden kérés független — a szerver nem őriz emléket a kliensről a kérések között.
  • State is carried by the client (cookie / token) or looked up server-side by a session ID.Az állapotot a kliens hordozza (cookie / token), vagy a szerver session ID alapján keresi ki.

Content-Type vs Accept?

  • Content-Type describes the body you're sending.A Content-Type azt írja le, amit küldesz a body-ban.
  • Accept tells the server which formats you can handle (content negotiation).Az Accept megmondja a szervernek, milyen formátumot tudsz kezelni (content negotiation).
04

Status Codeskódok

Three-digit codes signalling the outcome — 1xx info, 2xx success, 3xx redirect, 4xx client error, 5xx server error.Háromjegyű kódok, amelyek a kimenetet jelzik — 1xx info, 2xx siker, 3xx átirányítás, 4xx kliens-hiba, 5xx szerver-hiba.

200 OK201 Created204 No Content 301/302304 Not Modified 400401403404409422429 500502503504

401 vs 403?

  • 401 Unauthorized = not authenticated — missing/invalid credentials (“who are you?”).401 Unauthorized = nincs authentikálva — hiányzó vagy érvénytelen credential („ki vagy te?”).
  • 403 Forbidden = authenticated, but not allowed to do this.403 Forbidden = authentikálva van, de nincs joga ehhez a művelethez.

400 vs 422?

  • 400 — malformed request (bad JSON, wrong types).400 — hibás formátumú kérés (rossz JSON, rossz típusok).
  • 422 — syntactically valid but semantically invalid (validation failed).422 — szintaktikailag helyes, de szemantikailag érvénytelen (validáció bukott).

201 vs 200 vs 204?

  • 201 after a create (often with a Location header).201 létrehozás után (gyakran Location headerrel).
  • 200 success with a body; 204 success with no body (e.g. a DELETE).200 siker body-val; 204 siker body nélkül (pl. egy DELETE).

Watch outBuktatóNever return 200 with an error inside the body — clients, proxies and monitoring all trust the status line.Soha ne adj vissza 200-at úgy, hogy a hiba a body-ban van — a kliensek, a proxyk és a monitoring mind a status sorra támaszkodnak.

05

Authentication

Authentication proves who the caller is.Az authentication azt igazolja, hogy ki a hívó fél.

Session-based vs token-based auth?Session-alapú vs token-alapú auth?

  • Session: server stores session state; client holds an opaque session-ID cookie. Easy to revoke, but stateful.Session: a szerver tárolja a session állapotát, a kliensnél egy opaque session-ID cookie van. Könnyen visszavonható, de stateful.
  • Token (JWT): self-contained and stateless — scales horizontally, but hard to revoke before it expires.Token (JWT): önhordó és stateless — jól skálázódik horizontálisan, viszont lejárat előtt nehéz visszavonni.

Where do you store a token in the browser?Hol tárolod a tokent a böngészőben?

  • An httpOnly · Secure · SameSite cookie resists XSS token theft (but needs CSRF defence).Egy httpOnly · Secure · SameSite cookie véd az XSS-es token-lopás ellen (de kell mellé CSRF-védelem).
  • localStorage is readable by any injected script. Prefer a short-lived access token + refresh flow.A localStorage-ot bármelyik injektált script olvassa. Inkább rövid életű access token + refresh flow.

Watch outBuktatóA JWT is signed, not encrypted — anyone can base64-decode and read the payload. Never put secrets in the claims.A JWT aláírt, nem titkosított — bárki base64-dekódolhatja és elolvashatja a payloadot. Soha ne tegyél titkot a claimekbe.

06

Authorization

Authorization decides what an authenticated caller is allowed to do.Az authorization azt dönti el, hogy egy authentikált hívó mit csinálhat.

AuthN vs AuthZ?

Authentication = identity (login). Authorization = permissions (access). AuthN always comes first.Authentication = identitás (belépés). Authorization = jogosultságok (hozzáférés). Az AuthN mindig előbb jön.

RBAC vs ABAC vs scopes?RBAC vs ABAC vs scope-ok?

  • RBAC — permissions grouped into roles (admin, editor).RBAC — a jogosultságok szerepekbe csoportosítva (admin, editor).
  • ABAC — rules over attributes of user, resource and context; finer-grained.ABAC — szabályok a user, a resource és a kontextus attribútumai felett; finomabb szemcsézettség.
  • Scopes — coarse capability grants attached to a token (read:orders).Scope-ok — durvább szemcsézetű képesség-engedélyek a tokenhez kötve (read:orders).

Watch outBuktatóEnforce authorization on the server for every request. Hiding a button in React is UX, not security.Az authorization-t minden kérésnél a szerveren kell kikényszeríteni. Egy gomb elrejtése Reactben UX, nem biztonság.

07

Access Tokens

A credential — usually a Bearer token / JWT — that a client presents to prove an authenticated, authorized session, sent as Authorization: Bearer <token>.Egy credential — általában Bearer token / JWT —, amit a kliens felmutat egy authentikált, jogosult session igazolására, az Authorization: Bearer <token> headerben.

What's the structure of a JWT?Mi a JWT felépítése?

header.payload.signature, each base64url-encoded. The signature (HMAC or RSA/EC) verifies integrity so the claims can't be tampered with.header.payload.signature, mindegyik base64url-kódolva. Az aláírás (HMAC vagy RSA/EC) igazolja az integritást, így a claimek nem módosíthatók.

Access token vs refresh token?

  • Access: short-lived (minutes), sent on every call.Access: rövid életű (percek), minden híváskor megy.
  • Refresh: long-lived, stored securely, exchanged for a new access token without re-login. Rotate it to detect theft.Refresh: hosszú életű, biztonságosan tárolva, új access tokenre váltható újbóli belépés nélkül. Rotáld, hogy észrevehető legyen a lopás.

Watch outBuktatóA short access-token TTL limits the blast radius of a leak; pair it with a revocation list for high-value tokens.A rövid access-token TTL korlátozza egy szivárgás hatósugarát; értékes tokeneknél párosítsd revocation listával.

08

OAuth 2.0

A delegated-authorization framework — it lets an app act on a user's behalf without ever handling their password. Roles: resource owner (user), client (app), authorization server, resource server.Delegált authorization keretrendszer — lehetővé teszi, hogy egy alkalmazás a felhasználó nevében járjon el anélkül, hogy valaha kezelné a jelszavát. Szerepek: resource owner (user), client (app), authorization server, resource server.

Authorization Code flow with PKCE — the default for web, SPA and mobile clients.Authorization Code flow PKCE-vel — ez az alapértelmezés web, SPA és mobil kliensekhez.

Which grant type for a web / SPA / mobile login?Melyik grant type kell web / SPA / mobil loginhoz?

  • Authorization Code + PKCE — PKCE stops an intercepted code from being exchanged by an attacker on public clients.Authorization Code + PKCE — a PKCE megakadályozza, hogy egy lehallgatott code-ot a támadó beváltson public clienteknél.
  • Client Credentials for machine-to-machine (no user involved).Client Credentials gép-gép kommunikációhoz (nincs benne felhasználó).

OAuth 2.0 vs OpenID Connect?

  • OAuth 2.0 handles authorization (access).Az OAuth 2.0 az authorization-t (hozzáférés) kezeli.
  • OIDC layers authentication on top, adding an id_token (JWT) that describes the user.Az OIDC ráépíti az authentication-t, egy id_token-nel (JWT), ami a felhasználót írja le.

Watch outBuktatóAn access token proves access, not identity — don't use it to answer “who is this user?”. That's what OIDC's id_token is for.Az access token hozzáférést igazol, nem identitást — ne azzal válaszold meg, hogy „ki ez a felhasználó?”. Arra való az OIDC id_token-je.

09

Rate-Limiting

Capping how many requests a client can make per time window to protect capacity and keep usage fair.Annak korlátozása, hány kérést tehet egy kliens időablakonként — a kapacitás védelmére és a fair használatért.

Token bucket: refills at a steady rate, allows short bursts up to its size.Token bucket: egyenletes ütemben töltődik, és a méretéig enged rövid burstöket.

Common algorithms?Gyakori algoritmusok?

  • Fixed window — simple, but allows bursts at the window edge.Fixed window — egyszerű, de az ablak határán burstöt enged.
  • Sliding window — smooths those edge bursts.Sliding window — kisimítja ezeket a határon jelentkező burstöket.
  • Token bucket — steady refill, bursts up to bucket size. Leaky bucket — constant outflow.Token bucket — egyenletes utántöltés, burst a bucket méretéig. Leaky bucket — állandó kifolyás.

How do you signal limits to clients?Hogyan jelzed a limiteket a klienseknek?

429 Too Many Requests + Retry-After, plus X-RateLimit-Limit / Remaining / Reset headers.429 Too Many Requests + Retry-After, valamint X-RateLimit-Limit / Remaining / Reset headerek.

Watch outBuktatóAcross multiple instances, counters must live in shared storage (e.g. Redis) — otherwise every node limits independently and the real cap is N×.Több instance esetén a számlálóknak közös tárban kell lenniük (pl. Redis) — különben minden node külön limitál, és a tényleges korlát N-szerese lesz.

10

Throttling

Controlling the rate of processing — typically by delaying, queuing or degrading excess requests instead of rejecting them outright.A feldolgozás ütemének szabályozása — jellemzően a többletkérések késleltetésével, sorba állításával vagy degradálásával, ahelyett hogy egyenesen elutasítanánk őket.

Rate-limiting vs throttling?

  • Rate limiting usually rejects over-quota requests (429).A rate limiting jellemzően elutasítja a kvótán túli kéréseket (429).
  • Throttling shapes traffic — slows, queues or degrades it to stay within capacity.A throttling formálja a forgalmat — lassítja, sorba állítja vagy degradálja, hogy a kapacitáson belül maradjon.
  • The terms overlap; the distinction to state is intent: reject vs smooth.A két fogalom átfed; a lényeges különbség a szándék: elutasítás vs kisimítás.

Watch outBuktatóThrottling improves stability but adds latency — make sure clients have sane timeouts and exponential backoff.A throttling javítja a stabilitást, de latenciát ad hozzá — ügyelj rá, hogy a klienseknek legyen értelmes timeoutja és exponential backoffja.

11

Pagination

Splitting a large result set into pages instead of returning everything at once.Egy nagy találati halmaz lapokra bontása ahelyett, hogy egyszerre adnánk vissza mindent.

Offset drifts when data changes; a cursor points at the last item and stays stable.Az offset elcsúszik, ha változik az adat; a cursor az utolsó elemre mutat és stabil marad.

Offset vs cursor (keyset)?Offset vs cursor (keyset)?

  • Offset/limit (?page=3&limit=20) — simple, random access; but slow on deep offsets and can skip/duplicate rows when data shifts.Offset/limit (?page=3&limit=20) — egyszerű, tetszőleges lapra ugorhatsz; viszont mély offseteknél lassú, és adatváltozáskor sorokat hagyhat ki vagy duplázhat.
  • Cursor (?after=<cursor>) — stable and fast on huge datasets, but you can't jump to an arbitrary page.Cursor (?after=<cursor>) — stabil és gyors nagy adathalmazokon, de nem tudsz tetszőleges lapra ugrani.

Watch outBuktatóFor infinite scroll or high-write tables, prefer cursor pagination to avoid drift and slow deep-offset queries.Infinite scrollnál vagy sokat íródó tábláknál cursor pagination a jó választás — így elkerülöd az elcsúszást és a lassú, mély offsetes lekérdezéseket.

12

Caching

Storing responses so repeated reads skip the work. Layers: browser → CDN → reverse proxy → application → database.Válaszok tárolása, hogy az ismételt olvasások megspórolják a munkát. Rétegek: böngésző → CDN → reverse proxy → alkalmazás → adatbázis.

A matching ETag lets the server answer 304 and skip resending the body.Egyező ETag esetén a szerver 304-gyel válaszol, és nem küldi újra a body-t.

Cache-Control basics?Cache-Control alapok?

max-age (freshness), no-cache (revalidate before use), no-store (never cache), private vs public.max-age (frissesség), no-cache (használat előtt revalidálj), no-store (soha ne cache-eld), private vs public.

How do conditional requests work?Hogyan működnek a conditional requestek?

The server returns ETag / Last-Modified. The client re-requests with If-None-Match / If-Modified-Since; unchanged → 304 Not Modified with no body.A szerver ETag / Last-Modified headert ad vissza. A kliens If-None-Match / If-Modified-Since headerrel kérdez rá újra; ha nem változott → 304 Not Modified, body nélkül.

What's the hard part?Mi benne a nehéz?

Invalidation — via TTL expiry, write-through updates, or cache-busting (versioned URLs/keys).Az invalidáció — TTL lejárattal, write-through frissítéssel vagy cache-bustinggal (verziózott URL-ek/kulcsok).

Watch outBuktatóCache the right thing at the right layer — caching user-specific data in a shared/CDN cache leaks one user's data to another. Mark it private.A megfelelő dolgot cache-eld a megfelelő rétegben — felhasználó-specifikus adat közös/CDN cache-be tétele az egyik user adatát a másikhoz szivárogtatja. Jelöld private-nak.

13

Idempotency

A request that produces the same result no matter how many times it's sent. This is what makes safe retries possible.Olyan kérés, ami ugyanazt az eredményt adja, akárhányszor küldöd el. Ez teszi lehetővé a biztonságos retry-t.

An Idempotency-Key lets the server dedupe a retried write.Az Idempotency-Key lehetővé teszi, hogy a szerver deduplikálja az újraküldött írást.

Which methods are idempotent, and why does it matter?Mely metódusok idempotensek, és miért számít?

  • GET, PUT, DELETE, HEAD are; POST and PATCH generally aren't.A GET, PUT, DELETE, HEAD igen; a POST és a PATCH általában nem.
  • It matters for retries: after a timeout, a client can safely re-send an idempotent call with no side effects.A retry miatt számít: timeout után a kliens biztonságosan újraküldhet egy idempotens hívást mellékhatás nélkül.

How do you make POST idempotent (e.g. payments)?Hogyan teszel egy POST-ot idempotenssé (pl. fizetésnél)?

The client sends an Idempotency-Key header; the server stores key → result and returns the original response on any retry — so a double-submitted charge only bills once.A kliens küld egy Idempotency-Key headert; a szerver eltárolja a kulcs → eredmény párost, és minden retry-nál az eredeti választ adja vissza — így a duplán elküldött terhelés csak egyszer számláz.

Watch outBuktatóDELETE twice: first returns 204, second 404 — still idempotent, because the state (resource gone) is unchanged.Kétszeri DELETE: az első 204-et ad, a második 404-et — ettől még idempotens, mert az állapot (a resource nincs meg) változatlan.

14

Webhooks

Server-to-server HTTP callbacks — the provider POSTs an event to your URL the moment something happens. A “reverse API”.Szerver-szerver HTTP callbackek — a provider POST-ol egy eseményt a te URL-edre abban a pillanatban, amikor történik valami. Egyfajta „fordított API”.

Polling asks repeatedly; a webhook is pushed once, on the event.A polling ismételten kérdez; a webhook egyszer, az esemény pillanatában érkezik.

Webhooks vs polling?Webhook vs polling?

  • Polling repeatedly asks “any updates?” — wasteful and laggy.A polling ismételten rákérdez, hogy „van új?” — pazarló és késleltetett.
  • Webhooks push the event as it happens — efficient and near-real-time, but you must expose an endpoint and handle delivery failures.A webhook az esemény pillanatában küld — hatékony és közel valós idejű, viszont endpointot kell kitenned és kezelned kell a kézbesítési hibákat.

How do you secure and harden a receiver?Hogyan teszed biztonságossá és robusztussá a fogadót?

  • Verify a signature (HMAC of the payload) to prove it's really from the provider.Ellenőrizd az aláírást (a payload HMAC-je), hogy bizonyítsd: tényleg a providertől jött.
  • Return 2xx fast, process async; providers retry on non-2xx, so make handlers idempotent (dedupe by event ID).Adj vissza gyorsan 2xx-et, a feldolgozás menjen aszinkron; a providerek nem-2xx esetén újrapróbálnak, ezért a handler legyen idempotens (dedup event ID alapján).

Watch outBuktatóAssume at-least-once delivery and out-of-order events — dedupe on event ID and never rely on arrival order.Számíts at-least-once kézbesítésre és sorrenden kívüli eseményekre — dedupálj event ID alapján, és soha ne támaszkodj az érkezési sorrendre.

15

API Versioning

Managing change so existing clients don't break as the API evolves.A változás kezelése úgy, hogy a meglévő kliensek ne törjenek el az API fejlődése közben.

Where can the version live?Hol lehet a verzió?

  • URI path (/v1/orders) — most visible, trivial to route and test.URI path (/v1/orders) — a leglátványosabb, triviálisan routolható és tesztelhető.
  • Header / media type (Accept: application/vnd.api.v2+json) — cleaner URLs, harder to test by hand.Header / media type (Accept: application/vnd.api.v2+json) — tisztább URL-ek, kézzel nehezebben tesztelhető.
  • Query param (?version=2) — simple, less common.Query param (?version=2) — egyszerű, ritkábban használt.

Breaking vs non-breaking change?Breaking vs non-breaking változás?

  • Non-breaking (additive): new optional fields/endpoints — no version bump needed.Non-breaking (additív): új opcionális mezők/endpointok — nem kell verziót emelni.
  • Breaking: removing/renaming fields, changing types or semantics — needs a new version + deprecation window.Breaking: mezők törlése/átnevezése, típus- vagy szemantikaváltozás — új verzió kell + deprecation időablak.

Watch outBuktatóVersion only on breaking changes. Leaning on additive, backward-compatible evolution keeps you out of a version zoo.Csak breaking változásnál verziózz. Ha additív, visszafelé kompatibilis fejlődésre támaszkodsz, elkerülöd a verzió-dzsungelt.

16

OpenAPI

A standard, language-agnostic spec (YAML/JSON) that describes a REST API — its endpoints, params, schemas and auth. Formerly Swagger.Szabványos, nyelvfüggetlen specifikáció (YAML/JSON), ami leír egy REST API-t — endpointok, paraméterek, sémák és auth. Korábbi neve: Swagger.

What do you get from it?Mit nyersz vele?

Auto-generated docs, client/server SDK codegen, request validation, and contract/mock testing.Automatikusan generált dokumentációt, kliens/szerver SDK codegent, request validációt, valamint contract- és mock-tesztelést.

Contract-first vs code-first?

  • Contract-first: write the spec, generate stubs — teams agree on the interface before coding.Contract-first: megírod a specifikációt, generálsz stubokat — a csapatok kódolás előtt megegyeznek az interfészben.
  • Code-first: generate the spec from annotated code — convenient, but the spec can drift if it isn't enforced.Code-first: a specifikációt annotált kódból generálod — kényelmes, de a spec elcsúszhat, ha nincs kikényszerítve.

Watch outBuktatóAn OpenAPI doc is only useful if it stays in sync — validate the running API against the spec in CI.Egy OpenAPI dokumentáció csak akkor hasznos, ha szinkronban marad — validáld a futó API-t a spec ellen a CI-ban.

17

REST vs GraphQL

Two API styles. REST exposes many resource endpoints; GraphQL exposes one endpoint where the client asks for exactly the fields it needs.Két API-stílus. A REST sok resource-endpointot tesz elérhetővé; a GraphQL egyetlen endpointot, ahol a kliens pontosan a szükséges mezőket kéri le.

REST tends to over- and under-fetch; GraphQL returns exactly the requested shape in one call.A REST hajlamos over- és under-fetchre; a GraphQL egyetlen hívásban pontosan a kért alakot adja vissza.

Core trade-offs?Fő trade-offok?

  • REST: simple, cache-friendly (HTTP caching per URL) — but over/under-fetching and multiple round trips for related data.REST: egyszerű, cache-barát (HTTP caching URL-enként) — viszont over/under-fetching és több körút a kapcsolódó adatokért.
  • GraphQL: no over-fetch, one request for nested data, strong typed schema — but HTTP caching is harder, and you must guard the N+1 problem (DataLoader/batching) and query cost/depth.GraphQL: nincs over-fetch, egy kérés az ágyazott adatokra, erősen típusos séma — viszont a HTTP caching nehezebb, és védekezned kell az N+1 probléma (DataLoader/batching), valamint a query költsége/mélysége ellen.

When would you pick each?Mikor melyiket választanád?

  • REST: public APIs, simple CRUD, CDN caching.REST: publikus API-k, egyszerű CRUD, CDN caching.
  • GraphQL: rich clients (mobile) with varied data needs, or aggregating many sources.GraphQL: gazdag kliensek (mobil) változatos adatigénnyel, vagy több forrás aggregálása.

Watch outBuktatóGraphQL moves complexity to the server (resolvers, query-cost limits) — it isn't automatically “better”.A GraphQL a komplexitást a szerverre tolja (resolverek, query-cost limitek) — nem automatikusan „jobb”.

18

API Gateway

A single entry point in front of your services that handles cross-cutting concerns — routing, auth, rate limiting, TLS termination, transformation and aggregation.Egyetlen belépési pont a szolgáltatásaid előtt, ami a keresztmetsző feladatokat kezeli — routing, auth, rate limiting, TLS termináció, transzformáció és aggregáció.

Clients hit one gateway; it applies policy and routes to internal services.A kliensek egyetlen gateway-t hívnak; az alkalmazza a policy-t és routolja a belső szolgáltatásokhoz.

Why put a gateway in front of microservices?Miért teszünk gateway-t a microservice-ek elé?

It centralizes auth, rate limiting and logging so each service doesn't reimplement them, hides internal topology, and can aggregate several service calls into one client response.Központosítja az autht, a rate limitinget és a loggingot, így nem kell minden service-ben újraírni; elrejti a belső topológiát; és több service-hívást egyetlen kliens-válaszba aggregálhat.

Gateway vs BFF (Backend-for-Frontend)?

A BFF is a gateway tailored per client type (web vs mobile), shaping responses to that client's exact needs.A BFF egy kliens-típusra szabott gateway (web vs mobil), ami az adott kliens pontos igényeire formálja a válaszokat.

Watch outBuktatóThe gateway is a single point of failure and an extra latency hop — make it highly available and keep its logic thin (routing/policy, never business rules).A gateway single point of failure és plusz latencia-ugrás — tedd magas rendelkezésre állásúvá, és tartsd vékonyan a logikáját (routing/policy, soha ne üzleti szabályok).

Microservices

An architecture that splits an app into small, independently deployable services, each owning a bounded context and its own data.Olyan architektúra, ami az alkalmazást kicsi, egymástól függetlenül deployolható szolgáltatásokra bontja, mindegyik saját bounded contexttel és saját adattal.

Trade-offs vs a monolith?Trade-offok a monolithoz képest?

  • Pros: independent deploy/scaling, tech flexibility, fault isolation, team autonomy.Előnyök: független deploy/skálázás, technológiai rugalmasság, hibaizoláció, csapat-autonómia.
  • Cons: distributed-system complexity — network failures, data consistency, testing, observability.Hátrányok: elosztott rendszer komplexitása — hálózati hibák, adatkonzisztencia, tesztelés, observability.

How do services communicate and stay consistent?Hogyan kommunikálnak a szolgáltatások és hogyan maradnak konzisztensek?

  • Sync (REST/gRPC) or async (events / message queues).Szinkron (REST/gRPC) vagy aszinkron (események / message queue-k).
  • No distributed ACID across services — use the Saga pattern (local transactions + compensating actions) and accept eventual consistency.Nincs elosztott ACID a szolgáltatások között — használd a Saga patternt (lokális tranzakciók + kompenzáló műveletek), és fogadd el az eventual consistencyt.

Watch outBuktatóDon't reach for microservices on a small app — the operational overhead outweighs the benefits. Split when clear boundaries and scale justify it.Kis alkalmazásnál ne nyúlj microservice-ekhez — az üzemeltetési többletköltség felülmúlja az előnyöket. Akkor bonts, ha a tiszta határok és a méret indokolják.

Error Handling

Returning consistent, actionable errors so clients can react programmatically and humans can debug.Konzisztens, kezelhető hibák visszaadása, hogy a kliensek programozottan reagálhassanak, az emberek pedig debugolni tudjanak.

What does a good error response look like?Hogy néz ki egy jó hibaválasz?

The correct HTTP status + a consistent JSON body: a stable machine-readable code, a human message, and field-level details for validation. application/problem+json (RFC 7807) is a common shape.A helyes HTTP status + konzisztens JSON body: stabil, géppel olvasható code, emberi message, és mezőszintű részletek a validációhoz. Gyakori formátum az application/problem+json (RFC 7807).

What should you avoid?Mit kerülj el?

  • Leaking stack traces / internals (security risk).Stack trace / belső részletek kiszivárogtatását (biztonsági kockázat).
  • Returning 200 for failures.200 visszaadását hibák esetén.
  • Inconsistent error shapes across endpoints.Endpointonként eltérő hibaformátumot.

How do you debug across services?Hogyan debugolsz szolgáltatásokon átívelően?

Propagate a correlation / trace ID through every service and include it in error responses and logs.Vigyél végig egy correlation / trace ID-t minden szolgáltatáson, és tedd bele a hibaválaszokba és a logokba.

Watch outBuktatóDistinguish client errors (4xx — don't retry) from transient server errors (5xx — retry with backoff) so clients behave correctly.Különítsd el a kliens-hibákat (4xx — ne próbáld újra) az átmeneti szerver-hibáktól (5xx — retry backoffal), hogy a kliensek helyesen viselkedjenek.
