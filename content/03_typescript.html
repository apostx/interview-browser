Structural TypingStrukturális típusosság

TypeScript compares types by shape, not by name. If an object has the required members, it is assignable — regardless of which class or interface it was declared from.A TypeScript alak szerint hasonlítja a típusokat, nem név szerint. Ha egy objektumnak megvannak a szükséges tagjai, akkor értékül adható — függetlenül attól, melyik osztályból vagy interface-ből származik.

Assignable — shapes matchÉrtékül adható — az alak egyezik
interface Point { x: number; y: number }

class Vec { constructor(
  public x: number,
  public y: number
) {} }

// never declared as Point, still fits
const p: Point = new Vec(1, 2);
The downside — no nominal safetyAz árnyoldal — nincs névbeli védelem
type UserId  = string;
type OrderId = string;

function get(id: UserId) { /* … */ }

const order: OrderId = 'o-1';
get(order); // compiles — both are string

Structural vs nominal typing — what's the practical difference?Strukturális vs nominális típusosság — mi a gyakorlati különbség?

  • In nominal systems (Java, C#) a type matches only if you declared that relationship. In TypeScript the declaration is irrelevant — only the members count. This makes typing existing JS painless, and lets you type a function's parameter without importing the class the caller happens to use.Nominális rendszerekben (Java, C#) egy típus csak akkor illeszkedik, ha kimondtad a kapcsolatot. A TypeScriptben a deklaráció lényegtelen — csak a tagok számítanak. Ettől fájdalommentes a meglévő JS típusozása, és úgy tudsz paramétert típusozni, hogy nem importálod a hívó által épp használt osztályt.

How do you get nominal behaviour when you need it?Hogyan érsz el nominális viselkedést, ha kell?

  • A branded (or "opaque") type: intersect the primitive with a marker that has no runtime existence — type UserId = string & { readonly __brand: unique symbol }. Now a raw string won't pass, and a UserId won't be accepted where an OrderId is expected. It costs a cast at the boundary where you create one.Branded (vagy „opaque") típus: metszed a primitívet egy futásidőben nem létező jelölővel — type UserId = string & { readonly __brand: unique symbol }. Így egy nyers string nem megy át, és egy UserId sem fogadható el ott, ahol OrderId a várt. Cserébe egy cast kell a határon, ahol létrehozod.

Does an object with extra properties fit?Belefér egy plusz property-vel rendelkező objektum?

  • Yes, when it arrives through a variable — extra members are allowed. But an object literal assigned directly triggers excess property checking and errors. The check exists to catch typos; assigning through a variable is the documented escape hatch.Igen, ha változón keresztül érkezik — a plusz tagok megengedettek. De egy közvetlenül értékül adott objektum-literál aktiválja az excess property checkinget, és hibát dob. Az ellenőrzés az elgépeléseket célozza; a változón át adás a dokumentált kiskapu.

Watch outBuktatóStructural typing means {} and object are far wider than people expect — {} accepts anything except null/undefined, including numbers and strings. If you mean "some object", write Record<string, unknown>.A strukturális típusosság miatt a {} és az object sokkal tágabb, mint hinnéd — a {} a null/undefined kivételével mindent elfogad, számot és stringet is. Ha „valamilyen objektumra" gondolsz, írj Record<string, unknown>-t.

type vs interfacetype vs interface

Two ways to name a type. They overlap heavily for object shapes; the differences that matter are declaration merging, what each can express, and how errors read.Két mód egy típus elnevezésére. Objektum-alakoknál nagyrészt átfedik egymást; a lényeges különbségek: a declaration merging, hogy mit tudnak kifejezni, és hogyan olvashatók a hibaüzenetek.

interfacetype
Object shapesObjektum-alakok yesigen yesigen
Unions, tuples, primitivesUnionok, tuple-ök, primitívek nonem yesigen
Mapped & conditional typesMapped & conditional típusok nonem yesigen
Declaration mergingDeclaration merging yes — reopensigen — újranyitható no — duplicate errornem — duplikált hiba
ExtendingKiterjesztés extends intersection &metszet &

Which should you reach for?Melyiket válaszd?

  • A defensible default: interface for object shapes and public APIs, type for everything else — unions, tuples, function types, mapped and conditional types. The real answer in an interview is that it's mostly a consistency decision within a codebase, and being able to say why matters more than the pick.Védhető alapértelmezés: interface objektum-alakokra és publikus API-kra, type minden másra — unionok, tuple-ök, függvénytípusok, mapped és conditional típusok. Interjún az igazi válasz az, hogy ez leginkább konzisztencia-döntés egy kódbázison belül, és többet ér elmondani a miértet, mint magát a választást.

What is declaration merging, and when is it useful?Mi a declaration merging, és mikor hasznos?

  • Declaring the same interface twice merges the members instead of erroring. It's how you augment third-party types — adding user to Express's Request, or extending Window. It's also why a public API surface is often an interface: consumers can extend it without you republishing.Ugyanannak az interface-nek a kétszeri deklarálása összefésüli a tagokat hiba helyett. Így egészítesz ki külső típusokat — user hozzáadása az Express Request-jéhez, vagy a Window bővítése. Ezért is szokott a publikus API-felület interface lenni: a fogyasztók bővíthetik anélkül, hogy újra publikálnál.

Any difference in performance or error messages?Van különbség teljesítményben vagy hibaüzenetekben?

  • Interfaces are cached by name, so deep extends chains generally check faster than long intersection chains, and errors tend to report the interface name rather than expanding the whole shape. On a large codebase this is a real ergonomics difference, not a micro-optimisation.Az interface-eket név szerint cache-eli, így a mély extends láncok általában gyorsabban ellenőrződnek, mint a hosszú metszet-láncok, és a hibák inkább az interface nevét írják ki, mint a teljes kibontott alakot. Nagy kódbázison ez valódi ergonómiai különbség, nem mikro-optimalizáció.

Watch outBuktatóDeclaration merging cuts both ways: a typo'd interface name silently merges instead of failing, and a dependency can augment your interface without you noticing. type errors loudly on redeclaration, which is occasionally what you want.A declaration merging kétélű: egy elgépelt interface-név csendben összefésül hiba helyett, és egy függőség észrevétlenül kiegészítheti a te interface-edet. A type hangosan hibázik újradeklarálásnál, ami néha épp kívánatos.

03

Union & Intersection TypesUnion & metszet típusok

A union (A | B) is "one of these"; an intersection (A & B) is "all of these at once". Unions model alternatives and drive narrowing; intersections compose shapes together.Az union (A | B) azt jelenti, „ezek egyike"; a metszet (A & B) azt, „ezek mind egyszerre". Az unionok alternatívákat modelleznek és narrowingot hajtanak; a metszetek alakokat fűznek össze.

Loose — flags that can contradictLaza — egymásnak ellentmondó flagek
interface Res {
  ok: boolean;
  data?: User;
  error?: string;
}

// nothing stops ok:true with an error
// every read needs a defensive check
Discriminated unionDiszkriminált union
type Res =
  | { ok: true;  data: User }
  | { ok: false; error: string };

if (res.ok) res.data;   // narrowed
else        res.error;  // narrowed

What makes a union "discriminated", and why does it matter?Mitől „diszkriminált" egy union, és miért számít?

  • Every member shares a common property with a distinct literal type — the discriminant. Checking it narrows the whole object, so the compiler knows which members exist in each branch. It's the single most useful modelling pattern in TypeScript: it makes illegal states unrepresentable instead of merely discouraged.Minden tag oszt egy közös property-t eltérő literál típussal — ez a diszkriminátor. Ennek vizsgálata leszűkíti az egész objektumot, így a fordító tudja, melyik ágon mely tagok léteznek. Ez a TypeScript legjobban hasznosítható modellezési mintája: az illegális állapotokat kifejezhetetlenné teszi, nem csak „nem ajánlottá".

Why does a union of objects only let you access shared properties?Miért csak a közös property-ket érheted el egy objektum-unionon?

  • Because the value is only guaranteed to be one of the members, and the compiler doesn't know which. Only the intersection of their members is safe to read without narrowing first. This trips people up because it feels backwards — a union of types gives you an intersection of properties.Mert az érték csak arra garantált, hogy a tagok egyike, és a fordító nem tudja, melyik. Csak a tagjaik metszete olvasható biztonságosan előzetes narrowing nélkül. Ez azért zavaró, mert fordítottnak érződik — típusok unionja property-k metszetét adja.

When do intersections go wrong?Mikor romlik el egy metszet?

  • When members conflict. { a: string } & { a: number } makes a of type never — the type is technically legal but no value can satisfy it, and the error surfaces far from the declaration. Intersecting function types produces an overload, which is rarely what people intend.Amikor a tagok ütköznek. A { a: string } & { a: number } az a-t never típusúvá teszi — a típus formálisan legális, de érték nem elégítheti ki, és a hiba messze a deklarációtól jelenik meg. Függvénytípusok metszete overloadot ad, ami ritkán a szándék.

Watch outBuktatóA union of a type and any collapses to any, silently deleting the other member. And string | 'literal' collapses to string — to keep editor autocomplete for known values while still allowing any string, the idiom is 'a' | 'b' | (string & {}).Egy típus és az any unionja any-vé omlik, csendben törölve a másik tagot. A string | 'literal' pedig string-gé omlik — ha meg akarod tartani az autocomplete-et az ismert értékekre, miközben bármilyen string megengedett, az idióma az 'a' | 'b' | (string & {}).

04

Narrowing & Type GuardsNarrowing & type guardok

Narrowing is the compiler tracking a value's type through control flow — after if (typeof x === 'string'), x is a string inside that branch. A type guard is a function that participates in this.A narrowing az, hogy a fordító követi egy érték típusát a vezérlési folyamon át — egy if (typeof x === 'string') után az x az adott ágon string. A type guard egy függvény, amely részt vesz ebben.

Each check removes members from the union; the type at any point is what survived the checks above it.Minden ellenőrzés tagokat távolít el az unionból; a típus egy adott ponton az, ami túlélte a felette lévő ellenőrzéseket.

What narrows, in practice?Mi szűkít a gyakorlatban?

  • typeof for primitives, instanceof for classes, in for property presence, Array.isArray, equality against a literal (the discriminated-union case), and truthiness. Also switch on a discriminant, which narrows per case.typeof primitívekre, instanceof osztályokra, in property meglétére, Array.isArray, literál elleni egyenlőség (a diszkriminált union esete), és a truthiness. Továbbá switch a diszkriminátoron, ami case-enként szűkít.

What's a user-defined type guard, and what's the catch?Mi a user-defined type guard, és mi a csapda benne?

  • A function returning x is T — function isUser(v: unknown): v is User. The compiler trusts the annotation; it does not verify that the body actually checks what it claims. A guard whose body is wrong is a silent lie that propagates everywhere it's used, which is why guards over untrusted input are better generated by a schema validator (Zod, Valibot) than hand-written.Egy x is T-t visszaadó függvény — function isUser(v: unknown): v is User. A fordító megbízik az annotációban; nem ellenőrzi, hogy a törzs tényleg azt vizsgálja-e, amit állít. Egy hibás törzsű guard csendes hazugság, ami mindenhová továbbterjed, ahol használják — ezért érdemes a nem megbízható inputra vonatkozó guardokat séma-validátorral (Zod, Valibot) generálni, nem kézzel írni.

How do you enforce that a union is handled exhaustively?Hogyan kényszeríted ki egy union teljes körű kezelését?

  • In the default branch, assign the value to never: const _x: never = value. If a new union member is added later and left unhandled, the value is no longer never and this line fails to compile — turning a runtime surprise into a build error at every switch site.A default ágon add az értéket egy never-nek: const _x: never = value. Ha később új union-tag kerül be kezeletlenül, az érték már nem never, és ez a sor fordítási hibát dob — futásidejű meglepetésből build-hiba lesz minden switch-helyen.

Watch outBuktatóNarrowing is reset by anything the compiler can't follow: a callback, an await, or reading a mutable property twice. Narrow into a const local first. And typeof null === 'object' — the classic JS wart survives into TypeScript's narrowing.A narrowingot mindaz visszaállítja, amit a fordító nem tud követni: egy callback, egy await, vagy egy módosítható property kétszeri olvasása. Szűkíts előbb egy const lokálisba. És a typeof null === 'object' — a klasszikus JS-szeplő átöröklődik a TypeScript narrowingjába is.

05

unknown vs any vs neverunknown vs any vs never

any disables checking; unknown is its safe counterpart; never is the type with no values. Choosing between the first two is one of the clearest signals of how someone writes TypeScript.Az any kikapcsolja az ellenőrzést; az unknown a biztonságos párja; a never az a típus, amelynek nincs értéke. Az első kettő közti választás az egyik legárulkodóbb jel arra, hogyan ír valaki TypeScriptet.

any
const data: any = await res.json();

data.user.name;      // no error
data.totally.wrong;  // no error
// errors surface at runtime, elsewhere
unknown
const data: unknown = await res.json();

// data.user;  ← compile error
if (isUser(data)) {
  data.name;  // safe, narrowed
}

What exactly is the difference?Mi pontosan a különbség?

  • Both accept any value. The difference is what you may then do with it: any permits every operation and every assignment out, so one any silently spreads through everything it touches. unknown permits nothing until you narrow, so the danger stays contained at the boundary where it entered.Mindkettő bármilyen értéket elfogad. A különbség az, hogy utána mit tehetsz vele: az any minden műveletet és kifelé irányuló értékadást megenged, így egy darab any csendben szétterjed mindenen, amihez hozzáér. Az unknown semmit nem enged, amíg nem szűkítesz, így a veszély a belépési ponton marad.

Where does never show up?Hol bukkan fel a never?

  • As the return type of a function that never returns normally (throws, or loops forever); as the result of an impossible narrowing; and deliberately, in exhaustiveness checks. Seeing never where you expected data usually means a union was narrowed down to nothing — often a sign the checks above contradict each other.Egy normálisan soha vissza nem térő függvény visszatérési típusaként (dob, vagy örökké ciklizál); egy lehetetlen narrowing eredményeként; és szándékosan, exhaustiveness-ellenőrzésekben. Ha ott látsz never-t, ahol adatot vártál, az általában azt jelenti, hogy egy union semmivé szűkült — gyakran annak jele, hogy a fenti ellenőrzések ellentmondanak egymásnak.

Is any ever the right call?Van, amikor az any a helyes választás?

  • Occasionally — untyped legacy modules, or generic plumbing where the constraint is genuinely inexpressible. But it should be a deliberate, localized decision, not a default. noImplicitAny catches the accidental ones; an explicit any with a comment explaining why is defensible, an implicit one rarely is.Néha — típusozatlan legacy modulok, vagy olyan generikus vezetékezés, ahol a megkötés tényleg kifejezhetetlen. De legyen tudatos, lokalizált döntés, ne alapértelmezés. A noImplicitAny elkapja a véletleneket; egy indoklással ellátott explicit any védhető, egy implicit ritkán az.

Watch outBuktatóThe biggest any source isn't your code — it's JSON.parse, res.json(), and untyped dependencies, all of which hand you any by default. That's precisely the boundary where a runtime validator belongs (see Runtime vs Compile-time).A legnagyobb any-forrás nem a saját kódod — hanem a JSON.parse, a res.json() és a típusozatlan függőségek, amelyek alapból any-t adnak. Pontosan ez az a határ, ahová futásidejű validátor való (lásd Futásidő vs fordítási idő).

Generics & ConstraintsGenerikusok & megkötések

A generic is a type parameter — a placeholder filled in at each use site, letting one function or type work over many types while preserving the relationship between input and output.A generikus egy típusparaméter — helyőrző, amit minden használati helyen kitöltenek, így egy függvény vagy típus sok típuson működik, miközben megőrzi az input és output közti kapcsolatot.

Loses the typeElveszti a típust
function first(arr: unknown[]): unknown {
  return arr[0];
}

const n = first([1, 2, 3]);
// n: unknown — caller must re-narrow
Preserves the relationshipMegőrzi a kapcsolatot
function first<T>(arr: T[]): T | undefined {
  return arr[0];
}

const n = first([1, 2, 3]);
// n: number | undefined

What are constraints for?Mire valók a megkötések?

  • <T extends X> narrows what T may be, so the body can rely on it. The classic is <T, K extends keyof T>(obj: T, key: K) => T[K] — a property getter where the compiler knows the exact return type per key, and rejects a key the object doesn't have.A <T extends X> szűkíti, mi lehet T, így a törzs támaszkodhat rá. A klasszikus a <T, K extends keyof T>(obj: T, key: K) => T[K] — property-getter, ahol a fordító kulcsonként tudja a pontos visszatérési típust, és elutasít egy nem létező kulcsot.

When is a generic the wrong tool?Mikor rossz eszköz a generikus?

  • When the parameter appears exactly once. function log<T>(x: T): void gains nothing over (x: unknown) — a generic earns its keep only when it links two positions (input to output, or two inputs to each other). A type parameter used once is a more complicated way to write the constraint itself.Amikor a paraméter pontosan egyszer szerepel. A function log<T>(x: T): void semmit nem ad a (x: unknown)-hoz képest — a generikus akkor éri meg, ha két pozíciót összeköt (inputot outputtal, vagy két inputot egymással). Az egyszer használt típusparaméter csak bonyolultabb módja magának a megkötésnek.

What are default type parameters and inference good for?Mire jók az alapértelmezett típusparaméterek és az inference?

  • Defaults (<T = string>) keep the common case terse while allowing override — this is how Request<Params, ResBody, ReqBody> in Express stays usable. And inference means callers rarely write the parameter explicitly; if they routinely have to, the signature is usually wrong.Az alapértelmezések (<T = string>) tömören tartják a gyakori esetet, miközben felülírhatók — így marad használható az Express Request<Params, ResBody, ReqBody>-ja. Az inference miatt pedig a hívók ritkán írják ki a paramétert; ha rendszeresen ki kell írniuk, általában a szignatúra rossz.

Watch outBuktatóIn a .tsx file, const f = <T>(x: T) => x parses as JSX. Write <T,> or <T extends unknown> to disambiguate — a genuinely common stumbling block in React codebases.Egy .tsx fájlban a const f = <T>(x: T) => x JSX-ként parse-olódik. Írj <T,>-t vagy <T extends unknown>-t az egyértelműsítéshez — valóban gyakori buktató React-kódbázisokban.

Why derive instead of writing the second type by hand?Miért származtass, ahelyett hogy kézzel írnád a második típust?

  • Because hand-written copies rot. If User gains a field and PublicUser was Omit<User, 'passwordHash'>, the new field appears automatically — and, importantly, if someone renames passwordHash, the Omit fails to compile rather than silently leaking it.Mert a kézzel írt másolatok elrohadnak. Ha a User új mezőt kap, és a PublicUser egy Omit<User, 'passwordHash'> volt, az új mező automatikusan megjelenik — és ami fontosabb, ha valaki átnevezi a passwordHash-t, az Omit fordítási hibát dob, nem pedig csendben kiszivárogtatja.

How would you build one yourself?Hogyan építenél sajátot?

  • With a mapped type plus modifiers: type Partial<T> = { [K in keyof T]?: T[K] }. Add readonly, or remove modifiers with -? and -readonly. Being able to write Partial from memory is a reasonable proxy for understanding mapped types at all.Mapped típussal és módosítókkal: type Partial<T> = { [K in keyof T]?: T[K] }. Adj hozzá readonly-t, vagy vedd el a módosítókat -? és -readonly segítségével. A Partial fejből leírása jó közelítés arra, hogy érti-e valaki egyáltalán a mapped típusokat.

What's the catch with Omit?Mi a csapda az Omit-tal?

  • It doesn't validate that the key exists — Omit<User, 'typo'> compiles fine and quietly omits nothing. Pick does check. If you rely on Omit for security-relevant fields, add a test or use a checked variant that constrains K extends keyof T.Nem ellenőrzi, hogy a kulcs létezik-e — az Omit<User, 'typo'> hiba nélkül lefordul, és csendben semmit nem hagy ki. A Pick viszont ellenőriz. Ha biztonsági szempontból fontos mezőknél támaszkodsz az Omit-ra, tegyél mellé tesztet, vagy használj ellenőrzött változatot, ami K extends keyof T-t köt ki.

Watch outBuktatóPartial<T> is shallow — nested objects stay required. And it's frequently misused on function arguments: making every field optional to avoid writing a constructor turns a compile-time guarantee into a pile of runtime undefined checks.A Partial<T> sekély — a beágyazott objektumok kötelezők maradnak. És gyakran visszaélnek vele függvényargumentumoknál: minden mező opcionálissá tétele, hogy ne kelljen konstruktort írni, fordítási idejű garanciát cserél futásidejű undefined-ellenőrzések halmazára.

Mapped & Conditional TypesMapped & conditional típusok

Type-level programming: mapped types transform every member of a type; conditional types branch on a type relationship using T extends U ? X : Y.Típus-szintű programozás: a mapped típusok egy típus minden tagját átalakítják; a conditional típusok típus-kapcsolat alapján ágaznak el a T extends U ? X : Y formával.

Mapped — transform membersMapped — tagok átalakítása
type Nullable<T> = {
  [K in keyof T]: T[K] | null
};

// key remapping + template literals
type Getters<T> = {
  [K in keyof T & string
    as `get${Capitalize<K>}`]: () => T[K]
};
Conditional — branch on typesConditional — típus szerinti elágazás
type Unwrap<T> =
  T extends Promise<infer U> ? U : T;

type A = Unwrap<Promise<string>>; // string
type B = Unwrap<number>;           // number

What does infer do?Mit csinál az infer?

  • It captures a type from a pattern match — "if T looks like Promise<something>, bind that something to U". It's how ReturnType, Parameters and Awaited are implemented, and it's the mechanism behind most library types that feel magical.Típust ragad meg mintaillesztésből — „ha a T úgy néz ki, mint Promise<valami>, kösd azt a valamit U-hoz". Így van megvalósítva a ReturnType, a Parameters és az Awaited, és ez a mechanizmus áll a legtöbb varázslatosnak tűnő könyvtári típus mögött.

What is distributivity, and why does it surprise people?Mi a disztributivitás, és miért lep meg sokakat?

  • A conditional type over a naked type parameter distributes across a union: Unwrap<A | B> becomes Unwrap<A> | Unwrap<B>. Usually helpful, occasionally not — wrapping both sides in a tuple ([T] extends [U]) switches it off. This is the standard answer to "why did my conditional type return a union I didn't expect".Egy csupasz típusparaméter feletti conditional típus szétoszlik az unionon: az Unwrap<A | B> Unwrap<A> | Unwrap<B> lesz. Általában hasznos, néha nem — mindkét oldal tuple-be csomagolása ([T] extends [U]) kikapcsolja. Ez a bevett válasz arra, hogy „miért adott vissza a conditional típusom egy nem várt uniont".

When is this worth it, and when is it showing off?Mikor éri meg, és mikor csak mutatvány?

  • Worth it in library or framework code, and for deriving types from a single source of truth. Showing off in application code, where a clever conditional type produces error messages nobody on the team can read. The honest interview answer names both sides — knowing the feature and knowing when not to use it are different signals.Megéri könyvtári vagy framework-kódban, és amikor egyetlen igazságforrásból származtatsz típusokat. Mutatvány alkalmazás-kódban, ahol egy ügyes conditional típus olyan hibaüzeneteket gyárt, amiket a csapatból senki nem tud elolvasni. Az őszinte interjús válasz mindkét oldalt megnevezi — a képesség ismerete és a mellőzésének ismerete külön jelzés.

Watch outBuktatóDeeply recursive conditional types are a real compile-time cost and can hit the instantiation-depth limit. If your editor gets sluggish in one file, a recursive type is a prime suspect — tsc --generateTrace will point at it.A mélyen rekurzív conditional típusok valódi fordítási költséggel járnak, és beleütközhetnek az instantiation-depth limitbe. Ha egy fájlban belassul a szerkesztő, egy rekurzív típus a fő gyanúsított — a tsc --generateTrace rámutat.

Types at the BoundaryTípusok a határon

Runtime vs Compile-timeFutásidő vs fordítási idő

Types are erased at build time — none of them exist when the code runs. Anything crossing a boundary (HTTP, database, file, message queue) is unvalidated until you validate it yourself.A típusok build időben eltűnnek — futáskor egyikük sem létezik. Bármi, ami határon lép át (HTTP, adatbázis, fájl, message queue), validálatlan, amíg te nem validálod.

Annotating a fetch result as User asserts a fact about data you haven't checked. Only a runtime validator makes it true.Egy fetch-eredmény User-ként annotálása olyan tényt állít, amit nem ellenőriztél. Csak futásidejű validátor teszi igazzá.

What survives compilation, and what doesn't?Mi éli túl a fordítást, és mi nem?

  • Erased: types, interfaces, generics, as casts, private/public modifiers. Kept, because they emit real JavaScript: enum (a runtime object), classes, and parameter properties. This is why you can't do if (x instanceof MyInterface) — there's nothing there at runtime.Eltűnik: típusok, interface-ek, generikusok, as castok, private/public módosítók. Megmarad, mert valódi JavaScriptet generál: az enum (futásidejű objektum), az osztályok és a paraméter-property-k. Ezért nem írhatsz if (x instanceof MyInterface)-t — futásidőben nincs ott semmi.

So how do you get real safety at the boundary?Akkor hogyan érsz el valódi biztonságot a határon?

  • Parse, don't assert. A schema validator (Zod, Valibot, io-ts, AJV) checks the value at runtime and infers the static type from the same schema, so there's one source of truth. const user = UserSchema.parse(json) gives you a checked value and a typed one from a single declaration.Parse-olj, ne állíts. Egy séma-validátor (Zod, Valibot, io-ts, AJV) futásidőben ellenőrzi az értéket, és ugyanabból a sémából származtatja a statikus típust, így egyetlen igazságforrás van. A const user = UserSchema.parse(json) egyetlen deklarációból ad ellenőrzött és típusozott értéket is.

What's wrong with as?Mi a baj az as-szel?

  • It's an assertion, not a conversion — it changes what the compiler believes without changing the value or checking anything. It's occasionally necessary (branded types, narrowing the compiler can't follow), but each one is a place where a runtime error can originate with no compile-time trace. as unknown as T is a double assertion and almost always a smell.Ez állítás, nem konverzió — megváltoztatja, mit hisz a fordító, anélkül hogy az értéket megváltoztatná vagy bármit ellenőrizne. Néha szükséges (branded típusok, olyan narrowing, amit a fordító nem tud követni), de mindegyik olyan hely, ahonnan futásidejű hiba indulhat fordítási nyom nélkül. Az as unknown as T dupla állítás, és szinte mindig szagos.

Watch outBuktatóThis is the single most common source of production bugs in typed codebases: an API changes shape, the annotation still claims the old one, and TypeScript reports nothing because it never looked. Validate at every boundary you don't control — including your own database rows after a migration.Ez a típusozott kódbázisok leggyakoribb produkciós hibaforrása: az API alakja megváltozik, az annotáció még a régit állítja, és a TypeScript semmit nem jelez, mert sosem nézte meg. Validálj minden határon, amit nem te felügyelsz — beleértve a saját adatbázis-soraidat is egy migráció után.

Strictness & tsconfigStrictness & tsconfig

Compiler flags decide how much the type system actually protects you. Without strict, a large share of TypeScript's value is switched off — and most of the flags that matter are inside it.A fordító flagjei döntik el, mennyire véd valójában a típusrendszer. A strict nélkül a TypeScript értékének jelentős része ki van kapcsolva — és a fontos flagek nagy része ezen belül van.

FlagFlagCatchesMit fog el
strictNullChecksthe "cannot read property of undefined" class of buga „cannot read property of undefined" hibaosztályt
noImplicitAnyuntyped parameters silently becoming anya csendben any-vé váló típusozatlan paramétereket

Which single flag matters most?Melyik az egyetlen legfontosabb flag?

  • strictNullChecks, by a distance. Without it, null and undefined are members of every type, so the compiler can't warn about the most common runtime error in JavaScript. Everything else is refinement by comparison.A strictNullChecks, jelentős előnnyel. Nélküle a null és az undefined minden típus tagja, így a fordító nem tud figyelmeztetni a JavaScript leggyakoribb futásidejű hibájára. Minden más ehhez képest finomhangolás.

How do you enable strict on an existing loose codebase?Hogyan kapcsolod be a strictet egy meglévő laza kódbázison?

  • Incrementally — turning it on globally in a large project produces thousands of errors and gets reverted. Options: enable one flag at a time; use tsc --noEmit in CI as a ratchet so the count can only go down; or adopt per-directory configs and move the boundary. The point worth making in an interview is that it's a migration, not a config change.Fokozatosan — nagy projekten a globális bekapcsolás több ezer hibát ad, és visszavonják. Lehetőségek: egyszerre egy flag; tsc --noEmit a CI-ban racsniként, hogy a szám csak csökkenhessen; vagy könyvtárankénti configok és a határ tologatása. Interjún azt érdemes kimondani, hogy ez migráció, nem config-változtatás.

What does tsc actually guarantee about your build?Mit garantál valójában a tsc a buildedről?

  • Less than people assume. Type checking and transpiling are separable: esbuild, SWC and Babel strip types without checking them, which is why a project can build green and still be full of type errors. If your bundler doesn't check, tsc --noEmit has to run in CI or nothing does.Kevesebbet, mint hinnék. A típusellenőrzés és a transzpilálás szétválasztható: az esbuild, az SWC és a Babel ellenőrzés nélkül vágja le a típusokat, ezért fordulhat le zölden egy projekt, miközben tele van típushibával. Ha a bundlered nem ellenőriz, a tsc --noEmit-nek kell futnia a CI-ban, különben semmi nem ellenőriz.

Watch outBuktatónoUncheckedIndexedAccess is the flag people forget: without it, arr[999] is typed as present, and Record<string, User> claims every key exists. It's outside strict, so enabling strict doesn't give it to you.A noUncheckedIndexedAccess az elfelejtett flag: nélküle az arr[999] létezőként típusozódik, és a Record<string, User> azt állítja, minden kulcs létezik. Nincs benne a strict-ben, tehát a strict bekapcsolása nem adja meg.

Typing Node & ExpressNode & Express típusozása

Server-side TypeScript concentrates on three things: typing the request/response cycle, keeping validated input and static types in sync, and handling errors — which arrive as unknown.A szerveroldali TypeScript három dologra koncentrál: a request/response ciklus típusozása, a validált input és a statikus típusok szinkronban tartása, és a hibakezelés — a hibák ugyanis unknown-ként érkeznek.

Asserted — a lie the compiler believesÁllítás — hazugság, amit a fordító elhisz
app.post('/users', (req, res) => {
  const body = req.body as CreateUser;
  // req.body is `any`; the cast checks
  // nothing. Malformed input flows on.
  save(body);
});
Parsed — one source of truthParse-olva — egyetlen igazságforrás
const CreateUser = z.object({
  email: z.string().email(),
  age:   z.number().int().positive()
});
type CreateUser = z.infer<typeof CreateUser>;

app.post('/users', (req, res) => {
  const body = CreateUser.parse(req.body);
  save(body); // checked AND typed
});

Why is catch (e) typed unknown?Miért unknown típusú a catch (e)?

  • Because JavaScript lets you throw anything — a string, a number, undefined. Under useUnknownInCatchVariables (part of strict), TypeScript reflects that honestly, so you must narrow with e instanceof Error before reading e.message. Older code annotated it any and read .message off a string.Mert a JavaScriptben bármit dobhatsz — stringet, számot, undefined-ot. A useUnknownInCatchVariables alatt (a strict része) a TypeScript ezt őszintén tükrözi, így e instanceof Error-ral kell szűkítened, mielőtt olvasnád az e.message-et. A régebbi kód any-vel annotálta, és stringről olvasta a .message-et.

How do you type req.user added by auth middleware?Hogyan típusozod az auth middleware által hozzáadott req.user-t?

  • Declaration merging into Express's namespace — declare global { namespace Express { interface Request { user?: User } } }. Note the ?: it's genuinely absent before the middleware runs, and typing it as required is a lie that bites on unauthenticated routes. The stricter alternative is a separate AuthedRequest type used only on guarded handlers.Declaration merging az Express névterébe — declare global { namespace Express { interface Request { user?: User } } }. Figyeld a ?-et: a middleware lefutása előtt tényleg nincs ott, és kötelezőként típusozni olyan hazugság, ami a nem autentikált route-okon üt vissza. A szigorúbb alternatíva egy külön AuthedRequest típus, csak a védett handlereken.

What about types from the database layer?Mi a helyzet az adatbázis-réteg típusaival?

  • Prefer generated over hand-written: Prisma, Kysely and drizzle derive types from the schema, so a migration that renames a column breaks the build instead of production. Where types are hand-written (raw SQL, older ORMs), they are an assumption about the database — the same boundary problem as an HTTP response.Inkább generált, mint kézzel írt: a Prisma, a Kysely és a drizzle a sémából származtatja a típusokat, így egy oszlopot átnevező migráció a buildet töri el, nem a produkciót. Ahol kézzel írtak (nyers SQL, régebbi ORM-ek), ott a típus feltételezés az adatbázisról — ugyanaz a határprobléma, mint egy HTTP-válasznál.

Watch outBuktatóAn async Express handler that rejects does not reach your error middleware before Express 5 — the promise is unhandled and the request hangs. Wrap handlers, or upgrade. This is a favourite interview question precisely because the type system cannot warn you about it.Egy elutasító async Express handler az Express 5 előtt nem éri el a hibakezelő middleware-t — a promise kezeletlen marad, és a kérés lefagy. Csomagold be a handlereket, vagy frissíts. Pont azért kedvelt interjúkérdés, mert a típusrendszer nem tud figyelmeztetni rá.

Typing ReactReact típusozása

In React, TypeScript's job is mostly props, state and events — plus making invalid prop combinations impossible rather than merely documented.Reactben a TypeScript dolga főként a propok, a state és az események típusozása — plusz az érvénytelen prop-kombinációk lehetetlenné tétele, nem csak dokumentálása.

Documented, not enforcedDokumentált, nem kikényszerített
interface Props {
  variant: 'link' | 'button';
  href?: string;   // "required if link"
  onClick?: () => void;
}
// nothing stops variant='link' with no href
Impossible to misuseLehetetlen rosszul használni
type Props =
  | { variant: 'link';   href: string }
  | { variant: 'button'; onClick: () => void };

// omitting href on a link is now
// a compile error at the call site

How do you type children, events and refs?Hogyan típusozod a childrent, az eseményeket és a refeket?

  • Children as React.ReactNode. Events by their specific type — React.ChangeEvent<HTMLInputElement>, React.FormEvent<HTMLFormElement> — which gives you a typed e.target.value. Refs as useRef<HTMLDivElement>(null), remembering .current is nullable until mount.A children React.ReactNode. Az események a saját típusukkal — React.ChangeEvent<HTMLInputElement>, React.FormEvent<HTMLFormElement> —, ami típusozott e.target.value-t ad. A refek useRef<HTMLDivElement>(null), észben tartva, hogy a .current mountig nullable.

Why did React.FC fall out of favour?Miért ment ki a divatból a React.FC?

  • It used to add an implicit children prop to every component, including ones that should reject children — removed in React 18's types. It also complicates generic components. The current convention is to annotate the props parameter and let the return type infer: function Card(props: Props).Régen minden komponenshez implicit children propot adott, azokhoz is, amelyeknek el kellene utasítaniuk — a React 18 típusaiban eltávolították. A generikus komponenseket is bonyolítja. A mai konvenció: annotáld a props paramétert, és hagyd a visszatérési típust következtetni — function Card(props: Props).

How do you extend a native element's props?Hogyan bővíted egy natív elem propjait?

  • ComponentPropsWithoutRef<'button'> — it picks up every valid attribute, so consumers can pass aria-label, disabled or type without you enumerating them. Combine with Omit when you're overriding one, and use the WithRef variant when forwarding a ref.ComponentPropsWithoutRef<'button'> — felveszi az összes érvényes attribútumot, így a fogyasztók átadhatnak aria-label-t, disabled-t vagy type-ot anélkül, hogy felsorolnád őket. Kombináld Omit-tal, ha felülírsz egyet, és használd a WithRef változatot ref továbbításánál.

Watch outBuktatóuseState([]) infers never[], so every later push fails — annotate as useState<Item[]>([]). Same trap with useState(null), which infers null rather than T | null.A useState([]) never[]-et következtet, így minden későbbi push hibázik — annotáld useState<Item[]>([])-ként. Ugyanez a csapda a useState(null)-lal, ami null-t következtet T | null helyett.
