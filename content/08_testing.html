01

The Test PyramidA teszt-piramis

A model for how many tests of each kind to write: many fast unit tests at the base, fewer integration tests in the middle, very few slow end-to-end tests at the top.Egy modell arról, hány tesztet írj mindegyik fajtából: sok gyors unit teszt az alapon, kevesebb integrációs teszt középen, nagyon kevés lassú end-to-end teszt a csúcson.

Cost and runtime grow upward; confidence that the whole system works grows with them.A költség és a futásidő felfelé nő; a bizalom, hogy az egész rendszer működik, velük együtt nő.

Why not just write end-to-end tests?Miért ne írjunk csak end-to-end teszteket?

  • They're slow, flaky and expensive to maintain, and when one fails it doesn't tell you where the bug is. Unit tests are fast and pinpoint the failure; E2E tests prove the pieces actually work together. You need both, in different proportions.Lassúak, flaky-k és drágán karbantarthatók, és ha elbukik egy, nem mondja meg, hol a hiba. A unit tesztek gyorsak és pontosan megmutatják a hibát; az E2E tesztek bizonyítják, hogy a darabok tényleg működnek együtt. Mindkettő kell, eltérő arányban.

What's the “testing trophy” alternative?Mi a „testing trophy” alternatíva?

  • A newer model that puts the most weight on integration tests rather than unit tests, arguing they give the best confidence-per-effort for typical web apps. In a React/Node stack it's often the more practical shape.Egy újabb modell, amely a legnagyobb súlyt az integrációs tesztekre helyezi a unit tesztek helyett, azzal érvelve, hogy tipikus webappoknál ezek adják a legjobb bizalom/ráfordítás arányt. React/Node stacken gyakran ez a praktikusabb forma.

What actually makes a test valuable?Mitől értékes ténylegesen egy teszt?

  • It fails when the behaviour breaks and passes otherwise. A test that breaks on every refactor (too coupled to implementation) or that never fails (asserting nothing meaningful) is a maintenance cost with no return.Elbukik, amikor a viselkedés elromlik, és átmegy egyébként. Egy teszt, amely minden refaktornál eltörik (túlságosan az implementációhoz kötött), vagy amely sosem bukik el (semmi értelmeset nem állít), karbantartási költség hozam nélkül.

Watch outBuktatóChasing 100% coverage produces tests written to touch lines rather than to verify behaviour. Coverage tells you what isn't tested; it says nothing about whether what is tested is tested well.A 100%-os lefedettség hajszolása olyan teszteket szül, amelyek sorokat érintenek, nem viselkedést ellenőriznek. A lefedettség megmondja, mi nincs tesztelve; arról semmit nem mond, hogy ami tesztelve van, az jól van-e tesztelve.

02

Unit TestingUnit tesztelés

Testing one unit of behaviour in isolation — fast, deterministic, and run on every save. The base of the pyramid and the tests you'll write most often.Egy viselkedési egység izolált tesztelése — gyors, determinisztikus, és minden mentésnél fut. A piramis alapja, és ezekből írsz majd a legtöbbet.

What makes a good unit test?Mitől jó egy unit teszt?

  • Fast (milliseconds), isolated (no DB, network or filesystem), deterministic (same result every run), and focused — one behaviour per test, so a failure names the problem.Gyors (ezredmásodpercek), izolált (nincs DB, hálózat vagy fájlrendszer), determinisztikus (minden futásnál ugyanaz), és fókuszált — egy viselkedés tesztenként, hogy egy bukás megnevezze a problémát.

What is the Arrange-Act-Assert structure?Mi az Arrange-Act-Assert szerkezet?

  • Arrange the inputs and state, Act by calling the thing under test, Assert the outcome. Keeping the three visually separate makes tests readable and stops you from asserting mid-setup.Arrange — állítsd be az inputokat és állapotot, Act — hívd meg a tesztelt dolgot, Assert — ellenőrizd az eredményt. A három vizuális elkülönítése olvashatóvá teszi a teszteket és megakadályozza, hogy setup közben állíts.

Should you test implementation or behaviour?Az implementációt vagy a viselkedést teszteld?

  • Behaviour — the public contract. Tests that assert on internal state, private methods or call order break on every refactor even when the behaviour is unchanged, which trains the team to ignore failing tests.A viselkedést — a nyilvános szerződést. Azok a tesztek, amelyek belső állapotra, privát metódusokra vagy hívási sorrendre állítanak, minden refaktornál eltörnek, akkor is, ha a viselkedés változatlan — ez arra tanítja a csapatot, hogy figyelmen kívül hagyja a bukó teszteket.
Coupled to implementationImplementációhoz kötött
// breaks if you rename the helper
// or reorder internal calls
it('adds a user', () => {
  const spy = jest.spyOn(svc, '_validate');
  svc.addUser({ name: 'Ann' });
  expect(spy).toHaveBeenCalled();
  expect(svc._cache.size).toBe(1);
});
Tests behaviourViselkedést tesztel
it('adds a user', async () => {
  // Arrange
  const svc = makeService();
  // Act
  await svc.addUser({ name: 'Ann' });
  // Assert — the observable outcome
  expect(await svc.findByName('Ann'))
    .toEqual({ id: expect.any(String),
               name: 'Ann' });
});

Watch outBuktatóA test with no assertion, or one that only checks a mock was called, proves nothing about correctness. Ask of each test: “if I broke this feature, would this test fail?” If not, it isn't earning its keep.Egy teszt állítás nélkül, vagy amely csak azt ellenőrzi, hogy egy mockot meghívtak, semmit nem bizonyít a helyességről. Kérdezd meg minden tesztről: „ha elrontanám ezt a feature-t, elbukna ez a teszt?” Ha nem, akkor nem termeli ki magát.

Test Doubles: Mocks, Stubs & FakesTeszt-helyettesítők: mockok, stubok & fake-ek

Stand-ins for real dependencies — mocks, stubs, spies, fakes. They isolate the unit under test and make slow or unpredictable dependencies deterministic.Helyettesítők valódi függőségekhez — mockok, stubok, spy-ok, fake-ek. Izolálják a tesztelt egységet, és determinisztikussá teszik a lassú vagy kiszámíthatatlan függőségeket.

DoubleHelyettesítőWhat it doesMit csinálUse whenMikor
StubStubreturns canned valueselőre megadott értékeket ad visszayou need an input from a dependencyinput kell egy függőségtől
SpySpyrecords how it was calledrögzíti, hogyan hívtákthe call itself is the outcomea hívás maga az eredmény
MockMockpre-programmed with expectationselőre beprogramozott elvárásokkalverifying an interactioninterakció ellenőrzése
FakeFakea working lightweight implementationműködő könnyűsúlyú implementációin-memory DB, test servermemóriabeli DB, teszt-szerver

What should you mock — and what shouldn't you?Mit mockolj — és mit ne?

  • Mock things that are slow, non-deterministic or have side effects you don't want: network calls, time, randomness, payment providers. Don't mock the thing you're testing, pure functions, or your own data structures — mocking those tests nothing.Mockold azt, ami lassú, nemdeterminisztikus, vagy nem kívánt mellékhatása van: hálózati hívások, idő, véletlen, fizetési szolgáltatók. Ne mockold a tesztelt dolgot, a tiszta függvényeket, vagy a saját adatszerkezeteidet — ezek mockolása semmit nem tesztel.

Why prefer a fake over a mock where possible?Miért jobb a fake a mocknál, ahol lehet?

  • A fake behaves like the real thing, so the test exercises real logic and survives refactoring. A mock encodes your assumption about how the dependency behaves — and if that assumption is wrong, the test passes while production breaks.Egy fake úgy viselkedik, mint a valódi, így a teszt valódi logikát futtat és túléli a refaktorálást. Egy mock a te feltételezésedet kódolja arról, hogyan viselkedik a függőség — és ha ez a feltételezés hibás, a teszt átmegy, miközben a production eltörik.
Controlling time and networkIdő és hálózat kontrollálásadeterministic, no real I/O
// deterministic time
jest.useFakeTimers().setSystemTime(new Date('2026-01-01'));

// stub a module's network call
jest.mock('./api');
api.fetchUser.mockResolvedValue({ id: '1', name: 'Ann' });

// spy: the call IS the behaviour under test
const send = jest.fn();
await notifyUser({ send }, 'welcome');
expect(send).toHaveBeenCalledWith(
  expect.objectContaining({ template: 'welcome' })
);

Watch outBuktatóOver-mocking produces tests that pass against a fantasy version of your dependencies. If a test mocks everything the unit touches, it verifies only that your code calls the functions you told it to call — a tautology. Prefer real objects, then fakes, then mocks.A túl-mockolás olyan teszteket szül, amelyek a függőségeid képzeletbeli verziója ellen mennek át. Ha egy teszt mindent mockol, amit az egység érint, csak azt ellenőrzi, hogy a kódod azokat a függvényeket hívja, amiket mondtál neki — tautológia. Előbb valódi objektumok, aztán fake-ek, aztán mockok.

04

Integration TestingIntegrációs tesztelés

Testing several units together, including real collaborators — a route with its service and database, or a component with its store. Where most real bugs are actually caught.Több egység együttes tesztelése, valódi közreműködőkkel — egy route a szolgáltatásával és adatbázisával, vagy egy komponens a store-jával. Itt fogódik meg a legtöbb valódi bug.

Why do integration tests catch what unit tests miss?Miért fogják meg az integrációs tesztek azt, amit a unit tesztek nem?

  • Because most bugs live in the seams — wrong assumptions about a dependency's contract, serialization mismatches, incorrect SQL, missing middleware. Each unit can be individually correct while the composition is broken.Mert a legtöbb bug az illesztéseknél él — hibás feltételezések egy függőség szerződéséről, szerializációs eltérések, hibás SQL, hiányzó middleware. Minden egység lehet külön-külön helyes, miközben az összeállítás hibás.

How do you handle the database?Hogyan kezeled az adatbázist?

  • Use a real database of the same engine (Testcontainers or a dedicated test DB) rather than mocking the driver — mocked SQL proves nothing. Isolate tests by wrapping each in a transaction that rolls back, or by truncating between tests.Használj valódi, azonos motorú adatbázist (Testcontainers vagy dedikált teszt-DB) a driver mockolása helyett — a mockolt SQL semmit nem bizonyít. Izoláld a teszteket úgy, hogy mindegyiket tranzakcióba csomagolod, amely visszagördül, vagy truncate-elsz a tesztek között.

How do you test an Express API?Hogyan tesztelsz egy Express API-t?

  • With supertest against the app instance — no need to bind a port. You exercise the real router, middleware, validation and error handling, asserting on status codes and response bodies.supertest-tel az app instance ellen — nem kell portot foglalni. A valódi routert, middleware-t, validációt és hibakezelést futtatod, státuszkódokra és válasz-törzsekre állítva.
API integration test with supertestAPI integrációs teszt supertesttelreal router + real DB
import request from 'supertest';
import { app } from '../app.js';

describe('POST /api/orders', () => {
  it('rejects an unauthenticated request', async () => {
    await request(app)
      .post('/api/orders')
      .send({ item: 'book' })
      .expect(401);
  });

  it('creates an order for an authenticated user', async () => {
    const res = await request(app)
      .post('/api/orders')
      .set('Authorization', `Bearer ${testToken}`)
      .send({ item: 'book' })
      .expect(201);

    expect(res.body).toMatchObject({ item: 'book', status: 'pending' });
    // verify it actually persisted
    const saved = await db.orders.findById(res.body.id);
    expect(saved).not.toBeNull();
  });
});

Watch outBuktatóTests that share mutable state (a seeded row, a global counter) pass alone and fail when run together or in a different order. Each test must set up and tear down its own data — and never depend on execution order.Azok a tesztek, amelyek mutálható állapoton osztoznak (egy beszúrt sor, egy globális számláló), egyedül átmennek, és együtt vagy más sorrendben futtatva elbuknak. Minden tesztnek a saját adatát kell felépítenie és lebontania — és sosem függhet a végrehajtási sorrendtől.

05

End-to-End TestingEnd-to-end tesztelés

Driving the real application through the UI as a user would — a browser against a running stack. The highest confidence and the highest cost.A valódi alkalmazás vezérlése a UI-on át, ahogy egy user tenné — egy böngésző egy futó stack ellen. A legmagasabb bizalom és a legmagasabb költség.

What should E2E tests cover?Mit fedjenek le az E2E tesztek?

  • Only the critical user journeys — signup, login, checkout, the core workflow. A handful of paths that must never break. Everything else belongs lower in the pyramid where it runs faster and fails more precisely.Csak a kritikus felhasználói utakat — regisztráció, bejelentkezés, fizetés, az alapvető workflow. Néhány út, amely sosem törhet el. Minden más lejjebb tartozik a piramisban, ahol gyorsabban fut és pontosabban bukik.

Why are E2E tests flaky, and how do you fix it?Miért flaky-k az E2E tesztek, és hogyan javítod?

  • Mostly timing: the test acts before the app is ready. The fix is never a fixed sleep — wait for a condition (an element to be visible, a request to settle). Modern tools (Playwright, Cypress) auto-wait on assertions, which removes most flakiness.Főleg időzítés: a teszt cselekszik, mielőtt az app kész lenne. A javítás sosem fix sleep — várj egy feltételre (elem láthatóvá válása, kérés befejeződése). A modern eszközök (Playwright, Cypress) automatikusan várnak az állításoknál, ami a flakiness nagy részét megszünteti.

How should you select elements?Hogyan válaszd ki az elemeket?

  • By role or accessible name where possible, or a dedicated data-testid. Selecting by CSS class or DOM position couples the test to styling, so a purely visual change breaks the suite.Szerep vagy akadálymentes név szerint, ahol lehet, vagy dedikált data-testid-vel. A CSS osztály vagy DOM-pozíció szerinti kiválasztás a stílushoz köti a tesztet, így egy tisztán vizuális változás eltöri a suite-ot.

Watch outBuktatóA flaky test is worse than no test — the team learns to re-run until green, and real failures get ignored. Quarantine or fix flaky tests immediately; never let “just retry it” become the norm.Egy flaky teszt rosszabb, mint a semmilyen — a csapat megtanulja újrafuttatni, amíg zöld nem lesz, és a valódi hibák elsikkadnak. Karanténozd vagy javítsd azonnal a flaky teszteket; sose engedd, hogy a „futtasd újra” legyen a norma.

06

Testing React ComponentsReact komponensek tesztelése

Testing components the way a user interacts with them — querying by visible text and roles, firing real events, asserting on what's rendered rather than on internal state.Komponensek tesztelése úgy, ahogy egy user használja őket — látható szöveg és szerepek szerint keresve, valódi eseményeket kiváltva, a renderelt dologra állítva, nem a belső állapotra.

What's the guiding principle of React Testing Library?Mi a React Testing Library vezérelve?

  • The more your tests resemble how the software is used, the more confidence they give. So you query what the user perceives (text, labels, roles) and never reach into component internals, props or state.Minél jobban hasonlítanak a tesztjeid arra, ahogyan a szoftvert használják, annál nagyobb bizalmat adnak. Ezért azt keresed, amit a user érzékel (szöveg, címkék, szerepek), és sosem nyúlsz a komponens belsőségeibe, propjaiba vagy állapotába.

Which query should you reach for?Melyik query-t használd?

  • Prefer getByRole (which also checks accessibility), then getByLabelText for form fields, then getByText. Use getByTestId only as a last resort. findBy* is the async variant; queryBy* returns null instead of throwing, for asserting absence.Előnyben a getByRole (amely az akadálymentességet is ellenőrzi), aztán getByLabelText űrlapmezőkhöz, aztán getByText. A getByTestId-t csak végső esetben. A findBy* az aszinkron változat; a queryBy* null-t ad dobás helyett, hiány ellenőrzéséhez.

How do you test asynchronous UI?Hogyan tesztelsz aszinkron UI-t?

  • Use findBy* or waitFor, which retry until the assertion passes or times out. Mock the network at the boundary (MSW intercepts requests) rather than mocking your fetch wrapper — the component then exercises its real data flow.Használj findBy*-ot vagy waitFor-t, amelyek újrapróbálnak, amíg az állítás átmegy vagy lejár. A hálózatot a határon mockold (az MSW elfogja a kéréseket), ne a fetch wrapperedet — a komponens így a valódi adatfolyamát futtatja.
Component test with RTLKomponens-teszt RTL-lelqueries by role/label, not by class
import { render, screen } from '@testing-library/react';
import userEvent from '@testing-library/user-event';

it('shows an error when the email is invalid', async () => {
  const user = userEvent.setup();
  render(<SignupForm onSubmit={jest.fn()} />);

  await user.type(screen.getByLabelText(/email/i), 'not-an-email');
  await user.click(screen.getByRole('button', { name: /sign up/i }));

  // findBy* waits for the async validation to render
  expect(await screen.findByText(/enter a valid email/i))
    .toBeInTheDocument();
});

Watch outBuktatóAsserting on internal state or calling a component's methods directly means the test breaks whenever you refactor, even if the UI is identical. And userEvent is preferable to fireEvent: it simulates the full interaction (focus, keydown, input) rather than dispatching a single synthetic event.A belső állapotra állítás vagy egy komponens metódusainak közvetlen hívása azt jelenti, a teszt minden refaktornál eltörik, akkor is, ha a UI azonos. És a userEvent jobb a fireEvent-nél: a teljes interakciót szimulálja (fókusz, keydown, input), nem egyetlen szintetikus eseményt küld.

07

Test-Driven DevelopmentTest-Driven Development

Write a failing test, make it pass, then refactor — red, green, refactor. The tests drive the design rather than being written after the fact.Írj egy bukó tesztet, hozd zöldre, aztán refaktorálj — piros, zöld, refaktor. A tesztek vezérlik a tervezést, nem utólag íródnak.

What does TDD actually buy you?Mit ad valójában a TDD?

  • Testable design (code written test-first tends to have fewer hidden dependencies), a guaranteed-meaningful test suite (every test failed at least once), and a tight feedback loop. The tests are a by-product; the design pressure is the point.Tesztelhető tervezést (a teszt-először írt kódnak jellemzően kevesebb rejtett függősége van), garantáltan értelmes teszt-suite-ot (minden teszt legalább egyszer elbukott), és szoros visszacsatolási kört. A tesztek melléktermékek; a tervezési nyomás a lényeg.

Why must the test fail first?Miért kell a tesztnek először elbuknia?

  • A test that has never failed might be asserting nothing, testing the wrong thing, or not running at all. Seeing red first proves the test can detect the absence of the behaviour.Egy teszt, amely sosem bukott el, lehet, hogy semmit nem állít, rossz dolgot tesztel, vagy egyáltalán nem fut. A piros látása először bizonyítja, hogy a teszt képes észlelni a viselkedés hiányát.

Is TDD always appropriate?Mindig helyénvaló a TDD?

  • It fits well when the requirements are clear and the logic is meaningful. It fits poorly for exploratory work, UI layout experiments, or spikes where you don't yet know the shape of the solution — there, write tests once the design settles.Jól illik, ha a követelmények tisztázottak és a logika érdemi. Rosszul illik felderítő munkához, UI-elrendezési kísérletekhez, vagy spike-okhoz, ahol még nem tudod a megoldás formáját — ott írj teszteket, ha a terv letisztult.

Watch outBuktatóTDD isn't “write all the tests first.” It's one small test at a time, in a tight loop. Writing thirty tests up front against an unbuilt design produces the same brittle, coupled suite as writing them afterwards.A TDD nem azt jelenti, hogy „írd meg előbb az összes tesztet”. Egy kis teszt egyszerre, szoros körben. Harminc teszt előre megírása egy meg nem épített terv ellen ugyanolyan törékeny, összecsatolt suite-ot ad, mint az utólagos írás.

08

CI & Quality GatesCI & quality gate-ek

Running the suite automatically on every change, with rules about what may merge — the mechanism that keeps a test suite meaningful over time.A suite automatikus futtatása minden változtatásnál, szabályokkal arról, mi mergelhető — ez a mechanizmus tartja értelmesen a teszt-suite-ot hosszú távon.

What belongs in a CI pipeline?Mi tartozik egy CI pipeline-ba?

  • Install with a lockfile (npm ci), lint, typecheck, unit and integration tests, a build, then E2E against the built artifact. Order matters — run the fast, cheap checks first so failures surface in seconds, not minutes.Telepítés lockfile-lal (npm ci), lint, típusellenőrzés, unit és integrációs tesztek, build, majd E2E az elkészült artifact ellen. A sorrend számít — előbb a gyors, olcsó ellenőrzések fussanak, hogy a hibák másodpercek alatt kiderüljenek, ne percek alatt.

How do you keep the pipeline fast?Hogyan tartod gyorsan a pipeline-t?

  • Cache dependencies, run test files in parallel, shard the E2E suite across machines, and run only affected tests on a PR where the tooling supports it. A suite people wait ten minutes for is a suite people work around.Cache-eld a függőségeket, futtasd párhuzamosan a teszt-fájlokat, oszd szét az E2E suite-ot gépek között, és futtasd csak az érintett teszteket egy PR-en, ahol az eszköz támogatja. Egy suite, amire tíz percet várnak, olyan suite, amit megkerülnek.

What is a useful quality gate?Mi egy hasznos quality gate?

  • A green suite as a merge requirement, plus type checks and lint. Coverage thresholds are better as a “don't decrease” rule on changed files than as a global percentage, which incentivizes empty tests.Zöld suite merge-feltételként, plusz típusellenőrzés és lint. A lefedettségi küszöbök jobbak „ne csökkenjen” szabályként a változtatott fájlokon, mint globális százalékként, amely üres tesztekre ösztönöz.

Watch outBuktatóIf tests only run locally, they will eventually stop being run. And a suite that's allowed to stay red — “that one always fails” — has already stopped providing signal; fix or delete it.Ha a tesztek csak lokálisan futnak, előbb-utóbb abbamarad a futtatásuk. És egy suite, amely pirosan maradhat — „az mindig elbukik” —, már megszűnt jelzést adni; javítsd vagy töröld.
