Event Loop (Node)

Node runs your JS on a single thread; the event loop (via libuv) processes callbacks in ordered phases, one cycle at a time.A Node egyetlen szálon futtatja a JS-edet; az event loop (a libuv-on keresztül) rendezett fázisokban dolgozza fel a callbackeket, ciklusonként.

Each phase runs its callback queue; nextTick + Promise microtasks drain between them.Minden fázis lefuttatja a saját callback-sorát; a nextTick és a Promise microtaskok közöttük futnak le.

How does it differ from the browser event loop?Miben tér el a böngésző event loopjától?

  • Browser: task → drain microtasks → render. Node has multiple phases (timers, poll, check…), no rendering, and adds process.nextTick as a separate, higher-priority queue.Böngésző: task → microtaskok lefutnak → render. A Node-nak több fázisa van (timers, poll, check…), nincs renderelés, és a process.nextTick külön, magasabb prioritású sorként jelenik meg.

Where do Promises and nextTick run?Hol futnak a Promise-ok és a nextTick?

  • After each operation/phase transition, Node drains the process.nextTick queue first, then the Promise microtask queue — before continuing.Minden művelet/fázisváltás után a Node először a process.nextTick sort futtatja le, utána a Promise microtask-sort — csak azután megy tovább.

Watch outBuktató“Single-threaded” refers to your JS. libuv and the OS handle the actual I/O, so one Node process serves thousands of concurrent connections without a thread per request.Az „egyszálú” a te JS-edre vonatkozik. A tényleges I/O-t a libuv és az OS kezeli, így egyetlen Node process több ezer párhuzamos kapcsolatot szolgál ki anélkül, hogy kérésenként szálat indítana.

02

libuv & Thread Pool

libuv is the C library behind Node's event loop and async I/O. Most I/O is OS-level async; a few operations use a small thread pool.A libuv az a C könyvtár, ami a Node event loopja és aszinkron I/O-ja mögött áll. Az I/O nagy része OS-szintű aszinkron; néhány művelet egy kis thread poolt használ.

Only pool-backed ops (fs, DNS, some crypto/zlib) are limited by the thread count.Csak a pool-alapú műveleteket (fs, DNS, egyes crypto/zlib) korlátozza a szálak száma.

What actually uses the thread pool?Mi használja ténylegesen a thread poolt?

  • File system, DNS lookups (dns.lookup), some crypto (e.g. pbkdf2) and zlib. Network sockets do not — they're OS-async (epoll/kqueue/IOCP).A fájlrendszer, a DNS lookupok (dns.lookup), egyes crypto műveletek (pl. pbkdf2) és a zlib. A hálózati socketek nem — azok OS-aszinkronok (epoll/kqueue/IOCP).
  • Default pool size is 4 (UV_THREADPOOL_SIZE, up to 1024).Az alapértelmezett pool-méret 4 (UV_THREADPOOL_SIZE, akár 1024-ig).

Why does this matter?Miért számít ez?

  • CPU-heavy pool work (many crypto hashes) can exhaust the 4 threads and stall unrelated fs/DNS work — tune the pool or move to worker threads.A CPU-igényes pool-munka (sok crypto hash) kimerítheti a 4 szálat, és megakaszthat tőle független fs/DNS műveleteket — hangold a poolt vagy válts worker threadekre.

Watch outBuktatóNetwork requests scale far beyond the pool because sockets are OS-async — only pool-backed operations are capped by UV_THREADPOOL_SIZE.A hálózati kérések jóval a poolon túl skálázódnak, mert a socketek OS-aszinkronok — csak a pool-alapú műveleteket korlátozza az UV_THREADPOOL_SIZE.

nextTick / setImmediate / setTimeout

Three ways to defer work, each with different timing and priority.Három módszer a munka elhalasztására, mindegyik más időzítéssel és prioritással.

APIRunsMikor futPriorityPrioritás
process.nextTick before the loop continues (after current op)mielőtt a ciklus továbbmenne (az aktuális művelet után) highestlegmagasabb
Promise.then microtask, right after the nextTick queuemicrotask, közvetlenül a nextTick sor után highmagas

setImmediate vs setTimeout(0) — which fires first?setImmediate vs setTimeout(0) — melyik fut előbb?

  • Inside an I/O callback, setImmediate always wins (check comes right after poll). At the top level, the order is non-deterministic.Egy I/O callbacken belül a setImmediate mindig nyer (a check közvetlenül a poll után jön). Legfelső szinten a sorrend nem determinisztikus.

04

Blocking the Event LoopAz event loop blokkolása

Any long synchronous work blocks the single JS thread — heavy loops, sync crypto, parsing huge payloads, sync fs. No other request is served until it finishes.Bármilyen hosszú szinkron munka blokkolja az egyetlen JS szálat — nehéz ciklusok, szinkron crypto, hatalmas payloadok parse-olása, szinkron fs. Amíg be nem fejeződik, egyetlen másik kérés sem szolgálódik ki.

How do you avoid it?Hogyan kerülöd el?

  • Never use *Sync fs/crypto methods in request handlers; offload CPU-bound work to Worker Threads or a separate service; stream large data instead of buffering; paginate and limit.Soha ne használj *Sync fs/crypto metódusokat request handlerben; a CPU-igényes munkát told Worker Threadekre vagy külön szolgáltatásra; a nagy adatot streameld ahelyett, hogy pufferelnéd; lapozz és limitálj.

How do you detect it?Hogyan észleled?

  • Event-loop lag metrics (perf_hooks.monitorEventLoopDelay), an APM tool, or clinic.js.Event-loop lag metrikákkal (perf_hooks.monitorEventLoopDelay), APM eszközzel vagy a clinic.js-szel.

Watch outBuktatóOne slow synchronous handler degrades the whole process for every user — the single most common Node performance pitfall.Egyetlen lassú szinkron handler az egész processzt lerontja minden felhasználó számára — ez a leggyakoribb Node performance-buktató.

05

Workers, Cluster & Child Process

Three ways to use more than one thread or process — Worker Threads (CPU work), Cluster (scale a server across cores), Child Process (run other programs).Három módszer több szál vagy processz használatára — Worker Threads (CPU-munka), Cluster (szerver skálázása magokra), Child Process (más programok futtatása).

Cluster workers share the listening port; the OS/master load-balances connections.A cluster workerek közösen használják a figyelő portot; az OS/master osztja el a kapcsolatokat.

CPU-bound vs scaling — which do you reach for?CPU-igényes munka vs skálázás — melyikhez nyúlsz?

  • CPU-bound task (image processing, crypto) → Worker Threads (real threads, same process).CPU-igényes feladat (képfeldolgozás, crypto) → Worker Threads (valódi szálak, ugyanabban a processzben).
  • Use all cores for a web server → Cluster (forked processes sharing a port) — or run N containers behind a load balancer, common in production.Az összes mag kihasználása webszerverhez → Cluster (forkolt processzek közös porton) — vagy N konténer egy load balancer mögött, ez a gyakoribb éles környezetben.

Why not just use threads everywhere?Miért ne használjunk mindenhol szálakat?

  • Node's model is built for I/O-bound concurrency on one thread; threads/processes add memory + coordination cost. Use them for CPU work, not I/O.A Node modellje I/O-igényes párhuzamosságra épül egyetlen szálon; a szálak/processzek memória- és koordinációs költséget adnak. CPU-munkára használd őket, ne I/O-ra.

Watch outBuktatóCluster workers don't share memory — shared state needs an external store (Redis). In k8s, horizontal pod scaling usually replaces cluster.A cluster workerek nem osztanak memóriát — a közös állapothoz külső tár kell (Redis). Kubernetesben a horizontális pod-skálázás általában kiváltja a clustert.

06

CommonJS vs ES Modules

Two module systems. CJS (require/module.exports) — synchronous, Node's original. ESM (import/export) — the standard, static, async.Két modulrendszer. CJS (require/module.exports) — szinkron, a Node eredeti megoldása. ESM (import/export) — a szabvány, statikus, aszinkron.

Key differences?Fő különbségek?

  • CJS loads synchronously and resolves at runtime; ESM is static (imports hoisted, analyzable, tree-shakeable).A CJS szinkron tölt be és futásidőben oldja fel; az ESM statikus (az importok hoistolódnak, elemezhetők, tree-shakelhetők).
  • ESM has top-level await and import.meta.url; CJS has __dirname/__filename. ESM exports are live bindings; CJS exports are a value snapshot.Az ESM-ben van top-level await és import.meta.url; a CJS-ben __dirname/__filename. Az ESM exportok élő kötések; a CJS exportok érték-pillanatképek.

How do you pick per file?Hogyan választasz fájlonként?

  • .mjs or "type": "module" → ESM; .cjs or default → CJS. ESM can import CJS; CJS importing ESM needs dynamic import()..mjs vagy "type": "module" → ESM; .cjs vagy alapértelmezés → CJS. Az ESM tud CJS-t importálni; a CJS-ből ESM-hez dinamikus import() kell.

Watch outBuktatóYou can't require() an ESM module — mixing the two trips people up. New projects default to ESM.Egy ESM modult nem tudsz require()-tel behúzni — a kettő keverése sokakat megfog. Új projektek alapból ESM-et használnak.

07

Streams & Backpressure

Streams process data in chunks instead of loading it all into memory. Four types: Readable, Writable, Duplex, Transform.A streamek darabokban dolgozzák fel az adatot ahelyett, hogy az egészet memóriába töltenék. Négy típus: Readable, Writable, Duplex, Transform.

When the sink can't keep up, the source pauses until the buffer drains.Ha a nyelő nem bírja a tempót, a forrás szünetel, amíg a buffer ki nem ürül.

Why streams?Miért streamek?

  • Constant memory for arbitrarily large data (files, HTTP bodies, uploads), start processing before all data arrives, and composability via pipe/pipeline.Állandó memóriahasználat tetszőlegesen nagy adatnál (fájlok, HTTP bodyk, feltöltések), a feldolgozás megkezdhető, mielőtt az összes adat megérkezne, és komponálhatóság a pipe/pipeline révén.

What is backpressure?Mi az a backpressure?

  • When a writable can't keep up, its buffer fills. pipeline() pauses the source until the destination drains — preventing unbounded memory growth.Amikor egy writable nem bírja a tempót, megtelik a buffere. A pipeline() szünetelteti a forrást, amíg a cél ki nem ürül — így nem nő korlátlanul a memóriahasználat.

Watch outBuktatóUse stream.pipeline(), not raw .pipe() — pipeline handles backpressure and propagates errors + cleans up. Plain .pipe() silently drops errors.stream.pipeline()-t használj, ne sima .pipe()-ot — a pipeline kezeli a backpressure-t és továbbítja a hibákat + takarít. A sima .pipe() csendben elnyeli a hibákat.

08

EventEmitter

Node's pub/sub primitive: objects emit named events, listeners subscribe with .on(). Much of core (streams, HTTP server, process) is built on it.A Node pub/sub primitívje: az objektumok névvel ellátott eseményeket bocsátanak ki, a listenerek .on()-nal iratkoznak fel. A core nagy része (streamek, HTTP szerver, process) erre épül.

Is it sync or async?Szinkron vagy aszinkron?

  • Listeners run synchronously in registration order when emit() fires — it's not a queue; emit blocks until every listener returns.A listenerek szinkron futnak, regisztrációs sorrendben, amikor az emit() elsül — ez nem sor; az emit blokkol, amíg minden listener vissza nem tér.

Common pitfalls?Gyakori buktatók?

  • Leaks from listeners never removed (the “MaxListenersExceededWarning” at >10). Use .once() for one-shot and remove listeners on cleanup.Leakek a soha el nem távolított listenerekből (a „MaxListenersExceededWarning” 10 fölött). Egyszeri eseményre .once(), és cleanupkor távolítsd el a listenereket.

Watch outBuktatóAn 'error' event with no listener throws and can crash the process — always attach an error listener to emitters and streams.Egy listener nélküli 'error' esemény dob, és összeomlaszthatja a processzt — mindig tegyél error listenert az emitterekre és a streamekre.

Callbacks → Promises → async/await

The evolution of async in Node: error-first callbacks → Promises → async/await syntax over Promises.Az async fejlődése a Node-ban: error-first callbackek → Promise-ok → async/await szintaxis a Promise-ok fölött.

What problem did each solve?Melyik milyen problémát oldott meg?

  • Callbacks → “callback hell” (nesting, inversion of control). Promises → chaining and a single error path. async/await → synchronous-looking code with try/catch.Callbackek → „callback hell” (egymásba ágyazás, inversion of control). Promise-ok → láncolás és egyetlen hibaút. async/await → szinkronnak kinéző kód try/catch-csel.

How do async functions work?Hogyan működnek az async függvények?

  • An async function always returns a Promise; await pauses it until the Promise settles — without blocking the event loop, so other work runs meanwhile.Egy async függvény mindig Promise-t ad vissza; az await szünetelteti, amíg a Promise le nem zárul — az event loop blokkolása nélkül, így közben más munka is fut.

Watch outBuktatóawait in a loop serializes independent operations — use Promise.all for parallelism. And forgetting to await returns a pending Promise (a silent bug).A ciklusban lévő await sorosítja a független műveleteket — párhuzamossághoz Promise.all. Ha pedig elfelejted az await-et, egy függőben lévő Promise-t kapsz vissza (csendes bug).

Async Error HandlingAsync hibakezelés

Errors must be caught on the right path — a sync throw, a rejected Promise, or an error-first callback.A hibákat a megfelelő úton kell elkapni — szinkron throw, elutasított Promise, vagy error-first callback.

How do you catch each?Melyiket hogyan kapod el?

  • async/await → try/catch. Promises → .catch(). Callbacks → check the first arg. Streams/emitters → an 'error' listener.async/await → try/catch. Promise-ok → .catch(). Callbackek → ellenőrizd az első argumentumot. Streamek/emitterek → 'error' listener.

The dangerous cases?A veszélyes esetek?

  • Unhandled rejections (process.on('unhandledRejection')) — which crash by default in current Node — and uncaught exceptions.A kezeletlen rejectek (process.on('unhandledRejection')) — amik a mai Node-ban alapból összeomlást okoznak — és az elkapatlan exceptionök.

Watch outBuktatóA throw inside a setTimeout/callback isn't caught by an outer try/catch — the try already returned. Let a top-level handler log & exit; don't try to “resume” after an uncaught exception — process state is unknown.Egy setTimeout-ban/callbackben dobott hibát nem kap el a külső try/catch — a try már visszatért. Hagyd, hogy egy top-level handler logoljon és kilépjen; ne próbálj „folytatni” egy elkapatlan exception után — a processz állapota ismeretlen.

Concurrency Control

Running multiple async operations with the right amount of parallelism.Több aszinkron művelet futtatása a megfelelő mértékű párhuzamossággal.

Promise combinators?Promise kombinátorok?

  • all — all succeed, or reject on first failure. allSettled — wait for all, get each result/error. race — first to settle. any — first to fulfill.all — mind sikerül, vagy az első hibánál elutasít. allSettled — megvárja mindet, minden eredményt/hibát megkapsz. race — az első, ami lezárul. any — az első, ami sikeresen teljesül.

Why limit concurrency?Miért korlátozzuk a párhuzamosságot?

  • Firing 10,000 requests/queries at once exhausts sockets, connections and memory. Cap in-flight work with a limiter (p-limit) or batching.Ha egyszerre indítasz 10 000 kérést/lekérdezést, kimeríted a socketeket, a kapcsolatokat és a memóriát. Korlátozd a folyamatban lévő munkát limiterrel (p-limit) vagy batcheléssel.

Watch outBuktatóPromise.all rejects on the first failure, but the other promises keep running — they aren't cancelled. Use allSettled when you need every result regardless of individual failures.A Promise.all az első hibánál elutasít, de a többi promise tovább fut — nem szakadnak meg. Használj allSettled-et, ha minden eredmény kell, függetlenül az egyedi hibáktól.

SQL vs NoSQL

Relational (SQL) vs non-relational (NoSQL) databases — chosen by data model and access pattern.Relációs (SQL) vs nem-relációs (NoSQL) adatbázisok — az adatmodell és a hozzáférési minta alapján választunk.

TypeTípus ModelModell Best forMire jó ExamplesPéldák

Document flexible JSON documentsrugalmas JSON dokumentumok evolving schemas, nested dataváltozó sémák, ágyazott adat MongoDB
Key-value key → value, very fastkulcs → érték, nagyon gyors caching, sessions, high throughputcaching, sessionök, nagy átbocsátás Redis, DynamoDB

How do you choose?Hogyan választasz?

  • SQL by default for most apps (relations, transactions, mature tooling). Reach for NoSQL for a specific access pattern: schema flexibility, massive horizontal scale, or key-based/high-write workloads.A legtöbb alkalmazáshoz alapból SQL (relációk, tranzakciók, érett eszközkészlet). NoSQL-hez konkrét hozzáférési minta miatt nyúlj: séma-rugalmasság, hatalmas horizontális skála, vagy kulcs-alapú/sok írású terhelés.

Adatmodell alapján válassz, ne hype alapján; sok rendszer mindkettőt használja (polyglot persistence).

Database IndexingAdatbázis-indexelés

A data structure (usually a B-tree) that lets the DB find rows without scanning the whole table — like a book's index.Olyan adatszerkezet (általában B-fa), amivel az adatbázis a teljes tábla átolvasása nélkül találja meg a sorokat — mint egy könyv tárgymutatója.

When does an index help, and when does it hurt?Mikor segít az index, és mikor árt?

  • Helps: WHERE, JOIN, ORDER BY on the indexed column(s) — turns an O(n) scan into an O(log n) lookup.Segít: WHERE, JOIN, ORDER BY az indexelt oszlop(ok)on — az O(n) végigolvasásból O(log n) keresés lesz.
  • Hurts: every write must update the index → slower writes + storage. Unused indexes are pure cost.Árt: minden írásnak frissítenie kell az indexet → lassabb írás + tárhely. A használatlan index tiszta költség.

Composite index / column order?Összetett index / oszlopsorrend?

  • A multi-column index (a, b) serves queries on a or a + b (leftmost-prefix rule), but not on b alone.Egy többoszlopos (a, b) index kiszolgálja az a vagy a + b szerinti lekérdezéseket (leftmost-prefix szabály), de a csak b szerintit nem.

Watch outBuktatóIndexes don't help low-selectivity columns (e.g. a boolean) or leading-wildcard LIKE '%x'. Use EXPLAIN ANALYZE to confirm one is actually used.Az index nem segít alacsony szelektivitású oszlopoknál (pl. egy boolean) vagy vezető joker karakteres LIKE '%x' esetén. EXPLAIN ANALYZE-zal ellenőrizd, hogy tényleg használja-e.

Transactions & ACID

A transaction groups operations so they succeed or fail together. ACID = Atomicity, Consistency, Isolation, Durability.A tranzakció úgy csoportosítja a műveleteket, hogy együtt sikerüljenek vagy együtt bukjanak. ACID = Atomicity, Consistency, Isolation, Durability.

Isolation levelIzolációs szint PreventsMit előz meg
Read Uncommitted— (allows dirty reads)— (enged dirty readet)
Read Committeddirty reads · Postgres defaultdirty read · a Postgres alapértelmezése
Repeatable Read+ non-repeatable reads+ non-repeatable read

What does each ACID letter mean?Mit jelent az ACID minden betűje?

  • Atomicity — all-or-nothing. Consistency — constraints stay valid. Isolation — concurrent txns don't corrupt each other. Durability — committed data survives crashes.Atomicity — mindent vagy semmit. Consistency — a megszorítások érvényesek maradnak. Isolation — a párhuzamos tranzakciók nem rontják el egymást. Durability — a commitolt adat túléli az összeomlást.

Watch outBuktatóHigher isolation trades throughput for safety (more locking). And long-running transactions hold locks and bloat — keep them short.A magasabb izoláció átbocsátást áldoz a biztonságért (több lockolás). A hosszan futó tranzakciók pedig lockokat tartanak és bloatot okoznak — tartsd őket rövidre.

N+1 Query Problem

Fetching N parent rows, then firing one query per parent for its children → 1 + N queries instead of one.N szülősort kérsz le, majd szülőnként egy-egy lekérdezést indítasz a gyerekeikért → 1 + N lekérdezés egy helyett.

Invisible with 10 rows in dev; catastrophic with 10,000 in prod.10 sorral fejlesztésben láthatatlan; 10 000-rel élesben katasztrofális.

Where does it come from, and how do you fix it?Honnan jön, és hogyan javítod?

  • Classic with ORMs + lazy loading: loop over users, access user.posts → a query each.Klasszikus ORM + lazy loading eset: végigmész a usereken, eléred a user.posts-ot → mindegyikhez egy lekérdezés.
  • Fix: eager-load with a JOIN, or batch into one WHERE ... IN (...) (DataLoader batches + caches per request).Javítás: eager loading JOIN-nal, vagy batchelés egyetlen WHERE ... IN (...)-be (a DataLoader kérésenként batchel és cache-el).

Watch outBuktatóInspect query counts and use EXPLAIN — the fix is ORM eager-loading (include/join), not more indexes.Nézd meg a lekérdezésszámokat és használj EXPLAIN-t — a megoldás az ORM eager loadingja (include/join), nem több index.

16

Query Plans & EXPLAINQuery planek & EXPLAIN

The planner turns your SQL into an execution strategy, and EXPLAIN shows you which one it picked. SQL says what you want; the plan is how the database decided to get it — and that decision is cost-based, not fixed.A planner végrehajtási stratégiává alakítja az SQL-edet, az EXPLAIN pedig megmutatja, melyiket választotta. Az SQL azt mondja meg, mit akarsz; a plan azt, hogyan döntött az adatbázis — és ez a döntés költségalapú, nem rögzített.

Why EXPLAIN ANALYZE rather than plain EXPLAIN?Miért EXPLAIN ANALYZE a sima EXPLAIN helyett?

  • Plain EXPLAIN shows only the planner's estimate. ANALYZE actually runs the query and reports real timings and real row counts, so you can compare them. Why would the database ignore an index you created?Miért hagyná figyelmen kívül az adatbázis az általad létrehozott indexet?

  • Common causes: the query returns a large fraction of the table (a scan is genuinely cheaper); a function or cast on the column (WHERE lower(email) = … can't use a plain index on email); a type mismatch between the parameter and the column; leading-wildcard LIKE; or stale statistics after a bulk load. The fix depends on which — an expression index for the first, ANALYZE for the last.Gyakori okok: a lekérdezés a tábla nagy hányadát adja vissza (a scan tényleg olcsóbb); függvény vagy cast az oszlopon (a WHERE lower(email) = … nem tud sima email indexet használni); típuseltérés a paraméter és az oszlop között; vezető joker karakteres LIKE; vagy elavult statisztikák tömeges betöltés után. A javítás az októl függ — az elsőre kifejezés-index, az utolsóra ANALYZE.

How do you read a plan without drowning in it?Hogyan olvasol planet anélkül, hogy belefulladnál?

  • Read innermost-outward — nodes execute bottom-up. Look for three things in order: the node with the largest actual time, any estimate-vs-actual mismatch above ~10×, and unexpected Seq Scan on a big table. Also check loops — a fast node run 50,000 times is the real cost, and that is often an N+1 in disguise.Belülről kifelé olvasd — a csomópontok alulról felfelé hajtódnak végre. Három dolgot keress ebben a sorrendben: a legnagyobb tényleges idejű csomópontot, a ~10× feletti becslés-tényleges eltérést, és a váratlan Seq Scan-t nagy táblán. Nézd a loops értéket is — egy gyors, de 50 000-szer lefutó csomópont a valódi költség, és ez gyakran álcázott N+1.

Csomagold tranzakcióba, és görgesd vissza. Továbbá a planek eltérnek dev és produkció között, mert az adatmennyiségtől és a statisztikáktól függenek: egy 100 soron jól kinéző plan szinte semmit nem mond 10 millióról.

JOINs, Aggregation & CTEsJOIN-ok, aggregáció & CTE-k

The query constructs interviews actually test: which join keeps which rows, why filtering in WHERE versus ON changes the result, and when a CTE or window function replaces application-side looping.Azok a lekérdezés-szerkezetek, amiket az interjúk tényleg kérdeznek: melyik join mely sorokat tartja meg, miért változtat az eredményen a WHERE vagy ON szerinti szűrés, és mikor vált ki egy CTE vagy ablakfüggvény alkalmazásoldali ciklust.

Silently an INNER JOINCsendben INNER JOIN

What's the actual difference between the join types?Mi a tényleges különbség a join-típusok között?

  • INNER keeps only matching pairs. LEFT keeps all left rows, padding the right with NULL. RIGHT is the mirror image and is rarely worth using — flip the tables instead and keep every join in the codebase reading the same direction. FULL OUTER keeps both sides. CROSS is the cartesian product, occasionally deliberate (generating date series), usually a missing join condition.Az INNER csak az illeszkedő párokat tartja meg. A LEFT az összes bal oldali sort megtartja, a jobb oldalt NULL-lal tölti. A RIGHT ennek tükörképe, és ritkán érdemes használni — inkább fordítsd meg a táblákat, hogy a kódbázisban minden join ugyanabba az irányba olvasson. A FULL OUTER mindkét oldalt megtartja. A CROSS a descartes-szorzat, néha szándékos (dátumsorok generálása), általában hiányzó join-feltétel.

When does a CTE help, and does it cost anything?Mikor segít egy CTE, és van-e költsége?

  • CTEs (WITH x AS (…)) name intermediate steps, making a multi-stage query readable, and WITH RECURSIVE handles trees and graphs — org charts, category hierarchies, threaded comments — in one statement rather than a loop of queries. Historically in Postgres a CTE was an optimisation fence (materialised, never inlined); since Postgres 12 it can be inlined, and you can force either with MATERIALIZED / NOT MATERIALIZED. Knowing that shift is a good depth signal.A CTE-k (WITH x AS (…)) elnevezik a köztes lépéseket, olvashatóvá téve a többlépcsős lekérdezést, a WITH RECURSIVE pedig fákat és gráfokat kezel — szervezeti ábrák, kategória-hierarchiák, szálas kommentek — egyetlen utasításban, lekérdezés-ciklus helyett. A Postgresben a CTE korábban optimalizációs kerítés volt (materializált, sosem beágyazott); a Postgres 12 óta beágyazható, és a MATERIALIZED / NOT MATERIALIZED kulcsszóval bármelyik kikényszeríthető. Ennek a váltásnak az ismerete jó mélységi jelzés.

What do window functions give you over GROUP BY?Mit adnak az ablakfüggvények a GROUP BY-hoz képest?

  • GROUP BY collapses rows; a window function computes across a set of rows while keeping every row. That's how you get "each order plus its rank within the customer" or a running total in one pass. ROW_NUMBER() OVER (PARTITION BY user_id ORDER BY created_at DESC) filtered to = 1 is the standard "latest row per group" solution — a very common interview question, and much faster than a correlated subquery.A GROUP BY összevonja a sorokat; az ablakfüggvény sorok halmazán számol, miközben minden sort megtart. Így kapod meg egy menetben, hogy „minden rendelés a vevőn belüli rangsorával együtt", vagy a futó összeget. A ROW_NUMBER() OVER (PARTITION BY user_id ORDER BY created_at DESC) = 1-re szűrve a bevett „csoportonkénti legutolsó sor" megoldás — nagyon gyakori interjúkérdés, és jóval gyorsabb, mint egy korrelált alkérdés.

Watch outBuktatóAggregates skip NULL, and this bites: COUNT(col) counts non-null values while COUNT(*) counts rows, so a LEFT JOIN plus COUNT(*) reports 1 for parents with zero children. Use COUNT(child.id). Also, a join that multiplies rows silently inflates every SUM above it — aggregate in a subquery before joining.Az aggregátumok kihagyják a NULL-t, és ez visszaüt: a COUNT(col) a nem-null értékeket számolja, a COUNT(*) a sorokat, így egy LEFT JOIN plusz COUNT(*) 1-et jelent a gyermek nélküli szülőkre. Használj COUNT(child.id)-t. Továbbá egy sorokat sokszorozó join csendben felfújja a felette lévő összes SUM-ot — aggregálj alkérdésben, mielőtt joinolnál.

Locking & DeadlocksLockolás & deadlockok

Locks are how a database enforces isolation: a writer blocks other writers on the same row until it commits. A deadlock is two transactions each holding what the other needs — a cycle no amount of waiting resolves.A lockok révén kényszeríti ki az adatbázis az izolációt: egy író blokkolja a többi írót ugyanazon a soron, amíg nem commitol. A deadlock az, amikor két tranzakció mindegyike azt tartja, amire a másiknak szüksége van — olyan kör, amit semennyi várakozás nem old fel.

Deadlocks are detected, not prevented: the database picks a victim and aborts it. Handling that error is the application's job.A deadlockokat észlelik, nem előzik meg: az adatbázis kiválaszt egy áldozatot, és megszakítja. A hiba kezelése az alkalmazás dolga.

Optimistic or pessimistic locking — how do you choose?Optimista vagy pesszimista lockolás — hogyan választasz?

  • Pessimistic takes the lock up front (SELECT … FOR UPDATE) — correct under real contention, but holds the lock for the whole transaction and can queue everyone behind a slow request. Optimistic takes no lock and checks a version column on write (WHERE id = ? AND version = ?); if zero rows updated, someone beat you and you retry. Optimistic wins when conflicts are rare; pessimistic wins when they're common, because retry storms cost more than waiting.A pesszimista előre megszerzi a lockot (SELECT … FOR UPDATE) — valós versengésnél helyes, de a teljes tranzakció idejére tartja a lockot, és mindenkit besorolhat egy lassú kérés mögé. Az optimista nem lockol, hanem íráskor ellenőriz egy version oszlopot (WHERE id = ? AND version = ?); ha nulla sor frissült, valaki megelőzött, és újrapróbálsz. Az optimista akkor nyer, ha a konfliktus ritka; a pesszimista akkor, ha gyakori, mert az újrapróbálkozás-viharok többe kerülnek, mint a várakozás.

How do you actually avoid deadlocks?Hogyan kerülöd el ténylegesen a deadlockokat?

  • The one reliable rule: acquire locks in a consistent order everywhere — if every transaction touches accounts in ascending id order, the cycle can't form. Beyond that: keep transactions short, don't do network calls or user interaction inside one, and touch the fewest rows you can. Then still handle the deadlock error and retry, because you cannot eliminate them entirely.Az egyetlen megbízható szabály: mindenhol konzisztens sorrendben szerezd meg a lockokat — ha minden tranzakció növekvő id-sorrendben nyúl a számlákhoz, a kör nem alakulhat ki. Ezen túl: tartsd rövidre a tranzakciókat, ne legyen bennük hálózati hívás vagy felhasználói interakció, és a lehető legkevesebb sort érintsd. Aztán a deadlock-hibát így is kezeld és próbáld újra, mert teljesen kiiktatni nem lehet.

What are FOR UPDATE and SKIP LOCKED for?Mire való a FOR UPDATE és a SKIP LOCKED?

  • SELECT … FOR UPDATE locks the selected rows so a concurrent transaction can't change them before you write — the standard read-modify-write guard. FOR UPDATE SKIP LOCKED steps over rows someone else already locked, which turns a table into a work queue: many workers each grab different jobs with no coordination and no double-processing. It's the simplest correct job queue you can build without adding a broker.A SELECT … FOR UPDATE lockolja a kiválasztott sorokat, hogy egy párhuzamos tranzakció ne módosíthassa őket az írásod előtt — ez a bevett read-modify-write védelem. A FOR UPDATE SKIP LOCKED átlép a más által már lockolt sorokon, ami munkasorrá alakítja a táblát: sok worker mindegyike más feladatot ragad meg, koordináció és kettős feldolgozás nélkül. Ez a legegyszerűbb helyes job queue, amit broker hozzáadása nélkül építhetsz.

Watch outBuktatóAn ORM can create lock ordering you never wrote — saveAll on a collection may update rows in hash order, differing per process. Also, a deadlock error is retryable and a lock timeout usually isn't; treating them identically either loses work or hammers the database. And an unindexed foreign key makes a delete on the parent lock far more of the child table than expected.Egy ORM olyan lock-sorrendet hozhat létre, amit sosem írtál — egy kollekción futó saveAll hash-sorrendben frissíthet, ami processzenként eltér. Továbbá a deadlock-hiba újrapróbálható, a lock timeout általában nem; ha azonosan kezeled őket, vagy munkát veszítesz, vagy az adatbázist ostromolod. Egy indexeletlen idegen kulcs pedig a szülőn futó törléskor a vártnál sokkal nagyobb részét lockolja a gyermektáblának.

Which difference causes the most damage in practice?Melyik különbség okozza a legtöbb kárt a gyakorlatban?

  • Oracle treating '' as NULL. Logic that distinguishes "not provided" from "provided but blank" silently changes meaning on migration, and WHERE col = '' matches nothing in Oracle. And the transaction defaults?És a tranzakciós alapértelmezések?

  • Oracle opens a transaction implicitly and requires an explicit COMMIT; Postgres autocommits each statement unless you open a block. DDL is the sharper edge: Postgres can roll back a CREATE TABLE inside a transaction, Oracle commits implicitly around DDL — so a migration script that assumes all-or-nothing behaves very differently on each.Az Oracle implicit nyit tranzakciót, és explicit COMMIT-ot vár; a Postgres minden utasítást autocommitál, hacsak nem nyitsz blokkot. Do you need to rewrite application code, or just SQL?Az alkalmazáskódot is át kell írni, vagy csak az SQL-t?

  • Usually more than SQL. Identifier case folding differs (Postgres folds to lower, Oracle to upper, both fold unless quoted), so a driver returning ID instead of id breaks object mapping across the whole codebase. Date and NUMBER handling differ. And anything in PL/SQL — packages, triggers, stored logic — has no automatic equivalent; that's typically the largest and least visible part of the estimate.Általában többet az SQL-nél. Az azonosítók kis/nagybetű-kezelése eltér (a Postgres kisbetűre, az Oracle nagybetűre hajt, idézőjel nélkül mindkettő hajt), így egy id helyett ID-t visszaadó driver az egész kódbázisban töri az objektum-leképezést. A dátum- és NUMBER-kezelés eltér. És bármi, ami PL/SQL-ben van — csomagok, triggerek, tárolt logika — nem rendelkezik automatikus megfelelővel; jellemzően ez a becslés legnagyobb és legkevésbé látható része.

Watch outBuktatóBoth use MVCC, but Oracle's default isolation is Read Committed with statement-level consistency and no reader-blocks-writer, while Postgres implements Repeatable Read as snapshot isolation — so a workload tuned against one may see different anomalies on the other even at a nominally identical isolation level.Mindkettő MVCC-t használ, de az Oracle alapértelmezett izolációja Read Committed utasítás-szintű konzisztenciával és olvasó-nem-blokkol-írót viselkedéssel, míg a Postgres a Repeatable Read-et snapshot izolációként valósítja meg — így egy az egyikre hangolt terhelés a másikon névlegesen azonos izolációs szinten is más anomáliákat láthat.

Connection Pooling

Reusing a fixed set of open DB connections across requests instead of opening/closing one per query.Egy fix készletnyi nyitott adatbázis-kapcsolat újrahasznosítása a kérések között ahelyett, hogy lekérdezésenként nyitnánk/zárnánk egyet.

The pool caps total connections and skips the per-query TCP + auth handshake.A pool korlátozza az összes kapcsolatot, és megspórolja a lekérdezésenkénti TCP + auth handshake-et.

Why pool?Miért poolozunk?

  • Opening a DB connection is expensive (TCP + auth). A pool keeps connections warm; requests borrow and return them, capping the total the DB must handle.Egy adatbázis-kapcsolat megnyitása drága (TCP + auth). A pool melegen tartja a kapcsolatokat; a kérések kölcsönzik és visszaadják őket, így korlátozódik, mennyit kell az adatbázisnak kezelnie.

How do you size it?Hogyan méretezed?

  • Bounded by the DB's max connections and CPU — bigger isn't better. Too many connections thrash the DB; a modest pool per instance often wins.Az adatbázis max kapcsolatszáma és CPU-ja határolja — a nagyobb nem jobb. A túl sok kapcsolat megfojtja az adatbázist; instance-onként egy mérsékelt pool gyakran jobb.

Watch outBuktatóPool exhaustion (all borrowed, none returned) makes requests hang — usually from leaked connections or long queries. In serverless, use a pooler like PgBouncer to avoid connection storms.A pool kimerülése (minden kikölcsönözve, semmi visszaadva) függő kéréseket okoz — általában elszivárgott kapcsolatok vagy hosszú lekérdezések miatt. Serverlessben használj poolert (pl. PgBouncer), hogy elkerüld a kapcsolat-viharokat.

Graceful Shutdown

On a termination signal, stop taking new work, finish in-flight requests, close resources, then exit — instead of dropping connections mid-flight.Leállító szignálra ne vegyél fel új munkát, fejezd be a folyamatban lévő kéréseket, zárd le az erőforrásokat, majd lépj ki — ahelyett, hogy menet közben ejtenéd a kapcsolatokat.

What's the sequence?Mi a sorrend?

  • Catch SIGTERM → server.close() (stop accepting new connections) → let in-flight requests finish (with a timeout) → close DB pools / flush queues → exit(0). Force-exit if it hangs.Kapd el a SIGTERM-et → server.close() (ne fogadj új kapcsolatot) → hagyd befejeződni a folyamatban lévő kéréseket (timeouttal) → zárd le a DB poolokat / ürítsd a sorokat → exit(0). Ha beragad, kényszerítsd a kilépést.

Why does it matter?Miért számít?

  • In k8s/containers, deploys send SIGTERM. Without graceful shutdown you drop live requests and corrupt in-flight work on every rollout.Kubernetesben/konténerekben a deploy SIGTERM-et küld. Graceful shutdown nélkül minden rolloutnál élő kéréseket ejtesz el, és megsérül a folyamatban lévő munka.

Watch outBuktatóAlways include a hard timeout — a hung connection shouldn't block shutdown forever (the orchestrator will SIGKILL after the grace period anyway).Mindig tegyél bele kemény timeoutot — egy beragadt kapcsolat ne blokkolja örökre a leállást (az orchestrator a grace period után úgyis SIGKILL-t küld).

Logging & Observability

The three pillars: logs (events), metrics (numbers over time), traces (a request's path across services).A három pillér: logok (események), metrikák (számok az idő függvényében), trace-ek (egy kérés útja a szolgáltatásokon át).

Logging best practices in Node?Logging best practice-ek Node-ban?

  • Structured JSON logs (pino/winston), log levels, a correlation/request ID per request, and no secrets/PII. Write to stdout and let the platform ship them.Strukturált JSON logok (pino/winston), log szintek, kérésenként correlation/request ID, és semmi titok/PII. Írj stdout-ra, a szállítást bízd a platformra.

What are traces for?Mire valók a trace-ek?

  • Following one request across services (OpenTelemetry) — essential for debugging latency and errors in a distributed system.Egy kérés követésére a szolgáltatásokon át (OpenTelemetry) — elengedhetetlen a latencia és a hibák debugolásához elosztott rendszerben.

Watch outBuktatóconsole.log is synchronous and can block under load, and unstructured logs are hard to query — use a proper structured logger to stdout.A console.log szinkron, és terhelés alatt blokkolhat, a strukturálatlan logokat pedig nehéz lekérdezni — használj rendes strukturált loggert stdout-ra.

Env Config & 12-Factor

Config (secrets, URLs, flags) belongs in the environment, not in code — one build runs unchanged in every environment.A konfiguráció (titkok, URL-ek, flagek) a környezetbe való, nem a kódba — ugyanaz a build fut változtatás nélkül minden környezetben.

Key principles?Fő elvek?

  • Config in env vars (never committed); secrets in a secret manager (Vault / AWS Secrets Manager / KMS); validate config at startup and fail fast on missing vars.A konfig env változókban (soha ne commitold); a titkok secret managerben (Vault / AWS Secrets Manager / KMS); induláskor validáld a konfigot, és hiányzó változóra azonnal bukj el.

Why?Miért?

  • The same artifact promotes dev → staging → prod with only env differences, keeps secrets out of source control, and makes the app portable/containerizable.Ugyanaz az artifact megy dev → staging → prod útvonalon, csak a környezeti eltérésekkel; a titkok kimaradnak a verziókezelésből; az alkalmazás pedig hordozható/konténerezhető lesz.

Watch outBuktatóCommitting a .env with real secrets is a classic leak — gitignore it, rotate anything exposed, and inject secrets at deploy time.Valódi titkokat tartalmazó .env commitolása klasszikus szivárgás — tedd gitignore-ba, rotálj mindent, ami kikerült, és deploy időben injektáld a titkokat.
