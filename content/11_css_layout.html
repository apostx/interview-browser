Box Model & SizingDoboz-modell & méretezés

Every element is a box: content, padding, border, margin. The only question that matters in practice is what width actually measures — and the answer depends on box-sizing.Minden elem egy doboz: tartalom, padding, border, margó. A gyakorlatban az egyetlen fontos kérdés az, hogy a width valójában mit mér — és a válasz a box-sizing-tól függ.

content-box (default)(alapértelmezett)
.card {
  width: 300px;
  padding: 20px;
  border: 1px solid;
}
/* actual rendered width: 342px
   300 + 20*2 + 1*2 */
border-box
*, *::before, *::after {
  box-sizing: border-box;
}
.card { width: 300px; padding: 20px; }
/* actual rendered width: 300px
   padding eats into the content */

Why does nearly every codebase set border-box globally?Miért állít szinte minden kódbázis globális border-box-ot?

  • Because with content-box, adding padding changes an element's outer size, so any layout built from percentages breaks the moment you add breathing room. border-box makes width mean "the space this occupies", which is what you almost always intend.Mert content-box esetén a padding hozzáadása megváltoztatja az elem külső méretét, így minden százalékra épülő elrendezés eltörik abban a pillanatban, amikor levegőt adsz neki. A border-box révén a width azt jelenti, „ennyi helyet foglal el" — ez szinte mindig a szándék.

What is margin collapsing?Mi a margó-összeomlás?

  • Adjacent vertical margins merge into one — the larger wins, rather than summing. It also happens between a parent and its first/last child, which is why a child's margin-top can mysteriously push the parent down. Padding, a border, overflow: hidden, or a flex/grid container all stop it. Horizontal margins never collapse.A szomszédos függőleges margók egyetlenné olvadnak — a nagyobb nyer, nem adódnak össze. Szülő és első/utolsó gyermeke között is megtörténik, ezért tolhatja rejtélyes módon a gyermek margin-top-ja a szülőt lefelé. Padding, border, overflow: hidden vagy egy flex/grid konténer mind megállítja. A vízszintes margók sosem omlanak össze.

When do you reach for min-width / max-width instead?Mikor nyúlsz inkább a min-width / max-width után?

  • Whenever the element must adapt. max-width: 60ch on body text caps line length for readability while staying fluid on narrow screens; width: 60ch would overflow. The general rule: set constraints, not fixed sizes, and let the layout resolve the actual number.Amikor az elemnek alkalmazkodnia kell. A max-width: 60ch a szövegtörzsön olvashatósági okból korlátozza a sorhosszt, miközben keskeny képernyőn folyékony marad; a width: 60ch túlcsordulna. Az általános szabály: megkötéseket állíts, ne fix méreteket, és hagyd, hogy az elrendezés kiszámolja a tényleges számot.

Watch outBuktatóPercentage padding and margin resolve against the container's width — including padding-top. That's the old aspect-ratio hack, and it's a surprise if you meant height. Use aspect-ratio now.A százalékos padding és margó a konténer szélességéhez viszonyul — a padding-top is. Ez a régi képarány-trükk alapja, és meglepetés, ha magasságra gondoltál. Ma használj aspect-ratio-t.

Specificity & the CascadeSpecificitás & a kaszkád

When several rules target the same element, the winner is decided by origin, then specificity, then source order. Specificity is a three-part score — ids, classes, elements — compared left to right.Ha több szabály céloz ugyanarra az elemre, a győztest az eredet, majd a specificitás, majd a forrássorrend dönti el. A specificitás háromrészes pontszám — id-k, osztályok, elemek —, balról jobbra összehasonlítva.

Why is a higher class count not always the winner?Miért nem mindig a több osztály nyer?

  • Because the comparison is lexicographic, not additive. A single id (1,0,0) beats any number of classes — #nav a wins over a selector with ten classes. This is the core argument against ids in CSS: they create a specificity ceiling that later rules can only clear with another id or !important.Mert az összehasonlítás lexikografikus, nem összeadó. Egyetlen id (1,0,0) bármennyi osztályt ver — a #nav a nyer egy tízosztályos szelektorral szemben. Ez a fő érv az id-k CSS-beli használata ellen: specificitási plafont hoznak létre, amit a későbbi szabályok csak egy másik id-vel vagy !important-tal tudnak átlépni.

What do :is() and :where() change?Min változtat az :is() és a :where()?

  • Both group selectors, but :is() takes the specificity of its most specific argument, while :where() is always zero. That makes :where() the right tool for defaults and resets — a base style anyone can override with a single class, no specificity war.Mindkettő szelektorokat csoportosít, de az :is() a legspecifikusabb argumentumának specificitását veszi fel, míg a :where() mindig nulla. Ettől lesz a :where() a helyes eszköz alapértelmezésekhez és resetekhez — olyan alapstílus, amit bárki felülírhat egyetlen osztállyal, specificitási háború nélkül.

Which properties inherit?Mely tulajdonságok öröklődnek?

  • Typography-related ones mostly do — color, font-*, line-height, text-align, visibility. Box and layout properties don't: margin, padding, border, display, background. You can force either with the inherit and initial keywords.A tipográfiához kapcsolódók többnyire igen — color, font-*, line-height, text-align, visibility. A doboz- és elrendezési tulajdonságok nem: margin, padding, border, display, background. Bármelyik kikényszeríthető az inherit és initial kulcsszóval.

Watch outBuktató!important wins over specificity entirely, and the only thing that beats it is another !important with higher specificity — which is how stylesheets end up with escalating chains of them. Treat one as a bug report about the architecture, not a fix.Az !important teljesen felülírja a specificitást, és csak egy magasabb specificitású másik !important veri — így alakulnak ki az egyre eszkalálódó láncok a stíluslapokban. Tekints rá hibajelentésként az architektúráról, ne javításként.

FlexboxFlexbox

One-dimensional layout: items are distributed along a single axis — a row or a column — with control over how leftover space is shared and how items shrink when there isn't enough.Egydimenziós elrendezés: az elemek egyetlen tengely mentén oszlanak el — sor vagy oszlop —, szabályozva, hogyan osztozik a maradék helyen, és hogyan zsugorodnak, ha nincs elég.

The single most common confusion: justify-content and align-items swap meaning when flex-direction becomes column.A leggyakoribb félreértés: a justify-content és az align-items jelentése felcserélődik, amikor a flex-direction column lesz.

What does the flex shorthand actually mean?Mit jelent valójában a flex rövidítés?

  • flex: grow shrink basis. flex: 1 expands to 1 1 0% — grow to fill, shrink if needed, and ignore the content width when distributing. That last part matters: flex: 1 gives equal-width items, while flex: auto (1 1 auto) sizes them by content first and only shares the leftover. Reaching for the wrong one is why "equal columns" sometimes aren't.flex: grow shrink basis. A flex: 1 kibontva 1 1 0% — növekedj a kitöltésig, zsugorodj, ha kell, és hagyd figyelmen kívül a tartalom szélességét az elosztásnál. Ez az utolsó rész számít: a flex: 1 egyenlő szélességű elemeket ad, míg a flex: auto (1 1 auto) előbb tartalom szerint méretez, és csak a maradékon osztozik. A rossz választás miatt nem egyenlők néha az „egyenlő oszlopok".

Why won't a flex item shrink below its content?Miért nem zsugorodik egy flex elem a tartalma alá?

  • Because min-width defaults to auto on flex items, which means "at least the content's minimum size". A long unbroken string or a wide child then forces overflow that no width fixes. The remedy is min-width: 0 (or overflow: hidden) on the item — probably the single most-searched flexbox problem.Mert a flex elemeken a min-width alapértéke auto, ami azt jelenti, „legalább a tartalom minimális mérete". Egy hosszú, töretlen string vagy széles gyermek olyan túlcsordulást kényszerít ki, amit semmilyen width nem javít. A megoldás min-width: 0 (vagy overflow: hidden) az elemen — valószínűleg ez a legtöbbet keresett flexbox-probléma.

What are the everyday patterns worth knowing cold?Mely hétköznapi minták érdemesek arra, hogy vakon tudd?

  • Centering: display:flex; align-items:center; justify-content:center. Push one item away from the rest: margin-left: auto on it — auto margins absorb all free space, which is cleaner than space-between when only one item should move. Sticky footer: flex-direction: column on a full-height parent, flex: 1 on the main region.Középre igazítás: display:flex; align-items:center; justify-content:center. Egy elem eltolása a többitől: margin-left: auto rajta — az auto margók felszívják az összes szabad helyet, ami tisztább, mint a space-between, ha csak egy elemnek kell elmozdulnia. Ragadós lábléc: flex-direction: column a teljes magasságú szülőn, flex: 1 a fő területen.

Watch outBuktatóflex-wrap defaults to nowrap, so items squeeze instead of wrapping — a frequent cause of "why is my text one pixel tall". And gap works in flexbox now, so the old negative-margin technique is obsolete.A flex-wrap alapértéke nowrap, így az elemek összenyomódnak tördelés helyett — gyakori oka a „miért egy pixel magas a szövegem" jelenségnek. A gap ma már működik flexboxban is, tehát a régi negatív margós technika elavult.

04

CSS GridCSS Grid

Two-dimensional layout: you define rows and columns, then place items into them. Where flexbox distributes along one axis, grid controls both at once.Kétdimenziós elrendezés: sorokat és oszlopokat definiálsz, majd elemeket helyezel beléjük. Míg a flexbox egy tengely mentén oszt el, a grid egyszerre szabályozza mindkettőt.

Responsive with no media queryReszponzív media query nélkül
.grid {
  display: grid;
  gap: 1rem;
  grid-template-columns:
    repeat(auto-fit, minmax(240px, 1fr));
}
/* as many columns as fit at ≥240px,
   sharing leftover space equally */
Named areasElnevezett területek
.page {
  display: grid;
  grid-template-areas:
    "head head"
    "side main"
    "foot foot";
  grid-template-columns: 200px 1fr;
}
.sidebar { grid-area: side; }

Grid or flexbox — how do you decide?Grid vagy flexbox — hogyan döntesz?

  • Ask where the structure lives. If the container defines the shape and content fills it, that's grid — page scaffolding, card galleries, anything aligned in both directions. If the content determines the arrangement and you just need it distributed, that's flexbox — toolbars, button rows, nav items. They compose: a grid page whose cells are flex containers is the common shape.Kérdezd meg, hol lakik a szerkezet. Ha a konténer definiálja az alakot, és a tartalom kitölti, az grid — oldalváz, kártyagalériák, minden, ami mindkét irányban igazodik. Ha a tartalom határozza meg az elrendezést, és csak elosztásra van szükség, az flexbox — eszköztárak, gombsorok, navigációs elemek. Kombinálhatók: egy grid-oldal, amelynek cellái flex konténerek, a bevett forma.

What are fr, minmax() and auto-fit doing?Mit csinál az fr, a minmax() és az auto-fit?

  • fr is a share of the leftover space after fixed tracks and gaps are subtracted. minmax(240px, 1fr) means "never below 240, otherwise take a share". auto-fit creates as many tracks as fit and collapses the empty ones — which is what makes that one-line responsive grid work without any breakpoints. Use auto-fill instead when you want empty tracks preserved.Az fr a maradék hely egy hányada, miután a fix sávokat és a réseket levontuk. A minmax(240px, 1fr) azt jelenti, „soha 240 alá, egyébként vegyél egy részt". Az auto-fit annyi sávot hoz létre, amennyi elfér, és összecsukja az üreseket — ettől működik az az egysoros reszponzív grid töréspontok nélkül. Ha meg akarod tartani az üres sávokat, használj auto-fill-t.

How do you place an item explicitly?Hogyan helyezel el egy elemet explicit módon?

  • By line number — grid-column: 1 / 3 spans from line 1 to line 3 (two columns). span 2 is the relative form, and -1 means the last line, so grid-column: 1 / -1 is "full width" regardless of column count. Named areas are usually more readable for page-level structure.Vonalszám szerint — a grid-column: 1 / 3 az 1. vonaltól a 3.-ig tart (két oszlop). A span 2 a relatív forma, a -1 pedig az utolsó vonalat jelenti, így a grid-column: 1 / -1 „teljes szélesség" az oszlopszámtól függetlenül. Oldalszintű szerkezethez az elnevezett területek általában olvashatóbbak.

Watch outBuktatóGrid items inherit the same min-width: auto trap as flex items — a wide child overflows its track until you set min-width: 0. And percentage-based columns plus gap overflow, because the gap is added on top; fr accounts for gaps automatically, which is one more reason to prefer it.A grid elemek ugyanazt a min-width: auto csapdát öröklik, mint a flex elemek — egy széles gyermek túlcsordul a sávján, amíg nem állítasz min-width: 0-t. A százalékalapú oszlopok pedig a gap-pel együtt túlcsordulnak, mert a rés hozzáadódik; az fr automatikusan számol a résekkel, ami újabb ok a használatára.

PositioningPozicionálás

position decides what an element is offset from — and whether it still occupies space in the flow. Most positioning bugs are really about which ancestor is acting as the reference.A position dönti el, mihez képest tolódik el az elem — és hogy továbbra is helyet foglal-e a folyamban. A legtöbb pozicionálási hiba valójában arról szól, melyik ős szolgál referenciaként.

What does "nearest positioned ancestor" mean exactly?Mit jelent pontosan a „legközelebbi pozicionált ős"?

  • The closest ancestor whose position is anything other than static. If none exists, the element positions against the initial containing block — effectively the page. This is why the fix for a stray absolute element is almost always adding position: relative to the intended parent, with no offsets: it establishes the reference without moving anything.A legközelebbi ős, amelynek a position-je nem static. Ha nincs ilyen, az elem a kezdeti tartalmazó blokkhoz igazodik — gyakorlatilag az oldalhoz. Ezért az elkóborolt abszolút elem javítása szinte mindig az, hogy position: relative-t adsz a szánt szülőnek, eltolás nélkül: ez létrehozza a referenciát anélkül, hogy bármit elmozdítana.

Why does position: sticky so often do nothing?Miért nem csinál semmit gyakran a position: sticky?

  • Three usual causes: no threshold was set (it needs at least one of top/bottom/left/right); an ancestor has overflow: hidden, auto or scroll, which makes that the scroll container instead of the page; or the parent is exactly as tall as the sticky element, leaving no distance to travel. The second is the one that wastes the most time.Három szokásos ok: nincs küszöb beállítva (kell legalább egy top/bottom/left/right); valamelyik ősön overflow: hidden, auto vagy scroll van, amitől az lesz a görgetési konténer az oldal helyett; vagy a szülő pontosan olyan magas, mint a sticky elem, így nincs bejárható táv. A második viszi el a legtöbb időt.

How do you center an absolutely positioned element?Hogyan igazítasz középre egy abszolút pozicionált elemet?

  • Classic: top: 50%; left: 50%; transform: translate(-50%, -50%) — the percentages resolve against the parent, the transform against the element's own size, so it works without knowing dimensions. Modern alternative: inset: 0; margin: auto, which needs a defined size but avoids the transform (and its stacking-context side effect).Klasszikus: top: 50%; left: 50%; transform: translate(-50%, -50%) — a százalékok a szülőhöz, a transform az elem saját méretéhez viszonyul, így méretek ismerete nélkül működik. Modern alternatíva: inset: 0; margin: auto, ami definiált méretet igényel, de elkerüli a transformot (és annak stacking context mellékhatását).

Watch outBuktatóposition: fixed stops being fixed to the viewport if any ancestor has transform, filter, backdrop-filter or will-change — that ancestor becomes the containing block instead. A CSS animation on a wrapper can therefore break a modal that worked yesterday.A position: fixed megszűnik a viewporthoz rögzülni, ha bármely ősön transform, filter, backdrop-filter vagy will-change van — helyette az az ős lesz a tartalmazó blokk. Egy wrapperre tett CSS-animáció így eltörhet egy tegnap még működő modált.

Stacking Context & z-indexStacking context & z-index

z-index only compares siblings inside the same stacking context. A high value can't lift an element above something in a different context — which is why z-index: 9999 so often does nothing.A z-index csak az azonos stacking contexten belüli testvéreket hasonlítja össze. Egy magas érték nem tud elemet más contextben lévő dolog fölé emelni — ezért nem csinál gyakran semmit a z-index: 9999.

The dropdown's 9999 is compared only against its siblings. Against .modal, what counts is its parent's 1.A dropdown 9999-e csak a testvéreivel van összehasonlítva. A .modal-lal szemben a szülője 1-ese számít.

What creates a stacking context?Mi hoz létre stacking contextet?

  • The root element; any positioned element with a z-index other than auto; opacity below 1; transform, filter, backdrop-filter, will-change, isolation: isolate; and flex/grid children with a z-index. The surprising ones are opacity: 0.99 and transform, because they're usually added for visual reasons by someone not thinking about layering.A gyökérelem; bármely pozicionált elem auto-tól eltérő z-index-szel; az 1 alatti opacity; a transform, filter, backdrop-filter, will-change, isolation: isolate; és a z-index-szel rendelkező flex/grid gyermekek. A meglepők az opacity: 0.99 és a transform, mert ezeket általában vizuális okból teszi be valaki, aki nem a rétegzésre gondol.

How do you debug a layering problem?Hogyan debuggolsz egy rétegzési problémát?

  • Walk up the DOM from the element that won't come forward and find the first ancestor that creates a context — that's your real competitor, not the element you're fighting with. DevTools shows this in the Layers panel. The fix is usually structural: move the element out (a portal), or remove the ancestor's context-creating property.Menj fel a DOM-ban attól az elemtől, amelyik nem jön előre, és keresd meg az első contextet létrehozó őst — az a valódi versenytárs, nem az az elem, amivel küzdesz. A DevTools ezt a Layers panelen mutatja. A javítás általában szerkezeti: vidd ki az elemet (portál), vagy távolítsd el az ős contextet létrehozó tulajdonságát.

Why do modals get rendered through a portal?Miért portálon keresztül renderelődnek a modálok?

  • Precisely to escape this. Rendering into document.body puts the modal in the root stacking context, where it can actually sit above everything, and it also escapes any ancestor's overflow: hidden. That's why React's createPortal exists — it's a layout fix as much as a DOM-structure one.Pontosan azért, hogy elkerüljék ezt. A document.body-ba renderelés a gyökér stacking contextbe teszi a modált, ahol tényleg mindenek fölé kerülhet, és kikerül minden ős overflow: hidden-je alól is. Ezért létezik a React createPortal-ja — ez legalább annyira elrendezési javítás, mint DOM-szerkezeti.

Watch outBuktatóz-index is ignored entirely on position: static elements — no context, no effect, and no error to tell you. If a z-index appears to do nothing, check position before adding zeros.A z-index-et teljesen figyelmen kívül hagyja a position: static elemeknél — nincs context, nincs hatás, és nincs hibaüzenet sem. Ha egy z-index nem csinál semmit, előbb a position-t nézd meg, ne nullákat adj hozzá.

Responsive DesignReszponzív design

Layouts that adapt to the space available — increasingly through intrinsically flexible sizing rather than a ladder of breakpoints.Az elérhető helyhez alkalmazkodó elrendezések — egyre inkább belsőleg rugalmas méretezéssel, nem töréspontok létrájával.

Breakpoint ladderTöréspont-létra
.grid { grid-template-columns: 1fr; }
@media (min-width: 600px) {
  .grid { grid-template-columns: 1fr 1fr; }
}
@media (min-width: 900px) {
  .grid { grid-template-columns: repeat(3,1fr); }
}
Intrinsic — no breakpointsBelsőleg rugalmas — töréspont nélkül
.grid {
  display: grid;
  gap: 1rem;
  grid-template-columns:
    repeat(auto-fit, minmax(240px, 1fr));
}
/* adapts at every width, not three */

Which units should you actually use?Milyen egységeket használj valójában?

  • rem for type and spacing, because it scales with the user's browser font size — px ignores that preference, which is an accessibility problem, not just a stylistic one. ch for text measure, % and fr for layout shares, vw/vh sparingly. clamp(1rem, 2.5vw, 1.5rem) gives fluid sizing with hard limits and removes a lot of breakpoints on its own.rem a betűkhöz és térközökhöz, mert a felhasználó böngésző-betűméretével skálázódik — a px figyelmen kívül hagyja ezt a beállítást, ami akadálymentesítési probléma, nem csak stiláris. ch a szövegszélességhez, % és fr az elrendezési arányokhoz, vw/vh mértékkel. A clamp(1rem, 2.5vw, 1.5rem) folyékony méretezést ad kemény határokkal, és önmagában sok töréspontot kivált.

Why mobile-first?Miért mobile-first?

  • Writing base styles for the smallest screen and layering min-width queries on top means each query only adds. The reverse — max-width queries — requires each breakpoint to undo the previous one, and undoing is where the specificity fights and the forgotten resets come from.Ha az alapstílusokat a legkisebb képernyőre írod, és min-width query-ket rétegzel rá, minden query csak hozzáad. A fordítottja — max-width query-k — megköveteli, hogy minden töréspont visszavonja az előzőt, és a visszavonásból erednek a specificitási harcok és az elfelejtett resetek.

What do container queries change?Min változtatnak a container query-k?

  • They let a component respond to its own available width rather than the viewport's — container-type: inline-size on the parent, then @container (min-width: 400px). This is what media queries could never do: the same card can lay out one way in a sidebar and another in a main column, with no knowledge of the page. For component libraries it's the more correct tool, and it's now broadly supported.Lehetővé teszik, hogy egy komponens a saját rendelkezésre álló szélességére reagáljon, ne a viewportéra — container-type: inline-size a szülőn, majd @container (min-width: 400px). Ezt a media query-k sosem tudták: ugyanaz a kártya másképp rendeződik egy oldalsávban és másképp egy fő oszlopban, az oldal ismerete nélkül. Komponenskönyvtárakhoz ez a helyesebb eszköz, és mára széles körben támogatott.

Watch outBuktató100vh on mobile includes the area behind the browser's collapsing toolbar, so a "full height" section jumps as you scroll. Use 100dvh (dynamic viewport height) instead. And missing <meta name="viewport"> makes every media query behave as if the phone were a desktop.A 100vh mobilon beleszámítja a böngésző összecsukódó eszköztára mögötti területet, így egy „teljes magasságú" szakasz ugrál görgetés közben. Használj helyette 100dvh-t (dinamikus viewport magasság). A hiányzó <meta name="viewport"> pedig minden media query-t úgy működtet, mintha a telefon asztali gép lenne.

CSS PerformanceCSS teljesítmény

The browser turns style changes into work in three stages — layout, paint, composite. Which stage you trigger determines whether an animation runs at 60fps or stutters.A böngésző három szakaszban alakít munkává egy stílusváltozást — layout, paint, composite. Az, hogy melyik szakaszt váltod ki, dönti el, hogy 60fps-en fut-e az animáció, vagy akadozik.

So what do you animate?Akkor mit animálsz?

  • transform and opacity, essentially always. Moving something with transform: translateX() instead of left skips layout and paint entirely and runs on the compositor, off the main thread — so it keeps animating even while JavaScript is busy. Animating width or top forces a relayout every frame.transform-ot és opacity-t, gyakorlatilag mindig. Ha transform: translateX()-szel mozgatsz left helyett, teljesen kihagyod a layoutot és a paintet, és a compositoron fut, a fő szálon kívül — így akkor is animál, amikor a JavaScript dolgozik. A width vagy top animálása minden képkockánál újralayoutot kényszerít.

What is layout thrashing?Mi a layout thrashing?

  • Interleaving DOM reads and writes in a loop. Reading offsetHeight forces the browser to flush pending style changes and recalculate layout synchronously; doing that after every write means a full layout per iteration instead of one at the end. The fix is batching — read everything, then write everything.DOM-olvasások és -írások váltogatása ciklusban. Az offsetHeight olvasása arra kényszeríti a böngészőt, hogy szinkron módon kiürítse a függő stílusváltozásokat és újraszámolja a layoutot; ha ezt minden írás után megteszed, iterációnként teljes layout lesz egy helyett a végén. A megoldás a kötegelés — olvass be mindent, aztán írj mindent.

Does selector complexity matter?Számít a szelektorok bonyolultsága?

  • Far less than people assume — modern engines match selectors right-to-left and this is rarely the bottleneck. What does matter: unused CSS shipped to the client, render-blocking stylesheets in <head>, and layout-triggering animations. Optimising selectors is usually the wrong place to spend effort.Sokkal kevésbé, mint hinnék — a modern motorok jobbról balra illesztik a szelektorokat, és ez ritkán a szűk keresztmetszet. Ami viszont számít: a kliensnek kiküldött használatlan CSS, a <head>-ben lévő renderelést blokkoló stíluslapok, és a layoutot kiváltó animációk. A szelektorok optimalizálása általában rossz helyre fektetett energia.

Watch outBuktatówill-change promotes an element to its own layer, but leaving it on permanently consumes memory and can slow things down — apply it just before the animation and remove it after. It also creates a stacking context, which can break z-index and position: fixed further up the tree.A will-change saját rétegbe emeli az elemet, de tartósan rajta hagyva memóriát fogyaszt, és lassíthat is — közvetlenül az animáció előtt tedd rá, és utána vedd le. Stacking contextet is létrehoz, ami eltörheti a z-index-et és a position: fixed-et feljebb a fában.

Semantic HTML & AccessibilitySzemantikus HTML & akadálymentesítés

Using the element that means what you intend — which gives you keyboard behaviour, focus handling and screen-reader semantics for free, instead of rebuilding them on a div.Azt az elemet használod, ami azt jelenti, amit szánsz — ezzel ingyen kapod a billentyűzetes viselkedést, a fókuszkezelést és a képernyőolvasó-szemantikát, ahelyett hogy egy div-en építenéd újra.

Rebuilding what existsA meglévő újraépítése

     onclick="submit()">
  Save

     no role, not in tab order -->

Which semantics actually earn their keep?Mely szemantikai elemek térülnek meg valóban?

  • Landmarks (header, nav, main, footer, aside) give screen-reader users a way to jump around the page. A correct heading order (h1–h6, no skipping) is the document outline they navigate by. button versus a is the one people get wrong most: links navigate, buttons act — and swapping them breaks both keyboard behaviour and the user's expectations.A landmarkok (header, nav, main, footer, aside) a képernyőolvasót használóknak adnak módot az oldalon való ugrálásra. A helyes fejléc-sorrend (h1–h6, kihagyás nélkül) az a dokumentum-vázlat, ami mentén navigálnak. A button és az a az, amit a legtöbben elrontanak: a link navigál, a gomb cselekszik — a felcserélésük mind a billentyűzetes viselkedést, mind a felhasználó elvárásait eltöri.

What are the first rules of ARIA?Mik az ARIA első szabályai?

  • Don't use it if a native element will do — role="button" on a div announces correctly but still has no keyboard behaviour, so it's half a solution that looks complete. And don't change native semantics. ARIA is for things HTML has no element for (tabs, comboboxes, live regions), where you must also implement the expected keyboard interaction yourself.Ne használd, ha natív elem is megteszi — egy div-en a role="button" helyesen mondja be, de billentyűzetes viselkedése továbbra sincs, tehát fél megoldás, ami késznek látszik. És ne írd felül a natív szemantikát. Az ARIA azokra való, amikre a HTML-ben nincs elem (tabok, comboboxok, élő régiók), ahol az elvárt billentyűzetes interakciót is neked kell megvalósítanod.

What breaks keyboard users most often?Mi töri el leggyakrabban a billentyűzetes használatot?

  • Removing focus outlines with outline: none and replacing them with nothing — use :focus-visible to style them instead of deleting them. Also: click handlers on non-interactive elements, modals that don't trap focus or return it on close, and positive tabindex values, which scramble the natural tab order. Tabbing through your own feature once catches most of this.A fókuszkeret eltávolítása outline: none-nal, pótlás nélkül — használj inkább :focus-visible-t a stílusozáshoz, ne töröld. Továbbá: kattintáskezelők nem interaktív elemeken, modálok, amelyek nem fogják meg a fókuszt vagy nem adják vissza záráskor, és a pozitív tabindex értékek, amelyek összekeverik a természetes tab-sorrendet. Ha egyszer végigtabbolod a saját funkciódat, ennek nagy részét elkapod.

Watch outBuktatódisplay: none and visibility: hidden remove content from screen readers too — correct for genuinely hidden things, wrong for visually-hidden labels, which need a clip-based utility class instead. And placeholder text is not a label: it disappears on input and is often announced inconsistently.A display: none és a visibility: hidden a képernyőolvasók elől is elrejti a tartalmat — ez helyes a valóban rejtett dolgoknál, de hibás a csak vizuálisan rejtett címkéknél, amelyekhez clip-alapú segédosztály kell. A placeholder szöveg pedig nem címke: gépeléskor eltűnik, és gyakran következetlenül hangzik el.
