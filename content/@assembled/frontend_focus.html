Virtual DOM & Reconciliation

React renders a lightweight in-memory tree, diffs it against the previous one (reconciliation), and applies only the minimal real-DOM mutations.A React egy könnyű, memóriabeli fát renderel, összeveti az előzővel (reconciliation), és csak a minimálisan szükséges valódi DOM-módosításokat hajtja végre.

Same type + key → reuse and patch. Different type → tear down and rebuild.Azonos típus + key → újrahasznosítás és patch. Eltérő típus → lebontás és újraépítés.

What's the diffing heuristic?Mi a diffing heurisztika?

  • An O(n) heuristic, not a full tree diff. Two assumptions: different element types produce different trees; stable keys identify which children moved.Egy O(n) heurisztika, nem teljes fa-diff. Két feltevés: eltérő elemtípusok eltérő fát eredményeznek; a stabil key-ek azonosítják, mely gyerekek mozdultak el.

What actually causes a re-render?Mi okoz ténylegesen re-rendert?

  • A state change, a parent re-render, a consumed context value changing, or a subscribing hook firing.State-változás, a szülő re-rendere, egy fogyasztott context-érték változása, vagy egy feliratkozó hook eseménye.
  • Re-render ≠ DOM update — reconciliation may find nothing changed and touch no DOM.A re-render ≠ DOM-frissítés — a reconciliation azt is találhatja, hogy semmi nem változott, és hozzá sem nyúl a DOM-hoz.

Render vs commit phase?Render vs commit fázis?

  • Render: React calls your components and builds the tree — pure, and pausable in concurrent mode.Render: a React meghívja a komponenseidet és felépíti a fát — tiszta, és concurrent módban megszakítható.
  • Commit: applies DOM changes and runs effects — synchronous.Commit: alkalmazza a DOM-változásokat és lefuttatja az effekteket — szinkron.

Watch outBuktatóA re-render is relatively cheap; the cost is your render logic. Don't optimize re-renders you haven't profiled.A re-render viszonylag olcsó; a költség a te render-logikád. Ne optimalizálj olyan re-rendert, amit nem profiloztál.

02

Keys in ListsKey-ek listákban

key is the stable identity React uses to match elements across renders.A key az a stabil identitás, amivel a React renderek között összepárosítja az elemeket.

With index keys, inserting at the top shifts every key and misassigns DOM/state.Index-key esetén a lista elejére szúrás minden key-t elcsúsztat, és rosszul rendeli hozzá a DOM-ot/state-et.

Why not use the array index as a key?Miért ne használjuk a tömb indexét key-ként?

  • On insert/reorder/delete, indexes shift, so React reuses the wrong DOM node and component state — e.g. an input's value sticks to the wrong row.Beszúrásnál/átrendezésnél/törlésnél az indexek elcsúsznak, így a React rossz DOM-node-ot és komponens-state-et használ újra — pl. egy input értéke rossz sorhoz ragad.

What makes a good key?Mitől jó egy key?

  • Stable, unique among siblings, and tied to the data's identity (an ID) — not its position. It only needs to be unique within the list, not globally.Stabil, a testvérek között egyedi, és az adat identitásához (egy ID-hez) kötött — nem a pozíciójához. Csak a listán belül kell egyedinek lennie, globálisan nem.

Watch outBuktatóIndex keys are safe only for static, never-reordered lists. The classic bug: uncontrolled inputs in a reorderable list keyed by index.Az index-key csak statikus, soha nem átrendezett listáknál biztonságos. A klasszikus hiba: uncontrolled inputok index szerint key-elt, átrendezhető listában.

Rules of Hooks

Call hooks only at the top level — never in loops, conditions or nested functions — and only from React functions (components or custom hooks).A hookokat csak a legfelső szinten hívd — soha ne ciklusban, feltételben vagy beágyazott függvényben — és csak React függvényekből (komponensek vagy custom hookok).

Why the top-level rule?Miért van a top-level szabály?

  • React tracks hook state by call order, not by name. A conditional hook shifts the order between renders, so state gets misassigned to the wrong hook.A React a hook-state-et hívási sorrend alapján tartja nyilván, nem név szerint. Egy feltételes hook elcsúsztatja a sorrendet a renderek között, így a state rossz hookhoz kerül.

How do you do conditional logic then?Akkor hogyan csinálsz feltételes logikát?

  • Put the condition inside the hook (e.g. inside useEffect), or split into separate components — never wrap the hook call itself.A feltételt tedd a hookon belülre (pl. a useEffect-en belül), vagy bontsd külön komponensekre — soha ne magát a hook-hívást csomagold feltételbe.

Watch outBuktatóeslint-plugin-react-hooks catches most violations — don't disable it to “make it work.”Az eslint-plugin-react-hooks a legtöbb szabálysértést elkapja — ne kapcsold ki, hogy „működjön”.

04

Stale Closures

A closure that captured a value from an earlier render and keeps using that old snapshot.Olyan closure, ami egy korábbi renderből fogott be egy értéket, és továbbra is azt a régi pillanatképet használja.

Where does it bite?Hol üt vissza?

  • Event handlers, setTimeout/setInterval, and effects with missing deps — they all close over the render's snapshot of state and props.Event handlerekben, setTimeout/setInterval-ban, és hiányzó depekkel rendelkező effektekben — mindegyik a render state- és prop-pillanatképét zárja be.

How do you fix it?Hogyan javítod?

  • Add the value to the deps (re-create the closure), use the functional updater, or stash the latest value in a ref and read ref.current.Vedd fel az értéket a depek közé (újra létrejön a closure), használd a funkcionális updatert, vagy tedd a legfrissebb értéket egy refbe és olvasd a ref.current-et.

Watch outBuktatóA setInterval created in useEffect([]) that reads state will forever see the initial value — the textbook stale closure.Egy useEffect([])-ben létrehozott setInterval, ami state-et olvas, örökre a kezdeti értéket látja — ez a tankönyvi stale closure.

Structural TypingStrukturális típusosság

TypeScript compares types by shape, not by name. If an object has the required members, it is assignable — regardless of which class or interface it was declared from.A TypeScript alak szerint hasonlítja a típusokat, nem név szerint. Ha egy objektumnak megvannak a szükséges tagjai, akkor értékül adható — függetlenül attól, melyik osztályból vagy interface-ből származik.

Assignable — shapes matchÉrtékül adható — az alak egyezik
interface Point { x: number; y: number }

class Vec { constructor(
  public x: number,
  public y: number
) {} }

// never declared as Point, still fits
const p: Point = new Vec(1, 2);
The downside — no nominal safetyAz árnyoldal — nincs névbeli védelem
type UserId  = string;
type OrderId = string;

function get(id: UserId) { /* … */ }

const order: OrderId = 'o-1';
get(order); // compiles — both are string

Structural vs nominal typing — what's the practical difference?Strukturális vs nominális típusosság — mi a gyakorlati különbség?

  • In nominal systems (Java, C#) a type matches only if you declared that relationship. In TypeScript the declaration is irrelevant — only the members count. This makes typing existing JS painless, and lets you type a function's parameter without importing the class the caller happens to use.Nominális rendszerekben (Java, C#) egy típus csak akkor illeszkedik, ha kimondtad a kapcsolatot. A TypeScriptben a deklaráció lényegtelen — csak a tagok számítanak. Ettől fájdalommentes a meglévő JS típusozása, és úgy tudsz paramétert típusozni, hogy nem importálod a hívó által épp használt osztályt.

How do you get nominal behaviour when you need it?Hogyan érsz el nominális viselkedést, ha kell?

  • A branded (or "opaque") type: intersect the primitive with a marker that has no runtime existence — type UserId = string & { readonly __brand: unique symbol }. Now a raw string won't pass, and a UserId won't be accepted where an OrderId is expected. It costs a cast at the boundary where you create one.Branded (vagy „opaque") típus: metszed a primitívet egy futásidőben nem létező jelölővel — type UserId = string & { readonly __brand: unique symbol }. Így egy nyers string nem megy át, és egy UserId sem fogadható el ott, ahol OrderId a várt. Cserébe egy cast kell a határon, ahol létrehozod.

Does an object with extra properties fit?Belefér egy plusz property-vel rendelkező objektum?

  • Yes, when it arrives through a variable — extra members are allowed. But an object literal assigned directly triggers excess property checking and errors. The check exists to catch typos; assigning through a variable is the documented escape hatch.Igen, ha változón keresztül érkezik — a plusz tagok megengedettek. De egy közvetlenül értékül adott objektum-literál aktiválja az excess property checkinget, és hibát dob. Az ellenőrzés az elgépeléseket célozza; a változón át adás a dokumentált kiskapu.

Watch outBuktatóStructural typing means {} and object are far wider than people expect — {} accepts anything except null/undefined, including numbers and strings. If you mean "some object", write Record<string, unknown>.A strukturális típusosság miatt a {} és az object sokkal tágabb, mint hinnéd — a {} a null/undefined kivételével mindent elfogad, számot és stringet is. Ha „valamilyen objektumra" gondolsz, írj Record<string, unknown>-t.

type vs interfacetype vs interface

Two ways to name a type. They overlap heavily for object shapes; the differences that matter are declaration merging, what each can express, and how errors read.Két mód egy típus elnevezésére. Objektum-alakoknál nagyrészt átfedik egymást; a lényeges különbségek: a declaration merging, hogy mit tudnak kifejezni, és hogyan olvashatók a hibaüzenetek.

interfacetype
Object shapesObjektum-alakok yesigen yesigen
Unions, tuples, primitivesUnionok, tuple-ök, primitívek nonem yesigen
Mapped & conditional typesMapped & conditional típusok nonem yesigen
Declaration mergingDeclaration merging yes — reopensigen — újranyitható no — duplicate errornem — duplikált hiba
ExtendingKiterjesztés extends intersection &metszet &

Which should you reach for?Melyiket válaszd?

  • A defensible default: interface for object shapes and public APIs, type for everything else — unions, tuples, function types, mapped and conditional types. The real answer in an interview is that it's mostly a consistency decision within a codebase, and being able to say why matters more than the pick.Védhető alapértelmezés: interface objektum-alakokra és publikus API-kra, type minden másra — unionok, tuple-ök, függvénytípusok, mapped és conditional típusok. Interjún az igazi válasz az, hogy ez leginkább konzisztencia-döntés egy kódbázison belül, és többet ér elmondani a miértet, mint magát a választást.

What is declaration merging, and when is it useful?Mi a declaration merging, és mikor hasznos?

  • Declaring the same interface twice merges the members instead of erroring. It's how you augment third-party types — adding user to Express's Request, or extending Window. It's also why a public API surface is often an interface: consumers can extend it without you republishing.Ugyanannak az interface-nek a kétszeri deklarálása összefésüli a tagokat hiba helyett. Így egészítesz ki külső típusokat — user hozzáadása az Express Request-jéhez, vagy a Window bővítése. Ezért is szokott a publikus API-felület interface lenni: a fogyasztók bővíthetik anélkül, hogy újra publikálnál.

Any difference in performance or error messages?Van különbség teljesítményben vagy hibaüzenetekben?

  • Interfaces are cached by name, so deep extends chains generally check faster than long intersection chains, and errors tend to report the interface name rather than expanding the whole shape. On a large codebase this is a real ergonomics difference, not a micro-optimisation.Az interface-eket név szerint cache-eli, így a mély extends láncok általában gyorsabban ellenőrződnek, mint a hosszú metszet-láncok, és a hibák inkább az interface nevét írják ki, mint a teljes kibontott alakot. Nagy kódbázison ez valódi ergonómiai különbség, nem mikro-optimalizáció.

Watch outBuktatóDeclaration merging cuts both ways: a typo'd interface name silently merges instead of failing, and a dependency can augment your interface without you noticing. type errors loudly on redeclaration, which is occasionally what you want.A declaration merging kétélű: egy elgépelt interface-név csendben összefésül hiba helyett, és egy függőség észrevétlenül kiegészítheti a te interface-edet. A type hangosan hibázik újradeklarálásnál, ami néha épp kívánatos.

Box Model & SizingDoboz-modell & méretezés

Every element is a box: content, padding, border, margin. The only question that matters in practice is what width actually measures — and the answer depends on box-sizing.Minden elem egy doboz: tartalom, padding, border, margó. A gyakorlatban az egyetlen fontos kérdés az, hogy a width valójában mit mér — és a válasz a box-sizing-tól függ.

content-box (default)(alapértelmezett)
.card {
  width: 300px;
  padding: 20px;
  border: 1px solid;
}
/* actual rendered width: 342px
   300 + 20*2 + 1*2 */
border-box
*, *::before, *::after {
  box-sizing: border-box;
}
.card { width: 300px; padding: 20px; }
/* actual rendered width: 300px
   padding eats into the content */

Why does nearly every codebase set border-box globally?Miért állít szinte minden kódbázis globális border-box-ot?

  • Because with content-box, adding padding changes an element's outer size, so any layout built from percentages breaks the moment you add breathing room. border-box makes width mean "the space this occupies", which is what you almost always intend.Mert content-box esetén a padding hozzáadása megváltoztatja az elem külső méretét, így minden százalékra épülő elrendezés eltörik abban a pillanatban, amikor levegőt adsz neki. A border-box révén a width azt jelenti, „ennyi helyet foglal el" — ez szinte mindig a szándék.

What is margin collapsing?Mi a margó-összeomlás?

  • Adjacent vertical margins merge into one — the larger wins, rather than summing. It also happens between a parent and its first/last child, which is why a child's margin-top can mysteriously push the parent down. Padding, a border, overflow: hidden, or a flex/grid container all stop it. Horizontal margins never collapse.A szomszédos függőleges margók egyetlenné olvadnak — a nagyobb nyer, nem adódnak össze. Szülő és első/utolsó gyermeke között is megtörténik, ezért tolhatja rejtélyes módon a gyermek margin-top-ja a szülőt lefelé. Padding, border, overflow: hidden vagy egy flex/grid konténer mind megállítja. A vízszintes margók sosem omlanak össze.

When do you reach for min-width / max-width instead?Mikor nyúlsz inkább a min-width / max-width után?

  • Whenever the element must adapt. max-width: 60ch on body text caps line length for readability while staying fluid on narrow screens; width: 60ch would overflow. The general rule: set constraints, not fixed sizes, and let the layout resolve the actual number.Amikor az elemnek alkalmazkodnia kell. A max-width: 60ch a szövegtörzsön olvashatósági okból korlátozza a sorhosszt, miközben keskeny képernyőn folyékony marad; a width: 60ch túlcsordulna. Az általános szabály: megkötéseket állíts, ne fix méreteket, és hagyd, hogy az elrendezés kiszámolja a tényleges számot.

Watch outBuktatóPercentage padding and margin resolve against the container's width — including padding-top. That's the old aspect-ratio hack, and it's a surprise if you meant height. Use aspect-ratio now.A százalékos padding és margó a konténer szélességéhez viszonyul — a padding-top is. Ez a régi képarány-trükk alapja, és meglepetés, ha magasságra gondoltál. Ma használj aspect-ratio-t.

Specificity & the CascadeSpecificitás & a kaszkád

When several rules target the same element, the winner is decided by origin, then specificity, then source order. Specificity is a three-part score — ids, classes, elements — compared left to right.Ha több szabály céloz ugyanarra az elemre, a győztest az eredet, majd a specificitás, majd a forrássorrend dönti el. A specificitás háromrészes pontszám — id-k, osztályok, elemek —, balról jobbra összehasonlítva.

SelectorSzelektoridclassosztályelementelem
div001
.btn010
.card .btn:hover030
#nav .btn110
:where(.a, .b)000

Why is a higher class count not always the winner?Miért nem mindig a több osztály nyer?

  • Because the comparison is lexicographic, not additive. A single id (1,0,0) beats any number of classes — #nav a wins over a selector with ten classes. This is the core argument against ids in CSS: they create a specificity ceiling that later rules can only clear with another id or !important.Mert az összehasonlítás lexikografikus, nem összeadó. Egyetlen id (1,0,0) bármennyi osztályt ver — a #nav a nyer egy tízosztályos szelektorral szemben. Ez a fő érv az id-k CSS-beli használata ellen: specificitási plafont hoznak létre, amit a későbbi szabályok csak egy másik id-vel vagy !important-tal tudnak átlépni.

What do :is() and :where() change?Min változtat az :is() és a :where()?

  • Both group selectors, but :is() takes the specificity of its most specific argument, while :where() is always zero. That makes :where() the right tool for defaults and resets — a base style anyone can override with a single class, no specificity war.Mindkettő szelektorokat csoportosít, de az :is() a legspecifikusabb argumentumának specificitását veszi fel, míg a :where() mindig nulla. Ettől lesz a :where() a helyes eszköz alapértelmezésekhez és resetekhez — olyan alapstílus, amit bárki felülírhat egyetlen osztállyal, specificitási háború nélkül.

Which properties inherit?Mely tulajdonságok öröklődnek?

  • Typography-related ones mostly do — color, font-*, line-height, text-align, visibility. Box and layout properties don't: margin, padding, border, display, background. You can force either with the inherit and initial keywords.A tipográfiához kapcsolódók többnyire igen — color, font-*, line-height, text-align, visibility. A doboz- és elrendezési tulajdonságok nem: margin, padding, border, display, background. Bármelyik kikényszeríthető az inherit és initial kulcsszóval.

Watch outBuktató!important wins over specificity entirely, and the only thing that beats it is another !important with higher specificity — which is how stylesheets end up with escalating chains of them. Treat one as a bug report about the architecture, not a fix.Az !important teljesen felülírja a specificitást, és csak egy magasabb specificitású másik !important veri — így alakulnak ki az egyre eszkalálódó láncok a stíluslapokban. Tekints rá hibajelentésként az architektúráról, ne javításként.

The Test PyramidA teszt-piramis

A model for how many tests of each kind to write: many fast unit tests at the base, fewer integration tests in the middle, very few slow end-to-end tests at the top.Egy modell arról, hány tesztet írj mindegyik fajtából: sok gyors unit teszt az alapon, kevesebb integrációs teszt középen, nagyon kevés lassú end-to-end teszt a csúcson.

Cost and runtime grow upward; confidence that the whole system works grows with them.A költség és a futásidő felfelé nő; a bizalom, hogy az egész rendszer működik, velük együtt nő.

Why not just write end-to-end tests?Miért ne írjunk csak end-to-end teszteket?

  • They're slow, flaky and expensive to maintain, and when one fails it doesn't tell you where the bug is. Unit tests are fast and pinpoint the failure; E2E tests prove the pieces actually work together. You need both, in different proportions.Lassúak, flaky-k és drágán karbantarthatók, és ha elbukik egy, nem mondja meg, hol a hiba. A unit tesztek gyorsak és pontosan megmutatják a hibát; az E2E tesztek bizonyítják, hogy a darabok tényleg működnek együtt. Mindkettő kell, eltérő arányban.

What's the “testing trophy” alternative?Mi a „testing trophy” alternatíva?

  • A newer model that puts the most weight on integration tests rather than unit tests, arguing they give the best confidence-per-effort for typical web apps. In a React/Node stack it's often the more practical shape.Egy újabb modell, amely a legnagyobb súlyt az integrációs tesztekre helyezi a unit tesztek helyett, azzal érvelve, hogy tipikus webappoknál ezek adják a legjobb bizalom/ráfordítás arányt. React/Node stacken gyakran ez a praktikusabb forma.

What actually makes a test valuable?Mitől értékes ténylegesen egy teszt?

  • It fails when the behaviour breaks and passes otherwise. A test that breaks on every refactor (too coupled to implementation) or that never fails (asserting nothing meaningful) is a maintenance cost with no return.Elbukik, amikor a viselkedés elromlik, és átmegy egyébként. Egy teszt, amely minden refaktornál eltörik (túlságosan az implementációhoz kötött), vagy amely sosem bukik el (semmi értelmeset nem állít), karbantartási költség hozam nélkül.

Watch outBuktatóChasing 100% coverage produces tests written to touch lines rather than to verify behaviour. Coverage tells you what isn't tested; it says nothing about whether what is tested is tested well.A 100%-os lefedettség hajszolása olyan teszteket szül, amelyek sorokat érintenek, nem viselkedést ellenőriznek. A lefedettség megmondja, mi nincs tesztelve; arról semmit nem mond, hogy ami tesztelve van, az jól van-e tesztelve.
