01

Endpoint

An endpoint is a specific URL that exposes a resource or collection a client can act on — base URL + a resource path (e.g. https://api.shop.com/orders/42).Az endpoint egy konkrét URL, ami egy resource-t vagy collection-t tesz elérhetővé a kliens számára — base URL + resource path (pl. https://api.shop.com/orders/42).

How does REST resource design differ from RPC-style endpoints?Miben tér el a REST resource-design az RPC-stílusú endpointoktól?

  • REST models nouns: /orders/42, and the HTTP method carries the verb.A REST főneveket modellez: /orders/42, az igét pedig a HTTP metódus hordozza.
  • RPC bakes the action into the path: /getOrder?id=42, /cancelOrder.Az RPC a műveletet a path-ba égeti: /getOrder?id=42, /cancelOrder.

How should endpoints be named?Hogyan érdemes elnevezni az endpointokat?

  • Plural nouns for collections (/orders), path params for identity (/orders/42).Többes számú főnév a collection-ökhöz (/orders), path param az azonosításhoz (/orders/42).
  • No verbs in the path; filtering / sorting go in the query string (/orders?status=paid&sort=-created).Ige ne legyen a path-ban; a szűrés és a rendezés a query stringbe megy (/orders?status=paid&sort=-created).
  • Nest to show relationships (/users/7/orders), but avoid going much deeper than two levels.A kapcsolatokat nesteléssel jelezd (/users/7/orders), de két szintnél sokkal mélyebbre ne menj.

Watch outBuktatóThe endpoint is the URL; the operation is endpoint + method. GET /orders and POST /orders are two different operations on one endpoint.Az endpoint maga az URL; a művelet az endpoint + metódus. A GET /orders és a POST /orders két külön művelet ugyanazon az endpointon.

HTTP Methodsmetódusok

Methods are the verbs that define what action a request performs on a resource.A metódusok azok az igék, amelyek megadják, hogy a kérés milyen műveletet végez a resource-on.

MethodPurposeCélSafeIdempotent
GETRead a resourceResource olvasása✓✓
POSTCreate / non-idempotent actionLétrehozás / nem idempotens művelet——
PUTFull replaceTeljes csere—✓
PATCHPartial updateRészleges módosítás——*
DELETERemove a resourceResource törlése—✓

Safe vs idempotent — what's the difference?Safe vs idempotent — mi a különbség?

  • Safe = no server state change (GET, HEAD).Safe = nem változtat szerver-oldali állapotot (GET, HEAD).
  • Idempotent = repeating N times has the same effect as once (GET, PUT, DELETE).Idempotent = N-szer megismételve ugyanaz a hatás, mint egyszer (GET, PUT, DELETE).
  • Every safe method is idempotent; the reverse isn't true.Minden safe metódus idempotens; fordítva nem igaz.

PUT vs PATCH?

  • PUT replaces the whole resource — fields you omit get cleared/defaulted.A PUT a teljes resource-t lecseréli — a kihagyott mezők törlődnek vagy default értéket kapnak.
  • PATCH applies a partial change. *It isn't inherently idempotent (e.g. an increment op).A PATCH részleges módosítást alkalmaz. *Önmagában nem idempotens (pl. egy increment művelet).

Watch outBuktatóPOST is the catch-all for non-idempotent creates and RPC-ish actions that don't map cleanly onto the other verbs.A POST a gyűjtőmetódus a nem idempotens létrehozásokhoz és azokhoz az RPC-szerű műveletekhez, amelyek nem illenek tisztán a többi igére.

03

Rate-Limiting

Capping how many requests a client can make per time window to protect capacity and keep usage fair.Annak korlátozása, hány kérést tehet egy kliens időablakonként — a kapacitás védelmére és a fair használatért.

Token bucket: refills at a steady rate, allows short bursts up to its size.Token bucket: egyenletes ütemben töltődik, és a méretéig enged rövid burstöket.

Common algorithms?Gyakori algoritmusok?

  • Fixed window — simple, but allows bursts at the window edge.Fixed window — egyszerű, de az ablak határán burstöt enged.
  • Sliding window — smooths those edge bursts.Sliding window — kisimítja ezeket a határon jelentkező burstöket.
  • Token bucket — steady refill, bursts up to bucket size. Leaky bucket — constant outflow.Token bucket — egyenletes utántöltés, burst a bucket méretéig. Leaky bucket — állandó kifolyás.

How do you signal limits to clients?Hogyan jelzed a limiteket a klienseknek?

429 Too Many Requests + Retry-After, plus X-RateLimit-Limit / Remaining / Reset headers.429 Too Many Requests + Retry-After, valamint X-RateLimit-Limit / Remaining / Reset headerek.

Watch outBuktatóAcross multiple instances, counters must live in shared storage (e.g. Redis) — otherwise every node limits independently and the real cap is N×.Több instance esetén a számlálóknak közös tárban kell lenniük (pl. Redis) — különben minden node külön limitál, és a tényleges korlát N-szerese lesz.

04

Idempotency

A request that produces the same result no matter how many times it's sent. This is what makes safe retries possible.Olyan kérés, ami ugyanazt az eredményt adja, akárhányszor küldöd el. Ez teszi lehetővé a biztonságos retry-t.

An Idempotency-Key lets the server dedupe a retried write.Az Idempotency-Key lehetővé teszi, hogy a szerver deduplikálja az újraküldött írást.

Which methods are idempotent, and why does it matter?Mely metódusok idempotensek, és miért számít?

  • GET, PUT, DELETE, HEAD are; POST and PATCH generally aren't.A GET, PUT, DELETE, HEAD igen; a POST és a PATCH általában nem.
  • It matters for retries: after a timeout, a client can safely re-send an idempotent call with no side effects.A retry miatt számít: timeout után a kliens biztonságosan újraküldhet egy idempotens hívást mellékhatás nélkül.

How do you make POST idempotent (e.g. payments)?Hogyan teszel egy POST-ot idempotenssé (pl. fizetésnél)?

The client sends an Idempotency-Key header; the server stores key → result and returns the original response on any retry — so a double-submitted charge only bills once.A kliens küld egy Idempotency-Key headert; a szerver eltárolja a kulcs → eredmény párost, és minden retry-nál az eredeti választ adja vissza — így a duplán elküldött terhelés csak egyszer számláz.

Watch outBuktatóDELETE twice: first returns 204, second 404 — still idempotent, because the state (resource gone) is unchanged.Kétszeri DELETE: az első 204-et ad, a második 404-et — ettől még idempotens, mert az állapot (a resource nincs meg) változatlan.

05

Event Loop (Node)

Node runs your JS on a single thread; the event loop (via libuv) processes callbacks in ordered phases, one cycle at a time.A Node egyetlen szálon futtatja a JS-edet; az event loop (a libuv-on keresztül) rendezett fázisokban dolgozza fel a callbackeket, ciklusonként.

Each phase runs its callback queue; nextTick + Promise microtasks drain between them.Minden fázis lefuttatja a saját callback-sorát; a nextTick és a Promise microtaskok közöttük futnak le.

How does it differ from the browser event loop?Miben tér el a böngésző event loopjától?

  • Browser: task → drain microtasks → render. Node has multiple phases (timers, poll, check…), no rendering, and adds process.nextTick as a separate, higher-priority queue.Böngésző: task → microtaskok lefutnak → render. A Node-nak több fázisa van (timers, poll, check…), nincs renderelés, és a process.nextTick külön, magasabb prioritású sorként jelenik meg.

Where do Promises and nextTick run?Hol futnak a Promise-ok és a nextTick?

  • After each operation/phase transition, Node drains the process.nextTick queue first, then the Promise microtask queue — before continuing.Minden művelet/fázisváltás után a Node először a process.nextTick sort futtatja le, utána a Promise microtask-sort — csak azután megy tovább.

Watch outBuktató“Single-threaded” refers to your JS. libuv and the OS handle the actual I/O, so one Node process serves thousands of concurrent connections without a thread per request.Az „egyszálú” a te JS-edre vonatkozik. A tényleges I/O-t a libuv és az OS kezeli, így egyetlen Node process több ezer párhuzamos kapcsolatot szolgál ki anélkül, hogy kérésenként szálat indítana.

06

libuv & Thread Pool

libuv is the C library behind Node's event loop and async I/O. Most I/O is OS-level async; a few operations use a small thread pool.A libuv az a C könyvtár, ami a Node event loopja és aszinkron I/O-ja mögött áll. Az I/O nagy része OS-szintű aszinkron; néhány művelet egy kis thread poolt használ.

Only pool-backed ops (fs, DNS, some crypto/zlib) are limited by the thread count.Csak a pool-alapú műveleteket (fs, DNS, egyes crypto/zlib) korlátozza a szálak száma.

What actually uses the thread pool?Mi használja ténylegesen a thread poolt?

  • File system, DNS lookups (dns.lookup), some crypto (e.g. pbkdf2) and zlib. Network sockets do not — they're OS-async (epoll/kqueue/IOCP).A fájlrendszer, a DNS lookupok (dns.lookup), egyes crypto műveletek (pl. pbkdf2) és a zlib. A hálózati socketek nem — azok OS-aszinkronok (epoll/kqueue/IOCP).
  • Default pool size is 4 (UV_THREADPOOL_SIZE, up to 1024).Az alapértelmezett pool-méret 4 (UV_THREADPOOL_SIZE, akár 1024-ig).

Why does this matter?Miért számít ez?

  • CPU-heavy pool work (many crypto hashes) can exhaust the 4 threads and stall unrelated fs/DNS work — tune the pool or move to worker threads.A CPU-igényes pool-munka (sok crypto hash) kimerítheti a 4 szálat, és megakaszthat tőle független fs/DNS műveleteket — hangold a poolt vagy válts worker threadekre.

Watch outBuktatóNetwork requests scale far beyond the pool because sockets are OS-async — only pool-backed operations are capped by UV_THREADPOOL_SIZE.A hálózati kérések jóval a poolon túl skálázódnak, mert a socketek OS-aszinkronok — csak a pool-alapú műveleteket korlátozza az UV_THREADPOOL_SIZE.

Authentication vs AuthorizationAutentikáció vs autorizáció

Authentication proves who you are; authorization decides what you may do. They're separate steps, and conflating them is the root of many access-control bugs.Az autentikáció bizonyítja, ki vagy; az autorizáció eldönti, mit tehetsz. Két külön lépés, és az összemosásuk sok hozzáférés-vezérlési bug gyökere.

Where does each happen in a request?Hol történik mindegyik egy kérésben?

  • Authentication runs once at the edge of the request (validate the session or token, load the user). Authorization runs at every protected operation, against that user and the specific resource being touched.Az autentikáció egyszer fut a kérés elején (session vagy token validálása, user betöltése). Az autorizáció minden védett műveletnél fut, az adott userre és a konkrét érintett erőforrásra.

Common authorization models?Gyakori autorizációs modellek?

  • RBAC (role-based): permissions attached to roles — simple, coarse. ABAC (attribute-based): decisions from attributes of user/resource/context — flexible, more complex. Ownership checks: the most common need — is this row mine?RBAC (szerep-alapú): a jogosultságok szerepekhez kötve — egyszerű, durva. ABAC (attribútum-alapú): döntések a user/erőforrás/kontextus attribútumaiból — rugalmas, komplexebb. Tulajdonosi ellenőrzés: a leggyakoribb igény — az enyém ez a sor?

Watch outBuktatóA valid token means the caller is authenticated, not that they're allowed. Every endpoint still needs its own authorization check — “they had a token” has caused a great many data breaches.Egy érvényes token azt jelenti, a hívó autentikált, nem azt, hogy jogosult. Minden endpointnak kell saját autorizációs ellenőrzés — a „volt tokenje” nagyon sok adatszivárgást okozott.

Sessions vs TokensSession-ök vs tokenek

Two ways to remember a logged-in user: a server-side session referenced by an opaque cookie, or a self-contained signed token the client presents on each request.Két mód egy bejelentkezett user megjegyzésére: szerver-oldali session, amire egy átlátszatlan cookie hivatkozik, vagy egy önmagában álló aláírt token, amit a kliens minden kéréssel bemutat.

Sessions keep state on the server; tokens carry state in the credential itself.A session-ök a szerveren tartják az állapotot; a tokenek magában a hitelesítő adatban hordozzák.

StateÁllapotin a store (Redis/DB)egy tárban (Redis/DB)in the token itselfmagában a tokenben
RevocationVisszavonásimmediate — delete itazonnali — töröldhard — valid until expirynehéz — lejáratig érvényes
ScalingSkálázásneeds a shared storeközös tár kellstateless verificationstateless ellenőrzés
SizeMéretsmall opaque IDkis átlátszatlan IDlarger, sent every requestnagyobb, minden kéréssel megy

When would you choose each?Mikor melyiket választanád?

  • Sessions for classic web apps in one domain — simpler, revocable, and a cookie is easy to secure. Tokens for APIs consumed by mobile/third-party clients, or across services where a shared session store is awkward.Session klasszikus web appokhoz egy domainen — egyszerűbb, visszavonható, és egy cookie könnyen biztonságossá tehető. Token mobil/harmadik féltől használt API-khoz, vagy szolgáltatások között, ahol a közös session-tár kényelmetlen.

Watch outBuktatóThe revocation gap is the real trade-off — a stateless JWT stays valid until it expires, even after a logout or a ban. The usual fix (a server-side denylist) reintroduces the state you removed, which is why short-lived access tokens plus refresh tokens exist.A visszavonási rés a valódi trade-off — egy stateless JWT lejáratáig érvényes marad, még kijelentkezés vagy tiltás után is. A szokásos javítás (szerver-oldali denylist) visszahozza az eltávolított állapotot — ezért léteznek a rövid életű access tokenek plusz refresh tokenek.

CAP Theorem

During a network partition, a distributed system can guarantee at most two of Consistency, Availability, Partition tolerance — and since partitions are unavoidable, the real choice under one is C vs A.Hálózati partíció alatt egy elosztott rendszer a Consistency, Availability, Partition tolerance közül legfeljebb kettőt garantálhat — és mivel a partíció elkerülhetetlen, a valódi választás alatta C vs A.

P isn't optional — the choice is what to do during a partition: CP or AP.A P nem opcionális — a kérdés, hogy mit teszel partíció alatt: CP vagy AP.

What do C, A, P actually mean here?Mit jelent itt pontosan a C, A, P?

  • Consistency (linearizability): every read sees the latest write. Availability: every request gets a non-error response. Partition tolerance: the system keeps working despite dropped/delayed messages between nodes.Consistency (linearizability): minden olvasás a legfrissebb írást látja. Availability: minden kérés nem-hibás választ kap. Partition tolerance: a rendszer működik akkor is, ha a node-ok közti üzenetek elvesznek/késnek.

So which do you “pick”?Akkor melyiket „választod”?

  • Networks fail, so P is forced. The real decision is the partition response: CP (reject/block to stay consistent) or AP (serve possibly-stale data to stay available).A hálózat hibázik, szóval a P kényszer. A valódi döntés a partíció-válasz: CP (elutasít/blokkol a konzisztenciáért) vagy AP (esetleg elavult adatot szolgál ki az elérhetőségért).

Watch outBuktatóCAP only applies during a partition — a healthy network gives you both C and A. Real systems are tunable per-operation, not globally “CP” or “AP.”A CAP csak partíció alatt érvényes — egészséges hálón C és A is megvan. A valós rendszerek műveletenként hangolhatók, nem globálisan „CP" vagy „AP".

Consistency Models

The contract for what a read can observe after writes — from strongest to weakest.A szerződés arról, mit láthat egy olvasás az írások után — a legerősebbtől a leggyengébbig.

ModelModellGuaranteeGarancia
Linearizablereads see the latest committed write; acts like one copy (costly)az olvasás a legfrissebb committált írást látja; egyetlen példányként viselkedik (drága)
Sequentialall nodes see operations in the same order (not real-time)minden node ugyanabban a sorrendben látja a műveleteket (nem valós idejű)
Causalcausally-related ops seen in order; concurrent ops may differaz okságilag összefüggő műveletek sorrendben; a párhuzamosak eltérhetnek
Read-your-writesyou always see your own writesmindig látod a saját írásaidat
Eventualreplicas converge if writes stop; reads may be stale meanwhilea replikák konvergálnak, ha az írások leállnak; addig az olvasás elavult lehet

Why not always use strong consistency?Miért ne használjunk mindig strong consistency-t?

  • It requires coordination (consensus/quorum) → higher latency and lower availability. Weaker models scale better and stay available under partitions.Koordinációt igényel (consensus/quorum) → magasabb latencia és alacsonyabb elérhetőség. A gyengébb modellek jobban skálázódnak és partíció alatt is elérhetők maradnak.

Watch outBuktató“Eventual” doesn't mean “no rules” — causal + read-your-writes are practical middle grounds that avoid the worst anomalies.Az „eventual" nem azt jelenti, hogy „nincsenek szabályok" — a causal + read-your-writes praktikus köztes megoldások, amelyek elkerülik a legrosszabb anomáliákat.

The Test PyramidA teszt-piramis

A model for how many tests of each kind to write: many fast unit tests at the base, fewer integration tests in the middle, very few slow end-to-end tests at the top.Egy modell arról, hány tesztet írj mindegyik fajtából: sok gyors unit teszt az alapon, kevesebb integrációs teszt középen, nagyon kevés lassú end-to-end teszt a csúcson.

Cost and runtime grow upward; confidence that the whole system works grows with them.A költség és a futásidő felfelé nő; a bizalom, hogy az egész rendszer működik, velük együtt nő.

Why not just write end-to-end tests?Miért ne írjunk csak end-to-end teszteket?

  • They're slow, flaky and expensive to maintain, and when one fails it doesn't tell you where the bug is. Unit tests are fast and pinpoint the failure; E2E tests prove the pieces actually work together. You need both, in different proportions.Lassúak, flaky-k és drágán karbantarthatók, és ha elbukik egy, nem mondja meg, hol a hiba. A unit tesztek gyorsak és pontosan megmutatják a hibát; az E2E tesztek bizonyítják, hogy a darabok tényleg működnek együtt. Mindkettő kell, eltérő arányban.

What's the “testing trophy” alternative?Mi a „testing trophy” alternatíva?

  • A newer model that puts the most weight on integration tests rather than unit tests, arguing they give the best confidence-per-effort for typical web apps. In a React/Node stack it's often the more practical shape.Egy újabb modell, amely a legnagyobb súlyt az integrációs tesztekre helyezi a unit tesztek helyett, azzal érvelve, hogy tipikus webappoknál ezek adják a legjobb bizalom/ráfordítás arányt. React/Node stacken gyakran ez a praktikusabb forma.

What actually makes a test valuable?Mitől értékes ténylegesen egy teszt?

  • It fails when the behaviour breaks and passes otherwise. A test that breaks on every refactor (too coupled to implementation) or that never fails (asserting nothing meaningful) is a maintenance cost with no return.Elbukik, amikor a viselkedés elromlik, és átmegy egyébként. Egy teszt, amely minden refaktornál eltörik (túlságosan az implementációhoz kötött), vagy amely sosem bukik el (semmi értelmeset nem állít), karbantartási költség hozam nélkül.

Watch outBuktatóChasing 100% coverage produces tests written to touch lines rather than to verify behaviour. Coverage tells you what isn't tested; it says nothing about whether what is tested is tested well.A 100%-os lefedettség hajszolása olyan teszteket szül, amelyek sorokat érintenek, nem viselkedést ellenőriznek. A lefedettség megmondja, mi nincs tesztelve; arról semmit nem mond, hogy ami tesztelve van, az jól van-e tesztelve.
