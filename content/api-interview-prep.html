20 API Concepts

01

Endpoint

An endpoint is a specific URL that exposes a resource or collection a client can act on — base URL + a resource path (e.g. https://api.shop.com/orders/42).

How does REST resource design differ from RPC-style endpoints?

  • REST models nouns: /orders/42, and the HTTP method carries the verb.
  • RPC bakes the action into the path: /getOrder?id=42, /cancelOrder.

How should endpoints be named?

  • Plural nouns for collections (/orders), path params for identity (/orders/42).
  • No verbs in the path; filtering / sorting go in the query string (/orders?status=paid&sort=-created).
  • Nest to show relationships (/users/7/orders), but avoid going much deeper than two levels.

Watch outThe endpoint is the URL; the operation is endpoint + method. GET /orders and POST /orders are two different operations on one endpoint.

02

HTTP Methods

Methods are the verbs that define what action a request performs on a resource.

MethodPurposeSafeIdempotent
GETRead a resource✓✓
POSTCreate / non-idempotent action——
PUTFull replace—✓
PATCHPartial update——*
DELETERemove a resource—✓

Safe vs idempotent — what's the difference?

  • Safe = no server state change (GET, HEAD).
  • Idempotent = repeating N times has the same effect as once (GET, PUT, DELETE).
  • Every safe method is idempotent; the reverse isn't true.

PUT vs PATCH?

  • PUT replaces the whole resource — fields you omit get cleared/defaulted.
  • PATCH applies a partial change. *It isn't inherently idempotent (e.g. an increment op).

Watch outPOST is the catch-all for non-idempotent creates and RPC-ish actions that don't map cleanly onto the other verbs.

03

Request–Response

A stateless exchange: the client sends a request (method, path, headers, optional body); the server returns a response (status, headers, optional body).

A single request/response round trip.

Why is HTTP stateless, and how do we keep sessions?

  • Each request is independent — the server keeps no memory of the client between them.
  • State is carried by the client (cookie / token) or looked up server-side by a session ID.

Content-Type vs Accept?

  • Content-Type describes the body you're sending.
  • Accept tells the server which formats you can handle (content negotiation).
04

Status Codes

Three-digit codes signalling the outcome — 1xx info, 2xx success, 3xx redirect, 4xx client error, 5xx server error.

200 OK201 Created204 No Content 301/302304 Not Modified 400401403404409422429 500502503504

401 vs 403?

  • 401 Unauthorized = not authenticated — missing/invalid credentials (“who are you?”).
  • 403 Forbidden = authenticated, but not allowed to do this.

400 vs 422?

  • 400 — malformed request (bad JSON, wrong types).
  • 422 — syntactically valid but semantically invalid (validation failed).

201 vs 200 vs 204?

  • 201 after a create (often with a Location header).
  • 200 success with a body; 204 success with no body (e.g. a DELETE).

Watch outNever return 200 with an error inside the body — clients, proxies and monitoring all trust the status line.

05

Authentication

Authentication proves who the caller is.

Session-based vs token-based auth?

  • Session: server stores session state; client holds an opaque session-ID cookie. Easy to revoke, but stateful.
  • Token (JWT): self-contained and stateless — scales horizontally, but hard to revoke before it expires.

Where do you store a token in the browser?

  • An httpOnly · Secure · SameSite cookie resists XSS token theft (but needs CSRF defence).
  • localStorage is readable by any injected script. Prefer a short-lived access token + refresh flow.

Watch outA JWT is signed, not encrypted — anyone can base64-decode and read the payload. Never put secrets in the claims.

06

Authorization

Authorization decides what an authenticated caller is allowed to do.

AuthN vs AuthZ?

Authentication = identity (login). Authorization = permissions (access). AuthN always comes first.

RBAC vs ABAC vs scopes?

  • RBAC — permissions grouped into roles (admin, editor).
  • ABAC — rules over attributes of user, resource and context; finer-grained.
  • Scopes — coarse capability grants attached to a token (read:orders).

Watch outEnforce authorization on the server for every request. Hiding a button in React is UX, not security.

07

Access Tokens

A credential — usually a Bearer token / JWT — that a client presents to prove an authenticated, authorized session, sent as Authorization: Bearer <token>.

What's the structure of a JWT?

header.payload.signature, each base64url-encoded. The signature (HMAC or RSA/EC) verifies integrity so the claims can't be tampered with.

Access token vs refresh token?

  • Access: short-lived (minutes), sent on every call.
  • Refresh: long-lived, stored securely, exchanged for a new access token without re-login. Rotate it to detect theft.

Watch outA short access-token TTL limits the blast radius of a leak; pair it with a revocation list for high-value tokens.

08

OAuth 2.0

A delegated-authorization framework — it lets an app act on a user's behalf without ever handling their password. Roles: resource owner (user), client (app), authorization server, resource server.

Authorization Code flow with PKCE — the default for web, SPA and mobile clients.

Which grant type for a web / SPA / mobile login?

  • Authorization Code + PKCE — PKCE stops an intercepted code from being exchanged by an attacker on public clients.
  • Client Credentials for machine-to-machine (no user involved).

OAuth 2.0 vs OpenID Connect?

  • OAuth 2.0 handles authorization (access).
  • OIDC layers authentication on top, adding an id_token (JWT) that describes the user.

Watch outAn access token proves access, not identity — don't use it to answer “who is this user?”. That's what OIDC's id_token is for.

09

Rate-Limiting

Capping how many requests a client can make per time window to protect capacity and keep usage fair.

Token bucket: refills at a steady rate, allows short bursts up to its size.

Common algorithms?

  • Fixed window — simple, but allows bursts at the window edge.
  • Sliding window — smooths those edge bursts.
  • Token bucket — steady refill, bursts up to bucket size. Leaky bucket — constant outflow.

How do you signal limits to clients?

429 Too Many Requests + Retry-After, plus X-RateLimit-Limit / Remaining / Reset headers.

Watch outAcross multiple instances, counters must live in shared storage (e.g. Redis) — otherwise every node limits independently and the real cap is N×.

10

Throttling

Controlling the rate of processing — typically by delaying, queuing or degrading excess requests instead of rejecting them outright.

Rate-limiting vs throttling?

  • Rate limiting usually rejects over-quota requests (429).
  • Throttling shapes traffic — slows, queues or degrades it to stay within capacity.
  • The terms overlap; the distinction to state is intent: reject vs smooth.

Watch outThrottling improves stability but adds latency — make sure clients have sane timeouts and exponential backoff.

11

Pagination

Splitting a large result set into pages instead of returning everything at once.

Offset drifts when data changes; a cursor points at the last item and stays stable.

Offset vs cursor (keyset)?

  • Offset/limit (?page=3&limit=20) — simple, random access; but slow on deep offsets and can skip/duplicate rows when data shifts.
  • Cursor (?after=<cursor>) — stable and fast on huge datasets, but you can't jump to an arbitrary page.

Watch outFor infinite scroll or high-write tables, prefer cursor pagination to avoid drift and slow deep-offset queries.

12

Caching

Storing responses so repeated reads skip the work. Layers: browser → CDN → reverse proxy → application → database.

A matching ETag lets the server answer 304 and skip resending the body.

Cache-Control basics?

max-age (freshness), no-cache (revalidate before use), no-store (never cache), private vs public.

How do conditional requests work?

The server returns ETag / Last-Modified. The client re-requests with If-None-Match / If-Modified-Since; unchanged → 304 Not Modified with no body.

What's the hard part?

Invalidation — via TTL expiry, write-through updates, or cache-busting (versioned URLs/keys).

Watch outCache the right thing at the right layer — caching user-specific data in a shared/CDN cache leaks one user's data to another. Mark it private.

13

Idempotency

A request that produces the same result no matter how many times it's sent. This is what makes safe retries possible.

An Idempotency-Key lets the server dedupe a retried write.

Which methods are idempotent, and why does it matter?

  • GET, PUT, DELETE, HEAD are; POST and PATCH generally aren't.
  • It matters for retries: after a timeout, a client can safely re-send an idempotent call with no side effects.

How do you make POST idempotent (e.g. payments)?

The client sends an Idempotency-Key header; the server stores key → result and returns the original response on any retry — so a double-submitted charge only bills once.

Watch outDELETE twice: first returns 204, second 404 — still idempotent, because the state (resource gone) is unchanged.

14

Webhooks

Server-to-server HTTP callbacks — the provider POSTs an event to your URL the moment something happens. A “reverse API”.

Polling asks repeatedly; a webhook is pushed once, on the event.

Webhooks vs polling?

  • Polling repeatedly asks “any updates?” — wasteful and laggy.
  • Webhooks push the event as it happens — efficient and near-real-time, but you must expose an endpoint and handle delivery failures.

How do you secure and harden a receiver?

  • Verify a signature (HMAC of the payload) to prove it's really from the provider.
  • Return 2xx fast, process async; providers retry on non-2xx, so make handlers idempotent (dedupe by event ID).

Watch outAssume at-least-once delivery and out-of-order events — dedupe on event ID and never rely on arrival order.

15

API Versioning

Managing change so existing clients don't break as the API evolves.

Where can the version live?

  • URI path (/v1/orders) — most visible, trivial to route and test.
  • Header / media type (Accept: application/vnd.api.v2+json) — cleaner URLs, harder to test by hand.
  • Query param (?version=2) — simple, less common.

Breaking vs non-breaking change?

  • Non-breaking (additive): new optional fields/endpoints — no version bump needed.
  • Breaking: removing/renaming fields, changing types or semantics — needs a new version + deprecation window.

Watch outVersion only on breaking changes. Leaning on additive, backward-compatible evolution keeps you out of a version zoo.

16

OpenAPI

A standard, language-agnostic spec (YAML/JSON) that describes a REST API — its endpoints, params, schemas and auth. Formerly Swagger.

What do you get from it?

Auto-generated docs, client/server SDK codegen, request validation, and contract/mock testing.

Contract-first vs code-first?

  • Contract-first: write the spec, generate stubs — teams agree on the interface before coding.
  • Code-first: generate the spec from annotated code — convenient, but the spec can drift if it isn't enforced.

Watch outAn OpenAPI doc is only useful if it stays in sync — validate the running API against the spec in CI.

17

REST vs GraphQL

Two API styles. REST exposes many resource endpoints; GraphQL exposes one endpoint where the client asks for exactly the fields it needs.

REST tends to over- and under-fetch; GraphQL returns exactly the requested shape in one call.

Core trade-offs?

  • REST: simple, cache-friendly (HTTP caching per URL) — but over/under-fetching and multiple round trips for related data.
  • GraphQL: no over-fetch, one request for nested data, strong typed schema — but HTTP caching is harder, and you must guard the N+1 problem (DataLoader/batching) and query cost/depth.

When would you pick each?

  • REST: public APIs, simple CRUD, CDN caching.
  • GraphQL: rich clients (mobile) with varied data needs, or aggregating many sources.

Watch outGraphQL moves complexity to the server (resolvers, query-cost limits) — it isn't automatically “better”.

18

API Gateway

A single entry point in front of your services that handles cross-cutting concerns — routing, auth, rate limiting, TLS termination, transformation and aggregation.

Clients hit one gateway; it applies policy and routes to internal services.

Why put a gateway in front of microservices?

It centralizes auth, rate limiting and logging so each service doesn't reimplement them, hides internal topology, and can aggregate several service calls into one client response.

Gateway vs BFF (Backend-for-Frontend)?

A BFF is a gateway tailored per client type (web vs mobile), shaping responses to that client's exact needs.

Watch outThe gateway is a single point of failure and an extra latency hop — make it highly available and keep its logic thin (routing/policy, never business rules).

19

Microservices

An architecture that splits an app into small, independently deployable services, each owning a bounded context and its own data.

Trade-offs vs a monolith?

  • Pros: independent deploy/scaling, tech flexibility, fault isolation, team autonomy.
  • Cons: distributed-system complexity — network failures, data consistency, testing, observability.

How do services communicate and stay consistent?

  • Sync (REST/gRPC) or async (events / message queues).
  • No distributed ACID across services — use the Saga pattern (local transactions + compensating actions) and accept eventual consistency.

Watch outDon't reach for microservices on a small app — the operational overhead outweighs the benefits. Split when clear boundaries and scale justify it.

20

Error Handling

Returning consistent, actionable errors so clients can react programmatically and humans can debug.

What does a good error response look like?

The correct HTTP status + a consistent JSON body: a stable machine-readable code, a human message, and field-level details for validation. application/problem+json (RFC 7807) is a common shape.

What should you avoid?

  • Leaking stack traces / internals (security risk).
  • Returning 200 for failures.
  • Inconsistent error shapes across endpoints.

How do you debug across services?

Propagate a correlation / trace ID through every service and include it in error responses and logs.

Watch outDistinguish client errors (4xx — don't retry) from transient server errors (5xx — retry with backoff) so clients behave correctly.
